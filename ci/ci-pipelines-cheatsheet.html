01 CI Concepts & Pipeline Model

CI is the discipline of merging small changes frequently and validating each one automatically, so integration problems surface in minutes, not weeks.

What CI actually solves

  • Integration risk — without CI, divergent branches accumulate conflicts that compound over time ("integration hell"). Frequent merges + automated validation keep divergence small.
  • Fast feedback — a broken build/test is caught within minutes of the commit that caused it, while the change is still fresh in the author's head.
  • Deterministic quality bar — the same checks run for every change, removing "works on my machine" and reviewer inconsistency.
  • Shippable artifact — the end product of CI is a versioned, scanned, immutable artifact (image, package, binary) that CD promotes — CI never deploys to production itself in a clean split.

Canonical pipeline stages

build → test → scan → package → publish

StagePurposeTypical gate
BuildCompile / transpile / install depsNon-zero exit fails the pipeline
TestUnit → integration → (e2e in CD/staging)Coverage threshold, no regressions
ScanSAST, dependency/SCA, secret scan, license checkNo new CRITICAL/HIGH findings
PackageProduce the immutable artifact (container image, jar, wheel)SBOM generated, image signed
PublishPush to registry/artifact store with an immutable tagProvenance attestation attached

Trunk-based development & PR/merge checks

Trunk-based dev keeps everyone merging small commits into main/trunk at least daily, behind short-lived feature branches (or feature flags instead of long-lived branches). This is what makes fast CI feedback meaningful — you can't get fast feedback on a branch that diverges for three weeks.

  • PR/merge-request checks run the pipeline against the merge result (not just the branch tip) — GitHub's "merge queue" and GitLab's "merged results pipelines" both validate the hypothetical post-merge state before it lands.
  • Required status checks + branch protection make CI the actual gatekeeper, not a suggestion.

Artifact management, caching, fan-out/fan-in

  • Artifacts are the pipeline's persistent outputs (test reports, coverage, built binaries) — passed between stages/jobs and often retained for a fixed period for audit/debugging.
  • Cache is a performance optimization (dependency directories, build layers) — best-effort, restorable, and safe to lose; never treat cache as a source of truth for what shipped.
  • Fan-out/fan-in: one pipeline triggers N parallel jobs (matrix builds, per-package tests) that fan back into a single aggregate gate before deploy — this is how large monorepos keep CI fast without sacrificing full coverage.
Q&A "What's the difference between CI and CD?" — CI validates and produces an artifact on every change; CD takes that already-validated artifact and promotes it through environments. Conflating them (rebuilding per environment) breaks the "build once, promote the artifact" guarantee and is a common interview red flag to call out.

02 GitHub Actions

Event-driven automation native to GitHub. Model: repository events trigger workflows, which contain one or more jobs, which run ordered steps on a runner.

Core hierarchy

  • Workflow — a YAML file in .github/workflows/, triggered by one or more events.
  • Job — a set of steps that run on the same runner instance; jobs run in parallel by default unless needs orders them.
  • Step — a single command (run:) or reusable unit (uses: an Action).
  • Action — a reusable packaged unit (Docker, JS, or composite) versioned like a dependency.

Events / triggers

  • push, pull_request, pull_request_target (runs with base-repo secrets — a classic injection vector, see traps)
  • workflow_dispatch (manual), schedule (cron), release
  • workflow_call — makes a workflow reusable/callable from other workflows
  • repository_dispatch — external system triggers via API

Runners: hosted vs self-hosted

GitHub-hostedSelf-hosted
ProvisioningEphemeral VM per job, auto-provisionedYou provision (VM, container, bare metal)
StateClean every run — no residuePersistent by default — must be made ephemeral deliberately
Cost modelPer-minute billingYour infra cost, no per-minute fee
Use caseDefault choice, public/standard workloadsCustom hardware (GPU), network-restricted access, compliance
Security noteIsolated per jobFork PRs must never run on self-hosted runners with secrets — untrusted code gets a foothold on your infra

Matrix builds, caching, reusable workflows

  • Matrix (strategy.matrix) fans a job out across a cross-product of variables (OS × language version, etc.) — each combination runs as an independent parallel job.
  • actions/cache keys on a hash (e.g. lockfile checksum) with restore-key fallback; a cache miss just means a slower run, never a correctness issue — unless you cache something that should be revalidated (see cache poisoning trap).
  • Reusable workflows (workflow_call) let one workflow invoke another with typed inputs/secrets — the way to DRY across many repos.
  • Composite actions bundle several steps into one reusable action (single repo, simpler than a full reusable workflow).

Secrets & OIDC (keyless cloud auth)

Modern best practice: don't store long-lived cloud credentials as GitHub secrets at all. GitHub's OIDC provider issues a short-lived signed token per workflow run; AWS/GCP/Azure trust that token via a configured identity-provider/trust-policy and hand back temporary credentials scoped to that specific repo/branch. No static keys sit in the secrets store waiting to leak.

Permissions & concurrency

  • permissions: at workflow or job level scopes the auto-generated GITHUB_TOKEN — default should be contents: read, granting write/id-token only to the jobs that need it (least privilege).
  • concurrency: with a group key + cancel-in-progress: true avoids redundant/overlapping runs (e.g. superseded PR pushes) racing each other.
  • actions/upload-artifact / download-artifact pass build outputs between jobs; default retention is time-boxed (org/repo configurable).

Example workflow

name: CI

on:
  push:
    branches: [main]
  pull_request:

permissions:
  contents: read   # least-privilege default; id-token granted only to the deploy job below

concurrency:
  group: ${{ github.workflow }}-${{ github.ref }}
  cancel-in-progress: true

jobs:
  test:
    runs-on: ubuntu-latest
    strategy:
      matrix:
        node: [18, 20, 22]
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node }}
      - uses: actions/cache@v4
        with:
          path: ~/.npm
          key: npm-${{ hashFiles('package-lock.json') }}
      - run: npm ci
      - run: npm test -- --coverage

  deploy:
    needs: test
    runs-on: ubuntu-latest
    permissions:
      id-token: write
      contents: read
    steps:
      - uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: arn:aws:iam::123456789012:role/ci-deploy
          aws-region: us-east-1
          # no access-key-id / secret-access-key — OIDC issues a short-lived token
⚠ Trap pull_request_target runs with the base repo's secrets and permissions, even for a PR from a fork — combined with checking out and running the fork's untrusted code, this is the classic path to secret exfiltration. If you need to build fork PRs, use plain pull_request (no secrets, runs in a restricted context) or explicitly isolate what gets checked out.

03 GitLab CI

Config lives in a single .gitlab-ci.yml at repo root (or included from elsewhere). Jobs are assigned to stages that run sequentially; jobs within a stage run in parallel by default.

Stages & jobs

  • stages: defines the sequential phases (e.g. build, test, deploy).
  • Each job specifies stage:; jobs in the same stage run concurrently.
  • needs: lets a job skip waiting for its whole stage and instead depend on specific jobs — this is what enables a DAG instead of strict stage-by-stage execution.

rules vs the deprecated only/except

  • rules: is the modern, more expressive way to control whether/how a job runs (conditions, when:, variables) — it supersedes only/except.
  • only/except still work but are legacy and cannot combine conditions as flexibly as rules.

Runners & executors

ExecutorWhat it runs jobs in
shellDirectly on the runner host — fast but no isolation between jobs
dockerEach job in a fresh container from the specified image: — the common default
kubernetesEach job as a pod in a cluster — scales elastically, good for large fleets
docker-machine/autoscaleRunner provisions/tears down VMs on demand

Runners can be shared (instance-wide, GitLab.com's pool) or project/group-specific (dedicated, registered with a runner token).

Cache vs artifacts

  • cache: is keyed, best-effort, and meant for dependency directories to speed up subsequent runs — not guaranteed to exist.
  • artifacts: are the job's actual outputs, passed to later stages automatically and stored per GitLab's retention/expiry settings (expire_in:).

Includes, templates, environments

  • include: pulls in config from local files, other projects, remote URLs, or GitLab's own template library — the DRY mechanism analogous to Actions' reusable workflows.
  • environment: on a job ties it to a named deployment target, enabling environment-scoped variables, deployment history, and manual approval gates.

Example pipeline

stages:
  - build
  - test
  - deploy

variables:
  NODE_ENV: test

.build_template:
  image: node:20
  cache:
    key:
      files: [package-lock.json]
    paths:
      - .npm/

build:
  extends: .build_template
  stage: build
  script:
    - npm ci --cache .npm
    - npm run build
  artifacts:
    paths:
      - dist/
    expire_in: 1 week

test:
  extends: .build_template
  stage: test
  needs: [build]
  script:
    - npm test -- --coverage
  rules:
    - if: '$CI_PIPELINE_SOURCE == "merge_request_event"'
    - if: '$CI_COMMIT_BRANCH == "main"'

deploy_prod:
  stage: deploy
  needs: [test]
  environment:
    name: production
  script:
    - ./deploy.sh
  rules:
    - if: '$CI_COMMIT_BRANCH == "main"'
      when: manual
Q&A "How do you get a DAG instead of stage-by-stage waiting in GitLab CI?" — Add needs: listing specific job names on each job; GitLab then starts a job as soon as its named dependencies finish, ignoring the rest of its nominal stage.

04 Jenkins

The long-standing self-hosted automation server. Pipelines are defined as code in a Jenkinsfile, executed by a controller that dispatches work to agents.

Declarative vs Scripted Pipeline

DeclarativeScripted
SyntaxStructured pipeline { } block, fixed sectionsFull Groovy DSL, imperative
Learning curveLower, guided structureHigher, requires Groovy fluency
ValidationLinted up front, errors caught before running stagesErrors often surface mid-run
When to useDefault choice for the vast majority of pipelines todayComplex conditional/dynamic logic that declarative's script{} escape hatch can't cleanly express

Core building blocks

  • Jenkinsfile — the pipeline-as-code file, checked into the repo (or configured inline in older jobs — avoid this, it defeats "pipeline as code").
  • Agent (agent { }) — where a stage or the whole pipeline executes: a labeled static node, a Docker container, or a dynamically provisioned pod (Kubernetes plugin).
  • Stages/Steps — same conceptual shape as Actions/GitLab: named phases containing ordered commands.
  • Shared libraries — versioned Groovy code (vars/, src/) loaded into any Jenkinsfile via @Library, the mechanism for DRY across many pipelines/teams.
  • Plugin ecosystem — Jenkins' core strength and core liability: thousands of plugins provide integrations, but plugin compatibility/security maintenance is an ongoing operational burden.

Example Jenkinsfile (declarative)

pipeline {
    agent {
        kubernetes {
            yaml '''
              spec:
                containers:
                - name: node
                  image: node:20
            '''
        }
    }
    options {
        timeout(time: 30, unit: 'MINUTES')
        disableConcurrentBuilds()
    }
    stages {
        stage('Build') {
            steps {
                sh 'npm ci && npm run build'
            }
        }
        stage('Test') {
            steps {
                sh 'npm test -- --coverage'
            }
        }
        stage('Publish') {
            when { branch 'main' }
            steps {
                withCredentials([string(credentialsId: 'registry-token', variable: 'TOKEN')]) {
                    sh 'echo "$TOKEN" | docker login -u ci --password-stdin'
                }
            }
        }
    }
}

Why teams migrate off Jenkins

  • Controller/agent architecture demands ongoing self-managed infra (patching, scaling, plugin upgrades) vs. hosted-runner SaaS models.
  • Plugin sprawl creates version-compatibility matrices that become a maintenance tax, and plugins are a recurring CVE source.
  • Groovy scripted pipelines can become unmaintainable "pipeline spaghetti" without strong shared-library discipline.
  • Native GitHub/GitLab CI removes a whole separate system to operate when a team is already consolidated on one Git host.
  • Jenkins still wins where: complex on-prem/air-gapped requirements, deep legacy plugin integrations, or highly custom orchestration logic that hosted CI's YAML models don't express well.
Interview framing: "Jenkins isn't dead, it's a different trade-off." Expect the interviewer to want you to name the trade-off (control & extensibility vs. operational overhead), not declare a winner.

06 Pipeline Design Principles

Idempotency & ephemeral runners

A pipeline run should produce the same result regardless of what ran before it on that machine. Ephemeral, single-use runners (fresh container/VM per job) enforce this by construction — no leftover state from a previous job can leak into the next one.

Build once, promote the artifact

Build and package exactly once per commit; the identical artifact (same digest) moves through dev → staging → prod. Rebuilding per environment risks "it built differently in prod" and breaks provenance guarantees.

Semantic versioning & immutable tags

Tag artifacts by content-addressable digest or semver, never mutate a tag after publish. This is what makes rollback deterministic and what CD tooling (Argo/Flux image updaters) relies on to detect new versions.

SBOM & scan gates

Generate a Software Bill of Materials at package time (Syft, cdxgen, etc.) and gate publish on dependency/container scan results — this is what lets you answer "are we affected by CVE-X" in minutes instead of days.

Secrets handling

  • Never echo, print, or log a secret value — most CI engines mask known secret variables in log output, but only for values registered as secrets; string-concatenated or derived secrets can bypass masking.
  • Prefer OIDC/workload-identity federation over long-lived static keys wherever the cloud provider supports it — nothing to leak because nothing persists.
  • Scope secrets to the narrowest job/environment that needs them; a build job almost never needs deploy credentials.

07 Supply-Chain Security

CI is high-value attack surface: it has secrets, write access to registries, and runs arbitrary code from dependencies. This is squarely 2026 interview territory after several real-world incidents.

Pin third-party actions/dependencies by SHA

Referencing a third-party Action by a mutable tag (uses: some/action@v3) means the maintainer (or an attacker who compromises their account) can silently change what that tag points to — your pipeline then runs different code with zero diff in your own repo. Pin to a full commit SHA instead:

# risky — tag can be repointed after the fact
- uses: tj-actions/changed-files@v41

# safer — SHA is immutable, tag is just a human-readable comment
- uses: tj-actions/changed-files@a1b2c3d4e5f6... # v41.0.0
⚠ Trap — the tj-actions style compromise In a real supply-chain incident, a popular third-party Action was compromised and its version tags were repointed to a malicious commit that dumped CI secrets into build logs. Every workflow pinning that action by tag (not SHA) pulled the compromised code on its next run automatically — no code change, no PR, no review trail in the victim's own repo. Interviewers use this as the canonical example for "why pin by SHA."

Other supply-chain gates

  • Provenance/attestation (SLSA, in-toto) — cryptographically link an artifact back to the exact source commit and build steps that produced it.
  • Image signing (cosign/Sigstore) — sign artifacts at publish time; deploy-time policy (e.g. Kyverno/OPA) can refuse unsigned images.
  • Dependency review — block PRs that introduce a dependency with known critical vulnerabilities or a newly-changed maintainer/publish pattern.
  • Least-privilege tokens — narrow GITHUB_TOKEN/CI job token scopes so a compromised job can't push to unrelated repos or packages.

08 Testing in CI

Layering

Unit

Fast, isolated, no I/O. Run on every commit/PR. Should be the large base of the pyramid — hundreds to thousands, seconds to run.

Integration

Exercises real boundaries (DB, queue, internal service). Slower, fewer of them, usually still runs in CI with ephemeral dependencies (containers/testcontainers).

E2E

Full user-flow-level, against a deployed environment. Slowest and most brittle — often run post-deploy in CD/staging rather than blocking every commit.

Flaky test handling

  • Quarantine known-flaky tests into a non-blocking job/tag rather than silently retrying forever — retries hide real intermittent bugs (race conditions, shared state, network timing).
  • Track flake rate per test; a test that fails >X% of runs without a code change is a bug in the test (or the code under test), not noise to ignore.
  • Automatic retry-on-failure is acceptable for genuinely environmental flakiness (network blips) but should be capped and logged, never silent/unlimited.

Coverage gates & parallelism

  • Coverage thresholds (e.g. 80%) are a floor, not a target — 100% coverage with weak assertions is worse than 80% with meaningful ones. Interviewers probe for this nuance.
  • Parallelize test suites by splitting on historical timing data (not just file count) to balance shard duration — naive splitting leaves one shard as the bottleneck.

09 Traps & Gotchas Summary

⚠ Secrets in logs A secret concatenated into a derived string, base64-encoded, or split across variables can bypass the CI engine's log-masking, which typically only masks exact literal matches of registered secret values.
⚠ Unpinned third-party actions/images Mutable tags (@v3, :latest) on third-party Actions, Docker base images, or dependencies let upstream change what actually runs without any change in your own repo history.
⚠ Cache poisoning A cache keyed too broadly (e.g. by branch name instead of lockfile hash) can serve a malicious or stale dependency tree across unrelated builds — always key cache on content hashes, never trust cache for anything security-sensitive without revalidation.
⚠ latest tag in production pipelines Deploying/building against :latest makes builds non-reproducible and rollbacks meaningless — you can't roll back to "whatever latest was an hour ago."
⚠ Non-ephemeral self-hosted runners A long-lived self-hosted runner accumulates state (cached creds, leftover files, installed tooling) between jobs from different branches/PRs — a compromised or malicious job can leave a persistent foothold for the next job that lands on the same box.
⚠ Fork PRs on privileged triggers pull_request_target (Actions) or any trigger that grants base-repo secrets/write-access to a job that also checks out fork code is a direct path to secret exfiltration — separate "build untrusted fork code" from "have access to secrets."

10 Rapid Q&A Round

Q Why prefer OIDC over stored cloud credentials in CI?
Short-lived, per-run tokens scoped to a specific repo/branch/workflow — nothing static to rotate, leak, or forget to revoke. Removes an entire class of "old key still valid" incidents.
Q What's the difference between cache and artifacts?
Cache is a best-effort speed optimization you can lose without consequence; artifacts are the actual outputs of the job that later stages or humans depend on and are retained deliberately.
Q How do you keep a large monorepo's CI fast?
Fan out affected packages/services into parallel jobs (matrix or DAG via needs), cache aggressively on content hashes, and only run the full suite for changes that actually touch shared code — path-filtered triggers reduce redundant runs.
Q Declarative or scripted Jenkins Pipeline — which do you default to?
Declarative, because it's linted before execution and easier for a team to reason about; drop into a script{} block only for logic declarative genuinely can't express.
Q Your CI passed but the same commit fails in CD's staging deploy — what's your first hypothesis?
Environment drift or a rebuild — check whether CD rebuilt the artifact instead of promoting the exact digest CI produced; that's the single most common cause of "passed in CI, fails in CD" bugs.
