The Binary — What It Is // one process, all control loops, one leader
⚙️
kube-controller-manager anatomy
single binary embedding every control loop
What it isA single Go binary that runs all core controllers as goroutines inside one process. Not separate processes. All share the same leader election.
Where it runsStatic pod on each control-plane node (manifest in /etc/kubernetes/manifests/). Managed by kubelet directly, not by itself.
API surfaceOnly communicates with kube-apiserver. Never touches etcd directly. Never talks to kubelets or other nodes.
Startup sequence1. Parse flags 2. Build kubeconfig 3. Create shared informer factory 4. Try to acquire leader Lease 5. On win: start all controller goroutines 6. Start informers 7. Wait for cache sync
Leader electionSingle leader for the whole binary. Lease object in kube-system namespace. All controllers inside start/stop together. There is no per-controller leader.
cloud-ctrl-mgrCloud-specific controllers (Node lifecycle for cloud, LoadBalancer provisioning, Route management) were extracted into a separate cloud-controller-manager binary with its own leader election.
Common interview trap: "Can different controllers have different leaders?" NO. The entire kube-controller-manager binary has one Lease. All 30+ controllers inside it start and stop as a unit.
🔐
Leader Election — Lease Object
how HA control-plane stays sane
MechanismCandidates compete to PATCH a Lease object (leases.coordination.k8s.io) in kube-system. Atomicity guaranteed by apiserver + etcd MVCC.
Lease fieldsholderIdentity: who holds it. leaseDurationSeconds: 15s default. renewTime: leader heartbeats. acquireTime: when acquired.
RenewalLeader renews every leaseDuration/2 (~7s). If leader dies, Lease expires. Other candidates detect stale renewTime → attempt takeover.
Failover timeWorst case: leaseDurationSeconds (15s) + retryPeriod (~2s). New leader active within ~17s of leader death.
# Check who is current leader kubectl get lease kube-controller-manager \ -n kube-system -o json \ | jq '{holder: .spec.holderIdentity, renew: .spec.renewTime}'

Key Startup Flags
what actually controls behavior
--controllers* = all. -bootstrapsigner to disable one. deployment,replicaset for explicit allowlist. Use this to disable specific controllers.
--concurrent-*--concurrent-deployment-syncs=5 (default 5). Goroutines per controller for parallel reconciles. Tune up for large clusters.
--sync-periodFull resync interval. Default 10m–1h per controller. Even without events, cache resyncs → re-queues all objects. Catches drift.
--leader-electMust be true for multi-master HA. Default true. Set false only for single-master dev clusters.

SharedInformerFactory — The Nerve System // one watch per resource type, shared by all
📡
Why SharedInformerFactory Exists
solving the N×M watch problem
The problemWithout sharing: Deployment ctrl watches Pods, ReplicaSet ctrl watches Pods, GC ctrl watches Pods… = N watches for the same resource. N×load on apiserver.
The solutionOne informer per (resource, namespace) pair. All controllers that need Pods get the same Pods informer. One List+Watch to apiserver.
Factory patternfactory.Apps().V1().Deployments().Informer() — first caller creates. Subsequent calls return same instance. Go sync.Once guarantees.
Cache syncAt startup, controllers must call cache.WaitForCacheSync() before processing events. Acting on a stale/empty cache causes phantom reconciles and incorrect deletes.
ResyncFull List resync every resyncPeriod. Re-fires all OnUpdate handlers with current state even without real changes. Forces periodic reconcile of all objects — catches drift.
Event handlersEach controller registers AddEventHandler() on the informer. Multiple controllers can register on the same informer. Handlers just enqueue the object key — fast.
// Factory setup (once at startup) factory := informers.NewSharedInformerFactory(client, resyncPeriod) // Each controller gets SAME informer instance: podInformer := factory.Core().V1().Pods().Informer() deployInformer := factory.Apps().V1().Deployments().Informer() // Deployment controller registers on BOTH: deployInformer.AddEventHandler(cache.ResourceEventHandlerFuncs{ AddFunc: ctrl.enqueue, UpdateFunc: func(old, new interface{}) { ctrl.enqueue(new) }, DeleteFunc: ctrl.enqueue, }) // ReplicaSet controller also registers on podInformer // → same podInformer, different handler → no extra Watch call factory.Start(stopCh) // start all List+Watch factory.WaitForCacheSync(stopCh) // BLOCK until all synced
Never read from the apiserver directly inside a reconcile loop. Always read from the informer cache (via Lister). Cache reads are fast O(1) in-memory. API reads add latency + load to apiserver and etcd.

Indexers — Fast Lookups
how controllers find related objects
What they areCustom indexes on the in-memory cache. Pre-compute lookup tables at object add/update time. O(1) lookup instead of O(n) linear scan.
NamespaceIndexBuilt-in. cache.MetaNamespaceKeyFunc. List all pods in a namespace instantly. Used by every controller.
Custom indexExample: ReplicaSet ctrl indexes pods by ownerReference.uid. Given RS, find all its pods in O(1). Without index: scan all pods.
// Custom indexer: pods by owner UID podInformer.AddIndexers(cache.Indexers{ "ownerUID": func(obj interface{}) ([]string, error) { pod := obj.(*v1.Pod) var keys []string for _, ref := range pod.OwnerReferences { keys = append(keys, string(ref.UID)) } return keys, nil }, }) // Usage: find all pods owned by this RS pods, _ := podInformer.GetIndexer().ByIndex("ownerUID", rs.UID)
⚡
Work Queue Deep Dive
dedup · rate-limit · retry per controller
One per controllerEach controller has its own isolated work queue. Failures in one controller's queue don't block others.
Dedup windowObject X changes 100× in 1 second → only one reconcile triggered. The queue deduplicates by key. You always get the latest state from cache anyway.
BackoffFailed items go to rate-limited queue: 5ms → 10ms → 20ms… → 1000s max. This caps retry rate. NumRequeues(key) tells you how many times an item has failed.
Drain on shutdownQueue is drained on process shutdown. Inflight items complete. Pending items are dropped — they'll be re-queued when new leader starts and cache resyncs.

Deployment Controller
manages rollouts via ReplicaSet orchestration — does NOT touch Pods directly
WatchesDeployments (primary), ReplicaSets (owned), Pods (via RS owner ref, to update status only)
CreatesReplicaSets only. Never creates Pods. RS controller creates Pods.
RollingUpdateCreates new RS. Scales new RS up. Scales old RS down. Controlled by maxSurge (extra pods allowed) + maxUnavailable (pods can be down). Interleaved.
RecreateScale old RS to 0. Wait until all old pods gone. Then scale new RS. Downtime guaranteed.
RollbackOld RSes kept with spec.template preserved (up to revisionHistoryLimit). Rollback = scale old RS back up, scale new RS down.
Paused statespec.paused: true → controller does nothing. Use for staged rollouts: pause, do canary testing, resume.
DeadlineprogressDeadlineSeconds (default 600s). If rollout doesn't make progress in that window, Deployment condition Progressing=False. Does NOT roll back automatically.
RS namingdeploy-name-{pod-template-hash}. Hash is deterministic from pod template. Same template = same RS. Why: prevents duplicate RS creation on concurrent writes.
Revisiondeployment.kubernetes.io/revision annotation on Deployment + RS. Monotonically incrementing. Used for rollback targeting.
Rolling Update Math
replicas=10, maxSurge=25%, maxUnavailable=25%
→ max pods = 10 + ceil(10×0.25) = 13
→ min pods = 10 - floor(10×0.25) = 7
→ new RS scales: 0→3→6→8→10
→ old RS scales: 10→7→4→2→0
→ at no point <7 or >13 running pods
A Deployment with 0 old RSes and a failed rollout is stuck — the progress deadline fires but no auto-rollback occurs. You must kubectl rollout undo manually. Or set up your own alerting on Progressing=False.
🔢
ReplicaSet Controller
the actual pod factory
Core jobKeep len(ownedPods) == spec.replicas. Compare actual vs desired. Create or delete to reach desired count.
Pod ownershipFinds pods via label selector + ownerReferences. A pod matching selector but without ownerRef can be adopted. Pods with ownerRef to other RS are not touched.
Scale down choiceWhen deleting, prefers: unscheduled pods first, then pending, then least-recently-started. Not random. Minimizes disruption.
Pod creation burstBurstReplicas=500 default. At most 500 pods created per sync. Prevents API flooding on large RS scale-up.
Slow startPod creation uses slow-start algorithm: create 1, then 2, then 4, then 8… up to BurstReplicas. Avoids sending 500 parallel creation requests at once.
RS controller cannot roll out. It only counts pods. All rollout logic lives in the Deployment controller. If you use RS directly, updates don't roll out — they're instantaneous (delete all old, create all new).
🗃️
StatefulSet Controller
identity-preserving ordered pods
Unique identityEach pod gets: stable name (sts-0, sts-1), stable network identity (headless service), stable storage (PVC persisted across pod reschedule).
Ordered creationDefault (OrderedReady): pod N must be Running+Ready before pod N+1 starts. Parallel policy: create all at once (like RS).
Ordered deletionDeletes in reverse order: N-1 before N-2 before … 0. For scale-down. Guarantees graceful shutdown of highest-index first.
PVC lifecyclePVCs NOT deleted when pod is deleted. Survive pod restarts, reschedules. Only deleted if StatefulSet itself is deleted (with cascade) or manually.
Update strategyRollingUpdate: updates highest ordinal first, waits for Ready, then next. OnDelete: only update pod when you manually delete it.
PVC orphan trap: delete a StatefulSet without --cascade=foreground. Pods deleted, PVCs remain. A new StatefulSet with same name re-attaches the same PVCs. If you want clean slate, delete PVCs explicitly.
👾
DaemonSet Controller
one pod per eligible node, auto-tracking
Node trackingWatches both DaemonSet objects AND Node objects. When a node is added → controller reconciles → creates pod on new node within seconds.
EligibilityNode must pass DS nodeSelector, nodeAffinity, and tolerations. DS can tolerate node.kubernetes.io/not-ready — useful for network plugins (CNI must run before anything).
Rolling updatemaxUnavailable controls how many nodes updated at once. No surges — never creates extra pods. Updates in random node order (unlike StatefulSet which is ordered).
PreemptionDaemonSet pods are created with schedulerName: default-scheduler but with node name pre-set, bypassing the scheduler's Filter/Score. They can still be placed on unschedulable nodes.
OnDelete strategyFor CNI/network plugins: use OnDelete + manual node drain to control exactly when each node gets the new version. Avoids network disruption.
DaemonSet pods on control-plane nodes are blocked by default by the node-role.kubernetes.io/control-plane:NoSchedule taint. To run on control-plane too, add explicit toleration in DS spec.
⚡
Job Controller
run-to-completion semantics — entirely different from Deployment
Completion trackingTracks status.succeeded and status.failed. Job done when succeeded == spec.completions.
Parallelismspec.parallelism: how many pods run simultaneously. spec.completions: total successes needed. Indexed jobs: each pod gets a unique index (0…N-1) via env var.
backoffLimitMax pod failures before Job is marked Failed. Default 6. Retries use exponential backoff (10s→20s→40s…). Failed pod kept for log inspection.
activeDeadlineactiveDeadlineSeconds: wall-clock deadline from Job start. If hit, all running pods terminated. Different from backoffLimit (which is failure count, not time).
Pod failure policyKubernetes 1.27+: podFailurePolicy lets you specify: exit code X = ignore, exit code Y = FailJob (fast fail without retry). Prevents wasting retries on unrecoverable errors.
CleanupttlSecondsAfterFinished: auto-delete Job + pods N seconds after completion. Without it, finished Jobs accumulate forever.
Success policyIndexed job with successPolicy: job succeeds when specific set of indexed pods complete. E.g., only need indexes 0,1,2 to succeed in an MPI job.
Job pod restart trap: spec.template.spec.restartPolicy must be Never or OnFailure. Never use Always — Job controller and kubelet will fight forever, pod restarts infinitely, Job never completes.
🕐
CronJob Controller
schedule → Job objects → pods
MechanismNot event-driven. CronJob controller polls every 10s. Checks if any scheduled time passed since last run. If yes, creates a Job object.
Starting deadlinestartingDeadlineSeconds: if scheduler was down and missed runs, only catch up runs within this window. Without it: all missed runs are backfilled (can flood with Jobs on restart).
Concurrency policyAllow (default): multiple Jobs can run simultaneously. Forbid: skip if previous still running. Replace: kill previous, start new.
History limitssuccessfulJobsHistoryLimit + failedJobsHistoryLimit. Default 3+1. Old Jobs cleaned up automatically. Set to 0 for ephemeral jobs.
Last schedulestatus.lastScheduleTime is updated after every successful Job creation. Used to compute next run time. If controller crashes, it reads this on restart to resume.
CronJob schedule uses controller-manager's timezone by default (UTC). Kubernetes 1.27+: spec.timeZone: "Asia/Jerusalem" is supported. Before that: all cron times were UTC — burning cause of confusion.
Infrastructure Controllers — The Hidden Machinery // what runs without you thinking about it
🌐
NodeLifecycle Controller
detecting node failure + eviction policy
Health detectionMonitors node.status.conditions[Ready]. Updated by kubelet every nodeStatusUpdatePeriod (10s). If no update for node-monitor-grace-period (40s default) → mark Unknown.
Taint managerNodeLifecycle adds taint-based evictions. Node NotReady → add taint node.kubernetes.io/not-ready:NoExecute. Pods without toleration evicted after tolerationSeconds.
Rate limitingEvictions are rate-limited. If <55% of cluster nodes healthy: evictions stop entirely. Prevents entire cluster wipeout on mass network partition. Zone-aware: if only one zone fails, evictions in that zone also stop.
Eviction timelineNode NotReady → 40s → Unknown → 5min → eviction pod1 → pod2 → ... Rate: 0.1 pods/sec per zone by default.
PodDisruptionBudgetNodeLifecycle respects PDB during eviction. Won't evict if doing so would violate minAvailable/maxUnavailable. PDB is checked by the eviction API before deletion.
Node Unknown status + pods in Terminating for 5+ minutes = node was forcibly removed without draining. Pods are "stuck terminating" until node is deleted from API or forcibly cleaned: kubectl delete pod --force --grace-period=0
🔗
EndpointSlice Controller
the glue between Services and running Pods
What it doesWatches Services and Pods. For each Service, creates/updates EndpointSlice objects listing all Ready Pod IPs + ports that match the Service selector.
Slice sizingEach slice holds max 100 endpoints (configurable). Prevents one giant Endpoints object causing huge watch events. For 1000-pod Service: 10 EndpointSlices.
Ready filteringOnly pods that are: Running, podReady=True, and not have deletionTimestamp set are added to EndpointSlices. This is why pods drain traffic during termination.
Topology hintsCan add topology hints to slices so kube-proxy prefers endpoints in the same zone. Requires TopologyAwareHints feature gate + annotation on Service.
kube-proxykube-proxy watches EndpointSlices → programs iptables/IPVS rules. Service traffic path: ClusterIP hit → iptables/IPVS rule → one of the pod IPs in EndpointSlice. No DNS involved for ClusterIP routing.
Service with spec.selector: {} (empty, not absent) matches no pods. EndpointSlice will be empty. Service with spec.selector absent = headless/manual endpoints. Common typo that breaks traffic.
🗑️
GarbageCollector Controller
the cascade delete engine
What it doesWatches ALL resource types. Builds an in-memory owner graph. Detects orphaned objects (owner gone) and deletes them.
GraphBuilderSeparate goroutine maintains a DAG of all objects and their ownerReferences. Updated on every watch event. GC runs against this graph.
PropagationForeground: set foregroundDeletion finalizer on owner → GC deletes dependents → removes finalizer → owner gone. Background: delete owner → GC asynchronously deletes dependents.
Orphan policy--cascade=orphan: removes ownerReferences from all dependents, then deletes owner. Dependents survive as orphans.
CRD deletionDeleting a CRD triggers GC for all CR instances. GC respects finalizers on CRs. If CRs have finalizers, they block deletion. Common source of stuck namespace deletions.
GC is eventually consistent. Delete a Deployment: it's gone from etcd immediately, but RS and Pods may persist for seconds. Don't check "are the pods gone yet" in your CI script — wait with kubectl wait.
💾
PersistentVolume Controller
binding + provisioning + reclaim
Binding loopWatches PVCs (pending) and PVs (available). Finds best-fit PV for PVC. Best-fit = smallest PV that satisfies access modes, capacity, and storageClass. Binds via mutual references.
Dynamic prov.If no matching PV: calls external provisioner via StorageClass provisioner field. Provisioner creates PV + actual storage. PV controller then binds it to PVC.
PhasesPVC: Pending → Bound. PV: Available → Bound → on PVC delete: Released → (reclaim) → Available or deleted.
Reclaim policyRetain: PV goes to Released, data preserved, admin must manually reclaim. Delete: PV + storage deleted. Recycle: deprecated (was rm -rf /mountpath/*).
Pre-bindingclaimRef on PV pre-binds it to a specific PVC. Useful for StatefulSet storage migration — guarantee a specific PV is used for a specific pod.
PVC in Terminating forever: finalizer kubernetes.io/pvc-protection is on all PVCs. Removed only when no pod is actively mounting it. If pod is stuck Terminating, PVC stays Terminating too.
📊
HPA Controller
metrics-driven pod count adjustment
Poll intervalFetches metrics every --horizontal-pod-autoscaler-sync-period (default 15s). Not event-driven — purely polling.
Metric sourcesResource metrics (CPU/mem via metrics-server), Custom metrics (via custom.metrics.k8s.io), External metrics (via external.metrics.k8s.io e.g. Prometheus adapter, KEDA).
Formuladesired = ceil(current × currentMetricValue / targetValue). Across all pods: currentMetricValue = sum(podMetrics) / count(pods).
StabilizationDownscale: stabilization window = 300s default. Takes max recommended replicas in window. Prevents flapping. Upscale window: 0s (react immediately).
Unready podsPods not yet Ready are excluded from metric calculations. Prevents double-scaling while new pods spin up. Called "hot pods".
KEDA + native HPA conflict: if you deploy KEDA ScaledObject AND a manual HPA on the same target, they fight over .spec.replicas. KEDA creates/manages its own HPA internally — never add a separate one.

Full Reconcile Loop Anatomy
every controller follows this exact pattern
// ─── GENERIC CONTROLLER PATTERN ─── // Step 1: SETUP — informers + work queue (run once at startup) func NewController(factory SharedInformerFactory) *Controller { c := &Controller{queue: workqueue.NewRateLimitingQueue(...)} // Register event handlers → just enqueue the key factory.Apps().V1().Deployments().Informer().AddEventHandler( cache.ResourceEventHandlerFuncs{ AddFunc: c.enqueueDeployment, UpdateFunc: func(_, new interface{}) { c.enqueueDeployment(new) }, DeleteFunc: c.enqueueDeployment, // enqueue even on delete }) // Also watch owned RS — reconcile parent Deployment on child change factory.Apps().V1().ReplicaSets().Informer().AddEventHandler( cache.ResourceEventHandlerFuncs{ AddFunc: c.enqueueOwnerDeployment, // walk ownerRef → parent UpdateFunc: func(_, new interface{}) { c.enqueueOwnerDeployment(new) }, DeleteFunc: c.enqueueOwnerDeployment, }) return c } // Step 2: RUN — start N worker goroutines func (c *Controller) Run(workers int, stopCh <-chan struct{}) { // Wait for all informer caches to be populated if !cache.WaitForCacheSync(stopCh, c.deployListerSynced, c.rsListerSynced) { panic("cache sync failed") // fatal — no point running } for i := 0; i < workers; i++ { go wait.Until(c.runWorker, time.Second, stopCh) } <-stopCh } // Step 3: WORKER — pop from queue, reconcile, handle errors func (c *Controller) runWorker() { for { key, quit := c.queue.Get() if quit { return } err := c.syncDeployment(key) if err != nil { c.queue.AddRateLimited(key) // retry with backoff } else { c.queue.Forget(key) // success → reset backoff } c.queue.Done(key) // ALWAYS done, even on error } } // Step 4: SYNC — the actual reconcile logic func (c *Controller) syncDeployment(key string) error { ns, name, _ := cache.SplitMetaNamespaceKey(key) // READ from cache (never from API server) deploy, err := c.deployLister.Deployments(ns).Get(name) if errors.IsNotFound(err) { return nil } // deleted, OK // GET all owned ReplicaSets (from cache, via label selector) allRS, _ := c.rsLister.ReplicaSets(ns).List(labels.Everything()) owned := c.filterActiveRS(allRS, deploy) // COMPUTE desired state (pure, no side effects) desired := c.computeDesiredReplicas(deploy, owned) // APPLY — only write what needs to change return c.scale(deploy, desired, owned) }

The 6 Reconcile Invariants
rules every correct controller must obey
1
Idempotency
Running reconcile 10× on the same state must produce the same result as running it 1×. Never assume "I ran last time."
2
Read from cache only
All reads inside reconcile use the informer Lister, not the API. Lister reads are O(1) in-memory. API reads are N×RPCs that saturate apiserver.
3
Status ≠ Spec update
Use client.Status().Update() for status changes. Regular Update() won't touch status. Mixing them causes resourceVersion conflicts.
4
Handle 409 Conflict
Always wrap writes in RetryOnConflict. Re-fetch object with fresh resourceVersion, recompute, retry. Never ignore conflict errors.
5
Enqueue owner, not child
When a child object (Pod) changes, enqueue the parent owner (ReplicaSet), not the Pod itself. The owner's reconcile loop handles everything.
6
WaitForCacheSync before acting
Start worker goroutines only after ALL required informer caches are synced. Acting on empty cache = deleting objects that actually exist.

Controller-Manager Failure Modes
what breaks when this process dies
New leader takes 15sDuring failover: no reconciles happen. Pods crash → not replaced. Nodes fail → no taints applied. Endpoints → stale. Scale events → ignored.
Cache rebuild on startNew leader must List all objects. On large clusters (100k objects): List can take 10-30s. Reconciles only start after cache is fully populated.
Work queue lostWork queue is in-memory. Not persisted. On restart: queue is empty. Informer resync (within resyncPeriod) will re-queue everything. Nothing permanently lost.
Split brain dangerTwo leaders simultaneously (Lease bug, clock skew): both reconcile same objects → duplicate RS creations, conflicting patches. The 409 conflict mechanism partially protects but not fully.

ResourceQuota Controller
usage tracking — enforcement is in admission, not here
Role of controllerComputes and updates status.used on ResourceQuota objects. It does NOT enforce limits — that's the ResourceQuota admission plugin in apiserver.
What it tracksCounts all resources in namespace matching the quota's scope: pods, services, PVCs, ConfigMaps, Secrets, CPU requests/limits, memory requests/limits, etc.
ScopesQuota can be scoped to: BestEffort / NotBestEffort, Terminating / NotTerminating pods. Allows fine-grained limits per QoS class.
LimitRangeDifferent from ResourceQuota. LimitRange (admission plugin) sets defaults and max/min per container. Quota tracks namespace totals. Both needed for full governance.
Priority classesscopeSelector: matchExpressions: [{operator: In, scopeName: PriorityClassName, values: [high-priority]}] — separate quota for system-critical vs batch workloads.
ResourceQuota admission check is not transactional. Two concurrent creates can both pass the quota check, then both succeed, exceeding the limit. The controller then notices overuse and updates status.used — but the objects already exist. This is a known eventual consistency limitation.
📁
Namespace Controller
cascade deletion — more complex than it looks
What triggers itkubectl delete namespace foo → namespace gets deletionTimestamp. Namespace controller takes over.
Deletion sequence1. Set phase to Terminating. 2. List all resource types in namespace. 3. Delete each resource type (in parallel). 4. Wait until all gone. 5. Remove kubernetes finalizer. 6. Namespace deleted.
DiscoveryController uses API discovery (/api + /apis) to find all resource types that are namespace-scoped. Deletes ALL of them. Including your CRDs.
Stuck namespaceIf any resource in the namespace has a finalizer whose controller is dead, namespace stays Terminating forever. The finalizer must be cleared manually.
Fast-trackResources with ownerReferences to the namespace are deleted by GC controller in parallel with namespace controller. Speeds up large namespace deletion.
# Find what's blocking namespace deletion kubectl api-resources --verbs=list --namespaced -o name \ | xargs -I {} kubectl get {} -n stuck-ns \ --ignore-not-found 2>/dev/null # Nuclear option — remove finalizer kubectl get namespace stuck-ns -o json \ | jq '.spec.finalizers = []' \ | kubectl replace --raw \ /api/v1/namespaces/stuck-ns/finalize -f -

Diagnostic Runbook
when controllers aren't acting
1
Is there a leader?
kubectl get lease kube-controller-manager -n kube-system. If holderIdentity is stale (renewTime > 30s ago) — no active leader. Check if pods are running on all control-plane nodes.
2
Is the controller disabled?
Check --controllers flag in pod spec. Is your controller in the list? CRD-based controllers (from Operators) run in their own pods — not in kube-controller-manager at all.
3
Is the queue backed up?
Check workqueue_depth metric. If depth growing: reconcile is slow or failing. Check workqueue_retries_total — high retries = persistent error in sync logic.
4
RBAC issue?
Controller-manager uses system:kube-controller-manager service account. kubectl auth can-i update deployments --as=system:kube-controller-manager -n default. Custom controllers may have missing RBAC.
5
Admission webhook blocking writes?
Controller writes to apiserver go through admission. If a validating webhook rejects controller's create/patch: controller retries forever with backoff, never succeeds. Check --v=4 logs for 403/422 responses.
Vicious Interview Traps — Controller Edition // every question that catches people out
⚡
The Questions That Separate Seniors from Juniors
verbatim answers for each
QuestionWrong answer (junior)Right answer (senior)
"Does Deployment controller create Pods?" Yes, it manages pods No. Deployment ctrl creates ReplicaSets. ReplicaSet ctrl creates Pods. Two-step chain.
"Do controllers in kube-controller-manager have separate leaders?" Yes, each controller elects its own leader No. The entire binary has ONE Lease. All 30+ controllers start/stop as a unit.
"How does a controller read object state?" It calls the Kubernetes API (etcd) From the informer's in-memory cache via Lister. Never directly from apiserver during reconcile.
"What happens to the work queue when controller-manager crashes?" Work items are lost forever Queue is in-memory → lost. But informer resync on next leader startup re-queues everything. Idempotent reconcile means nothing is lost permanently.
"Why does kube-controller-manager need informers instead of polling the API?" For performance To avoid N×M watch explosion, provide O(1) local reads via in-memory cache, handle reconnects gracefully, and enable event-driven reconciliation with deduplication.
"A StatefulSet pod is deleted manually. What happens to its PVC?" PVC is deleted with the pod PVC survives. StatefulSet controller creates a new pod with the same name. New pod reattaches the existing PVC with all data intact.
"Node goes NotReady. When are its pods evicted?" Immediately when NotReady After node-monitor-grace-period (40s) → Unknown. Then after tolerationSeconds on NoExecute taint (default 300s) → eviction starts. Rate-limited. Stops if <55% nodes healthy.
"Can a controller talk directly to etcd?" Yes, via the etcd client library No. Controllers only talk to apiserver. Apiserver is the sole writer to etcd. Bypassing apiserver would skip admission, RBAC, and all watch subscriptions.
"What does GarbageCollector controller do that no other controller does?" It's just like other controllers It watches ALL resource types (not just one), builds an in-memory graph of ALL owner relationships, and manages cross-resource cascade deletion. It's the only controller with cluster-wide object graph awareness.
"DaemonSet bypasses the scheduler — is that true?" No, scheduler places DaemonSet pods Partially true. DaemonSet controller sets nodeName directly, bypassing Filter+Score. But the scheduler may still be involved for resource accounting in newer versions (NodeAffinity mode).
