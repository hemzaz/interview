01Helm Fundamentals & the Helm 2 → 3 Shift

Helm is a templating engine + package manager for Kubernetes manifests. A chart is the package; a release is a chart deployed with a specific set of values into a namespace.

The Tiller removal (Helm 2 → Helm 3)

Helm 2 ran a server-side component called Tiller inside the cluster. Tiller held a broad service account (often cluster-admin) and executed all install/upgrade logic server-side — a well-known security liability (anyone who could talk to Tiller's gRPC endpoint could act with Tiller's permissions, and there was no per-user RBAC differentiation).

Helm 3 removed Tiller entirely. The Helm CLI now talks directly to the Kubernetes API server using the user's own kubeconfig/RBAC credentials — same permission model as kubectl. Consequences:

  • No cluster-side component to install, secure, or upgrade.
  • RBAC is enforced per-user via normal Kubernetes auth, not a shared Tiller identity.
  • Release information moved from ConfigMaps in kube-system (Tiller's namespace) to Secrets in the release's own namespace.
  • Releases became namespace-scoped by name — two releases with the same name can exist in different namespaces (in Helm 2, release names were cluster-global).
  • helm template gained parity for pure rendering without any cluster contact.
  • 3-way strategic merge patch on upgrade (compares chart, live state, and previous release manifest) instead of Helm 2's naive 2-way diff — this is why Helm 3 correctly removes fields you deleted from a template, where Helm 2 would leave orphaned fields behind.
⚠ Trap

Candidates often say "Helm 3 added namespaces to charts." Wrong framing — Helm 2 had namespace support for resources, but release names were global across the cluster and all release metadata lived centrally in Tiller's namespace. Helm 3's real architectural change is removing the server-side component and moving to client-only + native RBAC; namespace-scoped release storage is a consequence of that, not the headline.

Q&A

Q: Why was Tiller considered a security risk?
A: It ran as a single in-cluster gRPC service, usually granted cluster-admin, with no fine-grained authorization of who could send it commands — any pod/user with network access to the Tiller service inherited its blast radius. Helm 3 eliminates the shared privileged identity by executing entirely client-side against the API server under the caller's own credentials.

02Chart Structure

Standard chart layout
mychart/
├── Chart.yaml           # metadata: name, version, appVersion, dependencies
├── values.yaml           # default configuration values
├── values.schema.json    # (optional) JSON Schema to validate values
├── charts/               # subcharts (dependencies), vendored or via `helm dependency update`
├── crds/                 # CRDs installed before templates render, never templated, never upgraded/deleted by Helm
├── templates/
│   ├── _helpers.tpl      # named templates ("partials"), prefixed with underscore = not rendered as a manifest
│   ├── NOTES.txt         # post-install/upgrade usage notes, templated, printed to stdout
│   ├── deployment.yaml
│   ├── service.yaml
│   ├── tests/
│   │   └── test-connection.yaml   # helm test hooks
│   └── ...
├── .helmignore           # patterns excluded when packaging
└── README.md
FilePurposeKey notes
Chart.yamlChart metadataapiVersion: v2 (Helm 3) required; version = chart version (SemVer), appVersion = version of the packaged app (free text, informational only)
values.yamlDefault values consumed by templatesOverridden by -f/--set at install/upgrade time
templates/Go templates rendered into K8s manifestsFiles starting with _ are not rendered directly (partials)
_helpers.tplNamed template definitions (define/include)Convention, not a magic filename — any _*.tpl works the same way
charts/Subchart dependencies (vendored .tgz or unpacked dirs)Populated by helm dependency update based on Chart.yaml's dependencies: block
crds/CustomResourceDefinitionsInstalled once, first, before any template rendering; Helm never upgrades or deletes CRDs in this directory — you manage that manually
⚠ Trap

"Why didn't my CRD update after I bumped the chart version?" — because crds/ is intentionally excluded from the normal template/upgrade/delete lifecycle to avoid Helm ever deleting a CRD (which would cascade-delete every CR instance in the cluster). If you need CRDs to evolve with releases, either manage them with a separate tool/process or put them in templates/ with explicit care (and accept they'll behave like normal — but riskier — resources).

03Templating: Go text/template + Sprig

Helm templates are Go's text/template syntax, extended with the Sprig function library (string, math, date, crypto, list/dict helpers) plus Helm-specific functions.

Built-in objects
ObjectContains
.ValuesMerged values (defaults + overrides)
.Release.Name, .Namespace, .IsInstall, .IsUpgrade, .Revision, .Service
.ChartContents of Chart.yaml: .Name, .Version, .AppVersion
.FilesNon-template files in the chart (.Get, .GetBytes, .Glob)
.CapabilitiesK8s/Helm version info, API availability (.APIVersions.Has)
.Template.Name / .BasePath of the currently rendering file
Named templates (_helpers.tpl)
{{- define "mychart.fullname" -}}
{{- printf "%s-%s" .Release.Name .Chart.Name | trunc 63 | trimSuffix "-" -}}
{{- end -}}

{{- define "mychart.labels" -}}
app.kubernetes.io/name: {{ .Chart.Name }}
app.kubernetes.io/instance: {{ .Release.Name }}
helm.sh/chart: {{ .Chart.Name }}-{{ .Chart.Version }}
{{- end -}}
Using them: include vs template
metadata:
  labels:
    {{- include "mychart.labels" . | nindent 4 }}

include returns a string you can pipe (e.g. into nindent); the older template action prints directly and cannot be piped. Prefer include in almost all cases.

Common Sprig helpers seen in interviews
{{ .Values.name | default "my-app" }}
{{ .Values.replicaCount | int }}
{{ list "a" "b" "c" | join "," }}
{{ .Values.tag | quote }}
{{ dict "key" "value" }}
{{ now | date "2006-01-02" }}
{{ .Values.config | toYaml | nindent 2 }}
{{ required "image.tag is required!" .Values.image.tag }}

04Templating Gotchas (favorite interview probes)

Whitespace control: {{- and -}}

A plain {{ }} action leaves surrounding whitespace/newlines untouched, which frequently produces blank lines or bad YAML indentation. The hyphen variants trim whitespace on that side:

{{- if .Values.enabled }}     # trims preceding whitespace/newline
key: value
{{- end }}                    # trims preceding whitespace/newline

{{- trims whitespace (including the newline) immediately before the tag; -}} trims whitespace immediately after. Get this wrong and you get invalid YAML (extra blank lines are usually harmless, but missing/extra indentation is not).

indent vs nindent

{{ .Values.labels | toYaml | indent 4 }}
# vs
{{- .Values.labels | toYaml | nindent 4 }}

nindent = "newline then indent" — it prepends a newline before indenting each line. Use nindent when injecting a multi-line block after a YAML key (labels:\n {{- ... | nindent 4 }}); use plain indent only when you already control the leading newline.

toYaml for passing through arbitrary structures

resources:
  {{- toYaml .Values.resources | nindent 2 }}

Lets users supply an arbitrary nested YAML block in values.yaml (e.g. full resources:, affinity:, tolerations: stanzas) without the chart author hand-templating every possible field.

required and default

image: {{ required "An image repository is required" .Values.image.repository }}
tag: {{ .Values.image.tag | default .Chart.AppVersion }}

required fails the render immediately with a clear message — much better than a cryptic downstream K8s API error from a missing field. default substitutes a fallback only when the value is Go's zero-value/empty (empty string, nil, 0, false) — this can bite you (see trap below).

Scoping: with, range, and the $ root

{{- with .Values.service }}
type: {{ .type }}
port: {{ .port }}
{{- end }}

{{- range .Values.env }}
- name: {{ .name }}
  value: {{ .value | quote }}
{{- end }}

{{- range .Values.env }}
- name: {{ .name }}
  value: {{ .value }}
  release: {{ $.Release.Name }}   {{/* $ = root context, since "." is rescoped inside range/with */}}
{{- end }}
⚠ Trap

Inside with or range, . is rebound to the iterated/scoped value — .Release.Name will fail because .Release no longer exists in that scope. You must capture the root context first ({{- $root := . }}) or reference the pre-defined $ variable, which Go templates always bind to the original root context regardless of scope changes.

⚠ Trap

default treats Go zero-values as "empty" — so {{ .Values.replicaCount | default 3 }} will replace an explicit replicaCount: 0 with 3, which is almost never what you want for a deliberate "scale to zero." Use {{ if hasKey .Values "replicaCount" }}/kindIs checks or restructure with a pointer-like sentinel when zero/false must be distinguishable from "unset."

05Values & Override Precedence

Highest wins. From lowest to highest priority:

  1. Chart's own values.yaml (the defaults shipped in the chart)
  2. Parent chart's values.yaml overrides for a subchart (scoped under the subchart's name)
  3. -f / --values files, in the order given on the command line (later files win, merged recursively/deep-merge)
  4. --set, --set-string, --set-json, --set-file flags (highest — always win over -f files)
Precedence in practice
helm upgrade myapp ./mychart \
  -f values.yaml \
  -f values-prod.yaml \
  --set image.tag=v1.2.3 \
  --set-string replicaCount=3

values-prod.yaml overrides values.yaml for overlapping keys (deep merge, not replace-whole-file); --set flags override everything from -f.

⚠ Trap

--set type coercion. --set parses values and tries to infer type: --set replicaCount=3 becomes the integer 3, --set enabled=true becomes boolean true. This breaks when you actually want a string that looks numeric/boolean, e.g. a tag: "01" or a numeric-looking Git SHA — Helm will silently coerce/strip it. Use --set-string to force string type: --set-string image.tag=00123. Also, arrays via --set use a fragile {a,b,c} syntax and commas inside values must be escaped (\,) — for anything nested or list-heavy, prefer -f files or --set-json.

⚠ Trap

Lists/arrays in -f value files are replaced wholesale, not merged element-by-element, when a later file also sets that key. Maps merge recursively; lists do not. This surprises people expecting "just append my extra item."

Q&A

Q: How do you pass values to a subchart?
A: Namespace them under the subchart's name (as declared as a dependency) in the parent's values.yaml: mysubchart:\n  replicaCount: 2. Use global: for values you want visible to every chart/subchart in the tree without namespacing.

06Release Lifecycle

Core commands
helm install myrelease ./mychart -n myns --create-namespace
helm upgrade myrelease ./mychart -n myns -f values-prod.yaml
helm upgrade --install myrelease ./mychart -n myns   # idempotent install-or-upgrade, common in CI

helm rollback myrelease 3 -n myns    # roll back to revision 3
helm history myrelease -n myns       # list all revisions
helm status myrelease -n myns        # current state, notes
helm list -n myns                    # list releases in a namespace
helm list -A                         # all namespaces
helm uninstall myrelease -n myns     # delete release + resources (history purged too by default; use --keep-history to retain — see below)
CommandWhat it does
helm installCreates revision 1 of a new release
helm upgradeRenders a new revision, 3-way merges against live + previous state, applies diff
helm upgrade --installInstalls if the release doesn't exist, upgrades if it does — the standard CI pattern
helm rollback <release> <rev>Re-applies a prior revision's rendered manifests as a new revision (rollback itself creates a new revision entry, it doesn't rewind history)
helm historyShows revision number, status (deployed/superseded/failed), chart version, app version, description
helm uninstallDeletes all release resources; release Secret history is deleted by default (use --keep-history to retain it for later helm history/rollback-from-uninstalled)
⚠ Trap

helm rollback creates a new revision — it does not delete forward history. If you're on revision 5 and roll back to revision 2, you land on revision 6 (content = revision 2's manifests). helm history will show all 6 entries. This matters for auditing and for anyone assuming rollback "erases" the bad deploy.

⚠ Trap

Helm rollback vs GitOps drift. If Argo CD or Flux owns this release (deploying from a Git-tracked HelmRelease/Application), running helm rollback by hand creates a live cluster state that diverges from Git — the GitOps controller will detect drift and either auto-heal (re-apply Git's desired state, silently undoing your manual rollback) or flag OutOfSync. The correct GitOps-native rollback is git revert the commit that introduced the bad change and let the controller reconcile — not a direct helm CLI call against a GitOps-managed release.

Failed upgrades

--atomic flag: if the upgrade fails, Helm automatically rolls back to the last successful revision. Without it, a failed upgrade can leave the release in a failed state requiring manual helm rollback.

helm upgrade myrelease ./mychart --atomic --timeout 5m

07Release Storage — Secrets In-Cluster

Helm 3 stores release state (rendered manifests, values used, chart metadata) as Kubernetes Secrets in the release's own namespace, one Secret per revision, named like:

sh.helm.release.v1.<release-name>.v<revision>

Type: helm.sh/release.v1. The Secret's data is a gzip-compressed, base64-encoded blob of the release manifest. This is why kubectl get secrets in an app namespace shows entries you never created directly — they're Helm's own bookkeeping, not application secrets.

Note

Helm 2 used ConfigMaps (and later, optionally, Secrets) stored centrally in Tiller's namespace (kube-system) rather than the release's own namespace. Helm 3's default backend is Secrets, colocated with the release — both a security improvement (encryption-at-rest where configured, tighter RBAC scoping) and a namespace-isolation improvement.

Storage backend is pluggable (--storage-driver or HELM_DRIVER env var supports secrets (default), configmaps, memory, or sql).

08Helm Hooks

Hooks let a chart run Kubernetes Jobs/other resources at specific points in the release lifecycle — outside the normal template render/apply flow.

HookFires
pre-installAfter templates are rendered, before any resource is created
post-installAfter all resources are created
pre-upgradeBefore any resource is upgraded during an upgrade
post-upgradeAfter all resources are upgraded
pre-deleteBefore any resource is deleted on uninstall
post-deleteAfter all resources are deleted
pre-rollback / post-rollbackAround rollback operations
testOn explicit helm test invocation only
Hook annotation example (DB migration Job)
apiVersion: batch/v1
kind: Job
metadata:
  name: {{ include "mychart.fullname" . }}-migrate
  annotations:
    "helm.sh/hook": pre-upgrade,pre-install
    "helm.sh/hook-weight": "-5"
    "helm.sh/hook-delete-policy": before-hook-creation,hook-succeeded
spec:
  template:
    spec:
      restartPolicy: Never
      containers:
        - name: migrate
          image: {{ .Values.image.repository }}:{{ .Values.image.tag }}
          command: ["./migrate.sh"]

Hook weights

helm.sh/hook-weight (string-encoded integer) controls ordering among multiple hooks bound to the same event — lower weights execute first. Negative weights are valid and commonly used to run things before weight-0/unweighted hooks.

Delete policies

PolicyBehavior
before-hook-creation (default)Delete the previous hook resource before creating a new one for this event
hook-succeededDelete the hook resource right after it succeeds
hook-failedDelete the hook resource if it fails
⚠ Trap

Hook resources are not tracked as part of the normal release manifest — they don't participate in helm uninstall's automatic cleanup unless you set a delete policy, and they're invisible to helm diff/3-way-merge logic that governs normal templates. This is by design (hooks are one-shot side effects, not desired-state resources), but it means orphaned hook Jobs are a common real-world cleanup issue if delete policies are misconfigured.

09Library Charts & Dependencies

Library charts

A chart with type: library in Chart.yaml contains only reusable named templates (define blocks) — it produces no Kubernetes resources of its own and cannot be installed standalone. Consuming charts declare it as a dependency purely to include its helper templates (common label sets, standard Deployment/Service scaffolding, etc.) — this is Helm's answer to "don't repeat yourself" across a fleet of charts.

Chart dependencies

Chart.yaml dependencies block
apiVersion: v2
name: myapp
version: 1.0.0
dependencies:
  - name: postgresql
    version: "12.x.x"
    repository: "https://charts.bitnami.com/bitnami"
    condition: postgresql.enabled
  - name: common
    version: "2.x.x"
    repository: "https://charts.example.com"

helm dependency update ./mychart   # fetches deps into charts/ as .tgz, writes Chart.lock
helm dependency build ./mychart    # rebuilds charts/ from an existing Chart.lock (no re-resolution)
helm dependency list ./mychart     # shows declared vs. present dependencies

condition ties a subchart's inclusion to a boolean value path (commonly <subchart>.enabled); tags offer a similar group-level toggle. Chart.lock pins resolved versions — check it into version control just like a lockfile in any package manager.

⚠ Trap

helm dependency update re-resolves version ranges against the repo index (can pull a newer patch/minor if your range allows it) and rewrites Chart.lock. helm dependency build is the reproducible, CI-safe operation — it trusts the existing lock file and only re-downloads. Forgetting this distinction leads to "works on my machine" dependency drift.

10helm template vs helm install

helm templatehelm install / upgrade
Cluster contactNone required (fully offline rendering)*Requires live API server connection
OutputRendered YAML to stdoutApplies resources to the cluster
Release trackingNone — no Secret is createdCreates/updates a release Secret
HooksRendered as regular manifests but NOT executed (unless --hook-only-style tooling wraps it)Executed at appropriate lifecycle points

* helm template can optionally validate against live cluster API capabilities with --validate/--is-upgrade, but its core rendering does not require a cluster.

Q&A

Q: How does Argo CD render a Helm chart without "installing" a Helm release?
A: Argo CD (and Flux's Helm controller, in template-only paths) shells out to effectively helm template to produce plain manifests, then applies/diffs those manifests itself using its own reconciliation engine — it does not use Helm's release/Secret bookkeeping at all in the default Argo CD Helm integration. This is a key reason helm history/helm rollback don't apply to an Argo CD-managed Helm app; Argo CD's own revision history (Git commits) is the source of truth instead.

11Chart Repositories & OCI Registries

Classic HTTP chart repos

helm repo add bitnami https://charts.bitnami.com/bitnami
helm repo update
helm search repo bitnami/postgresql
helm install mydb bitnami/postgresql

A classic repo is just a static index.yaml (chart metadata + download URLs) served over HTTP(S) — no special server logic required, can be hosted on any static file host (S3, GitHub Pages, etc.).

OCI registries (now the recommended default)

helm registry login registry.example.com -u user -p pass
helm push mychart-1.0.0.tgz oci://registry.example.com/charts
helm install myrelease oci://registry.example.com/charts/mychart --version 1.0.0
helm pull oci://registry.example.com/charts/mychart --version 1.0.0

Since Helm v3.8 OCI support graduated out of experimental and is enabled by default — charts are pushed/pulled as OCI artifacts through any OCI-compliant registry (Docker Hub, GHCR, ECR, ACR, Harbor, etc.), letting orgs reuse existing container registry infrastructure and auth instead of standing up a separate chart-repo server. helm search repo and index.yaml concepts don't apply to OCI — discovery works differently (tag listing via registry APIs), and there's no separate "repo add" step; you reference the OCI URL directly.

12Chart Testing & Linting

helm lint ./mychart                      # static analysis: YAML validity, required fields, best-practice checks
helm lint ./mychart -f values-prod.yaml  # lint against a specific values overlay
helm template ./mychart | kubeval -      # (external tool) validate rendered output against K8s schemas
helm install myrelease ./mychart --dry-run --debug   # render + validate against live API without applying
helm test myrelease -n myns              # runs any resources annotated helm.sh/hook: test

helm.sh/hook: test resources (conventionally Pods or Jobs under templates/tests/) are only ever run by an explicit helm test invocation — they're not part of install/upgrade at all. Common pattern: a Pod that curls the just-deployed Service and asserts a 200.

13Kustomize Fundamentals

Kustomize is a template-free configuration customization tool: you keep plain, valid Kubernetes YAML (a "base") and layer declarative patches on top (via "overlays") rather than parameterizing the YAML itself. It has been built into kubectl since v1.14 (kubectl apply -k, kubectl kustomize) with its own standalone kustomize CLI for more advanced/newer features.

base/ ├── kustomization.yaml ├── deployment.yaml └── service.yaml overlays/ ├── dev/ │ ├── kustomization.yaml # references ../../base, patches replicas down, adds -dev suffix │ └── patch-replicas.yaml ├── staging/ │ └── kustomization.yaml └── prod/ ├── kustomization.yaml # references ../../base, patches resources up, adds prod labels └── patch-resources.yaml
base/kustomization.yaml
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization
resources:
  - deployment.yaml
  - service.yaml
commonLabels:
  app.kubernetes.io/part-of: myapp
overlays/prod/kustomization.yaml
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization
namespace: prod
namePrefix: prod-
resources:
  - ../../base
patches:
  - path: patch-resources.yaml
    target:
      kind: Deployment
      name: myapp
replicas:
  - name: myapp
    count: 5
Apply
kubectl apply -k overlays/prod/
kubectl kustomize overlays/prod/    # render only, no apply
kustomize build overlays/prod/      # standalone CLI, same rendering (often newer features)
Note

The kubectl-bundled Kustomize version typically lags behind the standalone kustomize CLI release. For newer features (e.g. certain component behaviors, replacements transformer), use the standalone binary and feed rendered YAML to kubectl apply -f -.

14Patch Strategies: Strategic Merge vs JSON 6902

Strategic merge patch

YAML that looks like a partial version of the target resource; Kubernetes' strategic-merge semantics (which understand list merge keys like container name) decide how to combine it with the base.

# patch-resources.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp
spec:
  template:
    spec:
      containers:
        - name: myapp        # matched by name, merged into existing container entry
          resources:
            limits:
              memory: "512Mi"
              cpu: "500m"

JSON 6902 patch

An explicit list of RFC 6902 JSON Patch operations (add/remove/replace/move/copy/test) addressed by JSON pointer path — necessary when you need to target array elements by index, remove a field entirely, or do something strategic-merge can't express cleanly.

patches:
  - target:
      kind: Deployment
      name: myapp
    patch: |-
      - op: replace
        path: /spec/replicas
        value: 3
      - op: remove
        path: /spec/template/spec/containers/0/env/2
Strategic mergeJSON 6902
SyntaxPartial resource YAMLList of path-based operations
Best forAdding/overriding fields, merging by known list keys (container name, port name)Precise index-based edits, deletions, moves, anything strategic-merge can't target
ReadabilityHigher — looks like the resource itselfLower — path syntax, less self-documenting
CRD supportFalls back to standard merge (no merge-key awareness) unless the CRD defines patch strategiesWorks uniformly regardless of resource type
⚠ Trap

Patch target selectors matter — a patch with no match is silently a no-op in some Kustomize versions/configurations, or an explicit error in others. The target block matches on group, version, kind, name (supports regex), namespace, and labelSelector/annotationSelector. A typo in name or an overly narrow kind is one of the most common real-world Kustomize debugging sessions — always verify with kubectl kustomize . that the patch actually landed where intended before applying.

15Generators & Hash Suffixes

configMapGenerator and secretGenerator let you generate ConfigMaps/Secrets from literals, files, or env files, rather than hand-writing the resource.

configMapGenerator:
  - name: app-config
    literals:
      - LOG_LEVEL=info
    files:
      - config.properties
secretGenerator:
  - name: db-creds
    envs:
      - db.env
    type: Opaque
generatorOptions:
  disableNameSuffixHash: false   # default: false (hash suffix ON)

Why the hash suffix exists

Kustomize appends a content hash to the generated ConfigMap/Secret's name (e.g. app-config-8f2c6b9d4t). Any consuming Deployment/Pod spec that Kustomize also processes gets its volumeMounts/envFrom references automatically rewritten to the new hashed name. This is the mechanism that forces a rollout when config changes — because the ConfigMap/Secret name itself changes, the Pod template's spec changes, which Kubernetes deployment controllers detect as a genuine spec diff and roll new Pods. Without the hash, updating a ConfigMap in place would not trigger a rollout (Kubernetes doesn't watch ConfigMap/Secret content for that), leaving old Pods running stale config until they happen to restart.

⚠ Trap

This is one of the most-tested Kustomize interview facts: updating a plain ConfigMap referenced by name does not automatically restart Pods. secretGenerator/configMapGenerator's hash-suffix behavior is specifically the workaround — it changes the resource's identity so the owning workload's pod template spec changes too, and normal rolling-update mechanics take over. If disableNameSuffixHash: true is set, you lose this auto-rollout trigger and are back to manual restart (kubectl rollout restart) or another mechanism (e.g. a checksum annotation, common in Helm charts via checksum/config).

Q&A

Q: Helm doesn't have generators/hash-suffixes — how do Helm charts solve the same "config change needs a rollout" problem?
A: Common pattern: annotate the pod template with a checksum of the rendered ConfigMap/Secret, e.g. checksum/config: {{ include (print $.Template.BasePath "/configmap.yaml") . | sha256sum }} in the Deployment's spec.template.metadata.annotations. Since that's part of the pod template spec, any content change alters the checksum, alters the pod template, and triggers a rollout — conceptually the same trick as Kustomize's hash suffix, implemented via templating instead of generated resource identity.

16Common Transformers

FieldEffect
commonLabelsAdds labels to all resources and updates matching label selectors (Service selectors, Deployment matchLabels) so they stay consistent — this is more than a metadata stamp, it actively rewires selectors
commonAnnotationsAdds annotations to all resources (metadata only, no selector rewiring)
namePrefix / nameSuffixPrepends/appends a string to every resource name, and updates internal references (e.g. a ConfigMap reference in a Deployment) accordingly
namespaceSets/overrides the namespace on all namespaced resources
imagesRewrites container image name/tag/digest without touching the rest of the Deployment spec
replicasSets replica count for a named workload (cleaner than a full strategic-merge patch for this one field)
vars / replacementsInjects a value from one resource's field into another (e.g. propagate a generated Service name into a ConfigMap); replacements is the modern, more explicit successor to the deprecated vars
⚠ Trap

commonLabels in legacy Kustomize versions applies to immutable selector fields too (Service spec.selector, Deployment spec.selector.matchLabels) — which is fine on first apply but will hard-fail an update if you change commonLabels later, since those selector fields are immutable on existing objects. Newer Kustomize APIs introduced labels (with more granular control over which fields get touched) as the more flexible replacement for blanket commonLabels/commonAnnotations behavior.

17Kustomize Components

A Component (kind: Component) is a reusable, optional slice of configuration that overlays can opt into — unlike a base (which every overlay inherits wholesale), a component is designed to be mixed into multiple, unrelated overlays selectively (e.g. "add Istio sidecar config," "add a debug sidecar," "enable a feature flag's extra resources").

# components/debug-sidecar/kustomization.yaml
apiVersion: kustomize.config.k8s.io/v1alpha1
kind: Component
patches:
  - path: add-debug-container.yaml
    target:
      kind: Deployment

# overlays/dev/kustomization.yaml
resources:
  - ../../base
components:
  - ../../components/debug-sidecar

Key distinction from a base: components are additive, composable building blocks meant to be referenced from many different overlays (mix-and-match), while a base is the singular foundation an overlay is built from (typically one base per overlay chain).

Using them together

Two common combined patterns:

  • Helm renders, Kustomize patches: helm template ./chart | kustomize build - (or via a helmCharts/helmGlobals field in kustomization.yaml, supported by the standalone kustomize CLI) — useful when you consume a third-party Helm chart but need small environment-specific tweaks the chart's own values.yaml doesn't expose.
  • kustomization.yaml's native helmCharts field: lets Kustomize itself invoke Helm as a chart inflator, then apply Kustomize patches on the rendered output, all from one kustomize build invocation.
# kustomization.yaml with inline Helm chart inflation (standalone kustomize CLI, --enable-helm)
helmCharts:
  - name: redis
    repo: https://charts.bitnami.com/bitnami
    version: 18.1.5
    releaseName: myredis
    valuesFile: redis-values.yaml
patches:
  - path: patch-redis-resources.yaml
    target:
      kind: StatefulSet
      name: myredis-redis-master
Q&A

Q: If Kustomize has no templating, how do you avoid duplicating near-identical YAML across many overlays?
A: The base/overlay model itself is the DRY mechanism — put everything common in the base, express only the diff in each overlay's patches. Components extend this for cross-cutting, optional concerns. For genuinely parameterized needs (loops, conditionals, computed values), that's a signal to reach for Helm instead, or to generate the Kustomize YAML with an external templating step before it ever reaches Kustomize.

19GitOps Consumption: Argo CD & Flux

Argo CD

  • An Application resource's spec.source declares either a Helm chart (path + helm.valueFiles/helm.parameters) or a Kustomize directory (path + kustomize.namePrefix/images overrides etc.) — Argo CD auto-detects the tool from directory contents (presence of Chart.yaml vs kustomization.yaml) if not explicitly specified.
  • Internally, Argo CD's repo-server component runs the equivalent of helm template or kustomize build to produce plain manifests, then diffs those against live cluster state using its own reconciliation loop — it does not use Helm's release/Secret bookkeeping or invoke helm install/upgrade directly.
  • Multi-source apps can combine a Helm chart with a separate Kustomize-applied values overlay in one Application.

Flux

  • HelmRelease + HelmRepository/OCIRepository CRDs drive Helm charts through Flux's helm-controller, which does perform real Helm actions server-side (it manages actual Helm releases, storage, and history — closer to genuine helm upgrade semantics than Argo CD's render-and-diff approach).
  • Kustomization CRD (via kustomize-controller) applies a Kustomize-built directory tree, reconciling it against a GitRepository/OCIRepository source.
  • Flux commonly layers its own Kustomization CRD on top of plain manifests or even on top of Helm-rendered output as a generic "apply this path, patch these fields" mechanism — distinct from Kustomize's own kustomization.yaml patches, though conceptually similar.
⚠ Trap

Helm rollback vs GitOps drift, revisited: because Argo CD/Flux continuously reconcile toward what's declared in Git, any manual helm rollback, kubectl edit, or kubectl apply against a GitOps-managed resource is fighting a controller that will detect the drift and (depending on sync policy — especially selfHeal: true in Argo CD) revert it back to Git's declared state, often within seconds to minutes. The interview-safe answer: GitOps-managed workloads are rolled back by reverting the Git commit, not by imperative cluster commands.

Q&A

Q: Does Argo CD create real Helm releases you can see with helm list?
A: No (in the default/common integration) — Argo CD renders the chart to plain manifests and manages them through its own Application/sync state; there's no corresponding sh.helm.release.v1.* Secret and helm list will show nothing for it. Flux's helm-controller, by contrast, does drive genuine Helm releases and helm list/helm history will reflect them.

20Rapid-Fire Q&A

Q&A

Q: What's the difference between Chart.yaml's version and appVersion?
A: version is the chart's own SemVer, bumped whenever the chart's templates/structure change — it's what helm repo/OCI registries index and what helm dependency ranges match against. appVersion is free-form text describing the version of the application the chart deploys (e.g. the app's own release tag) — purely informational, not used by Helm's dependency resolution.

Q&A

Q: Can two Helm releases share the same name in the same cluster?
A: Not in the same namespace (release names are unique per-namespace in Helm 3), but yes across different namespaces — namespace + release name together form the unique key, unlike Helm 2 where release names were cluster-global.

Q&A

Q: How do you debug a chart that fails to render?
A: helm template ./chart --debug (shows the merged values used and full rendered output, or the exact template error with file/line), helm lint for structural issues, and helm install --dry-run --debug to also validate against the live API server's schema without actually applying.

Q&A

Q: What happens if you delete a resource's template from a chart and run helm upgrade?
A: Helm 3's 3-way merge diff (comparing the chart's new rendered manifest, the previous release's rendered manifest, and live cluster state) detects the resource is gone from the new render and deletes it from the cluster. Helm 2's naive 2-way diff (old chart vs new chart, ignoring live drift) could miss this in certain edge cases — one of the concrete correctness improvements from the Helm 3 rewrite.

Q&A

Q: Why prefer Kustomize's configMapGenerator over just editing a ConfigMap's data in place across environments?
A: Beyond DRY composition (base data + per-overlay literals/files), the generated hash suffix solves the "config changed but nothing restarted" problem for free — editing in place doesn't change the workload's pod template spec, so no rollout is triggered; the generator's name-change mechanism does trigger one automatically.

Q&A

Q: Is Kustomize a replacement for Helm?
A: Not fully — they solve overlapping but distinct problems. Kustomize has no packaging/versioning/distribution story and no templating logic; Helm has no first-class "environment overlay" concept as clean as bases/overlays and pulls in a full templating language even for trivial variance. Many orgs use Helm to consume/distribute reusable charts and Kustomize on top (or instead) for environment-specific overlay logic that doesn't need a chart's parameterization power.
