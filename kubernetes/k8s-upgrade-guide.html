The Order of Things
// this is the law · deviating causes split-brain and version skew
⚡ The Canonical Upgrade Order
0
Pre-flight: Read release notes + scan deprecated APIs
Run pluto detect-all-in-cluster, kubent, check kubectl deprecations. Never skip this step. Removed APIs cause immediate application failures.
1
etcd backup (if self-managed)
etcdctl snapshot save snapshot.db. Validate it: etcdctl snapshot status snapshot.db. Store off-cluster. This is your only rollback option. Managed K8s: provider handles this.
2
Upgrade first control plane node
On the primary control plane: kubeadm upgrade apply v1.31.0. This upgrades: apiserver, scheduler, controller-manager, etcd (if kubeadm-managed), CoreDNS, kube-proxy.
3
Upgrade additional control plane nodes
On each additional CP node: kubeadm upgrade node. Then upgrade kubelet/kubectl: apt-get update && apt-get install -y kubelet=1.31.0-1.1 kubectl=1.31.0-1.1.
4
Upgrade worker nodes — one at a time (rolling)
Cordon + drain node → upgrade kubelet on node → uncordon. Or use MNG/Karpenter drift for managed clusters. Worker kubelet must be ≤1 minor version behind API server.
5
Upgrade cluster addons + system components
CNI plugin, CoreDNS, kube-proxy, metrics-server, Ingress controllers, CSI drivers. Each has its own version matrix. Check compatibility before upgrading.
6
Update manifests / Helm charts to use new API versions
Replace deprecated API versions in all manifests. Redeploy affected resources. This is the step teams forget — apps that used policy/v1beta1 break silently until re-deployed.
7
Validate + monitor 24-72 hours
Run full validation suite. Monitor error rates, latency. Keep pre-upgrade etcd backup available. Check audit logs for removed API usage that slipped through.
📏 Version Skew Policy
Componentvs API ServerRule
kubelet±1 minorkubelet may be up to 3 minor versions older than kube-apiserver (2 before v1.28); must never be newer.
kube-proxy±1 minorSame as kubelet on each node
kubectl±1 minorYour laptop's kubectl can lag by 1
kube-schedulermay be up to 1 minor older; must not be newerSame binary, same node as apiserver

etcdsee notesSpecific etcd version per K8s version
Multi-CP during upgrade: for HA clusters, one CP node at a time. During transition, CPs can be at different minor versions — the skew policy allows CP N+1 to run alongside N briefly. This is expected and safe during rolling upgrade.
🚨 Golden Rules
One minor versionNever skip minor versions. 1.28 → 1.29 → 1.30. Skipping causes unknown behavior, unsupported upgrade paths, data migration failures.
CP before workersControl plane is always first. A newer kubelet talking to an older API server is unsupported and breaks. Workers can lag by one minor version.
No rollbackThere is no supported rollback. etcd data format changes between versions. The only "rollback" is restore from etcd snapshot (data loss since backup time).
Staging firstAlways upgrade a staging cluster using identical procedure. If it works → prod. If staging doesn't exist, create one before the upgrade.
Component upgradesAddons (CNI, ingress, CSI) must be compatible with the new K8s version. Check each addon's compatibility matrix before upgrading the cluster.
Upgrading to a version that removes APIs your apps use = instant outage for those apps on next restart. The apiserver returns 404 or 410 for removed API paths. Use pluto or kubent to scan before every upgrade.
⚠️ The Version Skew Window
Start: CP=1.29, Workers=1.29 ← all same version ✓
Step 1: CP=1.30, Workers=1.29 ← skew=1 (allowed) ✓
Step 2: CP=1.30, Workers=1.30 ← all upgraded ✓
──────────────────────────────────────────────
WRONG: CP=1.30, Workers=1.28 ← skew=2 (banned) ✗
WRONG: CP=1.28, Workers=1.30 ← workers newer ✗

// everything you must do before touching the cluster
☑️ Interactive Checklist
0/16 completed
Read the release notes for every version between current and target. kubernetes.io/blog/releases — check CHANGELOG for your specific jump.
CRITICAL
Scan for deprecated/removed APIs with pluto detect-all-in-cluster --target-versions k8s=v1.31.0 or kubent. Any hits = must fix before upgrade.
CRITICAL
Check API Server audit logs for warnings about deprecated API usage from external clients (CI/CD pipelines, monitoring tools, operators).
CRITICAL
Take etcd snapshot: etcdctl snapshot save /backup/etcd-$(date +%Y%m%d).db and validate with etcdctl snapshot status. Store off-cluster.
CRITICAL
Verify all PodDisruptionBudgets. Find blockers: kubectl get pdb -A. Check for maxUnavailable:0 or minAvailable==replicas — these will deadlock node drain.
CRITICAL
Check cluster health baseline. All nodes Ready: kubectl get nodes. All system pods Running: kubectl get pods -n kube-system. Don't upgrade an already broken cluster.
CRITICAL
Verify addon compatibility. Check CNI, Ingress controller, CSI drivers, CertManager, ArgoCD — all have K8s version compatibility matrices. Pin versions before upgrading.
IMPORTANT
Test in non-prod first. Use identical procedure. If staging cluster is on a different version, first bring it to match prod, then upgrade staging one version ahead.
IMPORTANT
Run kubeadm upgrade plan (self-managed): shows what will be upgraded and to which version. Shows any issues with current state.
IMPORTANT
Review webhook configurations. Admission webhooks failing = API calls blocked. Check all MutatingWebhookConfiguration and ValidatingWebhookConfiguration.
IMPORTANT
Scale up critical single-replica workloads to ≥2 replicas before upgrade. Add PodAntiAffinity so they spread across nodes. Node drain kills single-replica pods.
IMPORTANT
Notify stakeholders. Announce maintenance window. Estimate duration: ~10 min/CP + ~10 min/worker node. 20 nodes = 3-4 hour window.
IMPORTANT
Check image pull policies and availability. Nodes will pull new system images during upgrade. Ensure registry access and no quota limits.
RECOMMENDED
Snapshot node images if using cloud VMs. AMI snapshot = faster node recovery if something goes wrong on the node.
RECOMMENDED
Check RBAC / ServiceAccount token usage if upgrading across versions that change token projection or expiry. Legacy SA tokens may stop working.
RECOMMENDED
Document current state: kubectl get all -A -o yaml > pre-upgrade-state.yaml. Helm: helm list -A. Capture baseline metrics and alert thresholds.

🔍 Essential Pre-Upgrade Tools
plutoFairwindsOps/pluto — best in class. Scans live cluster, Helm releases, manifest files. Outputs: deprecated, removed, or replacement API versions. CI integration available.
kubentkubectl neat + kubent — alternative scanner. Less features than pluto but simpler to run. kubent --target-version 1.31.
kubectl deprecationskubectl get --raw /metrics | grep apiserver_requested_deprecated_apis — shows deprecated APIs currently being called, per resource + version + client.
popeyeScans cluster for misconfigurations: missing resources, deprecated APIs, failed pods, dangling ConfigMaps. popeye -s deploy,svc.
kubeadm upgrade planFor self-managed clusters: shows exact upgrade path, component versions, any issues. Run this before every upgrade.

⚠️ Pre-Upgrade Common Mistakes
Skipping notesRelease notes contain breaking changes in plain text. Teams that skip them discover the same issues in production. The notes are short. Read them.
Upgrading broken clustersIf nodes are NotReady or pods are CrashLoopBackOff before the upgrade — fix those first. Upgrade amplifies existing problems.
No staging upgradeUpgrading prod without staging = discovering issues in prod. At minimum, create a temporary test cluster with the same manifests.
Forgetting CI/CD toolsHelm, ArgoCD, Flux, Jenkins operators — all use K8s APIs. They will break if their versions don't support the new API server. Check tool compatibility too.

kubeadm
Self-managed · Most common bare-metal/VM approach
▶
FULL UPGRADE PROCEDURE
## On FIRST control plane node ## # 1. Update kubeadm apt-get update && apt-get install -y kubeadm=1.31.0-1.1 # Or yum/dnf for RHEL-based # 2. Verify upgrade plan kubeadm upgrade plan # 3. Apply upgrade (CP components) kubeadm upgrade apply v1.31.0 --yes # 4. Cordon this node kubectl drain <cp-node> \ --ignore-daemonsets \ --delete-emptydir-data # 5. Upgrade kubelet + kubectl apt-get install -y \ kubelet=1.31.0-1.1 \ kubectl=1.31.0-1.1 systemctl daemon-reload && systemctl restart kubelet # 6. Uncordon kubectl uncordon <cp-node> ## On ADDITIONAL CP nodes ## kubeadm upgrade node # Then: drain → upgrade kubelet → uncordon ## On WORKER nodes (repeat each) ## # From control plane: kubectl drain <worker> \ --ignore-daemonsets \ --delete-emptydir-data \ --grace-period=60 # On the worker node: apt-get install -y kubelet=1.31.0-1.1 systemctl daemon-reload && systemctl restart kubelet # Back on control plane: kubectl uncordon <worker>
What it upgradesapiserver, scheduler, controller-manager, etcd, CoreDNS, kube-proxy (system components). Does NOT auto-upgrade addons like CNI, Ingress controllers, CSI drivers.
RollbackNo native rollback. Only etcd snapshot restore. Restore procedure: stop all K8s services → restore etcd → restart services at old version. Data loss since snapshot time.
HA clustersDo NOT drain all CP nodes before upgrading. Upgrade one at a time: upgrade → drain → kubelet restart → uncordon → next CP node. etcd quorum must be maintained.
Validation cmdkubeadm upgrade plan shows current state and available upgrades. Always run before applying.
Never run kubeadm upgrade apply on secondary CP nodes. Use kubeadm upgrade node instead. apply is only for the first/primary CP node. Using apply on secondary nodes causes config conflicts.
Lock package versions with apt-mark hold kubelet kubeadm kubectl to prevent accidental upgrades. Unhold before upgrading, then re-hold after.

Managed K8s (EKS / GKE / AKS / DOKS)
Provider manages control plane · You manage nodes
▶
AWS EKS
# Upgrade control plane aws eks update-cluster-version \ --name my-cluster \ --kubernetes-version 1.31 # Update addons after CP aws eks update-addon \ --cluster-name my-cluster \ --addon-name vpc-cni \ --addon-version v1.19.0-eksbuild.1 # Update node group aws eks update-nodegroup-version \ --cluster-name my-cluster \ --nodegroup-name workers \ --kubernetes-version 1.31 # Monitor aws eks describe-cluster \ --name my-cluster \ --query 'cluster.status'
GKE
# Enable release channel or manual gcloud container clusters upgrade my-cluster \ --master \ --cluster-version 1.31.0-gke.100 # Upgrade node pool gcloud container clusters upgrade my-cluster \ --node-pool default-pool \ --cluster-version 1.31.0-gke.100 # List available versions gcloud container get-server-config \ --format "yaml(validMasterVersions)"
AKS
# Check available versions az aks get-versions \ --location eastus -o table # Upgrade cluster az aks upgrade \ --resource-group myRG \ --name my-cluster \ --kubernetes-version 1.31.0 # Upgrade only node pool az aks nodepool upgrade \ --resource-group myRG \ --cluster-name my-cluster \ --name agentpool \ --kubernetes-version 1.31.0
Managed CP upgrade is non-reversible. All managed providers (EKS, GKE, AKS) handle HA during CP upgrade but cannot rollback once started. GKE supports auto-upgrades on release channels. EKS requires explicit triggering. AKS has auto-upgrade channels (patch/stable/rapid).
🏗️
Terraform / IaC
Infrastructure as Code · Reproducible · State-managed
▶
# terraform-aws-eks module module "eks" { source = "terraform-aws-modules/eks/aws" version = "~> 20.0" cluster_version = "1.31" # bump this eks_managed_node_groups = { workers = { min_size = 2 max_size = 10 desired_size = 3 instance_types = ["m7g.xlarge"] } } } # Plan and review before applying terraform plan -target=module.eks terraform apply -target=module.eks # WARNING: state ordering matters # Upgrade CP first (cluster_version) # Then run apply again for node groups # Or use explicit targets: terraform apply \ -target=module.eks.aws_eks_cluster.this terraform apply \ -target=module.eks.aws_eks_node_group.workers
Ordering riskTerraform dependency graph may not match required upgrade order. If TF tries to upgrade node groups before CP is done, apply will fail or produce partial state. Use -target to control ordering.
State driftIf you manually upgrade via CLI then run TF, state drift causes TF to want to "revert" or re-trigger. Always upgrade via the same method (TF or CLI, not both).
Sensitive changescluster_version change triggers in-place update of the EKS cluster (via AWS API). It's not a destroy+create. But TF shows it as ~ (update) — verify in the plan.
Addons in TFManage addon versions as separate resources: aws_eks_addon. Update addon_version after CP upgrade. Pin exact versions to avoid surprises.
Never set lifecycle { ignore_changes = [cluster_version] } on an EKS cluster — you'll lose track of what TF wants to manage. Instead, use state to track the version and keep TF as the source of truth.

kops (Kubernetes Operations)
Self-managed K8s on AWS/GCP/DO · declarative cluster spec
▶
# Edit cluster spec to change version kops edit cluster --name mycluster.k8s.local # In the editor, change: # spec.kubernetesVersion: 1.30.0 # → # spec.kubernetesVersion: 1.31.0 # Preview changes (dry run) kops update cluster --name mycluster.k8s.local # Apply the changes kops update cluster --name mycluster.k8s.local --yes # Trigger rolling update of nodes kops rolling-update cluster --name mycluster.k8s.local # Rolling update with validation between batches kops rolling-update cluster \ --name mycluster.k8s.local \ --max-unavailable 1 \ --yes
kops stores cluster state in S3 (or GCS). The spec is declarative — change the version in the spec, apply, then rolling-update. kops handles the kubeadm bootstrap internally. Supports multiple instance groups (MIG) for different node types.

Rancher / RKE / RKE2
SUSE Rancher Platform · Multi-cluster management
▶
# RKE cluster.yml upgrade kubernetes_version: v1.31.0-rancher1-1 # Apply with rke rke up --config cluster.yml # RKE2: via systemd config # /etc/rancher/rke2/config.yaml: # system-default-registry: "" # Update RKE2 binary curl -sfL https://get.rke2.io | \ INSTALL_RKE2_VERSION=v1.31.0+rke2r1 sh - systemctl restart rke2-server # Rancher UI: Cluster → Edit → K8s Version # Rancher handles rolling update automatically
Rancher UIClick cluster → Edit → change Kubernetes Version dropdown. Rancher queues rolling upgrade. Watch progress in Cluster Events.
RKE vs RKE2RKE2 is the newer, security-focused runtime (replaces Docker with containerd, CIS benchmark by default). RKE1 is legacy. Both support rolling upgrades of nodes.
Multi-clusterRancher Fleet (GitOps) allows synchronized upgrades across multiple clusters from a single Git repo. Define cluster version in fleet.yaml.

GitOps Approach (ArgoCD / Flux)
Version-controlled upgrade config · Audit trail · Rollback via Git
▶
# Flux: cluster version as GitRepository # clusters/prod/cluster-config.yaml apiVersion: cluster.x-k8s.io/v1beta1 kind: Cluster spec: topology: version: v1.31.0 # bump in git workers: machineDeployments: - name: workers replicas: 3 # ArgoCD: helm values in git # eks-cluster/values.yaml cluster_version: "1.31" # PR-based upgrade workflow: # 1. Create branch: bump version in config # 2. Open PR → triggers CI validation (pluto, kubent) # 3. Merge PR → Flux/ArgoCD detects change # 4. ArgoCD syncs → triggers cluster upgrade API # 5. Upgrade progress visible in ArgoCD UI
BenefitsEvery upgrade is a Git commit. Full audit trail of who changed what when. "Rollback" = revert the commit + sync. CI can run pluto/kubent on every PR before merge.
CAPI integrationCluster API (cluster.x-k8s.io) makes K8s clusters themselves declarative objects. Changing spec.topology.version triggers rolling upgrade via the CAPI provider (AWS CAPA, GCP CAPG, etc.).
LimitationGitOps manages the desired state — but still relies on the underlying upgrade mechanism (kubeadm, EKS API, etc.). It doesn't replace those tools; it orchestrates them.
Add pluto scan to your CI pipeline. Every PR that changes a Helm chart or manifest should run pluto detect-files. Block merges with removed API usage. This prevents deprecated API surprises reaching production.

🚨 The Historically Painful Upgrades
1.16 (2019)Removed Deployment, DaemonSet, StatefulSet, ReplicaSet from extensions/v1beta1 and apps/v1beta. Many Helm charts still used old APIs. Mass breakage across the ecosystem. This set the precedent for API removal pain.
1.22 (2021)Ingress removed from extensions/v1beta1 and networking/v1beta1. Every app with an Ingress needed manifest updates. CRDs with v1beta1 broke. The most impactful removal release. ~18 months warning didn't help much in practice.
1.24 (2022)Dockershim removed. Any cluster node using Docker as container runtime needed to migrate to containerd or CRI-O. Caught many "set and forget" clusters off-guard. Pod-level Docker socket mounts broke.
1.25 (2022)PSP removed. PodSecurityPolicy was the defacto pod admission mechanism. Teams had to migrate to PSA, Kyverno, or OPA Gatekeeper before upgrading. No migration path = pods blocked at admission.

What Fails — And Why
// failure encyclopedia · causes · fixes · prevention
01
API Removal
App CrashLoops after upgrade: the server could not find the requested resource
Cause: app uses removed API version (e.g., extensions/v1beta1 Ingress). API server returns 404/410. Deployment controllers, Helm hooks, operators all affected.

Fix: update manifest apiVersion. If Helm: helm upgrade --force with updated chart. Prevention: pluto scan in CI before every upgrade.
02
PDB Deadlock
Node drain hangs forever during rolling upgrade
Cause: minAvailable: 2 on Deployment with 2 replicas, or maxUnavailable: 0. Both pods on nodes being drained simultaneously. PDB blocks eviction of both.

Fix: temporarily patch PDB: kubectl patch pdb my-pdb -p '{"spec":{"maxUnavailable":1}}'. Or scale up replicas first. Prevention: always check PDBs before upgrade. Ensure replicas > minAvailable.
03
Webhook Failure
All pod creation fails after upgrade: Internal error occurred: failed calling webhook
Cause: Admission webhook (Kyverno, OPA, custom) is down or incompatible with new K8s version. Webhook endpoint unreachable → API server blocks all matching API calls.

Fix: if failurePolicy: Fail — temporarily set to Ignore, fix the webhook, restore. Or delete the webhook temporarily: kubectl delete mutatingwebhookconfiguration my-webhook. Prevention: set webhooks to Ignore during upgrade windows.
04
etcd Failure
API server unresponsive: etcd cluster is unavailable
Cause: lost etcd quorum during CP upgrade. Upgrading 2 of 3 etcd members simultaneously. Or etcd version mismatch. For 3-node etcd: can only lose 1 member.

Fix: restore from snapshot. etcdctl snapshot restore snapshot.db --data-dir=/new/path. Then update etcd configs to point to new data dir. Prevention: upgrade etcd members one at a time. Validate health after each: etcdctl endpoint health.
05
Version Skew
kubelet refuses to register: version too old
Cause: worker kubelet is 2+ minor versions behind API server. K8s skew policy: kubelet must be ≤1 minor version behind. After upgrading CP by 2 versions, old workers are incompatible.

Fix: immediately upgrade worker nodes after CP. Prevention: never let workers fall 2 versions behind. Upgrade nodes within the same maintenance window as CP.
06
PSP Migration
All pods stuck in Pending after upgrading to 1.25+
Cause: cluster relied on PodSecurityPolicy (removed in 1.25). Without PSP admission controller, pods may be blocked by the new Pod Security Admission controller if namespaces have PSA labels set to restricted.

Fix: set namespace PSA labels appropriately, or configure PSA in audit-only mode. Prevention: migrate from PSP to Kyverno/OPA/PSA before upgrading to 1.25. Test PSA in warn/audit mode.
07
Addon Mismatch
DNS fails for new pods after upgrade
Cause: CoreDNS version incompatible with new K8s, or CoreDNS DaemonSet not updated. New pods can't resolve cluster DNS. Existing pods with cached DNS still work until TTL expires.

Fix: upgrade CoreDNS to a compatible version. Check configmap format hasn't changed. Restart CoreDNS pods. Prevention: upgrade CoreDNS as part of the upgrade process, not after.
08
CNI Breakage
Pods scheduled but stuck in ContainerCreating: CNI plugin error
Cause: CNI plugin (Calico, Cilium, Flannel) is incompatible with new kernel version or new K8s version. CNI fails to configure pod networking.

Fix: upgrade CNI to compatible version. Check CNI DaemonSet pods are Running: kubectl get pods -n kube-system -l k8s-app=calico-node. Prevention: check CNI compatibility matrix. Upgrade CNI before or during K8s upgrade.
09
StatefulSet Local PV
StatefulSet pod drain fails: pods won't terminate
Cause: StatefulSet pods using local PVs (hostPath, local PersistentVolume) cannot move to other nodes — the data is tied to the node. Drain blocks waiting for pod to be rescheduled elsewhere, which is impossible.

Fix: manually delete the pod kubectl delete pod --grace-period=0, then remove/replace the node. Prevention: use distributed storage (Longhorn, Rook-Ceph, cloud PVs) instead of local PVs for StatefulSets.
10
Service Account Tokens
Apps get 401 from API server after upgrade
Cause: 1.24 stopped auto-creating legacy SA secrets. Apps relying on kubernetes.io/service-account-token type secrets find them expired or missing. Bound tokens have shorter TTL (1h by default in 1.22+).

Fix: use projected volume tokens (serviceAccountToken volume source). Ensure AWS SDK / client-go >= version that handles token refresh. Prevention: audit applications using SA tokens before 1.24+ upgrade.
11
Dockershim
Nodes fail to join after 1.24 upgrade: container runtime not running
Cause: node still configured with Docker as CRI. Dockershim removed in 1.24. Kubelet cannot start without a compatible CRI. Nodes bootstrapped with old config stay broken.

Fix: migrate to containerd: systemctl stop docker; systemctl enable containerd; containerd config default > /etc/containerd/config.toml. Update kubelet --container-runtime-endpoint. Prevention: migrate all nodes to containerd before 1.24 upgrade.
12
Ingress
Ingress controller stops reconciling new Ingress objects
Cause: Ingress controller version doesn't support networking.k8s.io/v1 Ingress (required since 1.22). Old nginx-ingress watching the old API version sees nothing.

Fix: upgrade Ingress controller chart to latest. Old: kubernetes/ingress-nginx, new watches v1 automatically. Prevention: upgrade Ingress controller before or alongside K8s upgrade to 1.22+.

Post-Upgrade Playbook
// what to do immediately after · what to monitor · when to declare success
☑️ Post-Upgrade Checklist
0/14 completed
Verify all nodes are on target version: kubectl get nodes -o wide. All nodes should show the new version. No nodes stuck on old version.
IMMEDIATE
Verify all kube-system pods are Running: kubectl get pods -n kube-system. CoreDNS, kube-proxy, CNI daemonsets. No CrashLoopBackOff or Pending.
IMMEDIATE
Verify all application pods are healthy: kubectl get pods -A | grep -v Running. Any non-Running/Completed pods need investigation.
IMMEDIATE
Test DNS resolution from a pod: kubectl run test --rm -it --image=busybox --restart=Never -- nslookup kubernetes.default. CoreDNS must be working before declaring success.
IMMEDIATE
Run application smoke tests. Hit critical API endpoints, run health checks. Verify frontend-to-backend connectivity. Check PVC mounts work.
IMMEDIATE
Check HPA/KEDA is still scaling. HPA depends on metrics-server. KEDA depends on external triggers. Verify both are functional post-upgrade.
IMPORTANT
Verify CSI/storage works. Create a test PVC and pod, verify mount, delete. CSI drivers often need upgrading alongside K8s versions.
IMPORTANT
Check etcd health (self-managed): etcdctl endpoint health --cluster. All members should be healthy. Verify etcd version with etcdctl version.
IMPORTANT
Verify Ingress/LB endpoints. External traffic should reach services. Check ALB/NLB target health (managed cloud) or nginx-ingress pods.
IMPORTANT
Run pluto again post-upgrade. Check nothing slipped through. Any deprecated API now showing = must fix before next upgrade.
IMPORTANT
Check audit logs for new deprecation warnings. APIs deprecated in this version = need fixing before the next upgrade window.
IMPORTANT
Monitor error rates for 24-72 hours. Track 5xx rates, latency P99, pod restart rates. Compare to pre-upgrade baseline.
MONITOR
Update runbooks and documentation. Note the new version, what changed, any issues encountered. Future team members will thank you.
HOUSEKEEPING
Schedule next upgrade now. K8s releases every 4 months. You need to upgrade within 14 months of a release (before EOL). Don't let it pile up.
HOUSEKEEPING
📅 K8s Release Lifecycle
Day 0 — Release
Minor version released (every ~4 months)
v1.31.0 released. Patch releases (v1.31.1, .2...) follow monthly. Support window: 14 months from release date.
+4 months — Next minor
Next version released (v1.32)
Your current version is now 1 version behind. Upgrade before you fall 3 versions behind (EOL of your version).
+12 months — Deprecation notice
Minor version enters final 2 months
Last patch releases. Community support winds down. You MUST be planning your upgrade now.
+14 months — EOL
Version reaches End of Life
No more patches. No security fixes. No CVE responses. Managed providers (EKS/GKE/AKS) may auto-upgrade your cluster. Self-managed clusters are on their own.
Rule of thumb: upgrade 2-4 weeks after a minor release (let community find issues), but no later than 10 months after release. This gives you a comfortable runway before EOL.
⚠️ When To Declare the Upgrade Failed
Stop ifetcd quorum lost during CP upgrade. Restore from snapshot immediately — don't try to push through.
Stop ifAPI server returns 500 errors on all requests after upgrade. Something fundamental broke. Check kube-apiserver logs.
Stop ifMore than 20% of application pods are in error state after node upgrades. Rollback nodes (re-provision old AMI) and investigate.
Continue withOne or two applications in CrashLoop with a clear cause (deprecated API). Fix the app, not the cluster.
Restore commandetcdctl snapshot restore snapshot.db --data-dir=/var/lib/etcd-restore --initial-cluster-token="etcd-cluster-new" — the token must differ from old cluster to prevent split-brain.

📡 Prometheus Metrics to Watch
API server healthapiserver_request_total{code=~"5.."} — 5xx errors. Should be ~0. Spikes indicate apiserver issues.
API latencyhistogram_quantile(0.99, rate(apiserver_request_duration_seconds_bucket[5m])) — P99 should be <1s for most requests.
etcd latencyhistogram_quantile(0.99, rate(etcd_disk_backend_commit_duration_seconds_bucket[5m])) — should be <25ms. Spikes = disk I/O issue.
etcd leader changesincrease(etcd_server_leader_changes_seen_total[1h]) — should be 0 or near 0. Frequent leader changes = etcd instability.
Pod restartskube_pod_container_status_restarts_total — compare to pre-upgrade baseline. New restarts post-upgrade indicate broken apps.
Node conditionskube_node_status_condition{condition="Ready",status="true"} == num_nodes. Any node not Ready = problem.
Scheduler latencyscheduler_scheduling_algorithm_duration_seconds — elevated scheduling latency indicates scheduler load or configuration issues.
Deprecated API hitsapiserver_requested_deprecated_apis — non-zero after upgrade = apps still using old APIs. Must fix before next upgrade.
🔬 Comprehensive Validation Commands
▸ Control Plane Health
kubectl get nodes -o wideAll nodes Ready, correct version, correct OS image. No NotReady, SchedulingDisabled, or Unknown status.
kubectl get pods -n kube-systemAll system pods Running. Check CoreDNS (2+ replicas), kube-proxy (all nodes), CNI daemonset (all nodes).
kubectl version --shortServer version matches target. Client version within 1 minor of server.
kubectl get componentstatusesHealthy: etcd, scheduler, controller-manager. (Deprecated in newer K8s but still useful on older clusters)
kubectl get --raw='/healthz?verbose'API server healthz endpoint. Should return "ok". Shows which subsystems are healthy.
kubectl get --raw='/readyz?verbose'API server readyz (1.16+). More granular than healthz. Shows post-start-hooks and shutdown state.
kubectl get --raw='/livez?verbose'API server livez (1.16+). Liveness check — if this fails, the API server process itself is broken.
▸ Workload Health
kubectl get pods -A | grep -Ev "Running|Completed"Any non-Running/Completed pods. Pending, CrashLoopBackOff, ImagePullBackOff all need investigation.
kubectl get deployments -A | grep -v "True"Deployments not available. Available column should be True. If not, deployment is degraded.
kubectl get events -A --sort-by='.lastTimestamp' | tail -30Recent events — look for Warnings about Failed pulls, FailedScheduling, BackOff, Unhealthy probes.
kubectl top nodes && kubectl top pods -AResource usage — compare to pre-upgrade baseline. Elevated CPU/memory may indicate workloads in restart loops.
▸ Networking
kubectl run -it --rm nettest --image=nicolaka/netshoot --restart=Never -- bashInteractive network debug pod. Run: nslookup, curl service names, ping pod IPs cross-node.
kubectl exec -n kube-system <coredns-pod> -- nslookup kubernetes.defaultCoreDNS resolving internal names. kubernetes.default should resolve to the ClusterIP of the kubernetes service.
kubectl get svc -A | grep -v None | grep ClusterIPServices have valid ClusterIPs. No services with <none> IP (unless headless intentionally).
kubectl get endpoints -A | grep "<none>"Services with no endpoints = pods not matching selector. Label mismatch or no pods running.
▸ etcd Health (self-managed only)
etcdctl endpoint health --cluster --write-out=tableAll etcd members healthy. Check endpoint, health status, and latency. >25ms commit latency = disk issue.
etcdctl endpoint status --cluster --write-out=tableLeader elected, DB sizes within range. All members should have similar DB sizes. Leader should be stable.
etcdctl member list --write-out=tableAll members present and started. No unstarted members left from a failed join.
▸ Deprecated API Scan
pluto detect-all-in-clusterScan for deprecated API usage in live cluster. Covers all Helm releases and resources. Zero results = clean.
kubectl get --raw /metrics | grep apiserver_requested_deprecated_apisMetrics-based deprecated API hits. Shows which deprecated APIs are actively being called and by whom.
