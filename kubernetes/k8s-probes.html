The Three Probes
// what each one does · what happens on failure · independence
🔴 livenessProbe
Question"Is this container still alive and functional?" Detects deadlocks, infinite loops, corrupted state that process hasn't exited from.
On failureKubelet kills and restarts the container. RestartPolicy applies. Pod stays on the same node. Container restart count increments.
On successNothing happens — normal operation. The probe result is recorded but causes no action. Only failures trigger action.
Affects Endpoints?No. Liveness failure → container restart → brief period of Running but not Ready → briefly removed from EndpointSlice. But liveness itself doesn't gate traffic.
ThresholdfailureThreshold consecutive failures required before action. Default = 3. Each success resets the failure counter.
Liveness probes that are too aggressive kill healthy containers. If your app spikes under load and the probe times out → restart → more load → cascade. Set timeoutSeconds generously. Never probe a shared resource (DB) in liveness — DB down ≠ container needs restart.
🟢 readinessProbe
Question"Is this container ready to receive traffic?" Warm-up after start, temporary unavailability (cache loading, circuit breaker open).
On failurePod IP removed from EndpointSlice. No container restart. No pod kill. Traffic stops flowing to this pod. Container keeps running.
RecoveryWhen readiness probe succeeds again (after successThreshold consecutive successes), pod IP is re-added to EndpointSlice. Traffic resumes.
Affects Endpoints?Yes — this is the only probe that gates Service traffic. EndpointSlice controller watches pod Ready condition. Kubelet sets it based on readiness result.
successThresholdDefault = 1 for readiness. Liveness/startup successThreshold is always 1 (cannot be changed — spec rejects values > 1).
readinessProbe failing = pod invisible to Service, NOT removed. The pod still exists, still Running, but gets zero traffic. Use this intentionally for graceful shutdown: fail readiness first, then drain, then exit. kube-proxy takes ~seconds to update iptables after EndpointSlice change.
🟡 startupProbe
Question"Has the application finished starting up?" For slow-starting apps (JVM warmup, large config load, DB migration).
On successDisables itself. Liveness and readiness probes are then activated. Until startup succeeds, liveness and readiness are completely disabled — they don't run at all.
On failureSame as liveness: container killed and restarted. The maximum startup time = failureThreshold × periodSeconds. After that, kubelet kills the container.
Why it existsBefore startupProbe (1.16), you had to set initialDelaySeconds on liveness to the maximum possible startup time. Startup probe separates "startup budget" from "running health check frequency".
Max startup timefailureThreshold × periodSeconds. Example: 30 × 10s = 300s = 5 min budget before kubelet gives up and kills.
startupProbe replaces initialDelaySeconds for slow apps. Don't use both. If startupProbe is set, initialDelaySeconds on liveness/readiness is measured from after startup succeeds, not from container start. Actually — kubelet doesn't even run them until after startup passes.

Probe Mechanics (4 Types)
// HTTP · TCP · exec · gRPC — what kubelet actually does at the kernel level
🌐 httpGet
What kubelet doesKubelet makes an HTTP GET request from the node (not from inside the pod). It opens a TCP connection to podIP:port and sends a GET request. Uses Go's net/http.
SuccessResponse code ≥ 200 and < 400. Any 2xx or 3xx is success. 4xx and 5xx = failure. The body is ignored.
TimeoutKubelet sets a timeoutSeconds deadline on the HTTP client. If no response in time: failure (not retry). Counted as one failure toward failureThreshold.
HeadersCustom httpHeaders can be set. Kubelet adds default User-Agent: kube-probe/1.x. You can filter probes from real traffic via this header.
HTTPSscheme: HTTPS — kubelet uses TLS but skips certificate verification (InsecureSkipVerify: true). Certificates are not checked. No way to provide a CA cert to kubelet for probes.
livenessProbe: httpGet: path: /healthz port: 8080 scheme: HTTP # or HTTPS (no cert verify) httpHeaders: - name: X-Custom-Header value: probe initialDelaySeconds: 5 periodSeconds: 10 timeoutSeconds: 3 failureThreshold: 3
httpGet probe comes from node IP, not pod IP. If your app checks the client IP (rate limiting, auth) it will see the node's IP. NetworkPolicy must allow node IP → pod port. Check with X-Forwarded-For or by checking logs — probe requests show User-Agent: kube-probe.
🔌 tcpSocket
What kubelet doesKubelet attempts to open a TCP connection to podIP:port from the node. If the 3-way handshake completes (SYN + SYN-ACK + ACK): success. No data is sent.
SuccessTCP connection established. Kubelet immediately closes it with RST. The port is open and listening.
FailureConnection refused (RST in response to SYN) or timeout (no SYN-ACK within timeoutSeconds). Port is not open or host unreachable.
Use caseDatabases, message brokers, custom TCP servers. Anything that doesn't speak HTTP. Verifies the port is bound and accepting connections — but not that the app is functionally healthy.
Host fieldhost field optional — defaults to pod IP. Can be set to a hostname but resolves from node's DNS, not the pod's. Rarely needed.
readinessProbe: tcpSocket: port: 5432 # postgres port initialDelaySeconds: 10 periodSeconds: 5 timeoutSeconds: 2
tcpSocket is a weak healthcheck. A port can accept TCP connections while the app behind it is deadlocked. The probe confirms the port is open — not that queries return results. For databases, prefer an exec probe running a lightweight query.
RST vs timeout are different failures. RST = app refused connection (nothing listening). Timeout = network issue, firewall, or app overloaded and not accepting. Both count as one failure — but the cause differs. Check kubectl describe pod Events for the error message.
💻 exec
What kubelet doesKubelet calls containerd CRI ExecSync RPC → containerd → runc exec → runs command inside the container's existing namespaces. Shares PID NS, mount NS, network NS with container.
SuccessCommand exits with exit code 0. Any nonzero exit = failure. stdout/stderr output is ignored (not logged by kubelet unless debug verbosity).
Process treeThe exec command runs as a child process of the container's PID 1. It inherits the container's environment variables, working directory, and user context.
Resource costexec probes consume CPU and memory. For every probe execution: a process is forked, the binary loaded, and run. For high-frequency probes or large binaries: measurable overhead. Prefer HTTP when possible.
Timeout enforcementKubelet kills the exec process after timeoutSeconds via SIGKILL to the container's cgroup. Not a graceful kill — can leave side effects if probe does I/O.
livenessProbe: exec: command: - sh - -c - | redis-cli ping | grep -q PONG periodSeconds: 10 timeoutSeconds: 5 # Postgres readiness check: readinessProbe: exec: command: [pg_isready, -U, postgres]
exec probe shell injection risk. If command includes env vars ($MY_VAR) evaluated at probe time, an attacker controlling that env var can inject shell commands. Use full binary paths and avoid shell expansion in probe commands.
📡 gRPC
What kubelet doesKubelet makes a gRPC Health Check Protocol (grpc.health.v1) call to podIP:port. Calls Check() on the Health service. No TLS by default.
SuccessgRPC response status is SERVING. Any other status (NOT_SERVING, UNKNOWN, SERVICE_UNKNOWN) = failure. gRPC error (connection refused, deadline exceeded) = failure.
Service fieldservice field (optional) — passed as the service name in the health check request. Allows checking health of a specific gRPC service in a multi-service server.
App requirementApp must implement grpc.health.v1 Health service. Not automatic. Add google.golang.org/grpc/health package. If your app doesn't implement it, all gRPC probes fail with "unimplemented".
VersionGA in K8s 1.27. Requires feature gate GRPCContainerProbe=true on older clusters (1.24–1.26 were alpha/beta).
readinessProbe: grpc: port: 50051 service: my.grpc.Service # optional periodSeconds: 10 timeoutSeconds: 5
gRPC probe is the cleanest option for gRPC services. Beats exec (no process fork overhead) and avoids TCP probe's shallowness. Implement the health service in all gRPC apps.

All Parameters
// click each param to expand · defaults · tradeoffs
initialDelaySeconds
default: 0
Wait before running the first probe after container start.
When to useWhen app needs time to start before any probe runs. But prefer startupProbe for slow apps — initialDelaySeconds wastes time on fast-starting containers.
Applied whenClock starts when container state transitions to Running. Not when pod is created. Not when image pull starts.
Anti-patternSetting a large initialDelay "to be safe" delays readiness detection. A container that never becomes healthy wastes time before kubelet discovers the problem. Use startupProbe instead.
periodSeconds
default: 10 · min: 1
How often kubelet runs the probe.
TradeoffLower = faster failure detection, more overhead. Higher = less overhead, slower detection. For liveness: 10–30s fine. For readiness that guards traffic: 5–10s.
At scale1000 pods × periodSeconds=1 = 1000 probe requests/second from node kubelet processes. This is real load on your app. Don't set to 1 unless absolutely needed.
Min valueMinimum is 1 second. Setting periodSeconds: 0 causes a validation error. The spec doesn't allow sub-second probing.
timeoutSeconds
default: 1 · min: 1
Probe must respond within this many seconds or it counts as a failure.
Default dangerDefault is 1 second — extremely aggressive. A GC pause, load spike, or slow DNS lookup can cause your probe to time out. For JVM apps, set to 5–10s.
HTTP behaviorFor HTTP probes: net/http deadline set to timeoutSeconds. If server sends headers slowly, the deadline applies to receiving the full response, not just headers.
exec behaviorKubelet sends SIGKILL to the exec process's cgroup after timeout. Not SIGTERM. Immediate kill.
failureThreshold
default: 3 · min: 1
Number of consecutive failures before kubelet takes action.
Reset on successThe failure counter resets on any success. 2 failures + 1 success = counter back to 0. Must be failureThreshold consecutive failures to trigger action.
Liveness actionContainer killed (SIGTERM + SIGKILL after terminationGracePeriodSeconds). Then restarted per RestartPolicy.
Readiness actionPod marked Ready=False. EndpointSlice controller removes pod IP. No container kill. Can recover.
Startup actionContainer killed. Pod restarts. Note: startup probe has its own threshold independent of liveness.
successThreshold
default: 1 · liveness/startup: fixed 1
Consecutive successes required to mark probe as passing.
Readiness onlyOnly meaningful for readiness. successThreshold > 1 on liveness or startup is rejected by the API Server with a validation error. This is by design — liveness success always resets immediately.
Use caseReadiness: require 2-3 consecutive successes before adding back to EndpointSlice. Prevents flapping where a marginal app oscillates in/out of endpoints.
Re-add delayWith successThreshold: 2 and periodSeconds: 5: pod removed from endpoints requires 10s of success before re-added. Prevents briefly healthy pods from getting traffic.
terminationGracePeriodSeconds
per-probe default: pod-level value
Override pod-level termination grace period specifically for probe failures. K8s 1.25+.
Per-probe overrideSet on the probe spec itself (not pod-level). When liveness fails and triggers restart: uses this value instead of pod's spec.terminationGracePeriodSeconds.
Use caseShort grace for liveness-triggered restarts (you want fast recovery), longer grace for normal kubectl delete pod (you want graceful draining). Set probe-level to 5s, pod-level to 60s.
VersionAlpha in 1.21, beta in 1.25, GA in 1.28. Feature gate ProbeTerminationGracePeriod.

Timing Formulas
// when does action happen · worst case · best case
⏰ First probe fires at
T_first = initialDelaySeconds + periodSeconds
// initialDelay: wait after container Running
// then: first period before first check

Example: delay=5, period=10 → first probe at t=15s
If startupProbe is configured, liveness/readiness don't start their initialDelaySeconds clock until startup probe succeeds. The formula becomes: T_startup_success + initialDelaySeconds + periodSeconds.
💀 Liveness kills container at
T_kill = T_first + (failureThreshold - 1) × periodSeconds

// worst case from first probe to kill:
T_worst = failureThreshold × periodSeconds + timeoutSeconds

Example: period=10, fail=3, timeout=1
→ 3×10 + 1 = 31s from first failure to kill
Actual time includes timeoutSeconds × failureThreshold. If probe times out each time: each check takes timeoutSeconds. Total: (period - timeout) × (fail-1) + timeout × fail. Worst case when probe is always timing out.
🚀 Max startup budget
T_max_startup = failureThreshold × periodSeconds

// app must succeed within this window
// if not: container killed (same as liveness)

Example for 5-minute app:
failureThreshold = 30, periodSeconds = 10
→ 30 × 10 = 300s = 5 minute budget
Pattern for slow-start apps: Set startupProbe.failureThreshold × periodSeconds to your worst-case startup time. Then set liveness/readiness to aggressive values (period=5, threshold=3) safe in the knowledge they won't fire during startup.

Kubelet Engine Internals
// goroutines · prober manager · workers · how it actually runs
🔬 Prober Manager Architecture
🧵
ProberManager
Central goroutine manager. Maintains a map of probeKey → worker. Each probe (liveness/readiness/startup) per container gets its own worker. On pod add: spawns workers. On pod delete: signals workers to stop via channel close.
🔴
liveness worker
One goroutine per container with a liveness probe. Runs time.NewTicker(periodSeconds). On tick: calls prober.probe(). Result sent to livenessManager channel. If failureThreshold consecutive failures → writes to container kill channel.
🟢
readiness worker
Same structure as liveness worker. Results sent to readinessManager. The readiness result is stored in a sync.Map keyed by podUID/containerName. Kubelet reads this map when building pod status to set the Ready condition.
🟡
startup worker
Startup worker has a special flag: once probe returns success, it signals the other workers to start, then the startup worker goroutine exits cleanly. It never runs again for that container's lifecycle.
📊
statusManager
Separate goroutine that syncs pod status to API Server. Probe results don't immediately call API Server — they update an in-memory cache. StatusManager batches updates and patches pods/status subresource. Prevents probe-induced API Server storms.

Total latency from probe failure to traffic stop: probe period + statusManager batch interval (~several seconds) + EndpointSlice propagation + kube-proxy iptables sync. Not instantaneous. Plan for 5–15s of continued traffic after readiness fails.
💥 Liveness Kill Sequence
liveness threshold exceeded
↓ killContainer(container)
SIGTERM → container PID 1
↓ wait terminationGracePeriodSeconds
SIGKILL if still running
↓ container stops
containerRestartCount++
↓ RestartPolicy: Always → restart
container restarts (same pod, same node)
↓ startupProbe (if set) runs first
readiness/liveness resume after startup
CrashLoopBackOff is exponential. Restart delays: 10s → 20s → 40s → 80s → 160s → 300s (cap). Each restart counts even if caused by liveness probe. A misconfigured liveness probe can trap your app in CrashLoopBackOff.

⚡ Probe Ordering & Mutual Disable
ScenarioBehavior
Only startupProbe setUntil startup passes: container can be killed. After: nothing gates traffic or restarts. Pod stays Running permanently (even if deadlocked).
startupProbe + livenessProbeStartup runs first. Liveness completely disabled until startup succeeds. After: liveness runs on its own schedule.
startupProbe + readinessProbePod stays NotReady (no traffic) until startup passes. Then readiness runs. Traffic flows only when both startup passed AND readiness succeeds.
All three probesStartup first → disables liveness+readiness. After startup: liveness independently restarts, readiness independently gates traffic. They don't interact with each other.
No probes at allPod is Ready immediately when container is Running. Liveness never kills. Traffic flows even if app is deadlocked or still initializing.
Only readinessProbe failingPod stays Running, not restarted, not killed. Zero traffic. Stays like this indefinitely until probe passes or pod is deleted.

⚡ LIVENESS TRAPS
Never probe shared resources. DB down ≠ container needs restart. Liveness should check only this container's health — not external dependencies. Use readiness for dependency checks.
Timeout default (1s) is too aggressive for most apps. GC pause, slow disk flush, momentary load spike → liveness kills healthy app. Set 3–10s.
liveness + CrashLoopBackOff spiral: liveness kills → restart → liveness kills again before app warms up → CrashLoopBackOff. Fix: add startupProbe, increase initialDelaySeconds, or remove liveness entirely.
⚡ READINESS TRAPS
Traffic doesn't stop immediately on readiness failure. EndpointSlice propagation + kube-proxy sync = 5–15s lag. Don't rely on instant traffic stop for circuit breaking.
readiness failure ≠ pod removed. Pod keeps Running. kubectl get pods shows 0/1 Running — the "0/1" means 0 ready containers. Often confused with CrashLoopBackOff.
Graceful shutdown pattern: use a preStop hook to sleep 5–10s. This ensures kube-proxy has time to update rules before the process exits. Without this: in-flight requests get RST.
⚡ STARTUP TRAPS
startupProbe failure threshold must budget for worst-case startup. If app can take 90s to start, set failureThreshold × periodSeconds ≥ 90. Undershooting = random kills during cold start.
Don't use startupProbe + large initialDelaySeconds together. initialDelay on liveness/readiness is from after startup success. The combination is confusing and wasteful.
startupProbe uses same mechanism as liveness. Failure = container killed, not traffic-blocked. Make the startup probe check the simplest possible thing (port open, pid file exists).
