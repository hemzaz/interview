Key: Deployment ctrl creates RS, RS ctrl creates Pods. Deployment never creates Pods directly. The pod-template-hash label in the RS name (deploy-69d7bc8c7f) is a deterministic hash of the pod template — same template always produces same RS name, preventing duplicate RSes on concurrent writes.
Deployment ctrl ≠ Pod factory. It only manages ReplicaSets. Each RS owns its pods. Old RSes (0 replicas) are kept for rollback up to revisionHistoryLimit.

StatefulSet creates Pods directly — there is no intermediate ReplicaSet. Each Pod gets a stable ordinal identity (sts-0, sts-1) and a dedicated PVC that survives pod deletion. Next pod restart re-attaches the same PVC.
PVCs created by StatefulSet have no ownerReference to the STS. They are NOT garbage-collected when the STS is deleted. Must be cleaned up manually or the STS must be deleted with cascade=foreground and StorageClass reclaimPolicy=Delete.

🔀 Ingress Controller Role
Ingress objectDeclarative routing rules only. No traffic flows through the Ingress object itself — it's just config.
Ingress ctrlA pod running nginx/Envoy/Traefik that watches Ingress objects and configures itself. The controller is the actual proxy.
ingressClassspec.ingressClassName selects which Ingress controller handles this Ingress. Multiple controllers can coexist in same cluster.
Creating an Ingress with no IngressClass in a cluster with multiple ingress controllers = no controller picks it up, no traffic, no error. Silent failure.
🛡️ NetworkPolicy
Selects pods viaspec.podSelector (label match). Applies ingress/egress rules to matching pods.
No ownerRefNetworkPolicy does NOT own pods. It only selects them declaratively — same selector mechanism as Service.
Default: allow all. A NetworkPolicy only restricts if one exists for a pod. No policy = open. First deny-all policy makes all selected pods drop all traffic — including liveness probes from kubelet!

Reclaim policies: Retain = PV stays Released after PVC deleted (data safe, re-use manual). Delete = PV + cloud disk deleted when PVC deleted. Recycle = deprecated. PVC deletion always succeeds immediately — PV reclaim is async.
PVC in Terminating forever: finalizer kubernetes.io/pvc-protection blocks deletion until no pod is actively mounting it. If the pod is also stuck Terminating (e.g., node dead), PVC is stuck too. Solution: force-delete the pod, then the PVC finalizer clears automatically.
🔗 PVC ↔ PV Binding Rules
Binding matchPV controller finds best PV for PVC. Must satisfy: access modes, capacity ≥ requested, storageClassName match, volumeMode match.
Access modesRWO = ReadWriteOnce (one node). ROX = ReadOnlyMany (many nodes). RWX = ReadWriteMany (many nodes). RWOP = ReadWriteOncePod (one pod, 1.22+).
Best-fitPV controller picks smallest PV that satisfies all constraints. A 1Gi PVC won't consume a 1Ti PV if a 2Gi PV is available.
Static bindingAdmin pre-provisions PV with claimRef pointing to specific PVC. Ensures deterministic binding — used for data migration.
Dynamic prov.No matching PV exists → StorageClass provisioner called → creates PV + cloud disk (EBS, GCE PD) → PV controller binds to PVC.
Phase statesPVC: Pending→Bound. PV: Available→Bound→Released→Available (if Retain+manual reclaim) or deleted.
StatefulSet PVCs created via volumeClaimTemplates have an ownerRef pointing to the StatefulSet, but only in newer K8s versions (1.27+ with the PVC retention policy feature). In older versions the PVC has NO ownerRef and is never garbage-collected.

ConfigMap

As env varenvFrom.configMapRef.name — all keys become env vars. Or env[].valueFrom.configMapKeyRef for single key.
As volumevolumes[].configMap.name + volumeMounts — each key becomes a file in the mount path.

🔑 Secret → Pod

→

TypesOpaque (generic), kubernetes.io/tls (cert+key), kubernetes.io/dockerconfigjson (image pull), kubernetes.io/service-account-token (SA token).

📁 Namespace — Soft Boundary

Deletion cascadeDeleting a Namespace triggers the Namespace controller to delete ALL namespaced resources inside it. Phase transitions: Active → Terminating → gone (once all resources GC'd).
Cluster-scopedNodes, PVs, StorageClasses, ClusterRoles, ClusterRoleBindings, Namespaces themselves — not namespaced. They exist at cluster scope, accessible/visible across all namespaces.
ResourceQuotaNamespaced object that limits total resource consumption within the namespace. Enforced by admission plugin, tracked by ResourceQuota controller.
Stuck Terminating namespace = finalizer on a resource inside it. Namespace controller waits for all resources to disappear. If a CRD is deleted before its CRs, the CRs become unregistered orphans with no controller to remove their finalizers. Nuclear fix: patch finalizers on stuck objects.
🖥️ Node — Physical Boundary
Node
cluster-scoped
← nodeName
Pod
scheduled here
RelationshipPod references Node via spec.nodeName. Pod does NOT have an ownerReference pointing to the Node. The scheduler writes nodeName; kubelet on that Node watches for pods with its name and runs them.
Node objectCreated by kubelet itself (--register-node=true) or by cloud controller manager. Contains: addresses, conditions (Ready/DiskPressure/MemoryPressure/PIDPressure), allocatable/capacity resources, labels for node selection.
TaintsNode taints + Pod tolerations = the eviction/anti-scheduling policy layer. NodeLifecycle controller auto-adds taints when node goes NotReady or has resource pressure.
Pod eviction ≠ immediate. Node goes Unknown → 40s grace → taint node.kubernetes.io/unreachable:NoExecute added → pods tolerate it for tolerationSeconds (default 300s) → then eviction. Total: up to 340 seconds. Rate-limited. Stops if cluster-wide healthy ratio drops below 55%.
ownerReference Mechanics
// how garbage collection actually works
🔗 ownerReferences Array Structure
metadata:
  ownerReferences:
  - apiVersion: apps/v1
    kind: ReplicaSet
    name: my-deploy-69d7bc8c7f
    uid: a3f2b1c0-... ← matches RS .metadata.uid
    controller: true ← only one owner is "controller"
    blockOwnerDeletion: true
controller: trueAt most ONE owner can have controller: true. This is the "managing controller" — the one responsible for the object's lifecycle. Others are additional owners (rare).
blockOwnerDeletionIf true: owner gets foregroundDeletion finalizer when deleted — GC won't remove owner until all dependents with this flag are gone.
uid not nameownerRef uses UID not name. If you delete an RS and a new one with the same name is created, orphaned pods don't get re-adopted — UIDs don't match.
Cross-namespaceownerReferences cannot cross namespace boundaries. A pod in namespace A cannot have an owner in namespace B. Cluster-scoped resources (PV) cannot own namespaced resources.
🗑️ GC Propagation Policies
Background (default)Owner deleted immediately from etcd. GC controller asynchronously finds orphaned dependents (ownerRef UID no longer exists) → deletes them. Fast for owner, async cleanup.
ForegroundOwner gets foregroundDeletion finalizer. GC deletes dependents first (depth-first). Then removes finalizer → owner deleted. Owner visible in API with deletionTimestamp until all children gone.
Orphan--cascade=orphan: removes ownerReferences from all dependents, then deletes owner. Dependents survive as independent objects with no parent.
GC is eventually consistent. Delete a Deployment: RS + Pods persist for seconds. Never assume immediate cleanup in CI scripts — use kubectl wait --for=delete pod -l app=foo --timeout=60s.
