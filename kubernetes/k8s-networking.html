Pod Networking Fundamentals
// every pod gets a real routable IP · flat network model · no NAT between pods
🔌 The Kubernetes Network Model (4 rules)
Rule 1 — Pod↔Pod, no NAT
Every pod can reach every other pod using its real IP. No address translation. Pod A at 10.0.1.5 reaches Pod B at 10.0.2.7 directly.
Rule 2 — Node↔Pod, no NAT
Nodes can reach pods using pod IPs directly. No masquerade. Same applies in reverse — pods can reach nodes.
Rule 3 — IP seen = IP used
The IP a pod sees as its own (eth0) is the same IP other pods use to reach it. No masquerade within the cluster.
Rule 4 — Implemented by CNI
K8s doesn't implement networking itself. CNI plugins (Calico, Cilium, Flannel) are responsible for making these rules true on real hardware.
Pod CIDRConfigured per-cluster (e.g., 10.244.0.0/16 for Flannel, 192.168.0.0/16 for Calico). Each node gets a slice: node-1 = 10.244.0.0/24, node-2 = 10.244.1.0/24.
veth pairEach pod gets a virtual ethernet pair. One end (eth0) inside pod network namespace. Other end (vethXXXXX) on the node, connected to a bridge or direct routing.
Network NSEach pod has its own network namespace (created by pause container). All containers in the pod share the same namespace — same eth0, same IP, same port space.
pause container"infra" container. Holds the network namespace open. If your app container crashes and restarts, the network namespace persists — same IP, no network interruption.
Containers in the same pod share the network namespace. They communicate via localhost and see the same ports. Two containers in one pod cannot both listen on port 8080. Port conflicts are a pod-level concern, not a node-level one.
🏗️ How a Pod Gets Its IP
kubelet creates Pod
→ calls CNI binary via exec
→ CNI creates veth pair
→ moves one end into pod netns
→ assigns IP from node CIDR pool
→ sets up routes on node
→ returns IP to kubelet
→ kubelet writes to Pod status
CNI is called twice per container: ADD (bring up networking) on create, DEL (tear down) on delete. The spec is simple: binary reads stdin JSON, writes stdout JSON with assigned IP. No daemon required.
📡 Cross-Node Pod Traffic
Overlay (VXLAN/IPIP): Flannel/Weave. Pod packet encapsulated inside UDP (VXLAN) or IP-in-IP. Node knows which pod CIDRs live on which nodes via etcd/annotations.
Underlay (BGP): Calico. Pod IPs advertised as real routes via BGP. No encapsulation overhead. Requires network equipment to support BGP.
eBPF (bypass kernel): Cilium. Replaces iptables/kube-proxy entirely. Packets processed in eBPF programs at XDP/TC hook points. Lowest latency, best observability.

Service Types
// ClusterIP · NodePort · LoadBalancer · ExternalName · Headless
🗺️ Traffic Path: Internet → Pod
🟡 ClusterIP default
What it isVirtual IP inside the cluster only. Stable IP+port that load-balances to matching pods. Never accessible from outside.
IP realityClusterIP does not exist on any network interface. It's a fiction — iptables DNAT rules rewrite packets destined for it to a real pod IP before routing.
DNSCoreDNS resolves svc-name.namespace.svc.cluster.local → ClusterIP. In-cluster DNS always works regardless of actual pod IPs changing.
spec: type: ClusterIP selector: app: myapp ports: - port: 80 targetPort: 8080
ClusterIP=None = Headless. CoreDNS returns individual pod IPs (A records), not a single VIP. Used for StatefulSet DNS names (pod-0.svc.ns) and direct pod addressing.
🔵 NodePort
What it isOpens a static port (30000-32767) on every node. Traffic on nodeIP:nodePort forwards to the service. Includes a ClusterIP.
kube-proxyPrograms iptables rule on every node: packets arriving on the nodePort are DNAT'd to a random pod (any node, not just the local one).
Source IP lossBy default, kube-proxy masquerades the source IP. Use externalTrafficPolicy: Local to preserve client IP — but only local-node pods receive traffic.
spec: type: NodePort ports: - port: 80 targetPort: 8080 nodePort: 30080 # optional externalTrafficPolicy: Local
externalTrafficPolicy: Local + no local pods = dropped packets. If the node has no matching pod, packets are silently dropped. Health check port must be configured on LB to avoid this.
🟨 LoadBalancer
What it isProvisions an external cloud LB (ELB, GCP LB, Azure LB). Includes NodePort+ClusterIP. Cloud CCM watches and creates the LB.
ProvisioningCloud Controller Manager (CCM) calls cloud API to create LB pointing at all node NodePorts. status.loadBalancer.ingress[].ip populated when ready.
AnnotationsAll behavior controlled via cloud-specific annotations: service.beta.kubernetes.io/aws-load-balancer-type: nlb, internal vs external, cross-zone, SSL certs.
spec: type: LoadBalancer externalTrafficPolicy: Local loadBalancerSourceRanges: - "10.0.0.0/8" metadata: annotations: service.beta.kubernetes.io/aws-load-balancer-type: "nlb"
One LoadBalancer Service = one cloud LB = $$$. For HTTP/HTTPS, use one Ingress + one LB for all services. LoadBalancer type is L4-only — no host/path routing.
🟢 ExternalName + Headless
ExternalNameReturns a CNAME to an external DNS name. No proxying. Used to alias external services into the cluster DNS namespace. No ClusterIP allocated.
spec: type: ExternalName externalName: db.prod.example.com
HeadlessclusterIP: None. No VIP allocated. DNS returns pod IPs directly (A records). Client does its own LB/selection. Mandatory for StatefulSets.
spec: clusterIP: None # headless selector: app: mydb
StatefulSet pod DNS: pod-0.svc.namespace.svc.cluster.local → specific pod IP. Stable even across restarts. Only works with headless service.
ExternalName ignores ports and selectors. Also: CNAME lookup adds a DNS round-trip. TLS with ExternalName can fail SNI matching — the client sees the original service name, not the external hostname.

kube-proxy
// iptables vs IPVS · how ClusterIP actually works · connection tracking
⚙️ iptables vs IPVS mode
iptables (default)IPVS
MechanismNetfilter PREROUTING DNAT rulesLinux Virtual Server in-kernel LB
LB algorithmRandom (stateless probabilistic)RR, LC, DH, SH, SED, NQ — configurable
ScaleO(n) rule scan per packetO(1) hash table lookup
ServicesDegrades at ~10k servicesHandles 100k+ services
ConntrackFull conntrack for every flowIPVS conntrack (lighter)
Observabilityiptables -t nat -L -nipvsadm -Ln
Kernel req.Any modern kernelNeeds ip_vs modules loaded
IPVS mode requires kernel modules: ip_vs, ip_vs_rr, ip_vs_wrr, ip_vs_sh. Set kube-proxy mode: ipvs in ConfigMap. In iptables mode with 50k services, a single packet traverses 50k iptables rules — measurable latency.
kube-proxy is optional with Cilium. Cilium's eBPF mode replaces kube-proxy entirely — service load balancing handled in eBPF programs at the network hook point. Faster, no iptables, and works without the kube-proxy DaemonSet.
🔍 How DNAT Works
# Packet flow for ClusterIP 10.96.0.100:80 Pod sends to 10.96.0.100:80 ↓ iptables PREROUTING chain ↓ KUBE-SERVICES chain ↓ matches 10.96.0.100:80 KUBE-SVC-XXXX chain ↓ random select (33% each for 3 pods) KUBE-SEP-YYYY chain ↓ DNAT → 10.244.1.3:8080 (real pod IP) ↓ Normal IP routing to node ↓ arrives at pod
📊 Key kube-proxy facts
WatchesEndpointSlices (not Endpoints). Rewrites iptables rules on any change.
Connection stickyOnce a connection is DNAT'd to a pod, conntrack keeps it to the same pod. Even if pod is removed from endpoints. Existing connections don't break on pod scale-down.
Session affinitysessionAffinity: ClientIP — same client IP always hits same pod. Implemented via iptables recent module. timeoutSeconds default 10800.

⚙️ Pod DNS Configuration
# /etc/resolv.conf inside every pod: nameserver 10.96.0.10 # CoreDNS ClusterIP search default.svc.cluster.local \ svc.cluster.local \ cluster.local options ndots:5 # Custom DNS per pod: spec: dnsPolicy: None dnsConfig: nameservers: ["8.8.8.8"] searches: ["prod.svc.cluster.local"] options: - name: ndots value: "2"

DNS cache in CoreDNS is not per-namespace. All namespaces share the same CoreDNS pods. A buggy app doing DNS floods will impact other tenants. Set resource limits + rate limits on CoreDNS.
🔧 CoreDNS Architecture
DeploymentRuns as a Deployment (not DaemonSet). 2+ replicas behind a ClusterIP Service (kube-dns) at 10.96.0.10.
Plugin chainCorefile configures plugins: errors → health → kubernetes (watches API) → forward (upstream). Each plugin handles or passes the query.
kubernetes pluginWatches Services + Pods via informers. Answers cluster.local queries from in-memory cache. Zero etcd reads for DNS.
NodeLocal DNSCacheDaemonSet runs a DNS cache on each node. Pods bypass CoreDNS for most queries. Eliminates conntrack race condition and reduces CoreDNS load. Highly recommended at scale.
# Debug DNS from inside pod nslookup kubernetes.default nslookup myservice.mynamespace # Trace actual queries kubectl -n kube-system logs -l k8s-app=kube-dns

Ingress & Gateway API
// L7 routing · TLS termination · IngressClass · Gateway API (next-gen)
🔀 Ingress Resource
apiVersion: networking.k8s.io/v1 kind: Ingress metadata: annotations: # nginx-specific: nginx.ingress.kubernetes.io/rewrite-target: / nginx.ingress.kubernetes.io/ssl-redirect: "true" nginx.ingress.kubernetes.io/proxy-body-size: "50m" spec: ingressClassName: nginx # which controller handles this tls: - hosts: [api.example.com] secretName: tls-secret # kubernetes.io/tls type rules: - host: api.example.com http: paths: - path: /v1 pathType: Prefix # Exact | Prefix | ImplementationSpecific backend: service: name: api-svc port: { number: 80 }
pathType: ExactMatches exactly /v1 — no trailing slash, no subdirs.
pathType: PrefixMatches /v1, /v1/, /v1/users. URL segment split on /.
pathType: ImplSpecController-defined matching (nginx regex, Traefik rules). Not portable.
Ingress object does nothing by itself. Without an Ingress controller running in the cluster, Ingress resources are ignored. ingressClassName must match a deployed IngressClass. Multiple controllers can coexist — wrong class = silent no-op.
TLS secret must be in the same namespace as the Ingress. Cannot reference a Secret from another namespace. Use cert-manager to auto-provision Let's Encrypt certs and keep Secrets co-located.

🆕 Gateway API (replacing Ingress)
WhyIngress is too limited and too annotation-heavy. Gateway API is a proper K8s SIG with role-separated CRDs.
ResourcesGatewayClass (infra owner) → Gateway (cluster op) → HTTPRoute/TCPRoute/GRPCRoute (app dev). RBAC-separated concerns.
HTTPRouteHeader matching, weight-based splitting (canary: 10/90), URL rewrites, request mirroring — all first-class without annotations.
kind: HTTPRoute spec: rules: - backendRefs: - name: svc-v2 weight: 10 - name: svc-v1 weight: 90

CNI Plugins
// Flannel · Calico · Cilium · Weave · how they implement the network model
🟣 Flannel
ApproachOverlay network using VXLAN (default) or IPIP. Pod packets encapsulated in UDP tunnels between nodes.
Node CIDREach node allocated a /24 from cluster CIDR. Stored in node annotations. flannel daemon reads these to build routing table.
SimplicitySimplest CNI. No NetworkPolicy support. No encryption. Just gets pods talking. Good for dev/test clusters.
Backendvxlan (L3 overlay via UDP 8472), host-gw (pure L3 routing, no encap — nodes must be on same L2), wireguard (encrypted VXLAN).
Flannel has NO NetworkPolicy support. NetworkPolicy objects are silently ignored. Must pair with Calico in flannel+canal mode or switch CNI entirely to enforce policies.
🔵 Calico
ApproachBGP-based L3 routing. No overlay by default — pods get real IPs routed via BGP. Each node is a BGP peer. Optional VXLAN/IPIP overlay for non-BGP environments.
NetworkPolicyFull K8s NetworkPolicy + extended Calico GlobalNetworkPolicy (cluster-wide), HostEndpoint (secure node interfaces), order/priority rules.
DataplaneDefault: iptables. Optional: eBPF dataplane (replaces kube-proxy + iptables, faster). Or Windows HNS dataplane.
WireguardOptional in-transit encryption via WireGuard — transparent to pods. Enable with one annotation on node.
Calico is the most production-common CNI. BGP mode is fast (no encap overhead) but requires L3-aware network gear or a Route Reflector for large clusters.
🟢 Cilium
ApproacheBPF-native. No iptables. All packet processing in eBPF programs attached to veth/XDP hooks. Fastest and most observable.
kube-proxyCilium replaces kube-proxy entirely. Service load balancing via eBPF map lookups — O(1), no netfilter traversal.
L7 policyL7-aware NetworkPolicy — can enforce HTTP method/path, Kafka topic, gRPC method. Goes far beyond K8s NetworkPolicy spec.
HubbleBuilt-in observability: flow logs, service map, dropped packet reasons — all from eBPF maps without touching the dataplane.
RequirementLinux kernel ≥ 4.9 (recommended 5.10+). Full eBPF features need 5.15+. Can run alongside older kernels in legacy mode.
Cilium Hubble requires the Hubble relay deployment. The eBPF maps are per-node — Hubble relay aggregates across all nodes. Without it you only see flows for one node at a time.
📊 CNI Comparison
CNIPolicyPerfComplexity
FlannelNoneOverlayLow
CalicoFull+BGP/ebpfMed

AWS VPC CNI gives each pod a real VPC IP (ENI secondary IP). No overlay — traffic goes directly through VPC routing. Limited by node's max ENI/IP count. Best latency on AWS.
Switching CNI requires cluster rebuild. You cannot hot-swap a CNI plugin on a running cluster. All pods lose networking during transition. Plan for blue/green cluster migration.

NetworkPolicy
// ingress/egress rules · podSelector · namespaceSelector · default deny
🔒 NetworkPolicy Anatomy
apiVersion: networking.k8s.io/v1 kind: NetworkPolicy spec: podSelector: # applies to pods matching this matchLabels: { app: api } policyTypes: [Ingress, Egress] ingress: - from: - podSelector: # same namespace only matchLabels: { role: frontend } - namespaceSelector: # from specific namespace matchLabels: { env: prod } ports: - protocol: TCP port: 8080 egress: - to: - ipBlock: # external CIDR cidr: 10.0.0.0/8 except: [10.0.0.1/32] ports: - port: 5432 - ports: # allow DNS egress - port: 53 protocol: UDP
podSelector + namespaceSelector in same from array entry = AND. In separate entries = OR. This is the most common NetworkPolicy bug. from: [{podSelector: X, namespaceSelector: Y}] = pods matching X in namespace matching Y. Two separate entries = pods matching X in any namespace OR any pod in namespace matching Y.
⚠️ Default-Deny Patterns
# DEFAULT DENY ALL INGRESS in namespace spec: podSelector: {} # matches ALL pods policyTypes: [Ingress] # no ingress rules = deny all # DEFAULT DENY ALL EGRESS spec: podSelector: {} policyTypes: [Egress] # no egress rules = deny all # ⚠ blocks DNS too! Must add port 53 allow # ALLOW ALL (explicit permissive) spec: podSelector: {} ingress: - {} # empty = allow all sources
Default-deny egress blocks DNS. After applying a deny-all egress, all pod DNS lookups fail. Always add an egress rule allowing UDP/TCP port 53 to CoreDNS before applying egress default-deny.
🔑 NetworkPolicy Rules
Default openNo NetworkPolicy = allow all traffic. A policy only restricts when one exists for a pod. First deny-all creates a secure boundary.
AdditiveMultiple policies for same pod are unioned (OR). You cannot use one policy to override/restrict another. Policies only add permissions.
StatefulNetworkPolicy is connection-tracking aware. Allow TCP ingress on 8080 = reply packets automatically allowed (no egress rule needed for the response).
No deny rulesK8s NetworkPolicy only has allow rules. For deny-with-priority, need Calico GlobalNetworkPolicy or Cilium CiliumNetworkPolicy.
kubelet probesNodePort health checks come from node IP, not pod IP. If you block all ingress, kubelet liveness/readiness probes also blocked — pod gets killed.

Load Balancers, MetalLB & Cloud Integration
// cloud CCM · MetalLB · externalTrafficPolicy · health checks
☁️ Cloud LB Provisioning
CCM roleCloud Controller Manager watches type:LoadBalancer services. Calls cloud API (AWS/GCP/Azure) to provision LB. Writes external IP to status.loadBalancer.
AWS NLB vs CLBaws-load-balancer-type: nlb = NLB (L4, preserves client IP, TCP). Default = CLB (legacy). Use NLB for gRPC, WebSocket, static IP requirements.
Internal LBservice.beta.kubernetes.io/aws-load-balancer-internal: "true" → VPC-internal LB. No public IP. Access from within VPC only.
ETP: LocalexternalTrafficPolicy: Local on LB service → LB only routes to nodes with matching pods → client IP preserved (no SNAT) → LB must health-check the NodePort.
# AWS ALB (via AWS Load Balancer Controller) metadata: annotations: kubernetes.io/ingress.class: alb alb.ingress.kubernetes.io/scheme: internet-facing alb.ingress.kubernetes.io/target-type: ip alb.ingress.kubernetes.io/certificate-arn: arn:aws:acm:... alb.ingress.kubernetes.io/listen-ports: '[{"HTTPS":443}]'
ALB target-type: ip vs instance. ip = routes directly to pod IPs (requires VPC CNI). instance = routes to node NodePort. Use ip mode to avoid the extra hop and preserve client IP.
🏠 MetalLB (bare-metal)
What it solvesOn bare-metal/on-prem K8s, there's no cloud CCM. type:LoadBalancer stays Pending forever. MetalLB implements LB provisioning for physical clusters.
L2 modeOne node answers ARP/NDP for the assigned external IP. Traffic lands on that node, then kube-proxy distributes. Single-node bottleneck — no HA on the ingress node.
BGP modeEach node advertises external IPs via BGP. Router distributes traffic to all nodes via ECMP. True LB across nodes. Requires BGP-capable upstream router.
IPAddressPoolDefine address pools in IPAddressPool CRD. Services request from specific pool via metallb.universe.tf/address-pool annotation.
apiVersion: metallb.io/v1beta1 kind: IPAddressPool spec: addresses: ["192.168.1.240-192.168.1.250"] --- kind: L2Advertisement spec: ipAddressPools: [mypool]
MetalLB L2 mode leader election: only one node holds each IP at a time. Failover requires ARP/NDP withdrawal + re-advertisement — 10-30s gap. For HA, use BGP mode + ECMP.
📋 externalTrafficPolicy Deep Dive
ETP: Cluster (default)
✓ Traffic to any node, proxied to any pod
✓ All nodes healthy to LB
✗ Source IP lost (SNAT)
✗ Extra hop node→pod
✗ Uneven LB (hot nodes)
ETP: Local
✓ Client IP preserved
✓ No extra hop
✗ Only nodes with pods healthy
✗ Uneven if pods not spread
✗ Dropped on nodes without pods
internalTPinternalTrafficPolicy: Local (1.26 stable) — same concept for traffic arriving via ClusterIP from within the cluster. Forces traffic to local-node pods only.
ETP:Local + DaemonSet = perfect. DaemonSet ensures every node has a pod → every node is healthy to LB → client IP preserved + no extra hop. Ideal pattern for log shippers, proxies, per-node agents.

Service Mesh
// sidecar pattern · mTLS · traffic management · Istio · Linkerd
🔄 Sidecar Proxy Pattern
InjectionIstio mutating webhook automatically injects Envoy sidecar + istio-init container into pods in labeled namespaces (istio-injection: enabled).
iptables interceptInit container runs iptables rules in pod network NS: all inbound traffic redirected to Envoy port 15006, all outbound to 15001. App code is unaware of the proxy.
mTLSSPIFFE X.509 certs issued per service identity. Istiod acts as CA. Cert rotation every 24h by default. Zero-trust: services only trust each other with valid SPIFFE certs.

⚡ Ambient Mesh (Istio)
What it isSidecar-free service mesh. Istio 1.18+. Uses per-node ztunnel for mTLS + L4 policies, optional per-namespace waypoint proxy for L7.
BenefitNo sidecar injection = no pod restart needed, no per-pod overhead, gradual adoption. Traffic redirection is handled by istio-cni via iptables in the pod's netns; ztunnel is the node-level zero-trust L4/mTLS proxy.
Ambient mesh is GA in Istio 1.24+ but not all features are parity with sidecar. Check feature matrix before adopting ambient for critical workloads.
