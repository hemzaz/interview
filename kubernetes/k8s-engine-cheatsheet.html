Manifest Parsing + API Mechanics // what happens to your YAML
📄
YAML → Object lifecycle
How kubectl turns YAML into an API call
Decodekubectl parses YAML to JSON, deserializes into a typed Go struct (e.g., v1.Pod) using the API machinery decoder.
GVR / GVKGroup/Version/Resource determines the REST path. apps/v1/Deployment → /apis/apps/v1/namespaces/{ns}/deployments.
Patch typekubectl apply uses Strategic Merge Patch. Computes diff vs last-applied annotation. Server-Side Apply (SSA) sends full desired state; apiserver owns field management.
Dry run modes--dry-run=client validates locally. --dry-run=server runs full admission pipeline (mutating+validating webhooks) without etcd write.
Diffkubectl diff -f does a server-side dry run and shows diff. Uses kubectl.kubernetes.io/last-applied-configuration annotation to track your intent.
SMP vs JSON Merge Patch: SMP merges lists by a strategic key (containers by name). Plain JSON merge replaces entire list. Always use SMP or SSA for K8s resources.

Admission Pipeline Detail
Mutation first, then validation
1
Mutating Admission Webhooks
Run in parallel. Each can modify the object. Results merged back. Built-ins: DefaultStorageClass, MutatingAdmissionWebhook. Sidecar injectors live here.
2
Schema Validation + Defaulting
OpenAPI schema check. Built-in plugins: LimitRanger (injects CPU/mem defaults), ResourceQuota (checks namespace quota), PodSecurity (policy enforcement).
3
Validating Admission Webhooks
Run in parallel. Cannot modify — only approve or reject. OPA/Gatekeeper, Kyverno, custom policy engines hook here. Any rejection aborts entire request.
4
etcd Write
Serialized as protobuf under /registry/{resource}/{ns}/{name}. Returns resourceVersion. Triggers watch events to all subscribers.

Scheduling Constraints
How pods pick their nodes
nodeSelectorSimple equality match on node labels. Hardcoded. Fails scheduling if no node matches.
Node AffinityrequiredDuringScheduling = hard (like nodeSelector but richer). preferredDuringScheduling = soft scored preference.
Taints / TolerationsTaint on node repels pods. Toleration on pod allows it. Effects: NoSchedule, PreferNoSchedule, NoExecute (evicts).
Pod AffinityCo-locate with pods matching a label selector. topologyKey: topology.kubernetes.io/zone = same zone.
TopologySpreadmaxSkew: 1 — keep replicas evenly spread across zones. Replaces manual anti-affinity. Preferred modern approach.
Priority / PreemptionHigher priority pod can evict lower priority pods from nodes when cluster is full. Configured via PriorityClass.
Pod stuck Pending? Either no node passed Filter (check kubectl describe pod events) or cluster at capacity (check node allocatable vs requested).
⏱️
Scheduler Internals
Queue and cache model
Scheduling queueThree queues: activeQ (ready to schedule), backoffQ (failed, waiting backoff), unschedulableQ (blocked by policy).
SnapshotScheduler takes a read-only snapshot of node state at scheduling start. Prevents race conditions during the scheduling cycle.
Parallel scoringScore phase runs plugins in parallel across all feasible nodes. Results normalized to 0–100 and weighted.
Assume + BindScheduler assumes the pod is bound (optimistic) before the API write completes — keeps the cycle fast. Async bind is later confirmed.

🔢
Work Queue
Dedup, rate-limit, retry
Key formatItem enqueued is just namespace/name string. Not the object itself. Controller fetches fresh from cache when dequeuing.
DeduplicationIf object changes 10× before worker picks it up — only one reconcile runs. The queue deduplicates by key. Prevents thundering herd.
Rate limitingBuilt-in exponential backoff on failures. ItemExponentialFailureRateLimiter — default 5ms base, 1000s max. Prevents hot loop on repeated errors.
ForgetOn success, call queue.Forget(key) to reset failure count. If you don't, backoff accumulates forever even on success.
Done()Must call queue.Done(key) when finished processing — even on error. Otherwise item is permanently blocked from re-enqueueing.
// Canonical work queue pattern func runWorker() { for processNextItem() {} } func processNextItem() bool { key, quit := queue.Get() if quit { return false } defer queue.Done(key) // ALWAYS call Done err := reconcile(key) if err != nil { queue.AddRateLimited(key) // retry with backoff return true } queue.Forget(key) // reset failure count return true }
Forgetting to call queue.Done(key) causes the item to be stuck in "processing" state forever — the queue won't allow it to be re-added.

Reconcile Loop Pattern
The heart of every controller
// Universal reconcile pattern (controller-runtime) func (r *Reconciler) Reconcile(ctx, req) { // ① READ from local cache — NOT the API server obj := &v1.Deployment{} if err := r.Get(ctx, req.NamespacedName, obj); err != nil { return ignoreNotFound(err) // object deleted — OK } // ② Check if we should act (deletion in progress?) if !obj.DeletionTimestamp.IsZero() { return handleDeletion(ctx, obj) } // ③ Ensure finalizer registered if !containsFinalizer(obj, myFinalizer) { addFinalizer(obj); r.Update(ctx, obj); return } // ④ COMPUTE desired state (pure logic, no side effects) desired := buildDesiredState(obj) // ⑤ GET actual state of owned objects actual := getOwnedObjects(ctx, obj) // ⑥ DIFF and apply only what changed if diff := compare(actual, desired); diff != nil { r.Patch(ctx, desired, patch) // ← re-triggers watch } // ⑦ Update STATUS subresource obj.Status = buildStatus(actual) r.Status().Update(ctx, obj) // ⑧ Requeue after interval (drift correction) return ctrl.Result{RequeueAfter: 30 * time.Second}, nil }
Idempotency is non-negotiable. Reconcile must produce the same result whether it runs once or 100 times for the same input. Never assume state from a previous run.
Desired State vs Actual State // the core K8s mental model
⚖️
The State Triangle
spec vs status vs actual world
specWhat you want. Immutable intent. You write it, controller reads it.
statusWhat the controller observed. Written by controllers and kubelet. Never write status in your own code via regular Update — use Status().Update() subresource.
DriftController periodically requeues even without events (via RequeueAfter) to catch drift — e.g., someone manually deleted a Pod the controller owns.
Status subresource is separate from spec. A PATCH on the main object doesn't touch status. A PATCH on /status doesn't touch spec. Keeps concern separation clean.
🔗
Owner References + Garbage Collection
How cascading deletes work
OwnerReferenceEvery owned object has an ownerReferences array pointing to its parent. GC controller traverses this graph to determine cascade.
Foreground GCOwner gets foregroundDeletion finalizer. GC deletes dependents first, then removes finalizer. Blocking — owner stays until dependents gone.
Background GCOwner deleted immediately. GC orphan-adopts and deletes dependents asynchronously. Default for kubectl delete.
Orphan policy--cascade=orphan: owner deleted, dependents become orphaned. No GC. Useful for manual takeover.
FinalizersObject with finalizers gets deletionTimestamp set but not deleted. Controller sees deletionTimestamp → runs cleanup → removes finalizer → object disappears.

Debug Playbook // every scenario you'll face
🔍
Pod Pending / Not Scheduling
Scheduler can't place the pod
1
kubectl describe pod <name> → look at Events section. Scheduler writes reason there: 0/5 nodes available: 5 Insufficient cpu.
2
kubectl describe node <node> | grep -A8 Allocated → see actual vs allocatable. Is cluster full?
3
Check taints: kubectl get node -o json | jq '.items[].spec.taints'. Missing toleration? Check nodeSelector: does any node have the required labels?
4
Check PVC binding: if pod uses a PVC, kubectl get pvc — if Pending, storage class can't provision or zone mismatch with node affinity.
kubectl describe pod mypod | grep -A20 Events → 0/3 nodes available: 3 node(s) had untolerated taint kubectl get events --sort-by=.lastTimestamp -n mynamespace
💥
Pod CrashLoopBackOff / OOMKilled
Container keeps dying
1
kubectl logs <pod> --previous — get logs from the previous crashed container. Most crashes are visible here.
2
kubectl describe pod → check Last State: OOMKilled = memory limit too low. Error = app crash. ContainerCannotRun = image entry point issue.
3
Backoff timing: 10s → 20s → 40s → 80s → 160s → 300s max. Resets after 10m stable. Can't override — design for fast startup or use startupProbe for slow apps.
4
Check liveness probe config. Probe too aggressive + slow startup = kubelet kills before ready. Fix: increase initialDelaySeconds or add a startupProbe.
kubectl logs mypod --previous -c mycontainer kubectl describe pod mypod | grep -A5 "Last State" Last State: OOMKilled exitCode: 137
🌐
Service / Networking Issues
Pod running but not reachable
1
kubectl get endpoints <svc> — are there any? Empty = no pods match selector. Check svc.spec.selector vs pod.labels — must be exact match.
2
kubectl run tmp --image=busybox -it --rm -- wget -qO- http://<svc>:<port> — test DNS resolution + connectivity from inside cluster.
3
Check NetworkPolicy: default-deny ingress + no matching allow rule = requests silently dropped. kubectl get netpol -n <ns> and trace selectors.
4
Check readinessProbe: pod Running but not Ready = endpoint removed from Service. Pod receives no traffic. Check probe path + timeout.
kubectl get endpoints my-service → ENDPOINTS: <none> ← selector mismatch or pod not Ready kubectl get pod -l app=myapp --show-labels
🗑️
Stuck Terminating / Namespace
Finalizer traps
Root causeObject has a finalizer that a controller never removed. deletionTimestamp is set, object is frozen.
Find themkubectl get pod mypod -o json | jq .metadata.finalizers. If non-empty and controller is gone — you need to patch.
Force clearOnly do this if the owning controller is confirmed gone. Removes the finalizer, unblocks deletion.
# Remove finalizer to unblock deletion kubectl patch pod mypod \ -p '{"metadata":{"finalizers":[]}}' \ --type=merge # Stuck namespace — more complex kubectl get namespace stuck-ns -o json \ | jq '.spec.finalizers = []' \ | kubectl replace --raw \ /api/v1/namespaces/stuck-ns/finalize -f -
Clearing finalizers manually skips cleanup logic. Only do this when the owning controller is confirmed dead and you accept the risk of leaked resources.
⚡
resourceVersion Conflicts
409 Conflict — optimistic concurrency
What it isEvery write must include resourceVersion. If the object changed since you read it, apiserver returns 409 Conflict. Prevents lost updates.
Why it mattersTwo controllers updating same object simultaneously — only one wins. The other gets 409 and must re-read + retry. This is by design, not a bug.
Fix in codeUse retry.RetryOnConflict loop: re-Get the object (fresh resourceVersion), recompute patch, re-apply. Standard controller-runtime handles this automatically.
Watch latencyresourceVersion also used in Watch: WATCH /pods?resourceVersion=12345 resumes stream from that version. No List needed after a disconnect.
// Retry on conflict pattern import "k8s.io/client-go/util/retry" retry.RetryOnConflict(retry.DefaultRetry, func() error { // Always re-Get for fresh resourceVersion client.Get(ctx, key, obj) obj.Spec.Replicas = &newCount return client.Update(ctx, obj) })
