EKS Architecture
// managed control plane · data plane options · VPC topology
🎛️ Control Plane (AWS manages)
What's includedAPI server (HA, multi-AZ), etcd (encrypted, backed up), scheduler, controller-manager, cloud-controller-manager. You never SSH into these.
Cost$0.10/hr per cluster (~$73/month). No charge for worker nodes (EC2 billed separately). EKS Auto Mode adds $0.003/node/hr.
Endpoint modesPublic+Private (default) — internet-accessible API + in-VPC ENI. Private only — no internet access, requires VPN/DirectConnect. Public only — nodes reach API via internet.
OIDC providerEach cluster has an OIDC issuer URL. Used by IRSA to exchange ServiceAccount JWT tokens for IAM role credentials. Must enable explicitly.
Kubernetes versionAWS supports latest + 3 previous minor versions. 14 months standard support, 26 months extended (adds cost). After that: auto-upgrade happens.
Private endpoint clusters need VPN/Direct Connect for kubectl. CI/CD runners inside VPC can reach private endpoint — but your laptop can't without routing. Combine public+private: nodes use private ENI, you use public endpoint.
🖥️ Data Plane Options
ModeYou manageBest for
Managed Node GroupAMI version, instance type, desired countMost workloads. AWS handles drain on termination.
Self-Managed NodesEverything (ASG, bootstrap, AMI)Custom kernel, GPU drivers, special OS config.
FargateNothing (serverless pods)Bursty, isolated, no DaemonSet workloads.
EKS Auto ModeAlmost nothingKarpenter-based, AWS manages node config.
Managed Node Groups handle draining before terminating nodes. They don't automatically handle AMI upgrades — you trigger UpdateNodegroupVersion or use the console. MNG uses rolling update: launches new nodes, drains old ones.
🌐 Networking Deep Dive
VPC CNIaws-node DaemonSet. Assigns secondary IP addresses from your VPC CIDR to pods. Pods get real VPC IPs — no overlay. Max pods = (ENIs × (IPv4 per ENI − 1)) + 2 (e.g., t3.medium: 3 ENIs × (6 IPs per ENI − 1) + 2 = 17 pods max by default).
Prefix delegationAttach /28 prefixes to ENI instead of individual IPs → 16× more pod IPs per ENI. Required for larger pod densities. Enable: ENABLE_PREFIX_DELEGATION=true on aws-node.
Security groupsNode SG + Pod-level SGs (Security Groups for Pods addon). Pod SG assigned per workload via SecurityGroupPolicy CRD. Only Nitro instances support pod-level SG.
Service meshApp Mesh (AWS managed) or Istio/Linkerd (self-managed). Sidecar injection, mTLS, traffic management. Not required for EKS but common at scale.
Default VPC CNI max pods is very low. t3.medium: 3 ENIs × (6 IPs − 1) + 2 = 17 pods max. Enable prefix delegation or use larger instances. Check kubectl describe node | grep pods.

EKS Addons
// managed vs self-managed · click each addon for details
🟢 EKS Managed Addon — AWS handles upgrades, config stored in EKS
🟠 Self-Managed — Helm/manifests, you own the upgrade
vpc-cni (aws-node)
✅ EKS Managed Addon
AWS VPC CNI plugin. Assigns VPC IPs to pods. Foundation of all pod networking.
DaemonSet: runs on every node as aws-node. Calls EC2 API to manage ENIs and assign secondary IPs.

Key env vars: ENABLE_PREFIX_DELEGATION (more pods/node), AWS_VPC_K8S_CNI_EXTERNALSNAT (custom NAT).

Upgrade trap: upgrading CNI addon restarts aws-node DaemonSet — brief connectivity blip per node. Do during low-traffic window. CNI version must be compatible with K8s version.
CoreDNS
✅ EKS Managed Addon
In-cluster DNS. Every pod DNS query goes through CoreDNS. 2 replicas by default.
Scale: default 2 replicas handles ~300 nodes. Add replicas for larger clusters: kubectl scale deploy coredns --replicas=5 -n kube-system.

NodeLocal DNSCache: DaemonSet that caches DNS per-node. Eliminates conntrack race condition. Reduces CoreDNS load by 90%+. Highly recommended at scale.

Trap: CoreDNS upgrade during K8s upgrade can cause DNS failures if config format changes. Always check release notes.
kube-proxy
✅ EKS Managed Addon
Programs iptables/IPVS rules for Service networking. Runs on every node.
Mode: EKS uses iptables mode by default. Switch to IPVS for clusters with 1000+ Services (O(1) vs O(n) lookup).

Replacement: Cilium can replace kube-proxy entirely with eBPF. If using Cilium: set kubeProxyReplacement: true and don't install kube-proxy addon.

Version lock: kube-proxy version must match K8s server version within 1 minor version.
EBS CSI Driver
✅ EKS Managed Addon
PersistentVolume provisioner for EBS. Required for stateful workloads on EKS 1.23+.
Required since 1.23: in-tree EBS plugin removed. Must install CSI driver for PVs to work.

IRSA/Pod Identity required: driver needs IAM permissions to create/attach EBS volumes. Create IAM role with AmazonEBSCSIDriverPolicy and annotate ServiceAccount.

StorageClasses: gp2 (legacy), gp3 (recommended, 20% cheaper, more IOPS/throughput baseline). Set gp3 as default.

Limitation: EBS volumes are AZ-specific. Pod must schedule in same AZ as volume. Use EFS for cross-AZ storage.
EFS CSI Driver
✅ EKS Managed Addon
EFS (NFS) storage for ReadWriteMany workloads. Cross-AZ, elastic, shared filesystem.
Use when: multiple pods need shared storage, cross-AZ resilience needed, ML training data sharing.

Access points: mount specific subdirectory per team/namespace. Enforce UID/GID. Much cleaner than raw EFS mount.

Cost: EFS costs $0.30/GB/month (standard). Expensive for large datasets — consider S3 + Mountpoint-for-S3 instead.
Pod Identity Agent
✅ EKS Managed Addon
New (2023). Simpler alternative to IRSA for pod-level IAM roles. No OIDC required.
vs IRSA: Pod Identity uses EKS agent on node (DaemonSet). No OIDC configuration needed. Multiple clusters share same IAM role without trust policy changes.

Setup: associate IAM role + K8s ServiceAccount in EKS console/CLI. No annotation needed on ServiceAccount.

Limitation: requires EKS Pod Identity Agent addon on nodes. Doesn't work with Fargate. IRSA still works everywhere.
AWS Load Balancer Controller
🟠 Self-Managed (Helm)
Creates ALB from Ingress resources, NLB from LoadBalancer Services. Replaces legacy in-tree ELB provisioning.
Ingress → ALB: annotation kubernetes.io/ingress.class: alb. One ALB per Ingress by default. Use alb.ingress.kubernetes.io/group.name to share ALB across Ingresses (cost saving).

Target types: ip (routes to pod IP — no NodePort hop, requires VPC CNI) vs instance (routes to NodePort). Use ip mode.

IRSA required: controller needs EC2/ELBv2 permissions. Attach AWSLoadBalancerControllerIAMPolicy.
Cluster Autoscaler
🟠 Self-Managed (Helm)
Scales ASG/MNG based on pending pods. Legacy — Karpenter is preferred on AWS.
How it works: watches for Unschedulable pods → finds ASG that can satisfy → increments desired count. Scale-down: removes underutilized nodes after 10 min cooldown.

Version match: CA version must match K8s minor version exactly. 1.28 cluster needs CA 1.28.x.

Trap: CA and Karpenter together on same node group = conflict. Pick one. Karpenter is preferred for new clusters.
Karpenter
🟠 Self-Managed (Helm) — AWS-supported
Next-gen node provisioner. Direct EC2 Fleet calls, no ASG. Faster, smarter bin-packing.
Key CRDs: NodePool (replacement for InstanceProfile/ASG config) + EC2NodeClass (AMI, subnet, SG, instanceType filters).

Speed: node ready in ~45s vs CA ~3 min. Directly calls RunInstances API.

Consolidation: disruption.consolidationPolicy: WhenUnderutilized automatically bins nodes together. Significant cost savings.

Drift detection: if NodePool AMI changes, Karpenter drains and replaces nodes automatically. This is how node upgrades happen with Karpenter.
📋 Addon Upgrade Rules
Managed addonsDon't auto-upgrade. EKS notifies of available updates in console/API. You must explicitly update. During K8s control plane upgrade, addons may be marked "requires update" — update them after CP upgrade.
Conflict detectionIf you've manually modified addon-managed resources (e.g., edited CoreDNS ConfigMap), EKS marks the addon as DEGRADED and won't overwrite. Use --resolve-conflicts OVERWRITE to force.
Preservation mode--resolve-conflicts PRESERVE keeps your customizations but won't apply new default config from the upgrade. Risk: new config options missing.
Configuration schemaManaged addons support structured config via --configuration-values JSON. No more manual kubectl edits. Schema varies per addon version.
Addon version ≠ chart version. EKS addon versions (e.g., v1.16.4-eksbuild.2) are AWS-built images. The upstream Helm chart version is different. Don't mix documentation.
🔍 Addon Status Check
# List all addons and their status aws eks list-addons \ --cluster-name my-cluster # Describe specific addon aws eks describe-addon \ --cluster-name my-cluster \ --addon-name vpc-cni # Available versions for an addon aws eks describe-addon-versions \ --addon-name vpc-cni \ --kubernetes-version 1.31 # Update addon (after K8s upgrade) aws eks update-addon \ --cluster-name my-cluster \ --addon-name vpc-cni \ --addon-version v1.19.0-eksbuild.1 \ --resolve-conflicts OVERWRITE # Check for addon health issues aws eks describe-addon \ --cluster-name my-cluster \ --addon-name coredns \ --query 'addon.health'

Scaling
// HPA · VPA · KEDA · Cluster Autoscaler · Karpenter
📊 HPA — Horizontal Pod Autoscaler
What it doesScales pod replicas based on CPU, memory, or custom metrics. Does not scale nodes. Works in tandem with Cluster Autoscaler/Karpenter.
Metrics serverRequired for CPU/memory HPA. Install metrics-server addon. EKS doesn't include it by default. Without it: unknown metric status.
FormuladesiredReplicas = ceil(currentReplicas × currentMetric / targetMetric). Stabilization window: 5min down, 0s up (avoid rapid scale-down).
v2 APIUse autoscaling/v2. Supports multiple metrics, custom metrics (Prometheus via adapter), external metrics (SQS queue depth via KEDA).
apiVersion: autoscaling/v2 kind: HorizontalPodAutoscaler spec: scaleTargetRef: { name: myapp } minReplicas: 2 maxReplicas: 20 metrics: - type: Resource resource: { name: cpu, target: { type: Utilization, averageUtilization: 70 }} behavior: scaleDown: stabilizationWindowSeconds: 300
HPA + VPA on same deployment = conflict. VPA changes resource requests → HPA recalculates based on new requests. Use HPA only, or VPA in recommendation-only mode alongside HPA.
⚡ KEDA — Event-Driven Autoscaling
What it doesScale pods to zero and from zero based on external event sources: SQS queue depth, Kafka lag, Prometheus, DynamoDB, Cron schedules.
ScaledObjectCRD that wraps HPA. KEDA creates/manages HPA internally. You define triggers (event sources) and KEDA translates to HPA metrics.
Scale to zeroWhen queue is empty: replicas = 0. On new message: KEDA wakes one pod before HPA takes over. No traffic = zero cost. HPA alone can't scale to zero (min=1).
On AWSMost common trigger: aws-sqs-queue. Set queueLength threshold (target messages per pod). KEDA calls SQS API using IRSA.
kind: ScaledObject spec: minReplicaCount: 0 # scale to zero! maxReplicaCount: 100 triggers: - type: aws-sqs-queue metadata: queueURL: https://sqs.../queue queueLength: "5" # msgs per replica awsRegion: us-east-1 authenticationRef: { name: keda-aws-auth }
KEDA polling interval default is 30s. For latency-sensitive queues, set pollingInterval: 5. But each poll = SQS API call = cost. Balance freshness vs cost.

Cluster creatorThe IAM principal that creates the cluster is automatically cluster-admin. Other principals need explicit mapping in aws-auth or Access Entries. This is a top-3 "locked out" cause.
If you create EKS via Terraform with a role, and kubectl with a different profile — you're locked out. The role that ran terraform apply is the cluster admin, not your personal IAM user. Add yourself via aws-auth or use the same role.
🔑 aws-auth vs Access Entries
aws-auth (legacy)ConfigMap in kube-system. Maps IAM roles/users to K8s usernames/groups. Edited with kubectl — editing wrong format breaks ALL cluster access. No audit log.
Access Entries (new)EKS API-based approach (2024). Manage IAM principal→K8s RBAC mapping via AWS API. No kubectl needed. Auditable via CloudTrail. Preferred for new clusters.
# Legacy aws-auth ConfigMap kubectl edit configmap aws-auth -n kube-system mapRoles: - rolearn: arn:aws:iam::123:role/NodeRole username: system:node:{{EC2PrivateDNSName}} groups: [system:bootstrappers, system:nodes] - rolearn: arn:aws:iam::123:role/DevopsRole username: devops groups: [system:masters] # New: Access Entry (API-based) aws eks create-access-entry \ --cluster-name my-cluster \ --principal-arn arn:aws:iam::123:role/DevopsRole \ --type STANDARD aws eks associate-access-policy \ --cluster-name my-cluster \ --principal-arn arn:aws:iam::123:role/DevopsRole \ --policy-arn arn:aws:eks::aws:cluster-access-policy/AmazonEKSClusterAdminPolicy \ --access-scope type=cluster
aws-auth YAML format is strict. Extra spaces, wrong indentation, or missing quotes = all kubectl access breaks immediately. Use eksctl create iamidentitymapping instead of direct edits.
🎭 IRSA — IAM Roles for Service Accounts
How it worksPod's ServiceAccount has a JWT token → projected to pod at /var/run/secrets/eks.amazonaws.com/serviceaccount/token → AWS SDK exchanges it with STS using OIDC federation → gets temporary credentials.
Setup steps1. Enable OIDC provider on cluster. 2. Create IAM role with trust policy for OIDC issuer + service account name. 3. Annotate K8s ServiceAccount with role ARN.
# Trust policy for IRSA role { "Principal": { "Federated": "arn:aws:iam::123:oidc-provider/oidc.eks.us-east-1.amazonaws.com/id/EXAM" }, "Condition": { "StringEquals": { "oidc.eks...:sub": "system:serviceaccount:default:my-sa" } } } # ServiceAccount annotation annotations: eks.amazonaws.com/role-arn: arn:aws:iam::123:role/MyRole
IRSA trust policy has exact SA name. If you rename the ServiceAccount or change namespace, IRSA stops working. The sub condition is system:serviceaccount:{namespace}:{sa-name} — must match exactly.
🔓 Pod Identity (new 2023)
vs IRSANo OIDC setup needed. No trust policy with exact SA name. Associate IAM role to SA via EKS API — reusable across clusters without trust policy changes.
Pod Identity AgentDaemonSet on each node intercepts credential requests. Calls EKS API to get temp creds for the pod's associated IAM role. No STS OIDC exchange needed.
Setup1. Install Pod Identity Agent addon. 2. Create IAM role (no special trust policy). 3. Associate role with SA via aws eks create-pod-identity-association.
LimitationDoesn't work on Fargate. Requires the addon on every node (via DaemonSet). IRSA still preferred for maximum compatibility.
Both IRSA and Pod Identity can coexist. Pod Identity takes precedence if both are configured for the same ServiceAccount. Migrate gradually: new workloads on Pod Identity, existing on IRSA.
🛡️ Node IAM Role
Required policiesAmazonEKSWorkerNodePolicy — allows node to call EKS DescribeCluster. AmazonEKS_CNI_Policy — VPC CNI manages ENIs. AmazonEC2ContainerRegistryReadOnly — pull images from ECR.
IMDSv2Enforce IMDSv2 on node instances (hop limit = 1 blocks pods from using node metadata credentials). Critical security requirement. Pods must use IRSA/Pod Identity, not inherited node role.
Least privilegeNode role should have zero AWS service permissions beyond the 3 required. All AWS access from pods must go through IRSA/Pod Identity. Node role ≠ pod permissions.
# Enforce IMDSv2 on launch template aws ec2 modify-instance-metadata-options \ --instance-id i-xxx \ --http-put-response-hop-limit 1 \ --http-endpoint enabled \ --http-tokens required
If IMDSv2 hop limit is 2+, pods can steal node role credentials. Default hop limit is 2 for many AMIs. Set to 1 to block pod access to IMDS. Required for any multi-tenant cluster.

Pre-Flight Checklist
// run BEFORE starting upgrade · non-negotiable
📋
Check deprecated APIs. Run kubectl get --raw /metrics | grep apiserver_requested_deprecated_apis or use pluto detect-all-in-cluster. Any deprecated API in use = app breaks after upgrade.
📦
Check addon compatibility. aws eks describe-addon-versions --kubernetes-version 1.31. Some addon versions don't exist for the new K8s version.
🔍
Check EKS upgrade insights. aws eks list-insights --cluster-name myc --filter category=UPGRADE_READINESS. AWS pre-scans your cluster for known issues.
⏱️
Only upgrade ONE minor version at a time. 1.28 → 1.29 → 1.30. Cannot skip versions. Plan multiple maintenance windows if far behind.
📸
Backup critical state. Velero snapshot of PVs, ETCD backup is automatic but verify, export all manifests: kubectl get all --all-namespaces -o yaml.
🏥
Verify PodDisruptionBudgets. Misconfigured PDBs block node drain indefinitely. Check all PDBs: kubectl get pdb -A. maxUnavailable=0 with minAvailable=replicas = deadlock.
🔑
Confirm kubectl version. kubectl must be within ±1 minor version of server. Upgrade kubectl first if needed.
🌿
Check kube-apiserver audit logs. Look for warning level events about deprecated API usage by specific clients.
⚡
Test in non-prod first. Upgrade a staging cluster using the same procedure. If it works → proceed to prod.
📢
Notify teams. Announce maintenance window. Node upgrades drain workloads — brief disruption expected even with rolling updates.
# Quick pre-upgrade scan # 1. Check for deprecated APIs pluto detect-all-in-cluster --target-versions k8s=v1.31.0 # 2. EKS upgrade insights aws eks list-insights --cluster-name prod \ --filter '{"category": ["UPGRADE_READINESS"]}' # 3. Check PodDisruptionBudgets for blockers kubectl get pdb -A -o json | jq '.items[] | select(.spec.maxUnavailable == 0) | .metadata'

Control Plane Upgrade
// always first · AWS handles HA · ~10-25 minutes · no downtime
AWS managesAWS upgrades API servers one at a time, maintaining HA throughout. etcd is also upgraded. Zero downtime for the control plane itself — existing pods keep running.
DurationTypically 10–25 minutes. Can take longer for large clusters or when AWS is under capacity pressure. Cannot be rolled back once started.
API compatibilityK8s API server is backward compatible within 1 version. Existing workloads using the old API version continue to work temporarily. But deprecated APIs removed in new version break immediately.
Node version skewAfter CP upgrade: nodes are 1 version behind (e.g., CP=1.30, nodes=1.29). This is supported and expected. Kubelet can be 1 minor version behind API server per K8s spec. But don't leave nodes behind for weeks.
# Trigger control plane upgrade aws eks update-cluster-version \ --name my-cluster \ --kubernetes-version 1.31 # Monitor upgrade progress aws eks describe-cluster \ --name my-cluster \ --query 'cluster.{status: status, version: version}' # Or watch in real-time watch -n 30 'aws eks describe-cluster \ --name my-cluster \ --query "cluster.status" --output text'
Control plane upgrade cannot be paused or rolled back. Once started, it runs to completion. If there's a problem, EKS support must intervene. This is why pre-flight is critical.
One minor version at a time. EKS strictly enforces this. You cannot upgrade from 1.28 to 1.30 in one step. Each upgrade is an explicit API call.
3
Addon Upgrades
// update after control plane · BEFORE node groups · version compatibility is critical
OrderCP upgrade → addon upgrades → node group upgrades. Some addons require the new K8s version's APIs. Some addons have versions only available for the new K8s version.
kube-proxy firstkube-proxy version must be ≤ API server version. After CP upgrade, kube-proxy on nodes is old version — technically fine (1 version skew) but upgrade it before nodes.
vpc-cni timingvpc-cni upgrade restarts aws-node DaemonSet on every node. Brief per-node network disruption. Do during low-traffic window. New pods may lose connectivity for ~1-2 seconds per node restart.
CoreDNS upgradeCoreDNS upgrade recreates pods. Rolling update by default. Brief window where DNS capacity is reduced. Pre-scale CoreDNS replicas before upgrading if DNS-heavy workload.
# Find latest addon version for new K8s version aws eks describe-addon-versions \ --kubernetes-version 1.31 \ --addon-name vpc-cni \ --query 'addons[].addonVersions[0].addonVersion' # Update all addons for addon in vpc-cni kube-proxy coredns aws-ebs-csi-driver; do LATEST=$(aws eks describe-addon-versions \ --kubernetes-version 1.31 --addon-name $addon \ --query 'addons[].addonVersions[0].addonVersion' --output text) aws eks update-addon \ --cluster-name prod \ --addon-name $addon \ --addon-version $LATEST \ --resolve-conflicts OVERWRITE echo "Updating $addon to $LATEST" done
Self-managed addons (Helm) are NOT upgraded by EKS. Cluster Autoscaler, AWS LB Controller, Karpenter, KEDA — you must upgrade these manually. CA version must match K8s minor version exactly.
4
Node Group Upgrades
// last step · rolling replacement · PDB-aware · 1 new node at a time
MNG rolling updateAWS launches new node (new AMI), cordons it, waits for pods to start scheduling on it, then drains old node, terminates it. Respects PodDisruptionBudgets.
Max unavailableDefault maxUnavailable: 1. Can set maxUnavailable: 33% for faster upgrade at cost of more disruption. For production: 1 at a time.
Duration~5-10 min per node (launch + drain). 10-node group ≈ 50-100 min. 100-node group ≈ 8-16 hours. Plan accordingly.
Drain behaviorEKS MNG drain respects PDBs and terminationGracePeriodSeconds. Stuck drain (PDB blocks) waits indefinitely. Set drain timeout with --nodegroup-drain-timeout.
# Update MNG to new K8s version aws eks update-nodegroup-version \ --cluster-name prod \ --nodegroup-name workers \ --kubernetes-version 1.31 # Monitor progress aws eks describe-nodegroup \ --cluster-name prod \ --nodegroup-name workers \ --query 'nodegroup.{status:status,health:health}' # Force update (skip drain) - USE WITH CAUTION aws eks update-nodegroup-version \ --cluster-name prod \ --nodegroup-name workers \ --kubernetes-version 1.31 \ --force
Single-replica deployments without PDB = downtime during drain. When the node running a single-replica pod is drained, the pod is killed and rescheduled — brief outage. Always run ≥2 replicas for production workloads. Add PDB with minAvailable=1.
StatefulSets on local storage = drain failure. Pods with local PVCs (hostPath, local PV) cannot move nodes. They block drain. Must manually delete the pod if the PVC is not needed on the new node.
⚠
What Can Fail — And Why
// real-world failure modes during K8s upgrades
🧨
Deprecated API Removal
App uses extensions/v1beta1 Ingress, or policy/v1beta1 PodSecurityPolicy, or batch/v1beta1 CronJob. These APIs are completely removed in new versions. Apps fail with 404 or connection errors on startup. Fix before upgrade with pluto scan.
💥
PDB Deadlock Blocks Drain
minAvailable: 2 on a Deployment with 2 replicas and both replicas on nodes being drained = drain hangs forever. EKS waits for PDB to allow eviction, which never happens. Fix: temporarily patch PDB, or scale up replicas first.
🔒
Webhook Incompatibility
Admission webhooks (OPA Gatekeeper, Kyverno, custom webhooks) may fail with new API versions. If the webhook server is down or returns errors, all API calls matching its rules fail. Set failurePolicy: Ignore for non-critical webhooks during upgrade.
📦
Incompatible Addon Versions
Cluster Autoscaler 1.28.x on K8s 1.30 causes scheduling failures. AWS LB Controller old version may fail to create ALBs due to API changes. AWS CNI old version may fail to assign pod IPs. Always upgrade self-managed addons alongside K8s.
🌐
CNI Disruption During aws-node Restart
Upgrading vpc-cni restarts aws-node. During restart, new pods on that node get NetworkPluginNotReady. Existing pod connections are unaffected. Brief ~30s window per node. If aws-node crashes post-upgrade due to bug: all new pod scheduling on that node fails.
🎫
IRSA Token Rotation
Control plane upgrade can trigger service account token rotation. Pods with old tokens may get 401 from AWS STS. Usually self-resolving (AWS SDK auto-refreshes). But pods that cache credentials aggressively may need restart after upgrade.
6
Fargate Profile Upgrades
// different from node groups · rolling pod replacement
No node upgradeFargate pods don't run on nodes you manage. AWS upgrades the underlying Fargate infrastructure. You upgrade by rolling restarting your pods.
MethodAfter CP upgrade: delete Fargate pods in rolling fashion. AWS schedules replacement pods on new Fargate infrastructure with new K8s version. Or use kubectl rollout restart deployment.
Wait periodAfter CP upgrade, wait for AWS to propagate new Fargate runtime. Then roll pods. If you roll immediately, new pods may still launch on old runtime (usually OK, brief transition).
# Roll all Fargate deployments after CP upgrade kubectl get deployments -A -o json | \ jq -r '.items[] | "\(.metadata.namespace) \(.metadata.name)"' | \ while read ns name; do kubectl rollout restart deployment/$name -n $ns kubectl rollout status deployment/$name -n $ns done
Fargate doesn't support DaemonSets. Any addon that uses a DaemonSet (vpc-cni, kube-proxy, node-local-dns) doesn't run on Fargate nodes. Fargate handles networking and DNS differently — the EKS control plane manages this. You can't install custom DaemonSets on Fargate.
7
Karpenter Node Upgrades
// drift detection → automatic node replacement
Drift featureWhen EC2NodeClass AMI selector is updated (or K8s version changes), Karpenter marks existing nodes as drifted. It then replaces them: launch new node with new AMI → drain old node → terminate.
Drift updateAfter CP upgrade, update NodePool's amiFamily or let EKS auto-detect that current AMI is outdated. Karpenter handles the rolling replacement automatically per its disruption budget.
Disruption budgetdisruption.budgets in NodePool controls how many nodes can be disrupted simultaneously. Default: 10%. For prod: set explicit budget like nodes: 1.
# After CP upgrade: trigger drift on Karpenter nodes # Method 1: Update EC2NodeClass to use new AMI alias kubectl patch ec2nodeclass default \ --type merge \ -p '{"spec":{"amiSelectorTerms":[{"alias":"al2023@latest"}]}}' # Karpenter automatically detects drift and replaces nodes # Watch replacement progress kubectl get nodes -w # Manual drift trigger (force all nodes to replace) kubectl annotate nodes -l karpenter.sh/nodepool=default \ karpenter.k8s.aws/ec2nodeclass-hash-version=force-refresh
Karpenter drift is the cleanest upgrade experience. No manual node rotation needed. Just update the AMI reference and Karpenter handles the rest. Set disruption budgets to control blast radius.

💥 Breaking Changes (Things That Will Break)
⚙️
cgroups v2 — The Biggest Change
Many tools assume cgroups v1 paths (/sys/fs/cgroup/cpu/). cgroups v2 unified hierarchy (/sys/fs/cgroup/) breaks: older Java apps, custom monitoring agents that read cgroup files, some Prometheus node-exporter configs. Check: stat -fc %T /sys/fs/cgroup
📜
UserData / Bootstrap Format Change
AL2 bootstrap: /etc/eks/bootstrap.sh --kubelet-extra-args "...". AL2023 uses nodeadm YAML via nodeConfig.yaml. Self-managed nodes with custom UserData must be rewritten. MNG handles this automatically if you just change amiType.
🔒
IMDSv2 Required by Default
AL2023 enforces IMDSv2. Apps using IMDS v1 API (curl http://169.254.169.254/... without token header) fail silently. Fix: update SDK (any modern AWS SDK handles IMDSv2 automatically). Or set hop limit to allow v1 (not recommended).
🧪
Package Differences
Init scripts or custom AMI modifications using yum install may work but dnf is preferred. Some package names differ between AL2 and AL2023 repos. Custom userdata that installs OS packages must be tested.
🗺️ Migration Strategy
MNG (easiest)Change amiType from AL2_x86_64 to AL2023_x86_64 in the node group config, then trigger a version update. AWS handles the rest — new nodes launch with AL2023.
Karpenter (elegant)Update EC2NodeClass amiSelectorTerms to alias: al2023@latest. Karpenter's drift detection automatically replaces nodes with AL2023 AMIs.
Blue/green (safest)Create new MNG with AL2023. Gradually migrate workloads using taints/tolerations or node selectors. Keep AL2 nodes until all workloads verified on AL2023.
TestingRun critical workloads on AL2023 nodes first using node selector. Observe for 1-2 weeks before full migration. Check cgroup paths, metric collection, anything that reads /sys/fs/cgroup.

✅ Pre-Migration Checklist
🔍
Audit cgroups v1 dependencies. Search for hardcoded paths: grep -r "/sys/fs/cgroup/cpu" / in any custom scripts or operator code.
☕
Test JVM apps. Java 8 < 8u191 doesn't understand cgroups v2. Use modern JVM (Java 11+, Java 17+). Check -XX:+UseContainerSupport flag.
📊
Check monitoring agents. Datadog, Prometheus node-exporter, custom collectors that read cgroup metrics. Test on AL2023 in non-prod first.
📜
Review all UserData scripts. Any yum install commands, custom bootstrap hooks. AL2023 nodeadm format is different from AL2 bootstrap.sh.
🔑
Check IMDS usage. Any app using EC2 metadata service directly. AWS SDKs ≥ 2018 support IMDSv2. Older SDKs or curl-based IMDS calls need updating.
🐳

🔌

EKS Traps & Interview Questions
// operational gotchas that bite everyone
01
Access
Created EKS with Terraform role, now kubectl returns "Unauthorized"
The Terraform role is cluster admin, not your personal IAM user. Assume that role or add your principal to aws-auth/Access Entries. Use aws eks update-kubeconfig --role-arn arn:...terraform-role to use the right identity.
02
Upgrade
Can you upgrade EKS from 1.27 to 1.30 in one step?
No. EKS enforces one minor version at a time: 1.27 → 1.28 → 1.29 → 1.30. Each is a separate API call. Plan for 3 maintenance windows or a long single window with all 3 upgrades sequentially.
03
Networking
Pods can't communicate cross-AZ. What's wrong?
Security group rules. EKS VPC CNI uses pod IPs from the VPC CIDR. Cross-AZ traffic goes through routing — SG rules on the ENI must allow the pod CIDR range. Also check NACLs. Also: check if you accidentally have a NetworkPolicy blocking it.
04
IRSA
IRSA worked, then you renamed the Kubernetes namespace. Now it breaks.
IRSA trust policy includes exact namespace name. The sub condition is system:serviceaccount:{namespace}:{sa-name}. Update the IAM role trust policy with the new namespace name. Consider using Pod Identity which doesn't have this limitation.
05
Scaling
Pods are Pending but Cluster Autoscaler isn't scaling up. Why?
Multiple possible causes: (1) Pod has node affinity/selector that no ASG can satisfy. (2) Pod requests exceed any available instance type. (3) ASG already at max capacity. (4) CA doesn't have IAM permissions to call ASG API. (5) Pod is in a namespace CA is ignoring. Check CA logs: kubectl logs -n kube-system -l app=cluster-autoscaler.
06
Storage
PVC stuck in Pending after EKS 1.23 upgrade
In-tree EBS provisioner was removed in 1.23. You must install EBS CSI driver. Existing PVs still work (migrated). New PVCs with storageClass: gp2 fail until CSI driver is installed. Add aws-ebs-csi-driver managed addon before upgrading to 1.23.
07
Nodes
New nodes join cluster but not accepting pods. Status: "Not Ready"
Common causes: (1) aws-node (VPC CNI) not running on node — SG blocks port 443 to API server. (2) Node can't pull ECR images (ECR endpoint blocked, missing IAM). (3) Node IAM role not in aws-auth. (4) kubelet can't reach API server (private endpoint without route). Check: kubectl describe node {nodename}.
08
Drain
Node drain hangs indefinitely during upgrade
PodDisruptionBudget blocking eviction. Find the blocking pod: kubectl get events --field-selector reason=EvictionThrottled. Fix: scale up the Deployment temporarily (add 1 replica), let PDB allow eviction, then scale back. Or patch PDB maxUnavailable: 1 temporarily.
09
Fargate
DaemonSet pods don't run on Fargate nodes
By design. Fargate doesn't support DaemonSets. Each Fargate pod is a dedicated "micro-VM" — there's no node to run a DaemonSet on. CNI, kube-proxy, and log forwarding are handled by EKS control plane for Fargate. Use sidecar containers instead of DaemonSets on Fargate.
10
AL2023
Old Java app OOMKilled on AL2023 nodes but was fine on AL2
cgroups v2 changes JVM memory reporting. JVM < Java 8u191 doesn't understand cgroups v2 and thinks it has access to all node memory (no container limit), then gets OOMKilled. Fix: upgrade to Java 11+, or set -XX:MaxRAMPercentage=75 explicitly, or add -XX:+UseContainerSupport.
11
Karpenter
Karpenter over-provisions: launches 10 nodes for 10 pending pods
Consolidation not enabled or nodes are "tainted" as non-empty. Enable: consolidationPolicy: WhenUnderutilized. Also: Karpenter waits before consolidating (default: 10 empty seconds + underutilized for 1 min). Set consolidateAfter: 30s for faster consolidation in non-prod.
12
Addons
Managed addon stuck in "DEGRADED" state after upgrade
You modified addon-managed resources directly. EKS detected drift between desired addon config and actual resource state. Fix: aws eks update-addon --resolve-conflicts OVERWRITE to let EKS restore to its desired state. Or use PRESERVE to keep your changes (but new defaults won't apply).
