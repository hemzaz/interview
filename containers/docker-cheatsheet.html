📦 Image vs Container
ImageAn immutable, ordered stack of read-only layers plus a JSON config (env, entrypoint, exposed ports). It's a template — never runs by itself.
ContainerA running (or stopped) instance = the image's layers mounted read-only plus one thin writable layer on top, plus isolated namespaces/cgroups.
Many→oneN containers can run from the same image simultaneously, each with its own independent writable layer — the read-only layers are shared, not copied.
🔀 OverlayFS & Copy-on-Write
lowerdirThe stacked read-only image layers (can be many, merged in order).
upperdirThe container's writable layer — new/changed files land here.
mergedThe unified view the container's process actually sees as its root filesystem.
copy_upEditing a file that only exists in a lower (read-only) layer triggers OverlayFS to copy it up to upperdir first, then apply the write there. The original layer is never touched.
Deletion in an overlay: "deleting" a file that lives in a lower layer doesn't remove it from that layer — OverlayFS writes a whiteout marker in upperdir that hides it in the merged view. The bytes still physically exist in the lower, read-only layer.
🎯 Layer Caching & Cache Invalidation
Per instructionEach RUN/COPY/ADD instruction produces (or reuses) one layer. FROM, ENV, WORKDIR, ARG etc. only touch metadata — cheap.
Cache keyFor RUN: the instruction text itself. For COPY/ADD: a checksum of the copied file(s) content and metadata — not just the instruction string.
CascadeThe moment one layer misses cache, every layer after it also rebuilds — even if their own instructions and inputs are unchanged. Cache reuse is strictly a prefix match down the instruction list.
Ordering rulePut what changes least often first (base image, OS packages), what changes most often last (application source). Copy dependency manifests (package.json, go.mod, requirements.txt) and install before copying the rest of the source.
# good ordering — app code churn doesn't bust the dependency-install cache FROM node:22-slim WORKDIR /app COPY package.json package-lock.json ./ RUN npm ci --omit=dev # cached until package*.json changes COPY . . # app source — changes every commit CMD ["node", "server.js"]
Interview trap — cache busting: COPY . . placed before the dependency install means any source file edit invalidates the install layer too, forcing a full npm ci/pip install on every rebuild. This is the single most common Dockerfile performance mistake.
📝Dockerfile Masteryinstructions and their semantics
InstructionPurpose & key semantics
FROMBase image for the stage; may appear multiple times to start new build stages (multi-stage builds).
RUNExecutes a command and commits the filesystem result as a new layer. Shell form RUN cmd runs via /bin/sh -c. Exec form RUN ["cmd","arg"] runs the binary directly, no shell.
COPYCopies files/dirs from the build context (or another stage via --from) into the image. Simple, explicit, cacheable by content checksum.
ADDSuperset of COPY: can also fetch a remote URL and auto-extracts local .tar/.tar.gz archives into the destination. Prefer COPY — ADD's implicit extraction and remote-fetch behavior is a common source of surprises and (historically) unverified downloads.
WORKDIRSets (and creates if missing) the working directory for all following instructions and for the container's default process.
ENVSets an environment variable that is baked into the image and visible at runtime (docker inspect, the running process's env). Persists.
ARGBuild-time-only variable, supplied via --build-arg. Not present in the final image's runtime environment. Scope: an ARG declared before the first FROM only applies to the FROM line unless re-declared after it.
ENTRYPOINTThe fixed executable for the container. Exec form recommended (see callout below).
CMDDefault arguments (if ENTRYPOINT is set, exec form) or the default command (if no ENTRYPOINT). Overridden wholesale by args passed to docker run.
EXPOSEDocumentation only — declares which port(s) the app listens on. Does not publish anything; you still need -p/-P at runtime.
USERSets the UID/GID (or username) subsequent instructions and the container's process run as. Use a non-root numeric UID for production images.
HEALTHCHECKDefines a command Docker runs periodically to mark the container healthy/unhealthy/starting. Flags: --interval, --timeout, --start-period, --retries. HEALTHCHECK NONE disables an inherited one.
ONBUILDRegisters a trigger instruction that fires only when another Dockerfile uses this image in its own FROM. Convenient for base/framework images, but hides build steps from whoever reads the downstream Dockerfile — used sparingly.
.dockerignoreExcludes paths from the build context sent to the daemon/BuildKit — smaller/faster builds and prevents accidentally COPY-ing .git, node_modules, or secret files.
⚔️ ENTRYPOINT vs CMD — how they combine
ENTRYPOINTCMDEffective command
none["nginx","-g","daemon off;"]CMD runs as-is: nginx -g "daemon off;"
["nginx"] (exec)["-g","daemon off;"]CMD elements become args to ENTRYPOINT
nginx (shell)anythingCMD is ignored entirely — shell form ENTRYPOINT takes no CMD args
["nginx"] (exec)noneRuns nginx with no args
docker run img foo bar replaces CMD with foo bar, it never touches ENTRYPOINT — that's what --entrypoint is for.
🧨 Exec form vs shell form — why it matters
Shell formCMD myapp → really /bin/sh -c "myapp". The shell becomes PID 1, your app is a child. Signals sent to the container (e.g. SIGTERM from docker stop) go to the shell, which often does not forward them.
Exec formCMD ["myapp"] runs the binary directly as PID 1 — signals reach it immediately, no shell in between. No shell also means no env-var expansion/globbing/pipes unless you invoke a shell explicitly.
Fix for scriptsIf you must use a shell entrypoint script, end it with exec "$@" so the shell process is replaced by your app instead of staying as its parent.
Interview trap — PID 1 & zombie reaping: your app is PID 1 inside the container's PID namespace. Most language runtimes don't install default handlers for SIGTERM/SIGINT when running as PID 1, so docker stop waits out the full grace period then SIGKILLs. PID 1 is also responsible for reaping zombie processes — if your app spawns children and doesn't wait() on them, they pile up as zombies. Fix: run with docker run --init (injects tini as a tiny real PID 1 that forwards signals and reaps zombies), or use an exec-form entrypoint / exec "$@" in shell scripts.
🏗️Multi-Stage Buildsbuild tools stay out of the final image
🧩 The pattern
FROM golang:1.23 AS build WORKDIR /src COPY go.mod go.sum ./ RUN go mod download COPY . . RUN CGO_ENABLED=0 go build -o /out/app . FROM gcr.io/distroless/static AS runtime COPY --from=build /out/app /app USER nonroot:nonroot ENTRYPOINT ["/app"]
--targetdocker build --target build -t app:debug . stops the build at a named stage — handy for a debug image with the full toolchain, or CI test stages.
--fromCOPY --from= can name a previous stage, a stage index, or any external image: COPY --from=nginx:1.27 /etc/nginx/nginx.conf ./nginx.conf.
ResultThe final image contains only what's explicitly COPY --from'd in — no compiler, no source, no package manager, drastically smaller and smaller attack surface.
⚖️ Base image tradeoffs
BaseTradeoff
scratchTruly empty — no shell, no libc, no CA certs. Only works for statically-linked binaries. Smallest possible, zero debuggability.
distrolessLanguage runtime + libc + CA certs/tzdata, no shell, no package manager. Small attack surface, still runnable/debuggable via :debug tag variants (adds busybox).
alpine~5MB, full package manager (apk) and busybox shell. Uses musl libc, not glibc — see trap below.
-slim / debianglibc-based, small-ish, has a real shell + apt. Best compatibility with prebuilt binaries/wheels, larger than alpine.
Interview trap — musl vs glibc: Alpine's musl libc is not binary-compatible with glibc. Pre-compiled Python wheels (manylinux), Node native addons, and other binaries built against glibc can silently fail, hang, or behave differently (DNS resolution, locale, threading) under Alpine. "It works on debian-slim but breaks on alpine" is almost always this.
🔒Image Size & Securitythe classic secrets-in-layers trap
🚨 Secrets baked into layers
# WRONG — secret is recoverable from the image COPY id_rsa /root/.ssh/id_rsa RUN git clone git@repo && \ rm /root/.ssh/id_rsa # too late!
Layers are immutable diffs. The COPY layer still contains id_rsa in its own diff — deleting it in a later layer only hides it in the merged view via a whiteout. Anyone with the image can extract it: docker save img | tar -xO ... or simply docker history --no-trunc.
# RIGHT — BuildKit secret mount, never touches any layer RUN --mount=type=secret,id=ssh_key \ GIT_SSH_COMMAND="ssh -i /run/secrets/ssh_key" \ git clone git@repo # build with: docker build --secret id=ssh_key,src=$HOME/.ssh/id_rsa .
🛡️ Hardening checklist
Fewer layersChain related steps in one RUN with && so cleanup (rm -rf /var/lib/apt/lists/*) happens in the same layer as the install — cleaning in a later layer doesn't shrink the image.
Pin versionsPin the base image tag (ideally by digest) and package versions. Avoid floating tags in anything that ships.
Drop rootUSER 10001:10001 or a named non-root user; pairs with Kubernetes runAsNonRoot.
Cache mountsRUN --mount=type=cache,target=/root/.cache/pip persists package-manager caches across builds without ever landing in an image layer.
Multi-archdocker buildx build --platform linux/amd64,linux/arm64 -t repo:tag --push . — produces one manifest list tag resolving per-CPU-arch.
BuildKit has been the default builder since Docker 23.0 (previously opt-in via DOCKER_BUILDKIT=1). It parallelizes independent stages, understands --mount=type=secret / type=ssh / type=cache, and can emit build provenance and SBOM attestations.
🗂️Registries: Tags, Digests & Sharingthe mutable-tag trap
🏷️ Tags
A tag (myapp:1.4) is a mutable pointer, like a git branch — it can be reassigned to a different image digest at any time by anyone who can push. Pulling by tag today may not give you the same bits tomorrow.
🔑 Digests
myapp@sha256:abc123... is content-addressed and immutable — the digest is a hash of the manifest. Pin production deployments by digest for true reproducibility.
♻️ Layer sharing
Layers are stored and transferred by digest. If two images share an identical base layer, the registry and the local daemon store/pull it once.
Interview trap — :latest: it is not "the newest version" — it's simply the tag Docker applies by default when none is specified. It's just as mutable as any other tag, gives you no reproducibility guarantee, and is generally avoided in production manifests/Dockerfiles in favor of an explicit version (and ideally a pinned digest).
⚙️Container Runtime LandscapeDocker Engine · containerd · CRI-O · runc
🔽 The stack, top to bottom
1
Docker Engine — CLI + dockerd daemon + build (BuildKit), the full developer-facing platform. Internally delegates to containerd.
2
containerd — CNCF-graduated, CRI-compliant high-level runtime. Manages image pull/storage, container lifecycle, snapshots. Used directly by many Kubernetes nodes (no Docker Engine needed).
3
CRI-O — a lightweight, Kubernetes-only alternative to containerd: implements just enough CRI to run pods, nothing Docker-specific.
4
runc — the low-level OCI Runtime Spec reference implementation. Actually creates the namespaces/cgroups and execs the container process. (Alternatives: crun, gVisor/runsc, Kata Containers for VM-level isolation.)
📜 The specs that make this interoperable
OCI Image SpecDefines the image manifest, layer (tarball diff) format, and config JSON — any OCI-compliant tool can build/read images any other one produced.
OCI Runtime SpecDefines how to run a "filesystem bundle" (rootfs + config.json) as a container — what runc/crun/etc. implement.
CRIKubernetes' gRPC Container Runtime Interface — lets kubelet talk to any compliant runtime (containerd, CRI-O) without Docker-specific glue code.
dockershim removedKubernetes v1.24 removed the in-tree dockershim that translated CRI calls into Docker Engine API calls. Docker Engine itself doesn't speak CRI. Images built by Docker are unaffected (still OCI-compliant) — what broke was kubelet talking directly to dockerd; the community-maintained cri-dockerd adapter restores that path if needed.
Q: If Kubernetes dropped Docker support, do Docker-built images stop working on Kubernetes? A: No. Docker images are OCI images; containerd/CRI-O run them directly. Only the direct kubelet↔dockerd integration path was removed, not compatibility with Docker-produced images.
🌐Networking & Storagebridge/host/none · volumes/binds/tmpfs
🔌 Network drivers
DriverBehavior
bridge (default)Private virtual network (docker0); container gets its own IP; outbound via NAT; needs -p host:container to be reachable from outside.
hostShares the host's network namespace directly — no isolation, no port mapping, best raw performance. Avoid for untrusted workloads.
noneNo networking beyond loopback.
💾 Persistence options
TypeBehavior
volumeDocker-managed storage (under /var/lib/docker/volumes), lifecycle independent of any single container, portable, best practice for real persistent data.
bind mountMaps an arbitrary host path into the container. Host-dependent, great for local dev live-reload, riskier if the path is too broad.
tmpfsIn-memory only (Linux); never touches disk; wiped when the container stops — good for short-lived secrets.
🔍Scanning, SBOM & Signingsupply-chain concepts
🩺 Vulnerability scanning
Trivy / Grype scan every layer's OS packages and language dependencies against CVE databases, typically gating CI on high/critical findings before an image can be pushed or deployed.
📋 SBOM
A Software Bill of Materials lists exactly what's inside an image — packages, versions, licenses — in formats like SPDX or CycloneDX. BuildKit can emit an SBOM as a build attestation alongside the image.
✍️ Signing
cosign (Sigstore) signs an image's digest (not its mutable tag) and stores the signature as an OCI artifact next to the image, enabling admission-time verification that an image came from a trusted build and wasn't tampered with.
💬Rapid-Fire Q&Acommon follow-ups
Q: Container vs VM, in one sentence? A: A container shares the host kernel and is isolated by namespaces + cgroups; a VM virtualizes hardware and runs its own full guest kernel — heavier, stronger isolation boundary.
Q: Does `docker exec` work on a `scratch`-based image? A: Only if there's a shell in the image to exec into — `scratch` has none, so `docker exec sh` fails; you'd need a debug-variant image or a sidecar/ephemeral debug container instead.
Q: Why prefer distroless/minimal images over a full OS base, security-wise? A: Fewer installed packages means fewer CVEs to track, and no shell/package manager for an attacker to pivot with after a compromise.
Q: What does `docker commit` actually create? A: It snapshots a running container's current writable layer into a new read-only image layer on top of its original image — useful for debugging, not a build practice (no Dockerfile provenance).
Q: Why does ADD's URL-fetch feature get flagged in reviews? A: It downloads arbitrary remote content into the image at build time with no built-in integrity check (unless using newer `--checksum` syntax) — note a fetched URL is not auto-extracted, only local tar archives are — and isn't cached the same predictable way as COPY.
