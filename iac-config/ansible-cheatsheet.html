Architecture & Execution Model
// how ansible actually runs: control node → temporary python on managed node
⚙️ How Ansible Executes — Under the Hood
No agentTarget nodes need only Python 3 (or 2.7) and SSH access. No daemon running on managed nodes. Ansible connects, drops a temp Python file, runs it, deletes it.
Execution steps1. Parse inventory → 2. Gather facts (unless gather_facts: false) → 3. For each task: assemble module + args → SSH → copy to /tmp → execute → parse JSON result → delete temp file.
Forksforks=5 default = 5 hosts in parallel per task. One task completes on all hosts before next task starts (not task-per-host streaming). Set -f 20 for larger clusters.
serialserial: 1 or serial: "30%" in play: rolling update mode. Runs ALL tasks on N hosts before moving to next batch. Overrides forks for rolling deployments.
forks ≠ serial. forks = parallelism within a batch (tasks per host run in parallel). serial = batch size for rolling updates (how many hosts get the full play at once). They compose: serial: 2, forks: 20 = 2 hosts per batch, but those 2 run in parallel.
🔄 Idempotency Model
What it meansRunning a playbook N times has the same result as running it once. Modules check current state before acting.
changed vs okchanged = module made a change. ok = desired state already present, nothing done. Both are success.
Not automaticIdempotency is per-module, not automatic. command and shell modules always report changed unless you add changed_when / creates.
shell and command are never idempotent by default. They run the command every time and always show changed. Use creates: / removes: or changed_when: false to make them idempotent-aware.
📡 Connection Types
ssh (default)Uses OpenSSH. ControlMaster multiplexing enabled by default (ssh_args). Supports become for privilege escalation.
localRuns on control node itself. No SSH. Used for localhost tasks, API calls, local config generation.
winrmWindows targets via WinRM (HTTP/HTTPS). Requires pywinrm on control node. HTTP Basic/Kerberos/NTLM auth.
docker/kubectlRun tasks inside containers. connection: community.docker.docker / community.kubernetes.kubectl.

Inventory
// static · dynamic · group_vars · host_vars · precedence
📁 Inventory Formats
# ── INI format ────────────────── [webservers] web1.prod ansible_host=10.0.1.1 web2.prod ansible_host=10.0.1.2 [dbservers] db1.prod ansible_host=10.0.2.1 ansible_port=2222 [prod:children] # group of groups webservers dbservers [prod:vars] # group variables ansible_user=ubuntu ansible_python_interpreter=/usr/bin/python3 # ── YAML format ────────────────── all: children: webservers: hosts: web1.prod: ansible_host: 10.0.1.1 nginx_port: 443
🔄 Dynamic Inventory
What it isAn executable script (Python, Bash) or plugin that returns JSON. Called at playbook run time. AWS, GCP, Azure all have official plugins.
Plugin vs scriptInventory plugins (new way) — configure in inventory.yml with plugin: aws_ec2. Inventory scripts (legacy) — executable that accepts --list / --host flags.
CachingDynamic inventory is re-fetched on every run by default. Enable caching: cache: true, cache_plugin: jsonfile, cache_timeout: 3600.
# inventory.aws.yml plugin: amazon.aws.aws_ec2 regions: [us-east-1] filters: tag:Env: prod keyed_groups: - key: tags.Role prefix: role
Dynamic inventory groups are not static. A host's groups can change between runs. Don't hardcode group membership — use when: inventory_hostname in groups['webservers'].
📂 group_vars / host_vars
Directory layoutgroup_vars/all.yml — applies to all hosts. group_vars/webservers.yml — applies to webservers group. host_vars/web1.prod.yml — applies to one host.
Subdirectoriesgroup_vars/webservers/ directory works too — all .yml files inside are merged. Good for splitting large var sets.
Auto-loadedAnsible automatically loads group_vars/ and host_vars/ relative to the playbook file AND the inventory file location. Both are searched.
group_vars/all has the lowest precedence. host_vars overrides group_vars. More specific always wins — but see variable precedence (22 levels!) for the full ordering.
group_vars are loaded by group name, not inventory file name. Even if inventory is production.ini, the vars file must be group_vars/webservers.yml, not group_vars/production_webservers.yml.

role vars/main.yml (14) beats group_vars (6-7). Most people expect group_vars to override role vars. Wrong — role vars are much higher precedence. Use defaults/main.yml (level 2) for role defaults that users should be able to override.

⚠️ Variable Scope Traps
set_fact scopeset_fact sets a host-level variable that persists for the rest of the play (and subsequent plays in the same run). It does NOT persist across separate playbook runs.
register scoperegister sets a task-level variable scoped to the current host. Accessible in subsequent tasks for that host. result.stdout, result.rc, etc.
loop var scopeVariables set inside a loop are scoped to that task iteration. Can leak into when conditions on subsequent tasks — use loop_control.loop_var to rename if conflicting.
include_tasks variables don't bubble up. Variables set inside an included task file with vars: on the include_tasks directive are available inside that file but not in the parent. Use set_fact to return values.

Playbook Anatomy
// plays · tasks · blocks · loops · conditionals
📄 Full Playbook Structure
--- # Each element in the list is a "play" - name: Deploy web application hosts: webservers # inventory group become: true # sudo for all tasks become_user: root gather_facts: true # runs setup module first any_errors_fatal: false # don't stop all hosts on one failure max_fail_percentage: 20 # allow up to 20% host failures serial: "25%" # rolling: 25% of hosts per batch vars: app_version: "1.2.3" vars_files: - vars/secrets.yml pre_tasks: # run before roles - name: Drain from LB uri: { url: "http://lb/drain" } roles: # roles run here - role: nginx vars: { nginx_port: 443 } tasks: # after roles - name: Copy config template: src: nginx.conf.j2 dest: /etc/nginx/nginx.conf notify: Reload nginx # trigger handler post_tasks: # run after roles+tasks - name: Re-add to LB uri: { url: "http://lb/add" } handlers: # triggered by notify - name: Reload nginx service: { name: nginx, state: reloaded }
🧱 Blocks — Error Handling
- block: # try - name: Deploy app command: /deploy.sh - name: Verify health uri: { url: "http://localhost/health" } rescue: # catch on failure - name: Rollback command: /rollback.sh always: # finally - name: Send notification uri: { url: "http://slack/notify" }
rescue: runs only if block fails. always: runs regardless. Both rescue and always run on ALL hosts, not just the failed ones within the block.
🔁 Loops
# loop (modern, replaces with_items) - name: Install packages apt: { name: "{{ item }}", state: present } loop: [nginx, curl, git] # loop with dicts - user: name: "{{ item.name }}" groups: "{{ item.groups }}" loop: - { name: alice, groups: sudo } - { name: bob, groups: dev } # loop_control for label and index loop_control: label: "{{ item.name }}" # cleaner output index_var: idx # loop index loop_var: pkg # rename 'item'
loop with the package module is inefficient. Don't loop over package/apt/yum — pass a list directly to name: instead. One task for all packages vs N tasks for N packages.

Handlers
// notify · flush_handlers · listen · ordering traps
🔔 Handler Mechanics — Deep Dive
When they runHandlers run ONCE at the end of each play, not immediately when notified. Multiple tasks notifying the same handler result in it running only once.
DeduplicationIf 10 tasks all notify "Reload nginx", nginx reloads exactly once — at the end of the play. Tracked by handler name (string match).
OrderingHandlers run in the order they are defined, not the order they were notified. If handler B is defined before handler A, B runs first even if A was notified first.
Only on changedHandlers are only notified if the task that notified them reported changed. If a task is skipped, fails, or reports ok, the handler is NOT notified.
flush_handlersmeta: flush_handlers forces all pending handlers to run immediately at that point. Use when subsequent tasks depend on handler effects (e.g., restart service before verifying it).
listenHandlers can listen to a topic name. Multiple handlers can respond to one notify. Decouples task-handler coupling in roles.
# Multiple handlers sharing a topic handlers: - name: Restart nginx service: { name: nginx, state: restarted } listen: "web config changed" - name: Clear CDN cache uri: { url: "http://cdn/purge" } listen: "web config changed" tasks: - template: ... notify: "web config changed" # triggers BOTH
Handler on failed play does NOT run. If the play fails mid-way, pending handlers are abandoned. Use --force-handlers flag or move critical cleanup to always: block instead.
Handler names are global within a play. If a role defines a handler named "Restart nginx" and your playbook also defines one, they conflict. Prefix role handler names: myrole : Restart nginx.
🗂️ Execution Order in a Play
1. gather_facts (setup module)
↓ runs on all hosts in parallel (forks)
2. pre_tasks
↓ handlers flushed after pre_tasks
3. roles (in order listed)
↓ role dependencies run first
4. tasks
↓ roles + tasks share handler queue
5. handlers (notified from 2-4)
↓
6. post_tasks
↓ handlers flushed after post_tasks
7. handlers (notified from 6)
💡 Common Handler Patterns
Restart vs reloadUse reloaded state (SIGHUP) for config changes — zero downtime. Use restarted when process must fully restart (binary upgrade).
Conditional handlerAdd when: to handlers — they still only run if notified AND when condition passes.
Notify multiplenotify can be a list: notify: [Handler A, Handler B]. Both are added to the pending queue.

Roles & Collections
// directory structure · dependencies · include_role vs import_role
📂 Role Directory Structure
roles/nginx/ ├── defaults/ │ └── main.yml # lowest precedence vars ├── vars/ │ └── main.yml # high precedence vars ├── tasks/ │ └── main.yml # entry point ├── handlers/ │ └── main.yml ├── templates/ │ └── nginx.conf.j2 # Jinja2 templates ├── files/ │ └── ssl.crt # static files ├── meta/ │ └── main.yml # dependencies, galaxy info └── README.md
defaults/Intended to be overridden. Level 2 — lowest after command line. Role users set their own values in group_vars.
vars/Internal role vars. Level 14 — beats group_vars. Not intended for override. Constants the role needs.
meta/main.ymlDeclares role dependencies. Dependencies run before the role. Galaxy metadata (author, license, platforms).
tasks/main.ymlEntry point. Can import_tasks or include_tasks other files for organisation.
# meta/main.yml dependencies dependencies: - role: common - role: geerlingguy.mysql vars: mysql_root_password: "{{ vault_mysql_pass }}"
Role dependencies run multiple times if multiple roles depend on them. Use allow_duplicates: false in meta/main.yml to prevent a dependency from running more than once per play.

Ansible Vault
// encrypt vars · files · strings · multi-vault IDs
🔐 Vault Commands
# Encrypt an entire file ansible-vault encrypt secrets.yml ansible-vault encrypt_string 'mysecret' --name 'db_password' # Edit in place ansible-vault edit secrets.yml # Run playbook with vault ansible-playbook site.yml --ask-vault-pass ansible-playbook site.yml --vault-password-file .vault_pass # Multiple vault IDs (different passwords) ansible-vault encrypt secrets.yml --vault-id prod@prompt ansible-playbook site.yml \ --vault-id prod@.vault_prod_pass \ --vault-id dev@.vault_dev_pass # Rekey (change password) ansible-vault rekey secrets.yml \ --vault-id old@prompt \ --new-vault-id new@prompt
Vault password file path in ansible.cfg: vault_password_file = .vault_pass. Never commit the vault password file — add to .gitignore. The encrypted file CAN be committed safely.
📝 Vault in Variables
# Encrypt just the value, not the whole file # Run: ansible-vault encrypt_string 'secret123' db_password: !vault | $ANSIBLE_VAULT;1.1;AES256 38373932386435656337396166613131 ... # Recommended: separate vault file pattern # group_vars/all/vars.yml (unencrypted) db_password: "{{ vault_db_password }}" # group_vars/all/vault.yml (encrypted) vault_db_password: supersecret123
The vault_prefix pattern lets you grep for all vault vars (grep vault_), see variable names in plain text, and only encrypt the vault file — not the whole vars file. Best practice for large teams.
⚠️ Vault Traps
No runtime encryptVault only protects files at rest. Once decrypted during playbook run, values are in memory/logs. Use no_log: true on tasks handling secrets.
no_log caveatno_log: true suppresses task output but the value can still appear in debug messages, registered vars printed later, or exception tracebacks.
Git historyIf you accidentally commit plaintext secrets, encrypting later doesn't remove them from git history. Rotate the secrets.
vault ID mismatchIf file encrypted with vault ID prod but you pass dev password file, Ansible tries all provided vault passwords. If none match: fatal error.
ansible-vault view vs decrypt. view = temporary display, never writes. decrypt = writes plaintext to disk permanently. Never use decrypt on production files in shared repos.

Critical Trap Reference
// 18 traps every ansible engineer must know
01
Execution Order
Do handlers run immediately when notified?
No. Handlers run at the end of the play (or when meta: flush_handlers is called). Multiple notifications = single execution. If the play fails, pending handlers are skipped.
02
Idempotency
Is shell: rm -rf /tmp/old idempotent?
No. shell and command always show changed. Add removes: /tmp/old or changed_when: false (or use file: state=absent).
03
Variable Precedence
Does group_vars override vars/main.yml in a role?
No. Role vars/main.yml is precedence level 14; group_vars is 6-7. Role vars beat group_vars. Use defaults/main.yml (level 2) for overridable role defaults.
04
forks vs serial
Does forks: 10 mean 10 hosts get all tasks at once?
Not exactly. Forks = parallelism per-task across hosts. One task runs on 10 hosts simultaneously. The NEXT task starts only after the current task finishes on all hosts in the batch.
05
import vs include
Can you use loop with import_tasks?
No. import_* is static (parsed before run). loop is dynamic. Use include_tasks with loop. Same for import_role — cannot loop over it.
06
Tags
If a task has a tag, does ansible-playbook --tags mytag run pre_tasks?
No (by default). --tags skips untagged tasks including pre_tasks/post_tasks unless tagged always. Add tags: always to pre_tasks that must run even with tag filtering.
07
Vault
Does no_log: true fully protect secrets?
No. no_log suppresses task output but secrets can appear in subsequent debug tasks that print registered vars, in ARA/Tower job logs, or exception tracebacks.
08
Facts
Is ansible_hostname the same as inventory_hostname?
No. inventory_hostname = name in your inventory file. ansible_hostname = OS hostname. If inventory uses IPs or aliases, they differ. Use inventory_hostname for Ansible logic, ansible_facts.hostname for OS-level config.
09
Become
Does become: true on a play apply to all tasks including loops?
Yes, but: become_user defaults to root. If a specific task needs a different user, set become_user on that task. Task-level become: false overrides play-level.
10
Check Mode
Does --check (dry run) work with command/shell modules?
No. command and shell don't support check mode — they're skipped with a warning. This means check mode output for plays using shell commands is misleading. Use check_mode: false on a task to force it to run even in check mode.
11
Error Handling
Does a failed task on host A stop execution on host B?
No (by default). Failed hosts are removed from the active host list and execution continues on remaining hosts. Use any_errors_fatal: true to abort the entire play on any single failure.
12
register
If a task with register is skipped, is the var undefined?
No. Skipped tasks still register a result, but result.skipped is true and result.rc is absent. Always check result is not skipped before accessing task-specific keys.
13
Templates
Does template module always report changed?
No. Template compares rendered output to the existing file. If content is identical, it reports ok. Change in variables that don't affect the rendered output = no change. Handler not notified.
14
Delegation
When using delegate_to, which host's vars are used?
The original host's vars. delegate_to changes WHERE the task runs, not WHICH host's variables are used. Use hostvars['delegated-host']['var'] to access the delegate's vars. Add delegate_facts: true to assign facts to the delegate.
15
Loops
Is looping over apt to install 10 packages efficient?
No. Looping calls apt 10 times (10 separate package manager invocations). Pass a list to name: directly: name: [nginx, curl, git]. One task, one apt call.
16
Galaxy
Does ansible-galaxy role install pin versions by default?
No. Without a version, it installs the latest. Unpinned installs break CI when role authors push breaking changes. Always specify versions in requirements.yml.
17
SSH
Does Ansible require Python on the control node AND managed nodes?
Both — but differently. Control node needs Python (ansible itself). Managed nodes need Python for most modules. Exception: raw module uses only SSH/bash — used to bootstrap Python itself on bare systems.
18
Conditionals
Is when: myvar == true the same as when: myvar?
Not always. If myvar is the string "true" (not bool), when: myvar == true = False (type mismatch). Use when: myvar | bool for string booleans, or compare strings explicitly: when: myvar == "true".
