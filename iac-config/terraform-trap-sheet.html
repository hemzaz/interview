State
11 TRAPS
CRITICAL
ST-01Does terraform destroy destroy resources not in the state file?
✗ WRONGYes — it destroys everything defined in your .tf files.
✓ RIGHTNo. Terraform only knows about resources tracked in state. Resources created outside Terraform (console, CLI, other tools) are invisible and will never be touched — until you import them.
This is the fundamental trap. Terraform's world = state file. Reality can diverge silently. terraform plan -refresh-only detects drift without changing anything.
CRITICAL
ST-02Is it safe to manually edit the terraform.tfstate file to fix issues?
✗ WRONGYes — the state is just JSON, edit it carefully.
✓ RIGHTUse terraform state subcommands. Direct edits corrupt the serial number and MD5, break state locking semantics, and bypass validation. Use state mv, state rm, state replace-provider.
If you must edit raw state: terraform state pull > backup.tfstate first. Never edit the remote state file directly — it bypasses the lock. Only use terraform state push to restore.
CRITICAL
ST-03Does storing state in S3 automatically provide state locking?
✗ WRONGYes — S3 backend handles locking natively with zero configuration.
✓ RIGHTNo, locking must be enabled — but as of Terraform 1.11 (Feb 2025), you no longer need DynamoDB. Set use_lockfile = true in the S3 backend block to get native locking via S3 conditional writes. The older dynamodb_table param still works but is now deprecated/legacy, not required for new configs.
Terraform 1.11+: prefer use_lockfile = true for native S3 locking (no separate lock table to provision or pay for). DynamoDB-based locking still works and remains fine for existing setups, but is legacy going forward. Without either mechanism enabled: two CI jobs apply simultaneously → state corruption, duplicate resources, or destructive race conditions.
CRITICAL
ST-04Does terraform state rm delete the actual cloud resource?
✗ WRONGYes — removing from state destroys the resource.
✓ RIGHTNo. state rm removes the resource from Terraform's tracking only. The real resource keeps running untouched. Use this to "orphan" a resource from TF management — e.g., before deleting a module you want to keep.
Workflow for escaping TF management: terraform state rm aws_instance.foo → resource still exists, TF forgets it. Next plan will show it as a new resource to create (if still in .tf) or nothing (if removed from .tf).
HIGH
ST-05Can two workspaces in the same backend share a state file?
✗ WRONGYes, workspaces share state — that's the point.
✓ RIGHTNo. Each workspace has its own isolated state file. In S3: env:/workspace-name/path/to/terraform.tfstate. Resources in workspace A are completely invisible to workspace B.
Workspace ≠ environment best practice. Workspaces are for ephemeral environments or minor variations of same config. Different environments (prod/staging) should use separate state files and backends entirely.
HIGH
ST-06After renaming a resource block in .tf, will plan update it in place?
✗ WRONGYes — Terraform tracks by content, not by name.
✓ RIGHTNo. It will destroy the old name and create a new one. Resource address in state = type.name. Rename = new address = destroy+create. Use terraform state mv aws_instance.old aws_instance.new first.
Always run terraform plan after any rename. Seeing a destroy+create for a prod database is the moment you realize you needed state mv first. Especially dangerous with prevent_destroy = true — it won't save you here.
HIGH
ST-07Does prevent_destroy = true in lifecycle block prevent terraform destroy on that resource?
✗ WRONGYes — it protects the resource from any destroy operation.
✓ RIGHTIt only prevents destroy when the resource is in a config that TF is applying. Remove the block from .tf entirely (or remove the whole resource), then apply — now TF will happily destroy it.
prevent_destroy stops accidental in-plan destroys, not a security measure. It's a lint guard, not a lock. For true protection, use IAM deny policies on the underlying cloud resource.
HIGH
ST-08Does terraform refresh update your .tf config files with real cloud state?
✗ WRONGYes — refresh syncs .tf files with what's in the cloud.
✓ RIGHTNo. refresh only updates the state file, never .tf files. It reads actual cloud resource attributes → writes them to state. Your .tf is source of truth for desired state; state is TF's map of reality.
terraform refresh is deprecated in favor of terraform apply -refresh-only which shows a plan before changing state. Never use standalone refresh in automation — it modifies state without a plan review.
HIGH
ST-09Can you use terraform import to import multiple resources at once?
✗ WRONGYes — pass multiple IDs to import them all.
✓ RIGHTClassic terraform import = one resource per command. Terraform 1.5+ import block in .tf imports multiple resources in one apply. Also terraform plan -generate-config-out=generated.tf can scaffold the config.
Bulk import for pre-existing infra: use terraformer or tf-import tools to generate both state entries and .tf config from existing cloud resources. Classic import requires you to write the .tf config manually first.
MED
ST-10Does the state file serial number matter?
✗ WRONGNo — it's just a version counter, doesn't affect operations.
✓ RIGHTCritical for conflict detection. Each apply increments the serial. If two applies start from the same serial, the second write is rejected (stale state). This is how Terraform detects concurrent modifications without DynamoDB.
When force-pushing state (terraform state push -force): you override the serial check. This can overwrite a concurrent apply's changes. Only use -force when you're 100% certain no other apply is running.
MED
ST-11Does moving to a new backend automatically migrate the existing state?
✗ WRONGNo — you have to manually copy the state file to the new backend.
✓ RIGHTYes — terraform init after changing backend config prompts to migrate state automatically. It copies state from old backend to new and optionally deletes from old. Safe if done correctly.
Best practice: always take a manual backup before backend migration. terraform state pull > backup-$(date +%s).tfstate. Verify new backend has the state before deleting from old.

Plan & Apply
10 TRAPS
CRITICAL
PL-01Does a saved plan file (terraform plan -out=plan.tfplan) guarantee the apply is identical?
✗ WRONGNo — cloud state can drift between plan and apply.
✓ RIGHTYes — applying a saved plan file re-executes exactly the same operations, no refresh. This is the correct CI/CD pattern: plan in PR review → save plan → apply approved plan. Prevents "plan drift" between review and apply.
terraform apply plan.tfplan skips the interactive approval and executes the exact saved plan. The plan file is binary, not human-readable — use terraform show plan.tfplan to inspect it.
CRITICAL
PL-02Will changing a resource argument always show as an in-place update in the plan?
✗ WRONGYes — Terraform updates resources in place when possible.
✓ RIGHTDepends on whether the argument is "ForceNew" in the provider schema. Some attributes (e.g., EC2 AMI, RDS engine version) trigger destroy+recreate. Look for # forces replacement in the plan output — easy to miss.
Always scan plan output for the words "must be replaced" or "forces replacement". An innocent-looking change to a tag or name can silently trigger recreation of a stateful resource (RDS, ElastiCache, etc.).
HIGH
PL-03Does terraform apply -auto-approve skip the plan phase entirely?
✗ WRONGYes — auto-approve means no planning, just apply.
✓ RIGHTNo. -auto-approve only skips the interactive confirmation prompt. It still runs a full plan internally before applying. The danger: you can't review the plan before it executes. Never use in prod without a saved plan file workflow.
Safe CI pattern: terraform plan -out=plan.tfplan (reviewable) → approval gate → terraform apply plan.tfplan (no -auto-approve needed, saved plan is already approved).
HIGH
PL-04Does -target flag apply changes only to the targeted resource with no side effects?
✗ WRONGYes — targeted apply is surgical with no unintended changes.
✓ RIGHT-target skips dependency resolution and can create state inconsistencies. TF may apply the target without updating dependent resources, leaving state partially inconsistent. Use only for emergency triage, never routine ops.
HashiCorp themselves recommend against regular -target use. After any targeted apply, run a full terraform plan to verify state is consistent. Treat it like kubectl delete pod --force.
HIGH
PL-05Does terraform plan guarantee that apply will succeed?
✗ WRONGYes — plan validates everything, apply just executes the plan.
✓ RIGHTNo. Plan checks syntax, logic, and state — not cloud-side constraints. Apply can fail due to: IAM quota limits, cloud service quotas, API errors, race conditions, or resources that exist in cloud but not in state.
Common apply-only failures: resource already exists (manual creation bypassed TF), quota exceeded (plan doesn't check account quotas), dependency ordering (provider-side, not TF-side).
HIGH
PL-06Can you use count and for_each on the same resource block?
✗ WRONGYes — combine them for more complex resource generation.
✓ RIGHTNo. They are mutually exclusive. count creates resources addressed as resource[0], resource[1]. for_each creates resources addressed as resource["key"]. Using both on one block is a parse error.
Prefer for_each over count for anything non-trivial. With count: removing item from middle of list shifts indexes → destroys+recreates all subsequent resources. for_each addresses by key → stable.
HIGH
PL-07Switching a resource from count to for_each — is it safe without a state migration?
✗ WRONGYes — TF figures out the mapping automatically.
✓ RIGHTNo. It will destroy all count-indexed resources and recreate them with for_each keys. Use terraform state mv 'aws_instance.web[0]' 'aws_instance.web["server-a"]' to migrate addresses before switching.
This is a common refactor trap. The resource is identical in the cloud — only the state address changes. state mv updates the address in state without touching the real resource.
MED
PL-08Does terraform validate catch all configuration errors?
✗ WRONGYes — validate confirms the config is correct and will apply cleanly.
✓ RIGHTValidate checks syntax and internal references only — no provider API calls, no state access. It won't catch: invalid AMI IDs, wrong region names, missing IAM permissions, or resource attribute conflicts.
validate → plan → apply is the hierarchy of confidence. Each step catches more classes of errors. validate is fast and suitable for pre-commit hooks. plan needs provider credentials.
MED
PL-09Does depends_on only affect ordering of resource creation?
✗ WRONGYes — depends_on just sets the creation order.
✓ RIGHTdepends_on affects destroy order too (reversed), and forces full plan re-evaluation of the dependent resource whenever the dependency changes. Over-using it on modules slows plans dramatically.
Prefer implicit dependencies (reference expressions) over explicit depends_on. Implicit deps are precise. depends_on on a module forces re-plan of everything inside that module whenever anything in the dependency changes.
MED
PL-10Is a null_resource with triggers guaranteed to re-run its provisioner on every apply?
✗ WRONGYes — null_resource runs its provisioner on every apply.
✓ RIGHTOnly when a trigger value changes. If triggers map is static (no values change), the provisioner never re-runs. Use triggers = { always = timestamp() } for every-apply execution — but this forces replacement every time.
null_resource is deprecated in favor of terraform_data (1.4+) which has the same trigger semantics without the null provider dependency. Both are escape hatches — prefer native resources where possible.

Modules
9 TRAPS
CRITICAL
MO-01Can a child module access variables from its grandparent module directly?
✗ WRONGYes — variables propagate down the module tree automatically.
✓ RIGHTNo. Variables are only passed explicitly from parent to immediate child. Each module is a black box. Grandparent → parent → child: values must be threaded down manually at each level. No implicit inheritance.
This forces explicit, visible interfaces between modules — which is the design intent. Globals/environment don't exist in TF module scope. Only terraform.workspace and path.* are implicitly available.
HIGH
MO-02Does updating a module source version cause a .terraform.lock.hcl mismatch?
✗ WRONGYes — changing the module version requires terraform init -upgrade to fix a lock mismatch.
✓ RIGHTNo. The lock file doesn't track modules at all. .terraform.lock.hcl records only provider dependencies (versions + hashes). Module version changes are resolved at terraform init / terraform get and never produce a lock mismatch.
Lock file (.terraform.lock.hcl) locks provider versions only, for reproducible runs — commit it to git. Modules are recorded separately in .terraform/modules/modules.json, a runtime manifest (not a committed lock file). Just run terraform init after changing a module's source or version — nothing to reconcile.
HIGH
MO-03Can a module output a sensitive value without marking it sensitive = true?
✗ WRONGYes — sensitivity is only needed for variables, not outputs.
✓ RIGHTThe output will be shown in plaintext in terraform output and logs. Mark outputs containing secrets with sensitive = true. The value is still in state (in plaintext) — sensitive just suppresses display.
sensitive = true prevents display in CLI output and logs but does not encrypt the value in state. State still contains the plaintext secret. Encrypt your state backend (S3 SSE + KMS) to protect sensitive values.
HIGH
MO-04Can you call a module multiple times without issues using count inside the module?
✗ WRONGYes — count inside the module handles multiple instances.
✓ RIGHTUse count or for_each on the module call, not inside the module. Multiple module calls without count/for_each create naming conflicts (duplicate resource addresses). for_each on the call creates distinct instances.
module.vpc[0], module.vpc[1] (count) or module.vpc["prod"], module.vpc["staging"] (for_each). Each call gets an isolated scope with no resource address collision.
HIGH
MO-05Does source = "git::https://github.com/org/module.git" pin a specific version?
✗ WRONGYes — the git URL is a stable reference.
✓ RIGHTNo. Without a ref/tag it tracks HEAD — your infra can change on the next init. Always pin: git::https://...?ref=v1.2.3 or use the Terraform Registry with version = "~> 1.2".
Unpinned module sources in production are a ticking time bomb. A breaking change merged to the module's main branch deploys on your next terraform init. Always pin. Always.
MED
MO-06Can a module create resources in a different provider region than the root module?
✗ WRONGNo — modules inherit the root provider configuration.
✓ RIGHTYes, via provider aliases. Define provider "aws" { alias = "us_east" region = "us-east-1" } in root. Pass it to module: providers = { aws = aws.us_east }. Module declares required_providers and accepts it.
Multi-region deployments (Route53 + CloudFront + ACM in us-east-1, resources in eu-west-1) require provider aliases passed down to modules. Modules cannot create their own provider configurations — must receive them from caller.
MED
MO-07Does removing a module block from your .tf files remove the resources it managed?
✗ WRONGNo — resources stay until you explicitly delete them in cloud console.
✓ RIGHTYes — next terraform apply will destroy all resources the module managed. TF sees resources in state with no corresponding config → destroy. Use removed block (TF 1.7+) to detach without destroying.
Safe module removal workflow: terraform state mv resources to root module (or a new module) before deleting the old module block. Or use TF 1.7+ removed { from = module.old lifecycle { destroy = false } }.
MED
MO-08Is it a good practice to put all resources in a single large module?
✗ WRONGYes — one module = simple, everything in one place.
✓ RIGHTNo. Large monolithic root modules mean one plan locks everything, any error blocks all deployments, blast radius is entire infra. Separate by lifecycle: networking, databases, applications — each with own state.
At scale: VPC (rarely changes), RDS (changes per sprint), EKS apps (changes per hour) should have completely separate state files. Coupling them means deploying apps requires locking the VPC state.
MED
MO-09Can module outputs reference resources that use count without index?
✗ WRONGYes — TF resolves the count automatically in output references.
✓ RIGHTNo. You must specify the index or use * splat. output "ids" { value = aws_subnet.this[*].id } returns a list. aws_subnet.this.id without index is an error when count > 1.
Splat syntax: aws_subnet.this[*].id = list of all IDs. aws_subnet.this[0].id = first one. For for_each: { for k, v in aws_subnet.this : k => v.id } to output a map.

Providers
9 TRAPS
CRITICAL
PR-01Should you hardcode AWS credentials in the provider block for simplicity?
✗ WRONGYes — it's explicit and avoids environment variable confusion.
✓ RIGHTNever. Credentials in .tf files are committed to git and end up in state. Use: IAM roles (EC2/ECS/Lambda instance profile), IRSA (EKS), environment variables, or AWS profiles. Provider block should have no credentials at all.
Even if you gitignore .tf files: state file contains all resource attributes including any passed credentials. State is stored in S3 — needs encryption and strict access control regardless.
HIGH
PR-02Does upgrading a Terraform provider version always work without config changes?
✗ WRONGYes — providers are backwards compatible within major versions.
✓ RIGHTMinor and patch bumps are usually safe but can still change default attribute values or introduce new required arguments. Patch bumps can also change ForceNew attributes. Always read the provider changelog and run plan after upgrading.
AWS provider upgrades are notorious for unexpected plan changes: new required attributes with no defaults, resources that now ForceNew on a previously in-place attribute, new resources that shadow old ones.
HIGH
PR-03Can you use terraform {required_providers {}} in a child module to override the provider version?
✗ WRONGYes — each module can independently require its own provider version.
✓ RIGHTModules declare version constraints as hints, not requirements. The root module's required_providers controls the actual version used. Child module constraints are checked for compatibility but can't override root. One provider instance shared.
If a child module requires aws >= 5.0 and root pins aws = 4.x: error at init time. Module constraints narrow the acceptable range; the root must select a version satisfying all constraints in the tree.
HIGH
PR-04Does the kubernetes provider need the cluster to exist before you can plan EKS + k8s resources in the same config?
✗ WRONGNo — TF resolves dependencies and connects to the cluster after it's created.
✓ RIGHTYes. Provider config is evaluated at plan time, before resources are created. The cluster endpoint must exist when TF initializes the k8s provider. Standard solution: two separate applies, or use a data source + depends_on hack.
Best pattern: module 1 (EKS cluster) → separate state → module 2 (k8s resources) reads cluster endpoint via terraform_remote_state or data source. Trying to do both in one apply is a classic ordering failure.
HIGH
PR-05Does the helm provider manage Helm chart upgrades the same way helm upgrade does?
✗ WRONGYes — same behavior, just declarative.
✓ RIGHTThe Helm provider has different rollback semantics. TF doesn't know about Helm release history — it only sees the current desired state. A failed TF apply mid-upgrade can leave the release in a broken state that requires helm rollback manually.
Helm provider is convenient but consider: Helm manages its own state (releases in cluster), TF manages its own state. Two sources of truth. For complex Helm deployments, ArgoCD or Helmfile gives better lifecycle management.
MED
PR-06Does a data source run at plan time or apply time?
✗ WRONGAlways at plan time — that's why it's called a data source.
✓ RIGHTNormally at plan time. But if it depends on a resource not yet created, it runs at apply time after that resource exists. This means its value is unknown during plan, shown as (known after apply).
This causes cascading "known after apply" unknowns. A data source depending on a new resource makes all resources that use the data source also show unknown during plan. Plan output becomes less informative.
MED
PR-07Does terraform init always download providers from the public Terraform Registry?
✗ WRONGYes — registry.terraform.io is always the source.
✓ RIGHTYou can configure provider mirrors. terraform providers mirror caches to local dir. network_mirror or filesystem_mirror in ~/.terraformrc for air-gapped environments. Critical for enterprise/regulated infra.
Air-gapped CI: pre-cache providers with terraform providers mirror ./vendor, configure filesystem_mirror pointing to that path, set TERRAFORM_PLUGIN_CACHE_DIR. Providers won't need internet on init.
MED
PR-08Is it safe to run multiple terraform apply commands in parallel against different workspaces on the same backend?
✗ WRONGNo — workspaces share a lock so parallel applies will block.
✓ RIGHTWorkspaces use separate lock keys — parallel applies against different workspaces are safe and common. Lock contention only occurs within the same workspace. This is why workspaces enable parallel environment deployments.
Each workspace in S3/DynamoDB backend has its own state path and its own DynamoDB lock entry. workspace-dev and workspace-prod can apply simultaneously without blocking each other.
MED
PR-09Does the random_password resource generate a new password on every apply?
✗ WRONGYes — it's random, so it generates a new value each time.
✓ RIGHTNo. The result is stored in state and reused on subsequent applies. A new password is only generated if you taint the resource or its keepers input changes. The random value is deterministic once created.
The random provider is idempotent by design. random_password.db.result is stable across applies. To rotate: update a keeper value or terraform taint random_password.db (or terraform apply -replace in newer TF).

Security
7 TRAPS
CRITICAL
SE-01Is the Terraform state file safe to store in a public S3 bucket with encryption enabled?
✗ WRONGYes — server-side encryption makes it unreadable without the key.
✓ RIGHTNo. SSE encrypts at rest but S3 decrypts for any authenticated AWS caller. Anyone with S3 GetObject permission reads the state in plaintext. State contains: passwords, private keys, connection strings. Block all public access + strict bucket policy.
State bucket should have: Block Public Access enabled, bucket policy denying access except specific IAM roles, SSE-KMS (not SSE-S3) for key rotation control, versioning enabled for state recovery, access logging for audit.
CRITICAL
SE-02Does marking a variable as sensitive = true protect it from appearing in state?
✗ WRONGYes — sensitive variables are redacted from all TF output including state.
✓ RIGHTNo. Sensitive only suppresses CLI display. The value is stored in plaintext in state. Anyone with state read access sees it. Use a secrets manager (Vault, AWS Secrets Manager) + data source instead of passing secrets as variables.
Pattern: store secret in AWS Secrets Manager → data "aws_secretsmanager_secret_version" fetches it at apply time → pass to resource. Secret never appears in .tf files or CI logs. Still in state though — encrypt state with KMS.
HIGH
SE-03Can you use the same Terraform IAM role for both plan and apply in a CI/CD pipeline?
✗ WRONGYes — simpler to have one role for the whole pipeline.
✓ RIGHTYou should use separate roles: read-only for plan, write permissions for apply. Plan role: read infra + state. Apply role: assumed only after approval gate. Reduces blast radius of a compromised plan step.
Pattern: plan stage assumes TerraformPlanRole (S3:GetObject, ec2:Describe*, iam:Get*). After PR approval: apply stage assumes TerraformApplyRole (full write). Apply role never triggered on PRs from forks.
HIGH
SE-04Is committing .terraform/ directory to git harmful?
✗ WRONGCommitting it speeds up CI — no need to re-init.
✓ RIGHTThe .terraform/ directory contains provider binaries and potentially cached credentials. Always gitignore it. CI should run terraform init fresh. Provider binaries are large, platform-specific, and potentially tampered if committed.
Gitignore: .terraform/, *.tfstate, *.tfstate.backup, *.tfplan (binary, platform-specific), .terraform.tfstate. Commit: .terraform.lock.hcl (provider version pins).
HIGH
SE-05Does Terraform Cloud / Enterprise encrypt secrets in its own state storage?
✗ WRONGNo — Terraform Cloud state has the same plaintext issue as S3.
✓ RIGHTYes — HCP Terraform encrypts state at rest and in transit, and supports fine-grained access controls per workspace. But sensitive values are still accessible to workspace members with state read permission. Least-privilege workspace ACLs still required.
HCP Terraform also provides ephemeral run credentials (no long-lived keys), audit logging, Sentinel policy as code, and run triggers between workspaces. Self-managed S3 backends require all this to be built manually.
MED
SE-06Is it safe to output an RDS password from a Terraform module to pass to an application?
✗ WRONGYes — mark it sensitive and pass it as an env variable.
✓ RIGHTAvoid. Passing secrets through TF outputs means they're in state and potentially CI logs. Create the RDS instance with TF, but store the password directly in Secrets Manager and let the app fetch it at runtime — never route it through TF outputs.
Pattern: aws_db_instance with manage_master_user_password = true (RDS manages secret in Secrets Manager automatically). App reads from Secrets Manager directly. Password never touches TF state or CI output.
MED
SE-07Does terraform fmt fix security issues in your config?
✗ WRONGYes — fmt validates and formats for security compliance.
✓ RIGHTfmt only normalizes whitespace and formatting. No security checks. Use tfsec, checkov, or terrascan for security scanning. These catch open security groups, unencrypted buckets, missing logging, public RDS instances.
Security scanning pipeline: tfsec . or checkov -d . → block PRs with HIGH/CRITICAL findings → terraform validate → terraform plan. Run tfsec in pre-commit hooks for instant feedback.

Advanced
6 TRAPS
CRITICAL
AD-01Can a for_each use a value that is only known after apply (e.g., an output from a resource being created)?
✗ WRONGYes — TF resolves the value during apply when it becomes known.
✓ RIGHTNo. for_each key set must be known at plan time. TF needs to enumerate all resource instances before apply starts. Using a dynamic/unknown set causes: "The 'for_each' map includes keys derived from resource attributes that cannot be determined until apply." The classic blocker.
Solutions: (1) Use a static local set as the for_each key, reference the dynamic value inside the resource body. (2) Use count with a known length instead. (3) Split into two applies: first create the source resource, then use its output as for_each in the next apply.
CRITICAL
AD-02Is terraform taint the correct way to force resource recreation in modern Terraform?
✗ WRONGYes — taint marks a resource for recreation on next apply.
✓ RIGHTtaint is deprecated since Terraform 1.0. Use terraform apply -replace="aws_instance.web" instead. It shows the replacement in the plan for review before executing. taint modified state without showing a plan.
-replace is superior: it generates a proper plan showing the destroy+create, which you review before applying. taint mutated state immediately with no preview. The -replace flag is available from TF 0.15.2+.
HIGH
AD-03Does terraform output always return the current real-world value?
✗ WRONGYes — output queries the live cloud resource.
✓ RIGHTNo. terraform output reads from the state file, not the cloud. If the real resource was changed outside Terraform, the state is stale and output returns the old value. Run terraform apply -refresh-only first to sync state.
State = TF's last known snapshot of reality. It drifts whenever someone touches infra outside TF. -refresh-only is the safe way to sync: it shows you what drifted before updating state.
HIGH
AD-04Can you use lifecycle { ignore_changes = all } to let another tool manage a resource while keeping it in TF state?
✗ WRONGNo — ignore_changes only applies to specific listed attributes.
✓ RIGHTYes. ignore_changes = all makes TF adopt the resource into state without ever modifying it. TF created it (or imported it), another tool manages changes, TF tracks it for dependency graph but never diffs or modifies.
Use case: EKS managed node group where ASG scaling is managed by Karpenter/KEDA but you still want TF to track the resource and manage destroy. ignore_changes = [desired_size] for partial — all for full ownership handoff.
HIGH
AD-05Does terraform graph show the dependency graph of resources as they exist in the cloud?
✗ WRONGYes — it reflects the current cloud infrastructure dependencies.
✓ RIGHTNo. terraform graph shows the dependency graph from your .tf configuration, not cloud reality. Output is DOT format. Pipe to Graphviz: terraform graph | dot -Tsvg > graph.svg. Useful for debugging complex dependency cycles.
Dependency cycle errors are notoriously hard to debug. terraform graph | grep -v provider | dot -Tpng > deps.png visualizes the cycle. Also shows why a -target might miss dependent resources.
MED
AD-06Is toset() interchangeable with tolist() when using for_each?
✗ WRONGYes — both convert to a collection, for_each handles either.
✓ RIGHTNo. for_each requires a set (unique keys) or map. A list has integer indexes, not string keys. for_each = toset(var.names) works. for_each = var.names (list) → error. Sets also deduplicate automatically.
The error message is confusing: "The given value is not suitable for use as for_each argument." — it means you passed a list, not a set/map. Fix: wrap with toset(). Or use a map with meaningful keys for clearer resource addresses.
