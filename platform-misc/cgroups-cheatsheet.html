What cgroups actually is
// not a userspace tool — a kernel subsystem exposed as a virtual filesystem
⚙️ The Three Container Primitives
Containers are not a kernel primitive. They are built from three orthogonal Linux mechanisms combined by a runtime (containerd, runc):
cgroups
LIMITS
How much resource a group of processes can use.
namespaces
ISOLATION
What a process can see (PID, net, mount, UTS, IPC, user).
union fs
LAYERING
overlay2 / OverlayFS stacked image layers.
What cgroups doesGroups processes into a hierarchy and enforces resource limits per group. CPU throttling, memory limits, I/O bandwidth, PID counts — all enforced by the kernel, not by the runtime.
ImplementationA pseudo-filesystem mounted at /sys/fs/cgroup/. Creating a directory = creating a cgroup. Writing a PID to cgroup.procs = adding a process to the group. Reading files = seeing current usage.
Kernel enforcesLimits are enforced in kernel space — the process cannot exceed them regardless of how hard it tries. malloc() returns ENOMEM. CPU is throttled by the scheduler. Excessive memory = OOM kill.
Process + threadsAll threads of a process are in the same cgroup (they share the same task_struct cgroup pointer). You can't put one thread in a different cgroup than its parent process (v1). v2 relaxes this.
cgroups ≠ namespaces. A process in a PID namespace still counts toward the cgroup resource budget of its host. Docker containers have both: namespaces for isolation (what they see), cgroups for limits (how much they get).
🗂️ The Virtual Filesystem

🔑 Three Core Operations
Create groupmkdir /sys/fs/cgroup/mygroup — kernel auto-populates interface files inside.
Add processecho $PID > /sys/fs/cgroup/mygroup/cgroup.procs — process + all threads join the group.
Set limitecho 512M > /sys/fs/cgroup/mygroup/memory.max — kernel enforces immediately.
Read usagecat /sys/fs/cgroup/mygroup/memory.current — live kernel counter, no sampling.
Delete grouprmdir /sys/fs/cgroup/mygroup — only works when cgroup has zero processes.

🏚️ cgroups v1 — The Messy Reality
v1 Structure
/sys/fs/cgroup/
├─ cpu/ ← separate mount
│ ├─ system.slice/
│ │ └─ docker-abc.scope/
├─ memory/ ← different tree!
│ ├─ system.slice/
│ │ └─ docker-abc.scope/
├─ blkio/ ← yet another tree
├─ pids/
├─ cpuset/
└─ net_cls/ ...
v2 Structure
/sys/fs/cgroup/ ← ONE mount
├─ system.slice/
│ ├─ docker-abc.scope/ ← all controllers here
│ │ ├─ cpu.max
│ │ ├─ memory.max
│ │ ├─ io.max
│ │ └─ pids.max
│ └─ docker-def.scope/
└─ user.slice/
v1 problemEach subsystem has its own independent hierarchy. A process can be at different positions in different trees. The cpu group "A" and memory group "A" have no enforced relationship. Makes coordinated resource management impossible.
Cpuacct splitv1 has both cpu and cpuacct as separate subsystems. cpu = throttling/scheduling. cpuacct = accounting only. In v2 they're unified into one.
v1 thread-level inconsistency: some v1 controllers allow tasks (threads) to be in different cgroups than their process. This makes accounting inaccurate and is the primary reason v2 was designed.
✨ cgroups v2 — The Unified Hierarchy
One hierarchyAll controllers share a single hierarchy. A process can only be in one cgroup. All resource limits (CPU, memory, I/O) apply to the same group. Consistent, predictable.
Thread modev2 supports threaded mode for cgroups — threads can have different cgroups within the same process, but only for specific subsystems. Much more controlled than v1's chaos.
New controllersv2-only features: memory.oom.group (OOM kills entire cgroup not just one process), cpu.pressure / memory.pressure / io.pressure (PSI — pressure stall info), memory.swap.max (separate swap limit).
PSI (Linux 4.20)Pressure Stall Information — measures what % of time tasks were stalled waiting for CPU/memory/IO. memory.pressure gives you 10s/60s/300s averages. K8s uses this for eviction decisions.
K8s requirementMemoryQoS (--memory-manager-policy), PSI-based eviction, and new kubelet features require cgroups v2. Ubuntu 22.04+ and RHEL 9 default to v2. Explicitly check before upgrading.
# Check if v2 is active cat /sys/fs/cgroup/cgroup.controllers # cpuset cpu io memory hugetlb pids rdma misc ← v2 # Enable v2 on older systems (grub cmdline) # GRUB_CMDLINE_LINUX="systemd.unified_cgroup_hierarchy=1" # Read PSI memory pressure cat /sys/fs/cgroup/system.slice/memory.pressure # some avg10=0.50 avg60=0.20 avg300=0.05 total=12345 # full avg10=0.00 avg60=0.00 avg300=0.00 total=0
v1/v2 hybrid mode exists. Some distros mount v2 at /sys/fs/cgroup/unified/ while keeping v1 at /sys/fs/cgroup/. containerd and Docker must be explicitly configured for which to use. Mismatched configuration = resource limits silently ignored.

Subsystems (Controllers)
// what each controller limits, how it works under the hood
🔴 cpu controller
v1 filescpu.shares (relative weight), cpu.cfs_period_us + cpu.cfs_quota_us (hard limit)
v2 filescpu.weight (1-10000, replaces shares), cpu.max (quota period format)
CFS throttleCompletely Fair Scheduler. If process uses quota within a period, it's throttled until next period. Default period = 100ms.
Soft vs hardcpu.shares/cpu.weight = soft — only applies when CPU is contended. cfs_quota/cpu.max = hard — enforced always.
# v1: limit to 0.5 CPU (50% of 100ms period) echo 100000 > cpu.cfs_period_us echo 50000 > cpu.cfs_quota_us # v2: same — "50000 100000" echo "50000 100000" > cpu.max # v2: weight (default=100, range 1-10000) echo 200 > cpu.weight # 2× priority
CPU throttling ≠ CPU usage. A container can appear "idle" (low usage) but be throttled 80% of the time. Check cpu.stat: throttled_time and nr_throttled. K8s metric: container_cpu_cfs_throttled_seconds_total.
🔵 memory controller
v1 filesmemory.limit_in_bytes, memory.usage_in_bytes, memory.memsw.limit_in_bytes (mem+swap)
v2 filesmemory.max (hard limit), memory.high (soft limit, throttle), memory.min (guarantee), memory.current
memory.highv2 only. When usage crosses high: kernel aggressively reclaims pages + throttles allocations. Process can still allocate — just slower. Graceful degradation before OOM.
memory.minv2 only. Guaranteed memory — kernel won't reclaim below this threshold even under global pressure. Used by K8s Guaranteed QoS class.
oom.groupv2 only. When enabled, OOM killer kills all processes in the cgroup, not just the worst offender. Prevents zombie containers with partial-killed workloads.
# v2: set 512MiB hard limit echo "536870912" > memory.max # Or human units aren't supported — use bytes # Read current usage cat memory.current # bytes currently used cat memory.stat # detailed breakdown # anon — anonymous pages (heap/stack) # file — page cache (files) # slab — kernel slab allocations
memory.usage_in_bytes (v1) includes page cache. A container using 400MB "memory" might have only 100MB actual working set — the rest is filesystem cache. The OOM killer uses working set, not raw usage. K8s reports working_set_bytes = current - inactive_file.
🟣 io (blkio) controller
v1 nameblkio subsystem. v2 name: io controller. Completely different interface format.
v1 limitsblkio.throttle.read_bps_device, blkio.throttle.write_bps_device — device major:minor format. Only throttles direct I/O by default.
v2 limitsio.max: MAJ:MIN rbps=X wbps=Y riops=Z wiops=W. Limits both direct and buffered I/O. io.weight for proportional scheduling.
Buffered I/Ov1 blkio cannot reliably limit buffered writes — dirty pages are flushed by kernel writeback threads outside the cgroup. v2 io controller tracks writeback to the originating cgroup and limits it properly.
# v2: limit device 8:0 to 100MB/s read, 50MB/s write echo "8:0 rbps=104857600 wbps=52428800" > io.max # Read io stats cat io.stat # 8:0 rbytes=.. wbytes=.. rios=.. wios=.. dbytes=.. dios=..
This is why K8s requires cgroups v2 for reliable I/O throttling. v1 blkio only throttles direct I/O — containers doing buffered writes (almost all of them) can blow through their limits. v2 io controller closes this gap.
🟡 pids controller
PurposeLimits number of processes+threads in the cgroup. Prevents fork bombs from consuming all PIDs on the host.
Filespids.max (limit, or max for unlimited), pids.current (live count), pids.events (count of max-exceeded events)
K8s mappingspec.containers[].resources.limits.pids (alpha/beta feature gate SupportPodPidsLimit). Default: no limit. Dangerous on multi-tenant clusters.
# Set PID limit echo 1000 > pids.max # Monitor exhaustion events cat pids.events # max 5 ← hit limit 5 times
PID exhaustion is a denial-of-service vector. Default Linux PID limit is 32768 (or 4M on 64-bit). A container without a pids limit can fork-bomb the host, preventing all new processes including SSH connections. Always set pids.max in production.
🩷 cpuset controller
PurposePin processes to specific CPU cores and NUMA memory nodes. Eliminates cache bouncing and NUMA remote access.
Filescpuset.cpus (e.g., 0-3,8-11), cpuset.mems (NUMA nodes, e.g., 0-1)
Exclusivecpuset.cpus.exclusive: prevents other cgroups from using these cores. Used for real-time/latency-sensitive workloads.
# Pin to cores 0,1 only echo "0-1" > cpuset.cpus echo "0" > cpuset.mems
K8s Guaranteed QoS + --cpu-manager-policy=static: kubelet pins containers to dedicated cores via cpuset. No CPU sharing, near bare-metal performance.
⏸️ freezer controller
PurposePause/resume all processes in a cgroup atomically. Used for container checkpointing, live migration, and debugging.
v1freezer.state: write FROZEN or THAWED. Processes receive no CPU until thawed.
v2cgroup.freeze: write 1 (freeze) or 0 (thaw). cgroup.events reports frozen state.
# v2: freeze all container procs echo 1 > cgroup.freeze # processes get SIGSTOP-like suspension echo 0 > cgroup.freeze # thaw
Freezing a cgroup that holds a lock (e.g., holding a mutex) can deadlock other cgroups waiting for that lock. Freeze only for short diagnostic captures.
🍊 hugetlb + perf_event
hugetlbLimits usage of huge pages (2MiB/1GiB pages). Latency-sensitive apps (databases, ML inference) benefit from huge pages — reduces TLB misses.
perf_eventScopes perf events to a cgroup. perf stat -G cgroupname profiles only processes in that group.
rdma (v2)Limits RDMA/InfiniBand resources. Relevant for HPC + ML clusters with high-speed interconnects.
misc (v2)Generic resource counter for things that don't fit elsewhere (e.g., AMD SEV memory, device-specific counters).
🟢 net_cls / net_prio
net_clsTags outgoing packets with a class ID — allows tc (traffic control) to apply QoS rules per cgroup. v1 only — absent from v2.
net_prioSets per-interface network priority for processes in the cgroup. v1 only.
v2 networkNo network controller in v2. Network bandwidth limiting for containers must be done via tc/eBPF outside cgroups (e.g., CNI plugins, tc htb qdisc). This is an intentional design decision.
K8s does NOT enforce network bandwidth limits via cgroups. kubernetes.io/egress-bandwidth annotation is handled by CNI plugins (if supported), not by the kubelet/cgroup layer. Most CNIs don't implement it.

Kubernetes → cgroups Mapping
// how K8s requests/limits translate to kernel parameters
📐 requests / limits → cgroup files
K8s fieldv1 cgroup filev2 cgroup fileBehavior
cpu request cpu.shares cpu.weight Soft guarantee. Share of CPU when contended. Never throttled for this.
cpu limit cpu.cfs_quota_us cpu.max (quota) Hard throttle. Process frozen until next 100ms period if budget exhausted.
memory request (not enforced by cgroup) memory.min v1: scheduler hint only. v2: kernel won't reclaim below this. Actual guarantee.
memory limit memory.limit_in_bytes memory.max Hard OOM limit. Exceeding triggers OOM kill inside cgroup.
— — memory.high v2 only. ~= 0.9× limit. Throttles before OOM. K8s MemoryQoS sets this.
memory request ≠ memory guarantee in cgroups v1. With v1, memory.limit_in_bytes only sets the hard cap. The request is used only by the scheduler for bin-packing — the kernel doesn't enforce a minimum. v2's memory.min changes this.
cpu request translation: shares = max(2, request_milliCPU * 1024 / 1000). A container requesting 500m CPU = 512 shares. Default = 1024 shares = 1 CPU equivalent. When no contention: no throttling regardless of shares.

⚖️ QoS Class → cgroup + OOM
QoSConditionoom_score_adj
BestEffortNo requests or limits set1000 (first killed)
BurstableRequests < limits, or partial2–999

oom_score_adj is written to /proc/<pid>/oom_score_adj by kubelet. Combined with actual memory usage → oom_score. Highest score = first killed when host runs out of memory. Guaranteed QoS pods are effectively protected from host-level OOM.

📡 CPU Manager Policy — cpuset pinning
none (default)CFS shares all CPU cores across all containers. Processes migrate between cores freely. Cache thrashing possible under contention.
static policyGuaranteed QoS pods with integer CPU requests get dedicated cores. kubelet writes to cpuset.cpus — process pinned to exclusive cores. No sharing, no migration.
RequirementPod must be Guaranteed QoS + CPU request must be an integer (not 500m, must be 1, 2, etc.). kubelet tracks the CPU pool and allocates from it.
Topology managerGoes further: aligns cpuset, memory NUMA node, and device (GPU, RDMA) to the same socket. Prevents cross-NUMA penalties for latency-sensitive workloads.
# kubelet flag --cpu-manager-policy=static --topology-manager-policy=single-numa-node # Check what cores a container got cat /sys/fs/cgroup/.../cpuset.cpus # 2-3 ← dedicated cores 2 and 3
🌡️ Memory Manager + MemoryQoS
Memory managerK8s 1.22+. Like CPU manager but for NUMA memory. Pins Guaranteed pods to specific NUMA nodes, preventing cross-NUMA memory access.
MemoryQoSK8s 1.27+ (beta). Requires cgroups v2. Sets memory.high = 0.9 × memory limit. Pod gets throttled before OOM instead of killed abruptly.
memory.high effectWhen usage crosses memory.high: kernel reclaims page cache aggressively + slows new allocations. App sees higher latency but keeps running. Early warning signal.
PSI evictionkubelet watches memory.pressure PSI files. High memory pressure triggers pod eviction decisions. More granular than pure memory usage threshold.
MemoryQoS requires cgroups v2 AND explicit kubelet feature gate. On v1 clusters: memory limit is a hard cliff — process goes from normal to OOM with no warning. v2 with MemoryQoS gives a soft landing zone.

Container Runtime Stack

🏗️ The Full Stack: kubelet → cgroup
📦 containerd vs Docker
Docker (old)dockerd → containerd → containerd-shim → runc. K8s removed dockershim in 1.24. Docker added an unnecessary extra hop.
containerdDirect: kubelet → containerd (via CRI plugin) → containerd-shim-runc-v2 → runc. No dockerd. The default runtime since K8s 1.24.
CRI-OAlternative: kubelet → CRI-O → conmon → runc. Lighter than containerd, OpenShift default. Same runc underneath, different daemon.
gVisor (runsc)Replaces runc with a userspace kernel. Stronger isolation — syscalls intercepted. No shared kernel with host. Higher overhead, used for untrusted workloads.
Kata ContainersEach container gets a lightweight VM (QEMU/Firecracker). Full kernel isolation. cgroups still used — but inside the VM. Pod = VM boundary.

Debug Playbook
// diagnosing cpu throttle, oom, pid exhaustion from cgroup files
🌡️ CPU Throttling Diagnosis
# Find container cgroup path for a pod PODUID=$(kubectl get pod mypod -o jsonpath='{.metadata.uid}') CID=$(kubectl get pod mypod -o jsonpath='{.status.containerStatuses[0].containerID}' | cut -d/ -f3) CG="/sys/fs/cgroup/kubepods/burstable/pod${PODUID}/${CID}" # Check if CPU limit is set cat $CG/cpu.max # "50000 100000" → 0.5 CPU hard limit # "max 100000" → NO limit (unlimited) # Check throttle counters (reset on container restart) cat $CG/cpu.stat # nr_periods 10000 ← total 100ms windows # nr_throttled 4200 ← throttled 42% of time! # throttled_usec 420000000 # Prometheus: throttle ratio # rate(container_cpu_cfs_throttled_seconds_total[5m]) # / rate(container_cpu_cfs_periods_total[5m]) # > 0.25 → alert
High CPU throttling with low CPU usage is the classic trap. Container uses 5% CPU but gets throttled 60% of the time — because it bursts hard within a 100ms window. Solution: remove CPU limit (if trusted) or increase it. Throttling ≠ CPU usage.
💾 OOM Kill Investigation
# Was the container OOM killed? kubectl describe pod mypod | grep -A3 "Last State" # Last State: Terminated # Reason: OOMKilled # Exit Code: 137 (128 + SIGKILL) # Check OOM events in kernel log dmesg | grep -i oom | tail -20 # [123456.789] oom-kill:constraint=CONSTRAINT_MEMCG, # nodemask=(null),cpuset=..,mems_allowed=0, # oom_memcg=/kubepods/burstable/pod<uid>/<cid>, # task_memcg=..,task=java,pid=1234,uid=0 # What was actual memory usage at OOM? # (v1) read from memory.failcnt + memory.max_usage_in_bytes cat $CG/memory.events # low 0 high 0 max 15 oom 3 oom_kill 3 # Check working set vs limit gap cat $CG/memory.stat | grep -E "^(anon|file|slab)" # anon 480000000 ← heap — this is what triggered OOM # file 20000000 ← page cache (reclaimable)
OOMKilled = exit code 137. The container's memory limit was hit. Working set = memory.current - inactive_file. If working set approaches limit, increase limit or tune app GC/heap settings.
🔢 PID + Pressure Debugging
# PID exhaustion check cat $CG/pids.current # how many PIDs in use cat $CG/pids.max # limit cat $CG/pids.events # max = times limit was hit # System-wide PID check cat /proc/sys/kernel/pid_max # default 32768 or 4194304 cat /proc/sys/kernel/threads-max # PSI memory pressure (v2 only) cat $CG/memory.pressure # some avg10=5.23 avg60=1.80 avg300=0.50 total=52300 # full avg10=0.10 avg60=0.05 avg300=0.01 total=1000 # "some" = ≥1 task stalled. "full" = ALL tasks stalled # avg10 > 30 = severe memory pressure, consider eviction # cgroup v2 check script for a running pod for f in cpu.max memory.max memory.high pids.max; do printf "%-20s %s\n" "$f" "$(cat $CG/$f 2>/dev/null)" done
PSI "full" metric means ALL tasks in the cgroup are stalled — zero forward progress. This is the most severe indicator. full avg10 > 1% warrants immediate investigation. Used by systemd's oomd and K8s eviction manager.
High PID count with no app bug = thread pool leak or goroutine leak. Check pids.current trend. If growing without bound: ls /proc/<pid>/task/ to count threads, or use ps -eLf | wc -l.

Quick Reference
// key files, key commands, key numbers
CPU files (v2)
cpu.maxquota period. max 100000 = no limit. 50000 100000 = 0.5 CPU.
cpu.weight1–10000, default 100. Proportional share when contended.
cpu.statnr_throttled, throttled_usec. Read to detect invisible throttling.
cpu.pressurePSI for CPU. some avg10 = % time ≥1 task waited for CPU.
cpuset.cpusPinned cores, e.g. 0-3. Empty = all cores.
Memory files (v2)
memory.currentLive usage in bytes. Includes page cache.
memory.maxHard limit. OOM on exceed. max = unlimited.
memory.highSoft throttle. Kernel reclaims aggressively above this.
memory.minKernel won't reclaim below this. Actual guarantee.
memory.eventsoom_kill counter. Zero = no OOM kills ever. Watch this.
memory.statanon (heap/stack) + file (page cache) + slab.
I/O + PID files (v2)
io.maxMAJ:MIN rbps=X wbps=Y riops=Z wiops=W. Limits buffered + direct I/O.
io.statPer-device byte/ops counters. Includes writeback.
io.weight1–10000. Proportional I/O scheduling weight.
pids.maxMax processes+threads. max = unlimited.
pids.currentLive PID count. Watch for leaks.
pids.eventsmax = times hit limit. Nonzero = fork failures.
Universal files (v2)
cgroup.procsPIDs of processes in this cgroup. Write PID to add. Read to list.
cgroup.controllersActive controllers in this cgroup (space-separated).
cgroup.subtree_controlEnable controllers for children: echo "+memory +cpu" > ...
cgroup.eventspopulated (1 if any procs). frozen (1 if frozen).
cgroup.freezeWrite 1 to pause all procs. Write 0 to resume.
cgroup.killv2 Linux 5.14+. Write 1 = send SIGKILL to all procs in cgroup tree.
