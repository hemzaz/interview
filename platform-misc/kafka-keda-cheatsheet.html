Kafka Core Concepts // the fundamentals you'll be asked
📦
Topics, Partitions, Offsets
The data model
TopicLogical stream name. Split into partitions for parallelism. Partitions = max consumer parallelism.
PartitionOrdered, immutable log. Messages appended only. Each partition has exactly one leader broker at any time.
OffsetMonotonically increasing integer per partition. Consumer tracks "up to where I've read." Stored in __consumer_offsets topic.
OrderingGuaranteed within a partition only. Use consistent partition key (e.g., device_id) to keep events for same device ordered.
RetentionTime-based (retention.ms) OR size-based (retention.bytes). Compacted topics keep latest value per key forever.
More consumers than partitions in a group = idle consumers. Consumers don't share a partition. Plan partitions generously upfront — repartitioning requires rebalance.
🔁
Replication + Durability
How Kafka survives broker failures
RF (Replication Factor)How many broker copies exist. RF=3 means can lose 2 brokers and still serve. Standard in prod.
LeaderHandles all produce AND fetch requests for a partition. Elected by controller. Followers only replicate.
ISRIn-Sync Replicas — replicas caught up within replica.lag.time.max.ms. Only ISR members can become leader.
acks=allProducer waits for all ISR members to ack write. Slowest but zero message loss. Use for attribution.
min.insync.replicasMinimum ISR size to accept writes. Set to 2 with RF=3: min.insync.replicas=2. Rejects writes if only 1 broker alive — prevents silent data loss.
min.insync.replicas = ⌊RF/2⌋ + 1 max tolerance = RF − min.insync.replicas broker failures
👥
Consumer Groups
Horizontal scaling for consumers
Group IDConsumers sharing a group ID share the work. Each partition assigned to exactly one consumer per group.
RebalanceTriggered when consumer joins/leaves/crashes. All partition assignments re-computed. Processing pauses during rebalance — minimize it.
Static membershipgroup.instance.id — consumer retains its assignment across restarts without triggering rebalance. Critical for K8s rolling updates.
Commit modesAuto-commit (risky, may lose events). Sync commit after processing (safe, slower). Async commit (fast, must handle failures).
At-least-onceDefault semantic. Re-process after restart. Your consumer must be idempotent — dedup by event ID.
Committing offset before processing = at-most-once (can lose messages). Commit after = at-least-once (safe but requires idempotency).
Kafka Observability // metrics that matter in production
📊
Critical Metrics Hierarchy
Priority order for alerting
1
Consumer Group Lag (per partition)
Lag = LatestOffset − CommittedOffset. The most important metric. Per-partition granularity reveals hot partitions. Alert: lag growing monotonically OR lag > N.
2
UnderReplicatedPartitions
Must be 0 in production. Any value > 0 means a follower is behind — data at risk. Causes: broker overloaded, network partition, GC pause.
3
ISR Shrink Rate
Followers dropping out of ISR. Leading indicator before UnderReplicated. Caused by slow disk I/O, GC, network jitter.
4
Request Queue Time + Response Time
High request queue depth = broker CPU/IO saturated. Produce latency P99 > threshold = pipeline slowing.
5
Throughput: BytesIn/Out per broker
Reveals hot brokers (uneven partition leader distribution). Balance leaders across brokers with kafka-preferred-replica-election.

Monitoring Stack for Kafka
Tools + setup

AWS MSK MetricsCloudWatch metrics: KafkaDataLogsDiskUsed, UnderReplicatedPartitions, EstimatedTimeLag. MSK also pushes per-consumer metrics.
kminionModern Prometheus exporter for Kafka. Includes consumer group lag per-partition, topic offsets, consumer commit rates.
GrafanaDashboards: Consumer Lag by group/partition, Broker throughput, Produce/fetch latency P95/P99, ISR changes over time.

Kafka Production Tuning // the params that move the needle
⚙️
Producer Config
For high-throughput attribution pipeline
# Durability config (never lose ad events) acks: all min.insync.replicas: 2 retries: Integer.MAX_VALUE enable.idempotence: true # Throughput tuning batch.size: 65536 # 64KB batch linger.ms: 10 # wait 10ms to fill batch compression.type: lz4 # fast + good ratio buffer.memory: 33554432 # 32MB buffer
linger.ms vs latency tradeoff: higher = better batching = higher throughput but more latency. For attribution events: 5–20ms is acceptable.
⚙️
Consumer Config
For event-driven processing at scale
# Throughput + safety max.poll.records: 500 # records per poll max.poll.interval.ms: 300000 # 5min, avoid rebalance fetch.min.bytes: 1024 # wait for 1KB batch fetch.max.wait.ms: 500 # max wait for batch # Offset management (explicit) enable.auto.commit: false # manual commit = safe auto.offset.reset: earliest # new group → start fresh # K8s rolling update stability group.instance.id: pod-${POD_NAME}
max.poll.interval.ms: if processing takes longer than this, Kafka thinks consumer is dead → triggers rebalance. Increase if you have slow batch processing jobs.
⚙️
Broker Config
Operational tuning for large clusters
# Replication + durability default.replication.factor: 3 min.insync.replicas: 2 unclean.leader.election.enable: false # Log management log.retention.hours: 168 # 7 days log.segment.bytes: 1073741824 # 1GB segments log.cleanup.policy: delete # or compact # Network + I/O threads num.network.threads: 8 num.io.threads: 16 socket.send.buffer.bytes: 1048576
unclean.leader.election.enable=true allows out-of-sync replicas to become leader. Never do this for attribution data — you will lose messages.

ScaledObject YAML — Full Example
For Singular's attribution consumer deployment
apiVersion: keda.sh/v1alpha1 kind: ScaledObject metadata: name: attribution-consumer-scaler namespace: attribution-prod spec: scaleTargetRef: apiVersion: apps/v1 kind: Deployment name: attribution-consumer pollingInterval: 30 # check lag every 30s cooldownPeriod: 300 # wait 5m before scale-in minReplicaCount: 1 # never 0 (keep 1 warm) maxReplicaCount: 20 # max = partition count advanced: horizontalPodAutoscalerConfig: behavior: scaleUp: stabilizationWindowSeconds: 60 # fast up policies: - type: Percent value: 100 # double per 60s periodSeconds: 60 scaleDown: stabilizationWindowSeconds: 300 # slow down triggers: - type: kafka metadata: bootstrapServers: kafka-broker:9092 consumerGroup: attribution-workers topic: ad-events lagThreshold: "50000" # lag per replica activationLagThreshold: "1" # wake from 0 offsetResetPolicy: latest
maxReplicaCount should = partition count. More consumers than partitions = idle pods wasting memory. If you need to scale beyond current partitions — increase partitions first.
🧮
Scaling Formula
How KEDA computes desiredReplicas
desiredReplicas = ceil(totalLag ÷ lagThreshold) clamped between minReplicaCount and maxReplicaCount
Examplelag=124,500 ÷ threshold=50,000 → ceil(2.49) = 3 replicas
lag=0Scale to minReplicaCount. If min=0 → scale to zero (cost saving for batch jobs).
activationLagSeparate threshold to wake from 0 replicas. Avoids flapping at low lag.
lagThreshold is per replica, not total. With 3 replicas and threshold=50k, KEDA expects total lag ≤ 150k before scaling down.

Kafka + KEDA Operational Patterns // real production scenarios
🌊
Traffic Spike Pattern
Super Bowl / campaign launch scenario
T=0
Traffic 10× spikes (Nike launches campaign)
Kafka lag on ad-events grows from 5k → 500k in 2 minutes. Consumers can't keep up.
T+30s
KEDA polls lag, computes replicas
ceil(500,000 ÷ 50,000) = 10 replicas needed. KEDA updates HPA. HPA notifies scheduler.
T+90s
New pods running, rebalance completes
Kafka rebalances partitions across 10 consumers. Each gets 2 partitions (20 total). Lag starts draining.
T+10m
Lag back to zero, cooldown starts
Traffic normalizes. KEDA cooldown (300s) prevents premature scale-down. Avoids immediate rebalance.
⚠️
Pitfalls & Solutions
What goes wrong in production
Rebalance stormKEDA scales too aggressively → constant rebalancing → consumer downtime. Fix: increase stabilizationWindow + use static membership (group.instance.id).
Scale to 0 lagIf min=0 for prod, lag spike from cold start as pods spin up. Fix: min=1 or min=2 for always-on topics. Scale-to-zero only for overnight batch jobs.
Partition ceilingmaxReplicas > partition count = idle pods. KEDA will scale to min of (desired, partitions). Always cap maxReplicas at partition count.
Lag metric noiseBursting producers cause temporary lag. With short pollingInterval, KEDA over-scales. Fix: longer stabilization or Burrow for trend-based scaling instead of raw lag.
MSK IAM authOn AWS MSK, use SASL/IAM. KEDA needs a TriggerAuthentication with IAM role annotation on the service account. Easy to miss in initial setup.
KEDA creates and owns the HPA. If you also manually create an HPA for the same Deployment, they fight over replicas. Never mix manual HPA + KEDA on the same target.

HPA vs KEDA vs Cluster Autoscaler // know the differences
⚖️
Autoscaling Tools Comparison
When to use what
Tool What it scales Triggers on Scale-to-zero Best for Singular
Native HPA Pod replicas in Deployment/RS CPU%, memory%, custom metrics API No (min 1) CPU-bound services (API servers)
KEDA Pod replicas via managed HPA External event sources — Kafka lag, SQS depth, Prometheus, cron, etc. Yes (min 0) Kafka consumers, batch jobs, event processors
VPA Pod resource requests/limits Historical resource usage No Right-sizing long-running services (Druid, Redis)
Cluster Autoscaler Nodes in the cluster Unschedulable pods (pending) Slow (minutes) Node provisioning; pair with KEDA pod scaling
Karpenter Nodes (AWS native) Unschedulable pods + node utilization Fast + aggressive consolidation Replace Cluster Autoscaler on EKS — smarter instance selection
The full stack for Singular: KEDA drives consumer pod count based on Kafka lag → Karpenter provisions/deprovisions EC2 nodes based on pod demand → VPA ensures pods have correct resource requests for Karpenter to make good node decisions. Three-layer autoscaling.
Interview Scenarios // answer templates for Singular
💬
"Consumer lag spiked to 10M. Walk me through your response."
1
Check lag distribution per partition — is it all partitions or one hot partition? Hot = bad partition key. All = consumer too slow.
2
Check consumer health — CPU/memory/GC on consumer pods. Is processing time per message high? Check app logs for slow downstream calls.
3
Check if KEDA already triggered — look at HPA events, number of consumer pods. Did it scale? If not, check KEDA operator logs and ScaledObject status.
4
If KEDA maxed out: you're at partition count ceiling. Either increase partitions (requires topic recreation or replication) or horizontally scale the underlying processing (e.g., increase Druid ingestion capacity).
5
Post-mortem: Was this a traffic spike (normal), a slow downstream (Druid/Aerospike degraded), or a bad deploy? Add lag rate-of-change alert, not just raw threshold.
💬
"Design event-driven autoscaling for our attribution consumers."
1
Install KEDA operator in the cluster (Helm: keda/keda). It deploys the Metrics Adapter and the operator controller.
2
Identify partition count for each consumer's topic. Set maxReplicaCount = partition count. No point scaling beyond parallelism ceiling.
3
Set lagThreshold based on acceptable latency SLO. If we need attribution within 5 min and each consumer processes 10k/sec: threshold = 300s × 10k = 3M messages per replica.
4
Configure scale behavior: fast scale-up (100% per 60s), slow scale-down (5min stabilization). Enable group.instance.id in consumer to minimize rebalance cost.
5
Pair with Karpenter: as KEDA adds pods, Karpenter adds nodes. As pods drain, Karpenter consolidates and terminates underutilized nodes. Full elastic cost model.
Key Numbers to Remember // quote these in interviews
RF=3
Replication factor in prod. Tolerates 1 broker failure with min.insync=2.
50k
Typical KEDA lagThreshold per replica. Tune to your processing rate × acceptable lag time.
300s
KEDA cooldown period. Prevents scale-in before lag fully drains after a spike.
64KB
Good Kafka producer batch.size. Balance between memory and throughput gains from batching.
