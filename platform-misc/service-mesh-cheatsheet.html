Why Service Mesh?
// the problem · what K8s doesn't solve · what mesh actually gives you
🔒 The mTLS Problem
Without meshTraffic between pods is plaintext by default. A compromised pod can read any inter-service traffic. TLS requires every team to manage certs, rotation, CA trust. In practice: nobody does it consistently.
With meshThe proxy sidecar intercepts all traffic and automatically negotiates mTLS using SPIFFE/SVID certificates issued by the control plane. Zero application changes. Cert rotation is automatic. Every service-to-service connection is encrypted and authenticated.
SPIFFE/SPIREService identity standard used by all major meshes. Each workload gets a cryptographic identity: spiffe://cluster.local/ns/default/sa/myapp. Authorization policies use this identity — not IP addresses, which can be spoofed.
mTLS without mesh = every service manages its own TLS, cert rotation, CA trust chain. With mesh: one control plane, automatic issuance, zero-touch rotation. This alone justifies mesh for regulated environments.
📊 The Observability Problem
Without meshDistributed tracing requires adding SDK to every service (OpenTelemetry, Zipkin, Jaeger client). Teams skip it. You have no visibility into latency between services, retries, error rates — unless every team instruments their service.
With meshThe proxy captures all L7 traffic automatically: request rate, error rate, latency percentiles (P50/P99), per-service-pair. You get the "golden signals" for every service combination without a single line of instrumentation code.
TracingMeshes inject trace headers (B3, W3C TraceContext) automatically. Services that propagate headers get full traces. Services that don't at least show up in the mesh topology graph.
TopologyEvery major mesh generates a live service topology map showing which services talk to which, with real-time traffic rates and error rates. Invaluable for incident response and capacity planning.
🚦 The Traffic Problem
Canary without meshDeploy two Deployments, split traffic via Service weights or an Ingress annotation. Coarse-grained: you can't split by HTTP header, user ID, or percentage at the connection level.
With meshFine-grained L7 traffic control: split 5% of requests matching header x-beta: true to v2. Split by user segment. Mirror traffic to shadow service. All via CRDs, no code changes.
Retries / timeoutsDefine retry policy once as a CRD — applies to every caller of a service. Without mesh, every team implements retries differently (or not at all). Mesh enforces consistent network behavior.
Circuit breakerAutomatically stop sending traffic to a failing upstream. Configurable per service, no library integration. The proxy makes the decision — not the calling service's code.
Service mesh ≠ API gateway. Mesh handles east-west traffic (service-to-service). API gateways handle north-south (ingress → services). They complement each other. Don't replace your Ingress with a mesh — use both.

How It Works
// sidecar model · ambient mode · data plane · control plane · xDS · SPIFFE
🔧 The Sidecar Model (Classic)
How injection worksInit container runs first, rewrites iptables rules to redirect all inbound/outbound traffic to the sidecar proxy port. App container starts normally — it sends to localhost:80, iptables catches it, proxy handles the rest.
Auto-injectionLabel the namespace: istio-injection: enabled or linkerd.io/inject: enabled. A MutatingWebhookConfiguration intercepts pod creation and injects the sidecar container automatically. Zero manifest changes needed.
Resource costEach sidecar consumes CPU + memory. Rough baseline: Envoy (Istio) ~50-100MB RAM + ~0.1 vCPU at idle. Linkerd's Rust proxy ~10-20MB RAM. At 1000 pods: significant overhead. This drove the invention of ambient mesh.
🌊 Ambient Mesh (Istio v1.21+)
The problemSidecars add a container per pod, consume resources, complicate upgrades, and increase latency per hop. For large clusters (1000+ pods) the overhead is substantial.
Ambient approachNo sidecar per pod. Instead: a ztunnel DaemonSet on each node handles L4 (mTLS, SPIFFE identity). Optionally: a waypoint proxy per namespace/service handles L7 (retries, routing) — only deployed when you need L7 features.
LayersSecure overlay layer: ztunnel = always-on mTLS + identity for all pods on the node. L7 processing layer: waypoint proxy = opt-in per service when you need retries, traffic splitting, auth policies.
BenefitsDramatically lower resource usage. No pod restart needed to add to mesh. Easier upgrades (upgrade ztunnel DaemonSet, not every pod). Graduated adoption: start with L4 security, add L7 per service as needed.
StatusAmbient mode became stable in Istio 1.24 (late 2024). It's the recommended path for new Istio installations. Sidecar mode still supported and widely deployed.
Ambient mesh is a architectural shift. ztunnel runs as a DaemonSet — one per node, not one per pod. waypoint proxies are Envoy instances that sit in the path only for services that opt into L7 features. This is fundamentally different from the sidecar model.
Ambient + sidecar can coexist in a cluster during migration. You can run some namespaces in sidecar mode and others in ambient mode. This lets you migrate gradually — critical for large production clusters.
🎛️ Control Plane vs Data Plane
Control planeThe brain. Manages certificate issuance (CA), distributes configuration to proxies via xDS API, enforces policy, handles service discovery. Runs as a deployment — not in the data path.
Data planeThe body. The actual proxies (sidecars or ztunnel) that handle every packet. They enforce what the control plane configures. If the control plane is down, data plane still serves traffic using last-known config.
xDS APIEnvoy's configuration protocol. Control plane (istiod) pushes LDS (listeners), RDS (routes), CDS (clusters), EDS (endpoints) to each Envoy proxy. Any control plane that speaks xDS can manage Envoy proxies.
Failure modesControl plane down: proxies keep serving with last config. No new cert issuance, no config updates. Data plane down on a node: pods on that node lose mesh features (depending on mode). Neither failure = total outage.
🪪 SPIFFE & SVID Identity
SPIFFESecure Production Identity Framework for Everyone. An open standard for workload identity. Defines how workloads get cryptographic identities, independent of the runtime environment.
SVIDSPIFFE Verifiable Identity Document. A certificate with a URI SAN: spiffe://trust-domain/ns/namespace/sa/serviceaccount. Short-lived (1h by default), auto-rotated by the control plane.
AuthorizationPolicies reference SPIFFE identities: "allow spiffe://.../payments to call spiffe://.../orders on /checkout". Identity-based, not IP-based. IP addresses change. SPIFFE identities don't.
Zero TrustmTLS + SPIFFE = every connection authenticated + encrypted + authorized by policy. This is the technical foundation of Zero Trust networking inside a cluster.

Istio Deep Dive
// most feature-rich · ambient mesh (v1.24) · CRDs · VirtualService · AuthorizationPolicy
⛵ Architecture (Ambient Mode)
istiodSingle control plane binary. Combines: Pilot (xDS config distribution), Citadel (certificate authority), Galley (config validation). Issues mTLS certs, pushes config to all proxies via gRPC xDS stream.
ztunnelDaemonSet, one pod per node. Handles L4 for all pods on the node: mTLS, SPIFFE identity, connection-level metrics. Transparent to pods — no sidecar injection needed.
Waypoint proxyOpt-in Envoy instance per namespace/service for L7 features. Deploy: istioctl waypoint apply. All traffic to a service routes through its waypoint. Only needed when you want retries, traffic splitting, or L7 auth policies.
EnvoyThe data plane proxy (in sidecar mode). Most fully-featured L7 proxy. Supports gRPC, WebSocket, HTTP/1/2/3, TCP. High performance but higher resource footprint than Linkerd's Rust proxy.
KialiOfficial Istio observability dashboard. Service graph, traffic metrics, configuration validation, Jaeger tracing integration. Install separately but purpose-built for Istio.
# Install Istio (ambient mode — recommended) istioctl install --set profile=ambient # Enable ambient for a namespace kubectl label namespace default istio.io/dataplane-mode=ambient # Deploy waypoint for L7 features istioctl waypoint apply --namespace default # Check mesh status istioctl status istioctl proxy-status # Verify mTLS (sidecar mode) istioctl authn tls-check svc-a.default.svc.cluster.local
📋 Key CRDs
VirtualServiceTraffic routing rules: split by weight, match by header, retry policy, timeout. Applied to outgoing traffic from callers.
DestinationRulePolicy for traffic to a specific service: load balancing, circuit breaker thresholds, TLS mode, connection pool limits.
AuthorizationPolicyAllow/deny rules based on source identity (SPIFFE), method, path, headers. Zero-trust policy enforcement.
GatewayConfigure inbound/outbound traffic for the mesh edge. Exposes services outside the cluster through the ingress gateway.
PeerAuthenticationSet mTLS mode per namespace/workload: STRICT (mTLS required), PERMISSIVE (mTLS or plaintext), DISABLE.
# Traffic split: 90% stable, 10% canary kind: VirtualService spec: hosts: [myapp] http: - route: - { destination: { host: myapp, subset: stable }, weight: 90 } - { destination: { host: myapp, subset: canary }, weight: 10 } retries: attempts: 3 perTryTimeout: 2s retryOn: 5xx,reset,connect-failure # AuthorizationPolicy: zero trust kind: AuthorizationPolicy spec: action: ALLOW rules: - from: - source: principals: ["cluster.local/ns/payments/sa/payments-api"] to: - operation: methods: [GET] paths: ["/orders/*"]
⚖️ Istio Strengths & Weaknesses
✅ StrengthsMost feature-complete mesh. Best L7 traffic management. Ambient mode solves the resource overhead problem. Large community + CNCF graduated. Envoy's ecosystem.
⚠️ WeaknessesHistorically complex. Many CRDs, steep learning curve. Sidecar mode overhead at scale. Debugging Envoy config is hard. Configuration errors silently break things.
Best forLarge, complex microservices environments. Organizations needing full L7 traffic management, zero-trust auth policies, multi-protocol support (gRPC, HTTP/2). Ambient mode for large clusters.
Permissive mTLS mode is a trap. It's the default in many setups — allows plaintext traffic to bypass mTLS. Set to STRICT in production: kubectl apply -f peer-auth-strict.yaml. Otherwise you have mTLS theater — services think they're secure but aren't.

Linkerd Deep Dive
// simplicity first · Rust proxy · ultra-low overhead · CNCF graduated
🔗 Architecture
linkerd2-proxyLinkerd's data plane is not Envoy — it's a purpose-built Rust proxy. Ultra-lightweight: ~10MB RAM at idle vs Envoy's 50-100MB. Dramatically lower CPU overhead. Built specifically for HTTP/gRPC, TCP.
Control planeThree components: destination (service discovery, policy), identity (certificate CA, SPIFFE issuance), proxy-injector (webhook for sidecar injection). Minimal footprint.
PolicyAuthorization policies via Server and AuthorizationPolicy CRDs. Simpler than Istio but covering the common cases. Focused on "allow/deny based on identity" rather than complex routing rules.
No ambient modeLinkerd uses sidecars only (as of 2026). The Rust proxy is so lightweight that the sidecar overhead argument is largely moot — 10MB per pod is far less painful than 100MB.
Viz extensionLinkerd's observability dashboard. Real-time service graph, golden signals (success rate, requests/sec, P99 latency), live request tapping, and per-route metrics. Installed separately as an extension.
# Install Linkerd linkerd install --crds | kubectl apply -f - linkerd install | kubectl apply -f - linkerd check # Inject a namespace kubectl annotate namespace default \ linkerd.io/inject=enabled # Or inject a deployment kubectl get deploy myapp -o yaml | \ linkerd inject - | kubectl apply -f - # Observe live traffic linkerd viz stat deploy linkerd viz top deploy/myapp # real-time top linkerd viz tap deploy/myapp # live request tap # Check proxy health linkerd check --proxy -n default
Linkerd's viz tap is uniquely powerful. It shows you live HTTP requests passing through the mesh in real time — method, path, response code, latency. Zero additional tooling. Indispensable for debugging in production.
📋 Key CRDs (Linkerd SMI-compatible)
ServerDefines which ports/protocols a workload exposes. Required for policy to work. Acts as the "identity anchor" for a service.
AuthorizationPolicyAllow specific clients (by identity or namespace) to access a Server. Default: deny all in locked-down mode.
HTTPRouteGateway API compatible. Fine-grained HTTP routing: path matching, header matching, traffic splitting, retries, timeouts.
ServiceProfileLegacy CRD: per-route metrics, retries, timeouts per endpoint. Replaced by HTTPRoute in newer Linkerd versions.
# Server: define what myapp exposes apiVersion: policy.linkerd.io/v1beta3 kind: Server spec: podSelector: matchLabels: { app: myapp } port: 8080 proxyProtocol: HTTP/2 # AuthorizationPolicy: only allow payments service kind: AuthorizationPolicy spec: targetRef: group: policy.linkerd.io kind: Server name: myapp-server requiredAuthenticationRefs: - name: payments-auth kind: MeshTLSAuthentication --- # HTTPRoute: traffic split kind: HTTPRoute spec: rules: - backendRefs: - { name: myapp-stable, weight: 90 } - { name: myapp-canary, weight: 10 } timeouts: request: 10s retry: limit: 3 timeout: 2s
⚖️ Linkerd Strengths & Weaknesses
✅ StrengthsSimplest production mesh. Ultra-low resource overhead (Rust proxy). Much easier to operate than Istio. Fewer CRDs, smaller attack surface. Excellent UX (viz, tap). CNCF graduated.
⚠️ WeaknessesFewer features than Istio. No Envoy ecosystem. L7 features less mature. Multi-cluster support is good but simpler than Kuma. No ambient mode — sidecars required.
Best forTeams that want "mesh without the complexity." Kubernetes-only environments. Organizations where resource efficiency matters. Teams that want operational simplicity over feature depth.
Linkerd 2.x dropped Helm support in some versions — use the CLI for install. Also: Linkerd Enterprise is separate from the open-source version. Some features (FIPS compliance, certain extensions) are enterprise-only. Check before assuming parity.

Kuma Deep Dive
// multi-zone native · Envoy-based · universal mode · Kong-backed · CNCF
🦊 Architecture
Control planekuma-cp: the control plane. Runs in standalone mode (single cluster) or global + zone mode (multi-cluster). Zone control planes connect to a global CP. Most distinctive feature: multi-zone is first-class, not an afterthought.
Data planekuma-dp — an Envoy wrapper. Handles the xDS communication with kuma-cp and manages the Envoy process. Same rich L7 feature set as Istio (they both use Envoy) without Istio's historical complexity.
Universal modeKey differentiator. Kuma runs on Kubernetes AND on VMs/bare-metal. Same control plane, same policies, same observability. A single Kuma CP can manage services running on K8s pods and services running on legacy VMs simultaneously.
Multi-zoneBuilt for multi-cluster and multi-cloud. Global control plane coordinates multiple zone CPs across regions. Cross-zone traffic is automatically routed, with locality-aware load balancing and zone health awareness.
PoliciesKuma uses a policy model based on MeshTrafficPermission, MeshRetry, MeshTimeout, MeshHTTPRoute, etc. Policies can target by tag selectors across all zones simultaneously — very powerful for fleet-wide rules.
# Install Kuma (K8s mode) kumactl install control-plane | kubectl apply -f - kumactl install observability | kubectl apply -f - kubectl wait -n kuma-system \ deploy/kuma-control-plane --for=condition=Available # Enable namespace for mesh kubectl label namespace myapp kuma.io/sidecar-injection=enabled # Multi-zone: install global CP kumactl install control-plane \ --mode=global | kubectl apply -f - # Add a zone (another cluster) kumactl install control-plane \ --mode=zone \ --zone=us-east-1 \ --kds-global-address=grpcs://global-cp:5685 | \ kubectl apply -f - # Check zones kumactl get zones kumactl get dataplanes --mesh default
Kuma's tag-based policy targeting is powerful. Policies use Kuma's tag system: kuma.io/service: myapp, version: v2, env: prod. Apply a retry policy to "all services tagged env=prod" across all zones in one policy object.
📋 Key Policies
# MeshTrafficPermission (zero trust) kind: MeshTrafficPermission spec: targetRef: kind: MeshService name: orders from: - targetRef: kind: MeshService name: payments default: action: Allow # MeshRetry across all services in mesh kind: MeshRetry spec: targetRef: kind: Mesh # applies to ALL services to: - targetRef: kind: Mesh default: http: numRetries: 3 retryOn: [5xx, reset, connect-failure] perTryTimeout: 2s # MeshHTTPRoute: traffic splitting kind: MeshHTTPRoute spec: targetRef: kind: MeshService name: myapp to: - targetRef: { kind: MeshService, name: myapp } rules: - backendRefs: - { kind: MeshService, name: myapp-v1, weight: 90 } - { kind: MeshService, name: myapp-v2, weight: 10 }
⚖️ Kuma Strengths & Weaknesses
✅ StrengthsMulti-zone native. Universal mode (K8s + VMs). Fleet-wide policy targeting by tags. Simpler than Istio but more powerful than Linkerd for multi-cluster. Kong ecosystem integration.
⚠️ WeaknessesSmaller community than Istio or Linkerd. Less ecosystem tooling. Still Envoy under the hood — resource overhead similar to Istio. Less "battle-tested" at extreme scale.
Best forMulti-cluster, multi-cloud, or hybrid (K8s + VMs) environments. Organizations already using Kong API gateway (integration is seamless). Teams needing fleet-wide policy management across many clusters.
Kuma's "Universal mode" requires a Postgres database for the control plane state (in standalone or global mode for non-K8s). The K8s mode uses K8s CRDs as storage — no external DB. Make sure you're clear on which mode you're deploying before going to production.

⛵ Choose Istio When
Need full feature setComplex traffic routing (header-based, weighted, mirror), JWT authentication, Wasm plugin extensibility, advanced authorization policies.
Large scale500+ services or 1000+ pods — use ambient mode to eliminate per-pod overhead. Ambient ztunnel scales much better than per-pod sidecars.
CompliancePCI DSS, HIPAA, SOC2 environments that need proven, CNCF-graduated mesh with audit trail, fine-grained authorization policies, and mTLS enforcement.
Multi-protocolMixed HTTP/1, HTTP/2, gRPC, TCP workloads that all need consistent policy enforcement.
If you can afford the learning curve and need the full feature set — Istio ambient is the current state of the art. It eliminates the historical resource overhead argument against Istio.
🔗 Choose Linkerd When
Simplicity mattersSmall-to-medium cluster (under 500 pods). Team doesn't have dedicated platform engineers. Want mesh benefits without mesh complexity.
Resource constrainedEvery MB matters (edge clusters, small nodes). Linkerd's Rust proxy at 10-20MB vs Envoy's 50-100MB is a meaningful difference at scale.
K8s-onlyNo VMs, no multi-cluster (or simple multi-cluster), no need for Wasm extensibility. Pure Kubernetes environments.
Developer experiencelinkerd viz tap and linkerd viz top are genuinely excellent debugging tools. If developer productivity is a priority, Linkerd's UX wins.
Linkerd is the "start here" mesh for most teams. If you outgrow it, migrating to Istio is easier than starting with Istio and fighting its complexity from day one.
🦊 Choose Kuma When
Multi-cluster firstManaging 3+ clusters across regions or cloud providers. Kuma's global+zone CP model is purpose-built for this — not bolted on later like Istio's multi-cluster.
Hybrid (K8s + VMs)Legacy services running on VMs that need to communicate securely with K8s workloads. Kuma's Universal mode manages both from a single control plane.
Kong shopsAlready using Kong API Gateway? Kuma is made by the same company, integrates natively. Consistent data plane (Envoy) across ingress and mesh.
Fleet-wide policyNeed to apply a single retry policy to all services across all clusters simultaneously. Kuma's tag-based global policy targeting is uniquely powerful for this.

When, Why & Decision Guide
// when NOT to use mesh · the decision tree · migration strategy · real scenarios
🚦 When NOT To Use Service Mesh
3 services or fewerDon't add mesh to a monolith or tiny service set. The complexity and resource overhead vastly outweigh the benefits. You don't need mTLS automation for 2 services — just configure it manually.
Dev/test environmentsMesh in dev is overhead without benefit. Dev teams need fast iteration, not 10+ second cold starts from sidecar injection. Keep mesh in staging and above.
Simple internal trafficIf all your services run in a single trusted namespace and you have no compliance requirements, the network already provides isolation. Mesh is overkill.
Team isn't readyIf your team doesn't understand what a mesh does, adding one creates a new class of production incidents they won't know how to debug. Mesh is an operations multiplier — in both directions.
Performance critical L3Ultra-low latency workloads (HFT, real-time gaming servers, media streaming) may find even the ~0.2ms proxy overhead unacceptable. Measure first.
🌱 When TO Use Service Mesh
10+ microservicesThe complexity threshold. Once you have 10+ services communicating with each other, the operational value of unified observability, consistent retry policies, and automatic mTLS pays off immediately.
Compliance requirementsPCI DSS, HIPAA, SOC2, FedRAMP — any framework requiring encrypted in-transit data and audit trail of service-to-service communication. Mesh proves it with zero code changes.
Zero trust mandateOrganization moving to Zero Trust networking. Mesh provides the cryptographic identity layer (SPIFFE) and allows AuthorizationPolicies that enforce "least privilege" between services.
Multiple teams, multiple servicesWhen 5+ teams each own services and each team implements network concerns differently (retries, timeouts, auth) — mesh enforces consistency without requiring teams to change their code.
Canary / progressive deliveryIf you're using Argo Rollouts or Flagger for canary deployments, a mesh provides the precise traffic splitting they need at the L7 level.
🔍 Decision Tree

🛤️ Mesh Adoption Strategy
Phase 1: Observe
Install meshInstall with PERMISSIVE mTLS mode. Enable injection on one namespace. Zero changes to applications.
ObserveWatch the service topology appear. See where traffic is flowing, which services are communicating, baseline latencies and error rates.
Fix issuesSome apps have health checks that break with sidecar injection (timing, port conflicts). Fix before proceeding.
Phase 2: Secure
Enable namespaceInject remaining namespaces. Still PERMISSIVE — apps that don't support mTLS won't break.
Strict mTLSSwitch to STRICT per namespace as you verify each service handles mTLS correctly. PeerAuthentication policy per namespace.
AuthZ policiesStart with deny-all default, then add explicit ALLOW rules between services. Zero trust by default.
Phase 3: Control
Traffic policiesAdd retry policies, circuit breakers, timeouts via mesh CRDs. Remove the same logic from application code over time.
Canary infraConfigure Argo Rollouts or Flagger to use the mesh for traffic splitting. Progressive delivery now works at L7.
Tune & optimizeAmbient mode (Istio): migrate namespaces from sidecar to ambient for resource savings. Fine-tune timeouts based on observed P99 latencies.

Traps & Quiz
// 12 operational gotchas · interactive quiz
01
mTLS
Mesh is installed, mTLS "enabled" — but traffic is still plaintext. Why?
PERMISSIVE mode allows both plaintext and mTLS. This is the default in Istio to ease adoption. Services that don't have the sidecar can still communicate. For actual encryption: set PeerAuthentication to STRICT per namespace. Without STRICT, you have mTLS theater.
02
Health checks
Pods crash on startup after enabling sidecar injection. Health check failures.
Sidecar starts before the app is ready — or vice versa. iptables rules redirect traffic immediately, but the app isn't listening yet. Fix: set holdApplicationUntilProxyStarts: true in Istio, or add startupProbe with enough delay. Also: kubelet bypasses mesh for health checks by default (port-based) but some configs break this.
03
Observability
Mesh topology shows all traffic going through one service — looks like a star. Is that right?
Likely a missing sidecar on some services. Traffic between two sidecars shows both ends. If service B has no sidecar, all traffic to B appears to come from the last sidecar-enabled hop. Check: kubectl get pods -n myns -o jsonpath='{.items[*].spec.containers[*].name}' — every pod should have the proxy container.
04
AuthZ Policy
You apply a deny-all AuthorizationPolicy. Suddenly health checks from the kubelet fail too.
Kubelet health checks bypass the mesh — but some configurations don't. Istio automatically excludes kubelet health check ports from mTLS interception (via excludeInboundPorts). But if your deny-all policy is too aggressive, kubelet probes can be caught. Fix: explicitly allow ingress from notPrincipals for health check ports, or use ALLOW policies instead of DENY.
05
Istio CRDs
VirtualService is applied but traffic still goes 100% to the old service. No split.
VirtualService requires matching DestinationRule subsets. If you reference subset: canary in VirtualService but don't define that subset in a DestinationRule, Istio silently ignores the rule. Also: VirtualService must match the hostname exactly as called by clients. myapp ≠ myapp.default.svc.cluster.local in some contexts.
06
Resource overhead
Mesh deployed on 800-pod cluster. Node CPU spikes with no application load change. Why?
Envoy sidecar overhead. 800 Envoy sidecars × 50-100MB RAM + CPU for xDS config reconciliation when istiod pushes updates (e.g., service endpoint changes). Fix: (1) migrate to Istio ambient mode — eliminates per-pod Envoy, (2) tune PILOT_PUSH_THROTTLE to reduce istiod push rate, (3) use Linkerd's Rust proxy instead.
07
Tracing
Mesh shows traces but each trace has only one span — no parent-child relationships between services.
Applications aren't propagating trace headers. The mesh injects and reads B3/W3C trace headers, but the app must forward them between inbound and outbound requests. If Service A receives x-request-id: abc and makes a call to B without forwarding it, B's span appears as a new trace. Fix: use OpenTelemetry auto-instrumentation or manually propagate headers in each service.
08
Upgrade
After upgrading Istio control plane, some pods still show the old proxy version. Traffic works but metrics are inconsistent.
Sidecars don't auto-upgrade — pods must be restarted. Istio upgrades the control plane (istiod) but sidecars in running pods stay at the old version until those pods restart. Rolling restart: kubectl rollout restart deployment -n mynamespace. Ambient mode (ztunnel DaemonSet) upgrades all pods on a node simultaneously without pod restarts — another advantage of ambient.
09
Linkerd
Linkerd installed but linkerd viz tap shows no traffic for a deployment.
The namespace isn't annotated for injection, or pods predate the annotation. Check: kubectl get namespace myns -o yaml | grep inject. If the annotation was added after pods started, existing pods don't have the sidecar. Restart them: kubectl rollout restart deployment -n myns. Also verify with linkerd check --proxy -n myns.
10
Kuma multi-zone
Services in zone-A can't reach services in zone-B even though both zones are healthy.
Zone Ingress not configured or network connectivity between zones missing. Kuma requires each zone to have a ZoneIngress (a gateway for cross-zone traffic). Also: the network between zone CPs and the global CP must allow traffic on port 5685 (KDS). Check: kumactl get zone-ingresses and verify the global CP can reach each zone CP.
11
Ambient
Migrating namespace to Istio ambient mode — existing pods break. What happened?
Pods need to be restarted after enabling ambient for the namespace. The ztunnel intercepts traffic at the node network level, but pods that started before ambient was enabled on the namespace may not be registered with ztunnel correctly. Restart all pods in the namespace after labeling it for ambient. Also: check ztunnel logs for enrollment errors.
12
Common mistake
Team adds mesh "for observability" then uses Datadog for metrics, Jaeger for tracing, and ignores mesh dashboards. Did they need mesh?
Probably not just for observability. If you already have APM (Datadog/New Relic) with agents in every service, and a tracing library already integrated — mesh observability is redundant. The real mesh value propositions are: mTLS encryption, SPIFFE identity, zero-trust authorization policies, and L7 traffic control. Don't pay mesh overhead to duplicate what you already have.
