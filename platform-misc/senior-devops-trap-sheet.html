Kubernetes
17 TRAPS
CRITICAL
K8S-01Does the Deployment controller create Pods?
✗ WRONGYes — Deployment manages Pods directly.
✓ RIGHTNo. Deployment ctrl creates ReplicaSets. ReplicaSet ctrl creates Pods. Two-step chain. Deployment ctrl never touches a Pod object.
This is the #1 trap. The chain: Deployment → RS → Pod. Each step is a separate controller watching its own objects.
CRITICAL
K8S-02Do different controllers in kube-controller-manager elect different leaders?
✗ WRONGYes, each controller has its own leader election.
✓ RIGHTNo. The entire binary has ONE Lease object. All 30+ controllers start and stop as a unit. Per-controller leader election doesn't exist here.
cloud-controller-manager has its own separate Lease — because it's a separate binary.
CRITICAL
K8S-03Does a controller read object state from etcd?
✗ WRONGYes, via the etcd client library for freshest data.
✓ RIGHTNo. Controllers only read from the informer's in-memory cache via Lister. Cache = O(1). Direct API calls inside reconcile saturate apiserver.
Only apiserver writes to etcd. Controllers → apiserver → (apiserver caches + etcd). Controllers never touch etcd directly.
CRITICAL
K8S-04Does the scheduler start containers on the node?
✗ WRONGYes, it schedules pods to nodes = starts them.
✓ RIGHTNo. Scheduler only writes nodeName to the Pod spec. kubelet watches for pods with nodeName=thisNode and calls CRI to start containers.
Schedule = assignment only. Container lifecycle = kubelet's job via CRI gRPC → containerd → runc.
HIGH
K8S-05Can you configure a shorter CrashLoopBackOff delay?
✗ WRONGYes, via restartDelay or pod spec settings.
✓ RIGHTNo. Backoff is hardcoded: 10s → 20s → 40s → 80s → 160s → 300s max. Resets after 10 min stable. Use startupProbe for slow-start apps instead.
The backoff protects nodes from tight crash loops consuming CPU. No override. Design your app to fail fast or use startupProbe.
CRITICAL
K8S-06Does kubectl apply replace the full object?
✗ WRONGYes, it PUTs the full manifest, overwriting everything.
✓ RIGHTNo. apply uses Strategic Merge Patch (SMP) by default. Merges lists by strategic key (containers by name). SSA sends full desired state; apiserver manages field ownership.
kubectl replace = full PUT. kubectl apply = SMP patch. kubectl apply --server-side = SSA. Mixing them causes field manager conflicts.
HIGH
K8S-07Does deleting a pod automatically delete its PVC in a StatefulSet?
✗ WRONGYes, PVC is bound to the pod lifecycle.
✓ RIGHTNo. PVC persists. StatefulSet ctrl creates a new pod with the same name that reattaches the same PVC. Data survives pod deletion/reschedule.
PVCs are only deleted if you delete the StatefulSet with --cascade=foreground AND the storageClass reclaimPolicy=Delete. Or delete them manually.
HIGH
K8S-08A Service with selector: {} (empty) matches all pods?
✗ WRONGEmpty selector = select everything in namespace.
✓ RIGHTEmpty {} matches no pods. EndpointSlice will be empty. Absent selector field = headless/manual endpoints. A single typo kills all traffic.
This is a silent failure. Service exists, pods exist, no traffic flows. Always check kubectl get endpoints <svc> after creating a Service.
HIGH
K8S-09Does a ClusterIP exist on a real network interface?
✗ WRONGYes, it's assigned to a virtual network interface on each node.
✓ RIGHTNo. ClusterIP only exists in iptables/IPVS rules. ip addr show won't show it. Packets destined for ClusterIP are rewritten by netfilter to a real pod IP before routing.
kube-proxy programs iptables DNAT rules: ClusterIP:port → PodIP:port. The VIP is a fiction enforced by the kernel.
CRITICAL
K8S-10A pod is Running but gets no Service traffic. What's likely wrong?
✗ WRONGThe pod IP isn't registered. Check DNS.
✓ RIGHTreadinessProbe is failing. Pods not Ready are removed from EndpointSlices → no traffic. Also check label selector mismatch with kubectl get endpoints.
Running ≠ Ready. Running = container is alive. Ready = readinessProbe passing + included in Endpoints. They're independent.
HIGH
K8S-11Node goes NotReady — pods are evicted immediately?
✗ WRONGYes, immediately to protect workloads.
✓ RIGHTNo. 40s grace (node-monitor-grace-period) → Unknown → 300s taint toleration → eviction. Rate-limited. Evictions stop if <55% of cluster nodes are healthy.
Total worst-case: 340 seconds before pods move. Tune with --pod-eviction-timeout and tolerations on your pods.
HIGH
K8S-12An object with a deletionTimestamp set is being deleted?
✗ WRONGYes, the object is in the process of being deleted.
✓ RIGHTIt is frozen, not yet deleted. If finalizers list is non-empty, the object stays in etcd until a controller removes all finalizers. It can stay this way forever if the controller is dead.
Force clear: kubectl patch <obj> -p '{"metadata":{"finalizers":[]}}' --type=merge. Only do this if the owning controller is confirmed gone.
MED
K8S-13DaemonSet pods are always scheduled by kube-scheduler?
✗ WRONGYes, scheduler places all pods including DaemonSet pods.
✓ RIGHTDaemonSet ctrl sets nodeName directly, bypassing Filter+Score phases. It can place pods on unschedulable nodes (e.g., CNI needs to run before anything else can schedule).
DaemonSet pods on control-plane nodes blocked by node-role.kubernetes.io/control-plane:NoSchedule taint unless you add explicit toleration.
HIGH
K8S-14Job with restartPolicy: Always — will it ever complete?
✗ WRONGYes, it restarts failed pods until success.
✓ RIGHTNo. Job controller and kubelet fight forever. Pod crashes → kubelet restarts (Always) → Job ctrl sees running pod, never increments success count. Job never completes. Use Never or OnFailure.
This is a silent infinite loop. The pod appears healthy (always restarting), the Job never fails or succeeds. Resource leak.
MED
K8S-15Does a Deployment rollback create a new ReplicaSet?
✗ WRONGYes, it creates a new RS with the old template.
✓ RIGHTNo. Old RSes are preserved with their pod templates. Rollback just scales up the old RS and scales down the current one. No new RS created. Revision history kept up to revisionHistoryLimit.
kubectl rollout undo deploy/myapp --to-revision=3 scales RS with revision annotation 3. This is why old RSes with 0 replicas must be kept.
HIGH
K8S-16Does progressDeadlineSeconds trigger an automatic rollback?
✗ WRONGYes, Deployment auto-rolls back after the deadline.
✓ RIGHTNo. It only sets condition Progressing=False on the Deployment. You must manually kubectl rollout undo or detect the condition in your CD pipeline and act.
K8s has no built-in auto-rollback. Argo Rollouts, Flagger, or your own operator can implement it. Native Deployment does not.
HIGH
K8S-17Does updating a Deployment's status via regular Update() persist to etcd?
✗ WRONGYes, Update() writes the full object including status.
✓ RIGHTNo. Status is a separate subresource. Update() ignores status changes. Use client.Status().Update(). Mixing them causes resourceVersion conflicts and silent status loss.
The spec/status split is enforced by apiserver. It's intentional: spec = what you want, status = what the controller observed. They have separate access controls too.
Kafka
11 TRAPS
CRITICAL
KFK-01Is message ordering guaranteed across all partitions in a topic?
✗ WRONGYes, Kafka guarantees ordering within a topic.
✓ RIGHTOrdering guaranteed within a partition only. Across partitions: zero ordering guarantee. To order events per entity (device, user), use a consistent partition key — same key always hits same partition.
For attribution events: partition by device_id. All events for the same device → same partition → ordered. Different devices can be on different partitions.
CRITICAL
KFK-02More consumers than partitions in a group — do extra consumers help throughput?
✗ WRONGYes, more consumers = more parallel processing.
✓ RIGHTExtra consumers sit idle. Each partition is assigned to exactly one consumer per group. Consumer count beyond partition count = wasted resources. Partition count = max parallelism ceiling.
Before scaling consumers past partition count, increase partitions first. Partition count can be increased but not decreased without recreating the topic.
CRITICAL
KFK-03Should you commit offset before processing the message?
✗ WRONGYes, commit first = at-most-once, prevents reprocessing.
✓ RIGHTCommit AFTER processing = at-least-once (safe). Commit BEFORE = at-most-once (lose messages on crash). At-least-once requires idempotent consumers to handle duplicates.
For attribution data: always at-least-once. Deduplicate by event ID in the consumer or downstream store. A missing install event is worse than a duplicate.
CRITICAL
KFK-04Is unclean.leader.election.enable=true safe for an attribution pipeline?
✗ WRONGYes, it ensures availability when ISR is small.
✓ RIGHTNo. An out-of-sync replica becoming leader = guaranteed message loss. For financial/attribution data: always false. Accept unavailability over data loss.
Pair with: min.insync.replicas=2, acks=all, RF=3. The trio that prevents message loss at the cost of higher latency.
HIGH
KFK-05Do compacted topics retain full message history?
✗ WRONGYes, compacted = compressed but all messages kept.
✓ RIGHTNo. Compaction keeps only the latest value per key. Old values for the same key are deleted during compaction. Used for changelog/CDC patterns — current state, not event history.
A tombstone message (null value) for a key eventually removes that key entirely after compaction. Use delete policy for event history, compact for current-state snapshots.
HIGH
KFK-06Does Kafka provide exactly-once semantics by default?
✗ WRONGYes, Kafka is exactly-once from the start.
✓ RIGHTDefault is at-least-once. Exactly-once requires: enable.idempotence=true on producer + Kafka Transactions API. More complex, higher latency. Most systems use at-least-once + idempotent consumers.
Idempotent producer prevents duplicates from retries. Transactions extend this across multiple partitions/topics. Both are opt-in.
HIGH
KFK-07In KEDA: is lagThreshold the total lag before scaling triggers?
✗ WRONGYes, trigger when total consumer group lag exceeds threshold.
✓ RIGHTIt's per-replica. Formula: desired = ceil(totalLag / lagThreshold). threshold=50k with 3 replicas tolerates 150k total lag before scaling. Easy to miscalculate.
Also: maxReplicaCount should equal partition count. More consumer pods than partitions = idle pods. Scale up partitions before scaling past the ceiling.
HIGH
KFK-08Can you decrease the number of partitions in a Kafka topic?
✗ WRONGYes, same as adding — just specify fewer in the config.
✓ RIGHTNo. Partition count can only increase. Decreasing requires deleting and recreating the topic. Data in the removed partitions is lost. Plan partition count carefully upfront.
Rule of thumb: overprovision partitions. They're cheap. Consumer parallelism is bounded by partition count, and you can never reduce it later.
MED
KFK-09Consumer lag of 0 means your consumers are healthy?
✗ WRONGYes, lag=0 = fully caught up = healthy.
✓ RIGHTLag=0 means caught up, but consumers could be dead. If producers stop AND consumers stop, lag stays 0. Alert on both lag growth rate AND consumer heartbeat absence (ISR changes, missing group commits).
Use Burrow for trend-based lag alerting, not raw threshold. A dead consumer group looks identical to a caught-up consumer group from the lag metric alone.
HIGH
KFK-10Rolling restart of Kafka consumer pods triggers a rebalance — is there a way to avoid it?
✗ WRONGNo, every pod restart always triggers a full rebalance.
✓ RIGHTUse group.instance.id (static membership). Consumer retains partition assignment across restarts without rebalance if it reconnects within session.timeout.ms.
Critical for K8s rolling updates. Without static membership: every pod restart causes 2 rebalances (leave + rejoin). With static membership: zero rebalances during rolling update.
MED
KFK-11Does KEDA work alongside a manually created HPA on the same deployment?
✗ WRONGYes, they can both manage the same deployment simultaneously.
✓ RIGHTNo. They fight over spec.replicas and produce chaotic scaling. KEDA creates and owns its own HPA internally. Never add a separate HPA to a KEDA-managed deployment.
Delete any manual HPA before applying a ScaledObject. kubectl get hpa — if one exists for your target, KEDA and it will conflict.
Linux
10 TRAPS
CRITICAL
LNX-01Can a process catch or ignore SIGKILL (kill -9)?
✗ WRONGYes, a process can set a signal handler for any signal.
✓ RIGHTNo. SIGKILL is handled exclusively by the kernel. Process is not notified, cannot catch it, cannot block it, cannot ignore it. Immediate termination. No cleanup possible.
SIGTERM (15) = graceful — process can catch and clean up. SIGKILL (9) = unconditional. K8s sends SIGTERM first, waits terminationGracePeriodSeconds, then SIGKILL.
HIGH
LNX-02Can you kill a zombie process with kill -9?
✗ WRONGYes, -9 kills anything.
✓ RIGHTNo. A zombie is already dead — it has no running code to kill. It's an entry in the process table waiting for its parent to wait(). Kill the parent, or fix the parent's code to reap children.
Zombie processes consume only a PID slot, not CPU/memory. Dangerous only at scale — if you fill the PID table (default 32768), new processes can't start.
HIGH
LNX-03Does the OOM killer pick the process using the most memory to kill?
✗ WRONGYes, it kills the biggest memory consumer first.
✓ RIGHTOOM killer uses an oom_score (0-1000). Score based on memory usage, nice value, child processes, and oom_score_adj. Highest score is killed. Processes can set oom_score_adj=-1000 to be protected.
K8s sets oom_score_adj based on QoS class: BestEffort=1000 (first to die), Burstable=2-999, Guaranteed=-997 (protected). This is why resource limits matter.
HIGH
LNX-04Disk shows free space but processes fail to create files — what's wrong?
✗ WRONGFile system corruption — run fsck.
✓ RIGHTInode exhaustion. Each file consumes one inode regardless of size. A million tiny files (logs, temp files) exhausts inodes while blocks remain free. Check with df -i.
df -h shows block usage. df -i shows inode usage. Find directories with many small files: find / -xdev -printf '%h\n' | sort | uniq -c | sort -k 1 -n | tail -10
HIGH
LNX-05Does ulimit -n 65536 in your shell apply to all processes?
✗ WRONGYes, ulimit sets a system-wide limit.
✓ RIGHTulimit is per-process and inherited by children of that shell. System-wide limits are in /proc/sys/fs/file-max. Services need limits set in systemd unit (LimitNOFILE=) or Docker/K8s resource limits.
Common trap: you increase shell ulimit, but your Kafka broker (started by systemd) still has the old limit. Always check the process's actual limits: cat /proc/<pid>/limits.
MED
LNX-06Is a high number of TCP TIME_WAIT sockets a problem?
✗ WRONGYes, TIME_WAIT sockets waste resources and should be eliminated.
✓ RIGHTTIME_WAIT is normal and protective — prevents old packets from corrupting new connections. High count means high throughput, not a bug. Only a problem if you exhaust ephemeral ports (check ss -s).
net.ipv4.tcp_tw_reuse=1 allows reuse for outbound connections. net.ipv4.ip_local_port_range=1024 65535 widens ephemeral port range. Don't blindly enable tw_recycle (removed in Linux 4.12).
HIGH
LNX-07Does fork() immediately copy the parent's entire memory?
✗ WRONGYes, fork creates a full copy of all memory pages.
✓ RIGHTCopy-on-Write (CoW). Pages are shared read-only between parent and child. Physical copy only happens when either process writes to a page. Fork of a 4GB Redis is fast and cheap — until you start writing.
Redis BGSAVE uses fork() + CoW. The concern: if Redis is under heavy write load during BGSAVE, CoW copies can cause memory to double temporarily. Monitor used_memory during saves.
MED
LNX-08Are cgroups v1 and v2 interchangeable in modern systems?
✗ WRONGYes, v2 is just a newer version of the same thing.
✓ RIGHTv2 is a complete redesign. Unified hierarchy (one mount point vs many). Memory QoS requires v2. K8s requires v2 for MemoryQoS feature gate. containerd and runc both require v2 for full feature set.
Check: cat /sys/fs/cgroup/cgroup.controllers exists = v2. ls /sys/fs/cgroup/ shows cpu, memory, blkio separately = v1. Modern distros (Ubuntu 22+, RHEL 9) default to v2.
MED
LNX-09High iowait means your disks are the bottleneck?
✗ WRONGYes, iowait% directly indicates disk saturation.
✓ RIGHTiowait is idle time where at least one I/O is pending — not a direct disk metric. It's unreliable on multi-core systems. Check iostat -x for %util and await per device for real disk pressure.
High iowait on a 32-core machine with 1 pending I/O is meaningless. Use iostat -x 1, iotop, blktrace for real disk analysis.
MED
LNX-10Does nice value affect I/O scheduling priority?
✗ WRONGYes, nice controls all resource scheduling.
✓ RIGHTnice/renice affects CPU scheduling only. For I/O priority use ionice. ionice -c 3 <pid> = idle class (lowest I/O priority). Different schedulers for CPU vs I/O.
ionice -c 1 -n 0 <pid> = real-time I/O (highest). -c 2 -n 0 = best-effort high. -c 3 = idle (only gets I/O when nothing else needs it). Used for backup processes.
AWS / Cloud
10 TRAPS
CRITICAL
AWS-01Is AWS S3 eventually consistent for new object writes since always?
✗ WRONGYes, S3 is always eventually consistent — that's the trade-off.
✓ RIGHTSince December 2020, S3 is strongly consistent for all operations. Read-after-write consistency for PUTs and DELETEs. No more "wait 1s after PUT before reading" hacks needed.
Old S3 consistency logic baked into your codebase is now unnecessary. Still need eventual consistency thinking for S3 replication to other regions.
CRITICAL
AWS-02What's the Spot instance termination notice window?
✗ WRONGInstant — Spot can be terminated with no warning.
✓ RIGHT2-minute warning via instance metadata + EventBridge event. Poll http://169.254.169.254/latest/meta-data/spot/termination-time or use Spot interruption handler (e.g., aws-node-termination-handler) to drain gracefully.
For EMR: core nodes = On-Demand (data). Task nodes = Spot (compute only). Task node termination = Spark task fails + retries. Core node termination = data loss risk.
CRITICAL
AWS-03Does RDS Multi-AZ improve read performance?
✗ WRONGYes, standby can serve read traffic to reduce primary load.
✓ RIGHTNo. Multi-AZ standby is for failover only — it's not readable. For read scaling use Read Replicas. Multi-AZ = HA. Read Replicas = read scale-out. Completely different purposes.
Multi-AZ: synchronous replication, automatic failover (<60s), same region. Read Replicas: async replication, manual failover, can be cross-region. You can combine both.
HIGH
AWS-04Are Security Groups stateless like NACLs?
✗ WRONGBoth SGs and NACLs are stateless packet filters.
✓ RIGHTSecurity Groups are stateful. NACLs are stateless. SG: allow inbound → return traffic automatic. NACL: must explicitly allow both inbound AND outbound for the same connection. Most NACL bugs are missing outbound rules.
NACLs apply at subnet level, evaluated in rule number order, first match wins. SGs apply at instance level, all rules evaluated, most permissive wins.
HIGH
AWS-05Does a NAT Gateway in one AZ serve all AZs cost-efficiently?
✗ WRONGYes, one NAT Gateway per VPC is sufficient and cheapest.
✓ RIGHTCross-AZ data transfer costs $0.01/GB each way. With high egress, one NAT GW creates massive cross-AZ charges. Best practice: one NAT GW per AZ. Also: if that AZ fails, all AZs lose internet egress.
For Singular-scale (billions of events): cross-AZ data costs add up fast. Keep Kafka consumers in same AZ as brokers. Use VPC endpoints for S3/DynamoDB (free, no NAT needed).
HIGH
AWS-06Does attaching an IAM role to an EC2 instance give pods in EKS the same permissions?
✗ WRONGYes, the node role covers all workloads on the node.
✓ RIGHTNode IAM role = node-level permissions only. Use IRSA for pod-level permissions. IRSA: annotate ServiceAccount with IAM role ARN → pods get scoped STS credentials via projected token mount. Not the node role.
Node role with S3 access = every pod on every node gets S3 access. IRSA = only pods using the annotated SA get specific S3 access. Least privilege requires IRSA.
MED
AWS-07Does ALB preserve the original client IP in the request?
✗ WRONGYes, the client IP appears in the connection's source address.
✓ RIGHTALB terminates TCP — source IP is ALB's IP. Client IP in X-Forwarded-For header. NLB passes through — source IP is real client (unless using private link). Choose based on whether you need real-IP logging.
ALB: L7, HTTP-aware, WebSocket, gRPC, path/host routing. NLB: L4, ultra-low latency, preserves source IP, static IP, millions of req/sec. Pick based on protocol, not just IP preservation.
HIGH
AWS-08Savings Plans and Reserved Instances are the same commitment model?
✗ WRONGYes, both are 1-3 year commitments for a discount.
✓ RIGHTCompute Savings Plans apply to any instance family/region/OS automatically. RIs are tied to a specific instance type + AZ + OS. Savings Plans are more flexible; RIs give slightly higher discounts for exact match.
Best practice: Compute Savings Plans for baseline (covers EC2, Lambda, Fargate). Spot for variable/fault-tolerant. On-Demand for truly unpredictable bursts.
MED
AWS-09CloudWatch custom metrics are retained indefinitely?
✗ WRONGYes, AWS stores all metrics forever.
✓ RIGHTRetention depends on resolution. 1-second: 3 hours. 60-second: 15 days. 5-minute: 63 days. 1-hour: 15 months. High-resolution metrics are auto-rolled up. Plan your dashboards around retention limits.
For long-term metric storage: export to S3 via Metric Streams → Kinesis Firehose → S3. Or use a managed observability platform that handles retention separately.
HIGH
AWS-10Can EBS volumes be attached to multiple EC2 instances simultaneously?
✗ WRONGNo — EBS is always single-attach only.
✓ RIGHTio1/io2 volumes support Multi-Attach (up to 16 instances in same AZ). But: your application must handle concurrent writes — kernel-level file system locking is required. ext4 is not safe; use cluster-aware FS like GFS2.
Multi-Attach is not a substitute for shared file systems like EFS. It's for applications with their own distributed locking (Oracle RAC, etc.). Wrong use → data corruption.
Architecture
7 TRAPS
CRITICAL
ARC-01CAP theorem: can you achieve Consistency, Availability, and Partition tolerance simultaneously?
✗ WRONGYes, modern databases achieve all three with clever engineering.
✓ RIGHTNo. During a network partition, you must choose: serve stale data (AP) or reject requests (CP). "CA" systems don't exist in distributed environments — partitions always eventually happen.
Kafka = CP (ISR/acks=all prioritizes consistency). Cassandra = AP (tunable, defaults to availability). etcd = CP (Raft consensus). Know where your systems fall on this spectrum.
CRITICAL
ARC-02Is idempotency optional when using at-least-once delivery?
✗ WRONGAt-least-once just means events arrive, duplicates are rare.
✓ RIGHTAt-least-once = idempotency is mandatory, not optional. Duplicates are guaranteed to happen (retries, consumer restarts). Idempotent consumer = processing same event twice produces same result.
Idempotency patterns: natural idempotency (SET vs INCREMENT), deduplication ID stored in DB, upsert semantics, conditional writes with version checks.
HIGH
ARC-03Are Event Sourcing and CQRS the same pattern?
✗ WRONGYes, they're used together and mean the same thing.
✓ RIGHTDifferent patterns that complement each other. Event Sourcing = state derived from ordered event log (no mutable current state). CQRS = separate read model from write model. You can use either without the other.
Often combined: ES provides the write side (immutable event log), CQRS builds read-optimized projections (materialized views) from those events. Kafka + Druid = a form of this.
HIGH
ARC-04Does a circuit breaker prevent all downstream failures from affecting your service?
✗ WRONGYes, open circuit = no downstream calls = you're isolated.
✓ RIGHTOpen circuit prevents cascading failure but you must handle the fallback. States: Closed (normal) → Open (failing, reject immediately) → Half-Open (probe with one call). You need a fallback: cached value, default, queue for later.
Circuit breaker without fallback = fast fail with an error. That's still an error for your user. Define fallback behavior: serve stale cache, return 503, queue request, return partial data.
HIGH
ARC-05Blue-green, canary, and rolling deployments all minimize downtime equally?
✗ WRONGYes, they're all zero-downtime strategies.
✓ RIGHTDifferent tradeoffs. Blue-green = instant cutover, 2× resource cost, easy rollback. Canary = gradual traffic shift, real user validation, complex routing. Rolling = low resource overhead, mixed versions run simultaneously.
Mixed-version risk: rolling update means v1 and v2 pods handle traffic simultaneously. Your API and data schemas must be backward-compatible. Blue-green avoids this but doubles infra cost during deploy.
MED
ARC-06In a Saga pattern, does the orchestrator model require a message bus?
✗ WRONGYes, both choreography and orchestration use event buses.
✓ RIGHTOrchestrator pattern uses direct calls (sync or async) to services via a central coordinator. Choreography uses events — services react to each other's events. No central coordinator. Each has different failure and observability tradeoffs.
Orchestration: easier to reason about flow, single point of failure. Choreography: decoupled, hard to trace end-to-end flow. Complex Sagas are almost always orchestrated for debuggability.
HIGH
ARC-07Does database connection pooling always improve performance?
✗ WRONGYes, pooling is always better than per-request connections.
✓ RIGHTPool size must match DB capacity. Pool too large = DB overwhelmed (PostgreSQL: each connection = process, 1000 conns = 1000 processes). Rule: pool_size = (core_count × 2) + effective_spindle_count. PgBouncer for large K8s deployments.
K8s with 500 pods each holding a pool of 10 = 5000 DB connections. PostgreSQL max_connections default is 100. This is why PgBouncer (or RDS Proxy) is mandatory at scale.
