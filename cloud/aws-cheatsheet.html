01 IAM & STS

The single most interview-heavy topic. Know the vocabulary, the policy evaluation flow cold, and why roles beat long-lived keys.

Principals & identities

  • User — a long-lived identity for a human/app. Has credentials (console password, up to 2 access keys). Avoid for workloads.
  • Group — a collection of users; attach policies to the group, not each user. Groups cannot be nested and cannot be a principal in a policy.
  • Role — an identity with no long-term credentials. It's assumed to get temporary STS credentials. Used by EC2, Lambda, EKS pods, cross-account access, and federated users.
  • Instance profile — the container that lets an EC2 instance assume a role. One role per instance profile; the instance retrieves temporary creds from IMDS (use IMDSv2, token-based, to block SSRF).

Policy types — the two big families

TypeAttached toHas Principal?Answers…
Identity-basedUser / group / roleNo"What can this identity do?"
Resource-basedThe resource (S3 bucket, SQS, KMS key, role trust policy)Yes (required)"Who can act on this resource?"
Permission boundaryUser / role (as a ceiling)NoMax perms an identity can be granted
SCP (Organizations)OU / accountNoMax perms for everyone in the account (a guardrail, not a grant)
Session policyPassed at AssumeRoleNoFurther scopes down a session

Key nuance: within a single account, a resource-based policy can grant access on its own — the bucket policy alone suffices, no matching identity policy required. For cross-account access to S3, you need an allow on both the identity side (caller's account) and the resource side (bucket policy). Exception: some services (SNS, SQS, Lambda, KMS) let a resource policy that names the exact principal suffice on its own.

Policy evaluation logic — memorize this order

The one rule everyone asks

Explicit Deny > Explicit Allow > Implicit Deny (default). Access is denied by default. An Allow anywhere opens it. A single explicit Deny anywhere — in any policy type — overrides all Allows. There is no "override a deny."

Full decision flow for a request:

  1. Default = implicit deny.
  2. Evaluate all applicable policies (identity, resource, boundary, SCP, session).
  3. Is there an explicit Deny? → DENY (final, wins over everything).
  4. SCP allows the action? (Org accounts only) If not → deny.
  5. Permission boundary allows it? If not → deny.
  6. An explicit Allow exists (identity or resource)? → ALLOW.
  7. Otherwise → DENY (implicit).

Effective permissions = intersection of SCP ∩ permission boundary ∩ identity policy (union with resource policy), minus any explicit deny.

Example identity policy (least privilege, scoped)

{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "ReadOneBucket",
      "Effect": "Allow",
      "Action": ["s3:GetObject", "s3:ListBucket"],
      "Resource": [
        "arn:aws:s3:::my-app-data",
        "arn:aws:s3:::my-app-data/*"
      ],
      "Condition": {
        "IpAddress": { "aws:SourceIp": "203.0.113.0/24" }
      }
    }
  ]
}

ListBucket is a bucket-level action (resource = bucket ARN); GetObject is object-level (resource = bucket/*). Mixing these up is a classic failed-question.

Explicit Deny example (boundary / SCP style)

{
  "Effect": "Deny",
  "Action": "*",
  "Resource": "*",
  "Condition": {
    "StringNotEquals": { "aws:RequestedRegion": ["us-east-1", "eu-west-1"] }
  }
}

Deny-everything-outside-approved-regions. As an SCP this locks the whole account regardless of what admins later grant — the guardrail pattern.

STS & AssumeRole

  • STS (Security Token Service) issues temporary credentials (access key + secret + session token, minutes to hours). sts:AssumeRole, AssumeRoleWithWebIdentity (OIDC), AssumeRoleWithSAML.
  • A role has two policy sides: the trust policy (resource-based — who may assume it) and the permission policy (what the assumed session can do).
  • Cross-account: Account A's role trust policy names Account B's principal; B's identity needs sts:AssumeRole on that role ARN. Add ExternalId to defend against the confused-deputy problem for third parties.

Role trust policy (EC2 service principal)

{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Principal": { "Service": "ec2.amazonaws.com" },
    "Action": "sts:AssumeRole"
  }]
}

Federation & workload identity

  • OIDC federation — trust an external IdP (GitHub Actions, Google, EKS) so workloads exchange an OIDC token for AWS creds via AssumeRoleWithWebIdentity. No stored AWS keys. This is how GitHub Actions → AWS should be wired.
  • IRSA (IAM Roles for Service Accounts) — EKS maps a K8s service account to an IAM role via the cluster's OIDC provider. EKS sheet has depth
  • EKS Pod Identity — newer, simpler alternative to IRSA (an agent + association, no per-cluster OIDC trust wiring). Prefer for new clusters. EKS sheet has depth
Q&A

Access keys or roles for an app on EC2?

Roles, always. Attach an instance profile → the SDK pulls short-lived, auto-rotated creds from IMDSv2. Long-lived access keys baked into an AMI/env var are the #1 leaked-credential source. Static keys have one legit niche: on-prem/off-AWS systems that can't assume a role — and even those should move to IAM Roles Anywhere.

⚠ Trap

A permission boundary is a ceiling, not a grant. Attaching a boundary that allows s3:* gives zero permissions on its own — you still need an identity policy that allows the action. Effective perms = boundary ∩ identity policy.

⚠ Trap

SCPs never grant permissions — they only cap them. An SCP allowing an action does nothing unless an IAM policy also allows it. And the management (payer) account is not restricted by SCPs.

02 VPC Networking

Subnets, routing, gateways, and the stateful-vs-stateless firewall contrast that appears in nearly every networking interview.

Building blocks

  • VPC — a regional, logically isolated network. You pick a CIDR (e.g. 10.0.0.0/16 = 65,536 addresses). Can add secondary CIDRs.
  • Subnet — AZ-scoped slice of the VPC CIDR. AWS reserves 5 IPs per subnet (.0 network, .1 router, .2 DNS, .3 future, .255 broadcast).
  • Public subnet = its route table has a 0.0.0.0/0 route to an Internet Gateway. Private subnet = no direct IGW route.
  • Route table — controls where subnet traffic goes. Longest-prefix match wins. The local route (VPC CIDR) is always present and can't be removed.

Egress for private subnets: NAT

NAT GatewayNAT Instance
Managed?Fully managed by AWSYou run an EC2 instance
AvailabilityHighly available within one AZ (deploy 1/AZ for resilience)Single point of failure unless you script HA
BandwidthUp to 100 Gbps, auto-scalesBounded by instance size
CostHourly + per-GB data processing chargeInstance cost only (cheaper at low volume)
Security groupsN/A (no SG)Yes (it's an EC2)
Use caseDefault choice, productionEdge/legacy; can double as bastion or port-forward

NAT lets private instances make outbound connections (pull updates, hit APIs) while blocking inbound from the internet. NAT lives in a public subnet; private route tables point 0.0.0.0/0 → NAT.

⚠ Trap — the cost gotcha

NAT Gateway bills both an hourly rate and a per-GB data-processing fee on all traffic through it. Pulling large images or backups through NAT gets expensive fast. For S3/DynamoDB/ECR traffic, use VPC endpoints to bypass NAT entirely and cut the bill.

Security Groups vs NACLs — the classic contrast

Security GroupNetwork ACL
Operates atInstance / ENI levelSubnet level

Q&A

Why does a stateless NACL need ephemeral ports open?

A client connecting to port 443 receives the reply on a high ephemeral port (1024–65535). Because NACLs don't track connection state, the outbound NACL rule must explicitly allow that ephemeral range or replies get dropped. Security groups handle this automatically because they're stateful.

Mental model: SG = bouncer who remembers who they let in. NACL = a rulebook checked in both directions with no memory. In practice SGs do 95% of the work; reach for NACLs for coarse subnet-wide deny rules (e.g. block a bad CIDR).

Connecting VPCs & on-prem

  • VPC Peering — 1:1 private link between two VPCs. Non-transitive (A↔B and B↔C does not give A↔C). CIDRs must not overlap. Fine for a handful of VPCs.
  • Transit Gateway (TGW) — hub-and-spoke router connecting many VPCs, VPNs, and Direct Connect. Transitive, scales to thousands of attachments, route tables per attachment. Use when peering becomes a mesh nightmare (n² connections).
  • Direct Connect — dedicated physical line to AWS (consistent low latency, not encrypted by default). Site-to-Site VPN — IPsec over the internet (encrypted, quick to stand up).

VPC Endpoints — private access to AWS services

Gateway EndpointInterface Endpoint (PrivateLink)
ServicesS3 and DynamoDB onlyMost AWS services + your own/partner services
HowA route-table entry (prefix list)An ENI with a private IP in your subnet
CostFreeHourly + per-GB
SecurityEndpoint policySecurity group + endpoint policy

Endpoints keep traffic on the AWS private network — no IGW, no NAT, no public internet. A gateway endpoint to S3 is the standard way to avoid NAT data-processing charges for S3 access from private subnets.

DNS & visibility

  • Enable enableDnsSupport + enableDnsHostnames for the Amazon DNS resolver (the .2 address, or 169.254.169.253) to work and to get private DNS names.
  • VPC Flow Logs — capture accepted/rejected IP traffic metadata (5-tuple, action, bytes) to CloudWatch Logs / S3. Primary tool for "why is this connection failing?" — check for REJECT.

03 Compute — EC2 & Auto Scaling

Instance families, storage-attach models, and how ASGs give you elasticity and self-healing.

EC2 basics

  • Instance families — T burstable (dev, low baseline, CPU credits), M general purpose, C compute-optimized, R/X memory-optimized, I/D storage-optimized, P/G/Inf/Trn GPU/ML accelerators. Naming: m7g.large = family M, gen 7, g=Graviton (ARM), size large.
  • AMI — Amazon Machine Image, the template (OS + config) an instance boots from. Bake golden AMIs for fast, reproducible launches.

EBS vs Instance Store

EBSInstance Store
PersistencePersists beyond stop/terminate (if not delete-on-termination)Ephemeral — wiped on stop/terminate
AttachNetwork-attached block storage, detach/reattachPhysically attached to the host
PerformanceHigh, provisionable (io2)Very high, lowest latency (NVMe local)
UseBoot volumes, databases, anything that must surviveScratch, caches, buffers

Auto Scaling Groups (ASG)

  • Launch template (preferred over legacy launch configs) — defines instance type, AMI, SGs, user data, IAM profile. Supports versioning and mixed instances / spot.
  • Min / Desired / Max capacity. ASG keeps Desired healthy instances running across the AZs/subnets you specify.
  • Health checks — EC2 (instance status) or ELB (target group health). Prefer ELB checks so an app that hangs but keeps the OS alive still gets replaced.
  • Self-healing — an unhealthy instance is terminated and replaced automatically to meet Desired.

Scaling policies

  • Target tracking — "keep avg CPU at 50%." Simplest, most common.
  • Step scaling — add N instances per CloudWatch alarm threshold band.
  • Scheduled — scale on known patterns (business hours).
  • Predictive — ML forecasts load and pre-scales.

Purchasing options — cost lever

ModelDiscountCommitmentBest for
On-DemandBaselineNoneSpiky/unpredictable, dev
SpotUp to ~90% offNone, but can be reclaimed with 2-min warningFault-tolerant, stateless, batch, CI
Reserved (RI)Up to ~72% off1 or 3 yr, specific instance typeSteady legacy baselines

Interview framing

The senior answer: Savings Plans / RIs for the steady baseline, On-Demand for the variable middle, Spot for fault-tolerant burst. Spot in an ASG mixed-instances policy with multiple types/AZs minimizes reclaim impact.

Pick the right one by OSI layer and feature need. ALB vs NLB is the one they'll drill.

Target groups & health checks

  • Target group — the set of backends (instances, IPs, Lambda, or containers) an LB routes to. ALB routing rules point to target groups.
  • Health checks — LB probes a path/port; unhealthy targets are pulled from rotation. Tune threshold + interval so you don't flap on brief blips.
  • Sticky sessions — ALB can pin a client to a target via a cookie (needed for stateful apps that don't externalize session).
Q&A

You need a load balancer with a fixed IP to whitelist in a partner's firewall. Which one?

NLB. It gives you a static IP (one per AZ, and you can assign Elastic IPs). ALB only exposes a DNS name whose IPs change. Common pattern: NLB (static IP) → ALB (L7 routing) behind it if you need both.

05 Storage & Data

S3 dominates. Know storage classes, security controls, and the HA-vs-scaling contrast on RDS.

S3 — object storage

  • Bucket (globally unique name) holds objects (key + value + metadata). Flat namespace; "folders" are just key prefixes. Effectively unlimited; object max 5 TB (multipart upload required > 5 GB).
  • Strong read-after-write consistency for all operations since Dec 2020 — a GET right after a PUT/overwrite/delete now always returns the latest data. (Old "eventual consistency for overwrites" caveat is gone — say this if asked.)
  • Versioning — keeps every version; protects against overwrite/delete. Deletes add a delete marker. Pair with MFA-delete for critical buckets.
  • Lifecycle rules — transition objects between classes or expire them on age (e.g. → IA at 30d → Glacier at 90d → delete at 365d).

Storage classes

ClassUse / retrieval
StandardHot data, frequent access, low latency
Intelligent-TieringUnknown/changing access — auto-moves tiers, no retrieval fee
Standard-IA / One Zone-IAInfrequent access; One Zone = cheaper, single AZ (less durable)
Glacier Instant RetrievalArchive, needs ms access
Glacier Flexible RetrievalArchive, minutes–hours retrieval
Glacier Deep ArchiveCheapest, 12h retrieval, compliance/cold

S3 security

  • Encryption at rest: SSE-S3 (AWS-managed keys, default), SSE-KMS (KMS keys — audit trail, rotation, per-key access control), SSE-C (you supply keys). All new objects are encrypted by default now.
  • Bucket policies vs ACLs: prefer bucket policies (IAM JSON, fine-grained). ACLs are legacy — AWS now disables them by default (Object Ownership = "Bucket owner enforced"). Don't reach for ACLs in an interview.
  • Block Public Access — account- and bucket-level master switch; on by default. The correct guard against accidental public buckets.
  • Pre-signed URLs — time-limited URL that grants temporary GET/PUT to a private object using your credentials. The right way to hand out temporary download/upload access without making the bucket public.
  • Static website hosting — S3 can serve a static site; front with CloudFront for HTTPS, caching, and OAC (Origin Access Control) so the bucket stays private.
⚠ Trap

"Make the object publicly downloadable" is almost never the answer — it usually means pre-signed URL (temporary, scoped) or CloudFront. And Block Public Access on = ignores any policy/ACL that would make the bucket public, even if someone sets one. That's a feature, not a bug.

EBS & EFS

  • EBS volume types: gp3 (default SSD — decouple IOPS/throughput from size, cheaper than gp2), gp2 (legacy SSD, IOPS tied to size), io2 / io2 Block Express (highest IOPS, durability, multi-attach for clustered DBs), st1 (throughput HDD, big sequential), sc1 (cold HDD). EBS is single-AZ, single-instance (except io2 multi-attach).
  • EFS — managed NFS, multi-AZ, mounts to many instances at once, elastic. Use when you need a shared POSIX filesystem across instances. (FSx = managed Windows/Lustre/NetApp FS.)

RDS — managed relational

The contrast they want: Multi-AZ ≠ Read Replica
Multi-AZRead Replica
PurposeHigh availability / failoverRead scaling (offload read queries)
ReplicationSynchronous to standbyAsynchronous (can lag)
Standby serves traffic?No — passive until failover (Multi-AZ instance); Multi-AZ cluster adds readable standbysYes — actively serves reads
FailoverAutomatic, ~60–120s, DNS flips to standbyManual promotion to standalone primary

They solve different problems and are often used together. Multi-AZ = durability/uptime; Read Replicas = throughput. Neither is a backup.

  • Aurora — AWS's MySQL/PostgreSQL-compatible engine; storage auto-scales, up to 15 low-lag read replicas, 6-way replication across 3 AZs, faster failover. Aurora Serverless v2 auto-scales capacity.
  • Backups — automated backups (point-in-time restore within retention window, up to 35 days) vs manual snapshots (kept until you delete, shareable across accounts/regions).

DynamoDB (brief)

  • Fully managed serverless NoSQL key-value / document store. Single-digit-ms latency at any scale.
  • Partition key (+ optional sort key) design drives performance — avoid hot partitions. On-demand or provisioned capacity. Global tables = multi-region active-active. DynamoDB Streams for change capture.
  • Reach for it over RDS when you need massive scale, predictable latency, and a flexible schema — not for complex joins/transactions-heavy relational workloads.

06 Route 53 & DNS

AWS's managed DNS — record types, routing policies, and the alias-vs-CNAME distinction that trips people up.

Record types

  • A / AAAA — name → IPv4 / IPv6. CNAME — name → another name. MX — mail. TXT — verification/SPF/DKIM. NS / SOA — zone delegation.
  • Alias record — Route 53-specific; points a name at an AWS resource (ALB, CloudFront, S3 website, another R53 record). Free queries, resolves at the zone apex, auto-tracks the target's IP.
⚠ Trap — Alias vs CNAME

A CNAME cannot exist at the zone apex (example.com) — DNS forbids it. To point the naked/apex domain at an ALB or CloudFront, you must use an Alias A record. Alias also has no per-query charge and works at apex; CNAME only works on subdomains.

Routing policies

PolicyBehavior
SimpleOne record, one (or multi-value) answer. No health check logic.
WeightedSplit traffic by percentage across records — canary/blue-green, A/B.
Latency-basedRoute to the region with lowest latency for the user.
FailoverPrimary/secondary — health check fails → serve the backup (active-passive DR).
GeolocationRoute by user's continent/country (compliance, localization).
GeoproximityRoute by geographic distance with a bias knob.
Multivalue answerReturn multiple healthy IPs (client-side round-robin + health checks).

Health checks monitor endpoints and can gate failover/weighted/latency records so DNS stops handing out a dead endpoint.

07 Ops Glue

Observability, config/secrets, encryption, and who's responsible for what.

CloudWatch

  • Metrics — time-series (CPU, latency, custom). Namespaces + dimensions.
  • Logs — log groups/streams; the CloudWatch agent ships app + OS logs (memory/disk aren't default EC2 metrics — need the agent).
  • Alarms — threshold on a metric → SNS/Auto Scaling/action. Composite alarms combine conditions.
  • EventBridge (event bus) + CloudWatch Logs Insights (query language) for events and ad-hoc log analysis.

Systems Manager (SSM)

  • Session Manager — shell into instances with no SSH, no open port 22, no bastion (IAM-gated, audited). Big security win.
  • Run Command / Patch Manager / State Manager — fleet automation & patching.
  • Parameter Store — config + secrets (see below).

Secrets Manager vs Parameter Store

Secrets ManagerSSM Parameter Store
PurposeSecrets, first-classConfig + secrets (SecureString)
RotationBuilt-in automatic rotation (Lambda, native RDS integration)No native rotation
CostPer secret / per API call (paid)Standard tier free; advanced tier paid
UseDB creds, API keys needing rotationFeature flags, config, cheap secrets

Rule of thumb: if it needs rotation (DB passwords) → Secrets Manager. Plain config or cost-sensitive → Parameter Store. Both integrate with KMS and IAM.

KMS

  • Managed keys for encryption across AWS. Envelope encryption: KMS encrypts a data key that encrypts your data. Master key never leaves KMS.
  • Key policies (resource-based) + IAM control access; every use is logged in CloudTrail. Customer-managed keys support rotation and cross-account grants; AWS-managed keys are simpler but less flexible.

Shared Responsibility Model

AWS Security OF the cloud

Hardware, global infra, the hypervisor, managed-service internals, physical security, patching of managed services.

You Security IN the cloud

IAM & least privilege, OS/app patching on EC2, SG/NACL config, encryption choices, data classification, correct S3 access settings.

The line shifts by service model: for EC2 you patch the OS; for RDS/Lambda/S3 AWS handles more of the stack — but your data, IAM, and network config are always yours.

08 Well-Architected Framework

Name the six pillars — a favorite "do they know the framework" checkbox.

  1. Operational Excellence — run & monitor systems, IaC, automate, learn from failure.
  2. Security — least privilege, defense in depth, encryption, traceability.
  3. Reliability — recover from failure, scale horizontally, test recovery (Multi-AZ, backups, health checks).
  4. Performance Efficiency — use the right resource/type, scale, go serverless where it fits.
  5. Cost Optimization — right-size, Savings Plans/Spot, kill idle, measure spend.
  6. Sustainability — minimize environmental impact (efficient regions, utilization).

Mnemonic: "OSCAR-S" won't quite spell — just remember Ops, Security, Reliability, Performance, Cost, Sustainability. Six since Sustainability was added.

09 Trap Rapid-Fire

The gotchas interviewers use to separate memorizers from operators.

⚠ SG vs NACL statefulness

Security groups are stateful (return traffic implicit). NACLs are stateless — you must open ephemeral ports for return traffic. Getting this backwards is an instant tell.

⚠ NAT gateway cost / egress

NAT charges per-hour and per-GB processed. Route S3/DynamoDB/ECR through VPC endpoints to skip NAT. NAT is for outbound only — it never allows unsolicited inbound.

⚠ IAM explicit-deny precedence

Explicit Deny always wins — over any Allow, in any policy type. Default is implicit deny. SCPs and boundaries only limit, never grant.

⚠ Multi-AZ ≠ Read Replica

Multi-AZ = HA/failover (synchronous standby, not serving reads in the classic setup). Read Replica = read scaling (async, serves reads). Neither is a backup.

⚠ S3 public access

Block Public Access is on by default and overrides any policy/ACL that would expose the bucket. For temporary access use pre-signed URLs, not a public bucket. ACLs are legacy — use bucket policies.

⚠ Mutable AMI / :latest drift

Referencing "the latest AMI" or a mutable tag means two launches from the "same" template can boot different images → config drift and un-reproducible fleets. Pin explicit AMI IDs (or SSM Parameter Store references) and bake immutable golden images.

⚠ CNAME at zone apex

You can't put a CNAME on the naked domain. Use a Route 53 Alias A record for apex → ALB/CloudFront/S3.

⚠ Roles > access keys

Never bake long-lived access keys into instances/AMIs/env. Use IAM roles + IMDSv2 on EC2, OIDC federation for CI/CD. Static keys are the top leaked-secret vector.

⚠ VPC peering is non-transitive

A↔B and B↔C does not give A↔C. Need a mesh? Use Transit Gateway, which is transitive and scales.

⚠ ALB has no static IP

ALB exposes a changing DNS name. Need a fixed IP to whitelist? Use NLB (static/Elastic IP), optionally fronting an ALB.

Q&A — favorite closer

"How would you give a Lambda access to an S3 bucket in another account?"

Lambda's execution role (identity policy) allows the S3 action and the bucket's resource policy in the other account allows this role's ARN. Cross-account needs an allow on both sides. Encrypt with a KMS key whose key policy also grants the role. Mentioning both sides + KMS scores the point.
