01 systemd & unit management

systemd is PID 1 on nearly every modern distro. It supervises services, mounts, sockets, and timers as declarative units, resolving dependencies as a graph rather than a linear boot script.

Unit types

UnitExtensionPurpose
Service.serviceA managed process (daemon, one-shot script)
Socket.socketSocket-activated start (systemd listens, spawns service on first connection)
Timer.timerScheduled activation of a matching .service, cron-like
Target.targetSynchronization point / grouping (e.g. multi-user.target), analogous to old runlevels
Mount.mountA filesystem mount point, usually auto-generated from /etc/fstab
Path.pathActivates a unit when a file/dir changes (inotify-based)

systemctl essentials

# lifecycle
systemctl start nginx
systemctl stop nginx
systemctl restart nginx
systemctl reload nginx          # re-read config without dropping connections (if supported)
systemctl status nginx          # state, recent logs, cgroup, main PID

# boot-time behavior
systemctl enable nginx          # creates symlink in a .wants/ dir — does NOT start it now
systemctl enable --now nginx    # enable + start in one shot
systemctl disable nginx
systemctl is-enabled nginx
systemctl is-active nginx

# after editing a unit file
systemctl daemon-reload         # re-parse unit files; required before changes take effect

# inspection
systemctl list-units --type=service --state=running
systemctl list-unit-files --state=enabled
systemctl cat nginx.service     # print the effective unit file (incl. drop-ins)
systemctl show nginx -p MainPID,ActiveState,Restart
systemctl list-dependencies nginx
systemctl --failed              # list all failed units
systemctl mask nginx            # symlink to /dev/null — cannot start even manually
⚠ Trap
Forgetting systemctl daemon-reload after editing a unit file. Without it, systemctl restart uses the cached unit definition — the change silently doesn't apply.

Unit file anatomy

[Unit]
Description=My API service
After=network-online.target postgresql.service
Wants=network-online.target
Requires=postgresql.service

[Service]
Type=notify
ExecStart=/usr/bin/myapi --config /etc/myapi/config.yml
ExecReload=/bin/kill -HUP $MAINPID
Restart=on-failure
RestartSec=5s
User=myapi
Group=myapi
LimitNOFILE=65536
Environment=NODE_ENV=production

[Install]
WantedBy=multi-user.target
DirectiveMeaning
After=Ordering only — start after the listed unit, but does not pull it in as a dependency
Requires=Hard dependency — if the required unit fails/stops, this unit is stopped too
Wants=Soft dependency — pulls the unit in at start, but failure of the wanted unit doesn't stop this one
Restart=no (default), on-failure, on-abnormal, always, on-success
ExecStart=The command to run; exactly what "starting" means depends on Type=

Type= values

TypeBehavior
simple (default)Process started by ExecStart= is the main process; systemd considers the unit started immediately
forkingProcess forks and the parent exits; systemd expects a PID file (PIDFile=) or tracks the child via cgroup to know the real main PID
oneshotProcess is expected to exit before systemd considers the unit "active"; commonly paired with RemainAfterExit=yes
notifyProcess signals readiness itself via sd_notify() (READY=1) — most accurate for services with async startup
dbusUnit is considered started when the specified D-Bus name appears on the bus
Q&A
Q: Your service unit has Type=simple but the binary daemonizes (double-forks) internally. What happens?
A: systemd loses track of the real child process — the tracked PID exits immediately (the parent that forked), so systemd thinks the service died even though the daemon is still running. Fix: use Type=forking with a correct PIDFile=, or better, run the binary in the foreground and use Type=simple/notify.

02 journalctl & logging

journalctl queries the structured, binary systemd journal (/var/log/journal/ if persistent, otherwise volatile in /run/log/journal/).

journalctl -u nginx.service            # logs for one unit
journalctl -u nginx -f                 # follow, like tail -f
journalctl -u nginx --since "10 min ago"
journalctl -u nginx --since "2026-07-06 08:00" --until "2026-07-06 09:00"
journalctl -p err                      # priority filter: emerg/alert/crit/err/warning/notice/info/debug
journalctl -p err -b                   # errors since current boot
journalctl -b -1                       # logs from previous boot (needs persistent journal)
journalctl --list-boots                # enumerate boot IDs
journalctl -k                          # kernel messages only (like dmesg)
journalctl -o json-pretty -u nginx -n 5
journalctl --disk-usage                # how much space the journal is using
journalctl --vacuum-time=2weeks        # prune old entries
journalctl --vacuum-size=500M
Tip
Persistent journal requires /var/log/journal/ to exist (mkdir -p /var/log/journal && systemd-tmpfiles --create --prefix /var/log/journal) and Storage=persistent in /etc/systemd/journald.conf. Otherwise logs vanish on reboot.

03 Timers vs cron

systemd timercron
LoggingIntegrated with journal (journalctl -u foo.service)Usually mail or syslog only, easy to miss
DependenciesCan require other units (network, mounts) via After=/Requires=None — just fires at the time
Missed runsPersistent=true catches up missed runs after downtimeMissed run is simply skipped
RandomizationRandomizedDelaySec= to spread loadNot built in
Resource controlInherits cgroup limits (CPU/mem quotas) like any unitNone natively
SyntaxOnCalendar= expressions, more verboseTerse 5-field crontab syntax, ubiquitous
 
# /etc/systemd/system/backup.timer
[Unit]
Description=Run backup daily

[Timer]
OnCalendar=daily
Persistent=true
RandomizedDelaySec=300

[Install]
WantedBy=timers.target

# pairs with backup.service (Type=oneshot), same base name
systemctl enable --now backup.timer
systemctl list-timers               # next/last run times for all timers

04 The USE method

Brendan Gregg's systematic approach: for every resource (CPU, memory, disk, network), check three things.

MetricDefinitionExample
Utilization% of time the resource was busy servicing workCPU 95% busy
SaturationDegree of extra work queued that the resource can't serviceRun-queue length, disk I/O queue depth
ErrorsCount of error eventsNIC drops, disk retries, ECC memory errors

USE gives you a checklist to avoid tunnel vision — cycling through every resource in a fixed order rather than randomly poking at whatever looks suspicious first.

Reading vmstat CPU columns

ColumnMeaning
usUser-space CPU time
syKernel/system CPU time
idIdle
waWaiting on I/O (CPU idle but a process is blocked on disk)
stStolen — time a hypervisor took from this VM for other tenants (high st on a VM = noisy-neighbor problem)
Q&A
Q: High %wa in top — is that a CPU problem?
A: No — it means CPUs are idle but processes are blocked waiting on I/O. It's a signal to look at disk (iostat) or network storage, not to add CPU.

06 Load average — what it really measures

Load average (the three numbers from uptime or /proc/loadavg: 1/5/15-minute exponentially-decayed averages) counts the number of processes in the run queue (state R, runnable/running) plus processes in uninterruptible sleep (state D, usually blocked on I/O). It is not a CPU-only metric.

uptime
 14:22:01 up 12 days,  3:41,  2 users,  load average: 8.42, 5.10, 2.05
cat /proc/loadavg
8.42 5.10 2.05 3/812 24193   # 3rd field: running/total processes; last field: most recent PID
⚠ Trap
A load average of 8 on a 4-core box does not automatically mean "CPU-bound." If most of that load is processes stuck in D state (disk-blocked), the real bottleneck is storage I/O, not CPU. Always cross-check with vmstat's r/b columns and ps -eo stat,comm to see the state breakdown before concluding "we need more CPU."

07 Memory: tools & signals

free -h
              total        used        free      shared  buff/cache   available
Mem:           31Gi       6.1Gi       412Mi       210Mi        25Gi        24Gi
Swap:         8.0Gi          0B       8.0Gi
⚠ Trap — buff/cache misread
buff/cache is the kernel's page cache (file contents cached in RAM) and buffer cache. It is reclaimable on demand — the kernel frees it instantly under memory pressure. Seeing "only 412Mi free" is not a low-memory emergency; look at the available column instead, which already accounts for reclaimable cache. This trips up almost every junior engineer the first time they see free -h.
ToolShows

top/htopPer-process RSS/VSZ, %MEM, sortable
pidstat -r 1Per-process memory faults/RSS over time
vmstat 1si/so columns = swap-in/swap-out rate (should be ~0 on a healthy box)
sar -rHistorical memory utilization
/proc/meminfoRaw kernel memory counters (MemAvailable, Dirty, Slab, etc.)
smem, ps_memProportional set size (PSS) — accounts for shared memory correctly across processes

08 OOM killer & swap

When the kernel can't satisfy an allocation and has no more reclaimable memory, the OOM killer selects a victim process to kill based on a computed oom_score (heavier memory users score higher, and the score can be biased).

dmesg -T | grep -i 'killed process'
[Sun Jul  6 09:14:02 2026] Out of memory: Killed process 24193 (java) total-vm:8241232kB...

cat /proc/<pid>/oom_score          # computed likelihood of being killed (higher = more likely)
cat /proc/<pid>/oom_score_adj      # -1000 (never kill) to 1000 (kill first); user/admin adjustable
echo -500 > /proc/<pid>/oom_score_adj   # make this process less likely to be OOM-killed
Q&A
Q: A process died with no application-level error, no log entry, nothing. Where do you look first?
A: dmesg/journalctl -k for an OOM kill message. The application never got a chance to log anything — SIGKILL from the OOM killer is instant and uncatchable.

Swap & swappiness

swapon --show                       # active swap devices/files
cat /proc/sys/vm/swappiness         # 0-100, default 60: kernel's eagerness to swap out anonymous pages
sysctl vm.swappiness=10             # lower = prefer reclaiming cache over swapping app memory; runtime only
# persist in /etc/sysctl.conf or /etc/sysctl.d/*.conf: vm.swappiness=10

Low swappiness (e.g. 10) is common on database/latency-sensitive hosts to avoid swapping hot application memory in favor of dropping page cache instead. Swapping itself isn't inherently bad — sustained si/so activity under load, however, is a strong latency-degradation signal.

09 Disk I/O: tools & signals

iostat -xz 1
Device   r/s   w/s  rkB/s  wkB/s  await  r_await  w_await  aqu-sz  %util
nvme0n1  120   340   4800  22000   3.20     1.10     4.80    0.85   68.40
ColumnMeaning
%util% of time the device had at least one I/O in flight (utilization — but can be misleading for SSD/NVMe with parallel queues; near 100% may still have headroom)
awaitAverage time (ms) for I/O requests to be served, including queue wait — the real latency signal
aqu-szAverage queue size — saturation indicator
r/s, w/sReads/writes per second (IOPS)

Other tools: pidstat -d 1 (per-process disk I/O), sar -d (historical), iotop (top-style, per-process, needs root or CAP_NET_ADMIN-like privilege), blktrace/biosnoop (BPF-based, deep block-layer tracing).

10 Network: performance signals

sar -n DEV 1              # per-interface rx/tx throughput, historical + live
ip -s link show eth0      # interface counters: errors, drops, collisions
cat /proc/net/dev         # raw per-interface counters
ethtool -S eth0            # NIC driver stats (ring buffer drops, etc.)

Errors/drops on an interface (RX-DRP, RX-ERR, overruns) point to saturation at the NIC/ring-buffer level, often invisible in application-level metrics — always check for errors, not just throughput.

DNS resolution order

Name resolution order for most services is governed by /etc/nsswitch.conf:

hosts:  files dns

This means: check /etc/hosts first, then fall back to DNS as configured in /etc/resolv.conf (nameserver list, search domains). Modern systems often route this through systemd-resolved (/run/systemd/resolve/resolv.conf symlinked from /etc/resolv.conf) which adds caching and per-interface DNS — check with resolvectl status.

nftables / iptables basics

# iptables (legacy, still very common)
iptables -L -n -v                      # list rules, numeric, verbose (packet/byte counters)
iptables -A INPUT -p tcp --dport 22 -j ACCEPT
iptables -P INPUT DROP                 # default-deny policy
iptables-save > /etc/iptables/rules.v4 # persist rules

# nftables (modern replacement, single framework for IPv4+IPv6+bridge+arp)
nft list ruleset
nft add table inet filter
nft add chain inet filter input { type filter hook input priority 0 \; }

13 Filesystems & disk management

df -h                       # filesystem-level usage, by mount point
df -i                       # inode usage instead of blocks
du -sh /var/log/*           # directory-level usage, walks the tree
du -h --max-depth=1 /var
⚠ Trap — df says full, du doesn't add up
df reports space allocated at the filesystem level; du reports space used by files currently reachable in the directory tree. If a process holds a file descriptor open on a file that has been unlinked (deleted), the disk blocks are not freed until the last file descriptor closes — but the file no longer appears in any directory listing, so du won't count it while df still shows the space as used. Diagnose with: 
lsof +L1                      # list open files with a link count of 0 (deleted but still open)
COMMAND   PID USER   FD   TYPE DEVICE SIZE/OFF NLINK NODE NAME
java    24193 app   17w   REG  259,1   4.2G       0 9931 /var/log/app.log (deleted)
Fix: restart/HUP the offending process to release the descriptor, or truncate via > /proc/<pid>/fd/17 if you can't restart it yet.

Inode exhaustion

A filesystem can report free space (df -h) while being completely unable to create new files, because it ran out of inodes (each file/dir consumes one, regardless of size — common with huge numbers of tiny files, e.g. session caches, mail spools).

df -i /var
Filesystem      Inodes  IUsed   IFree IUse% Mounted on
/dev/sda1      6553600 6553600      0  100% /var

LVM basics

Physical Volume (PV)→ Volume Group (VG)→ Logical Volume (LV)→ Filesystem 
pvcreate /dev/sdb1
vgcreate vg_data /dev/sdb1
lvcreate -L 50G -n lv_app vg_data
mkfs.ext4 /dev/vg_data/lv_app

pvs; vgs; lvs                     # list PVs/VGs/LVs
lvextend -L +20G /dev/vg_data/lv_app
resize2fs /dev/vg_data/lv_app     # grow the ext4 filesystem to fill the extended LV

LVM's value: resize volumes online, take snapshots, and span a logical volume across multiple physical disks — decoupling the filesystem from any single physical device.

ACLs (finer-grained than ugo)

Capabilities — precision alternative to full root/setuid

Linux capabilities split root's monolithic privilege into discrete units, letting a binary get exactly the privilege it needs instead of full root via setuid.

getcap /usr/bin/ping
/usr/bin/ping cap_net_raw=ep

setcap cap_net_bind_service=+ep /usr/bin/myserver
# lets a non-root process bind to ports < 1024 without running as root or being setuid

capsh --print                     # show current process's capability sets
Q&A
Q: Why does /bin/ping work as a non-root user?
A: Either it's setuid-root (older systems), or — the modern, more precise approach — it carries cap_net_raw via setcap, granting just the raw-socket privilege ICMP needs without full root.

15 Signals, process states, limits

Common signals

SignalNumberDefault actionCatchable?
SIGHUP1Terminate (often repurposed by daemons to mean "reload config")Yes
SIGINT2Terminate (Ctrl-C)Yes
SIGKILL9Terminate immediatelyNo — cannot be caught, blocked, or ignored
SIGTERM15Terminate gracefully (default for kill)Yes
SIGSTOP19Pause processNo — uncatchable, like SIGKILL
SIGCONT18Resume a stopped processYes
SIGUSR1/SIGUSR210/12User-defined — apps commonly wire these to custom actions (log rotation, etc.)Yes
 
kill -15 24193             # SIGTERM — ask nicely, let the process clean up
kill -9 24193              # SIGKILL — kernel terminates immediately, no cleanup possible
kill -HUP 24193             # common convention: reload config without restart
pkill -f "myapp.*worker"    # kill by matching process name/args pattern
killall nginx
⚠ Trap — SIGKILL is uncatchable, and that has consequences
SIGKILL (and SIGSTOP) bypass the process entirely — there's no signal handler, no cleanup, no flushing buffers, no closing files gracefully. Always try SIGTERM first and give the process a grace period; reach for SIGKILL only when the process is unresponsive. It also means a process stuck in uninterruptible sleep (state D) cannot be killed at all with SIGKILL until the blocking kernel operation (usually I/O) completes — the signal is queued but not delivered until the process returns to a signal-checkable state.

Process states

StateCodeMeaning
Running/RunnableRActively executing or waiting for a CPU slot on the run queue
Sleeping (interruptible)SWaiting on an event (I/O, timer, signal); can be woken by a signal
Uninterruptible sleepDWaiting on I/O at a point where the kernel cannot safely interrupt it (e.g. mid-syscall to a device); does not respond to signals, including SIGKILL, until the I/O completes
ZombieZProcess has exited but its parent hasn't yet called wait() to reap the exit status; consumes a PID slot and an entry in the process table, but zero other resources
StoppedTSuspended by SIGSTOP/SIGTSTP or being traced
⚠ Trap — zombie vs D-state
A zombie is dead already — it holds no memory, no open files, nothing to reclaim except a process-table slot; the fix is to make the parent reap it (fix the parent's code, or if the parent itself is broken/dead, the zombie gets reparented to PID 1/init which reaps it). A D-state process is very much alive, actively blocked on I/O, consuming real resources, and cannot be killed until the kernel operation completes — often the real symptom of a stuck NFS mount, failing disk, or hung driver. Interviewers use this pair to check whether you conflate "can't kill it" with "it's already dead." 
ps -eo pid,ppid,stat,comm | awk '$3 ~ /Z/'    # find zombies
ps -eo pid,stat,wchan:32,comm | awk '$2 ~ /D/' # find D-state + what kernel function they're blocked in

nice / renice

nice -n 10 ./batch_job          # start with lower priority (range -20 most favorable to 19 least)
renice -n 5 -p 24193             # adjust priority of a running process
ps -eo pid,ni,comm                # view current niceness

ulimit / limits.conf

ulimit -a                         # show all current shell limits
ulimit -n 65536                   # raise open-file-descriptor limit for this shell session
ulimit -Hn                        # hard limit (only root can raise)

# /etc/security/limits.conf — persistent, applies via PAM at login
appuser   soft   nofile   65536
appuser   hard   nofile   131072

# for systemd services, ulimits are set per-unit instead:
# [Service] LimitNOFILE=65536
⚠ Trap
/etc/security/limits.conf only affects processes started through a PAM login session (interactive shells, su -, SSH). A systemd-managed service ignores it entirely — its limits come from LimitNOFILE=/etc. in the unit file (or systemd's DefaultLimitNOFILE= default). "I set limits.conf but my service still hits too-many-open-files" is a very common misconfiguration.

16 Systematic troubleshooting flow

"The box is slow" — a repeatable order beats random poking. Walk it top-down; note at which step you find a saturated/erroring resource before drilling deeper.

1. Load→ 2. CPU→ 3. Memory→ 4. Disk→ 5. Network
StepCommand(s)Question being answered
1. Loaduptime, cat /proc/loadavgIs load high? Is it climbing? (context only — don't conclude "CPU-bound" yet)
2. CPUvmstat 1, mpstat -P ALL 1, topIs us/sy pegged? Is one core maxed while others idle (single-threaded bottleneck)? Is %wa high (points to disk, not CPU)?
3. Memoryfree -h, vmstat 1 (si/so), dmesg (OOM)Is available low? Is the system actively swapping? Any recent OOM kills?
4. Diskiostat -xz 1, df -h, df -iIs await/%util high? Out of space or out of inodes?
5. Networkss -s, sar -n DEV 1, ip -s linkErrors/drops on an interface? Saturated bandwidth? Connection backlog?

Once a saturated/erroring resource is found, switch to the per-process view for that resource (pidstat, top sorted accordingly) to identify which process is responsible, then drill in with strace/lsof//proc/<pid> as needed.
