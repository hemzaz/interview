CPU Execution

How a core turns instructions into work — cycle, pipeline, cache

🔄The Instruction Cyclewhat one clock tick's worth of work looks like
⏱️ Clock & Cycle Basics
ClockA cycle is one tick of the clock. 3 GHz = 3 billion cycles/sec ≈ 0.33 ns each.
IPCModern cores retire several instructions per cycle (superscalar) — IPC > 1.
CPICycles-per-instruction. A cache miss can push effective CPI way up while the core stalls.
RegistersThe fastest storage — 0 cycles, on-core. Everything below is the memory hierarchy.
🧬 Fetch · Decode · Execute · Mem · Writeback
Every instruction flows through the same classic stages. On real cores these run overlapped (see pipeline below), but the logical order holds:
StageWhat happens
FetchRead next instruction from I-cache at the program counter
DecodeTranslate bytes into micro-ops; read source registers
ExecuteALU does the math / branch / address calc
MemoryLoad or store to D-cache if the op touches memory
WritebackWrite the result back into the register file

The win: while instruction 1 is in Execute, instruction 2 is Decoding and instruction 3 is being Fetched. Throughput approaches one instruction finished per cycle even though each takes 5 cycles end-to-end.
🌿 Branch Hazard
The CPU doesn't know which way an if goes until Execute — but it's already fetched the next instructions. Branch prediction guesses; a mispredict flushes the pipeline (~15–20 cycles lost).
🔗 Data Hazard
Instruction 2 needs a result instruction 1 hasn't written yet. Fixed by forwarding (bypass the result early) or, failing that, a stall (bubble).
🎲 Out-of-Order
Big cores execute ready instructions out of program order and retire them in order — hiding latency by finding independent work while one op waits on memory.

Read the gap: RAM is ~100× slower than L1. If your data isn't in cache, the core stalls for hundreds of cycles doing nothing. This is why data layout and access patterns often beat clever algorithms.
🎯 Locality — why cache works
TemporalRecently used data is likely reused soon → keep it close.
SpatialNearby addresses are likely accessed together → fetch a whole cache line (64 B), not one byte.
PrefetchHardware spots stride patterns and pulls lines in before you ask.
⚡ Practical consequences
• Sequential array walk ≫ linked-list chase (pointer chasing thrashes cache).
• Struct-of-arrays can beat array-of-structs for hot loops.
• False sharing: two threads writing different vars in the same 64 B line ping-pong the line between cores — a silent killer.
Interview trap: "Your loop is slow but the CPU is at 100%." High utilization doesn't mean useful work — the core may be stalled on memory, burning cycles waiting on RAM. Check cache-miss counters (perf stat), not just CPU %.
