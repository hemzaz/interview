01 Virtual memory & address spaces

Every process sees a private, contiguous address space. The kernel + MMU translate those virtual addresses to physical frames on the fly, so processes are isolated and physical RAM can be shared, overcommitted, and paged.

Why virtual memory exists

  • Isolation: a process cannot read/write another process's physical RAM — each has its own page tables.
  • Abstraction: code links against fixed virtual addresses regardless of where it physically lands (enables relocation, shared libraries).
  • Overcommit & sharing: more virtual memory can be handed out than physical RAM; identical pages (libc, COW pages) are shared.
  • Protection: per-page permissions (R/W/X) enforce W^X, guard pages, read-only text.

User vs kernel space split (x86-64)

Address space is split into a lower user half and an upper kernel half. With standard 4-level paging (48-bit virtual addresses), the canonical split is:

RegionRange (x86-64, 48-bit)Notes
User space0x0000000000000000 – 0x00007fffffffffff~128 TiB per process; text/data/heap/stack/mmap live here
Non-canonical holemiddleUnusable — bits 48–63 must sign-extend bit 47
Kernel space0xffff800000000000 – 0xffffffffffffffffShared across all processes, mapped only in kernel mode

With 5-level paging (57-bit VA, opt-in) the user half grows to ~128 PiB. The kernel half is the same physical kernel mapping in every process's page table — accessible only when CPU is in ring 0.

ASLR (Address Space Layout Randomization)

The kernel randomizes the base of the stack, mmap region (shared libs), heap (brk), and — for PIE binaries — the executable itself, at each exec(). Defeats attacks that need fixed addresses (ROP gadgets, GOT overwrite). Controlled by /proc/sys/kernel/randomize_va_space: 0=off, 1=stack/mmap/vDSO, 2=full (default). Disable per-process with setarch -R for debugging.

02 Process memory segments

A process image is a set of virtual memory areas (VMAs). Inspect them live via /proc/<pid>/maps.

# See a live process's address space
cat /proc/self/maps
# 55f... r-xp  ...  /usr/bin/cat        <- text
# 55f... rw-p  ...  [heap]
# 7f7... r-xp  ...  /usr/lib/libc.so.6  <- mmap'd shared lib
# 7ff... rw-p  ...  [stack]

text/data/bss are the classic ELF segments fixed at link time. The heap grows via brk/sbrk; large allocations and shared libraries go in the mmap region between heap and stack.

03 Paging, page tables & the MMU

Physical RAM and virtual space are both divided into fixed-size pages. The MMU walks per-process page tables to translate a virtual page number to a physical frame number on every memory access.

Pages

  • Base page size is 4 KiB on x86-64 and most architectures (ARM64 supports 4/16/64 KiB configured at kernel build).
  • Virtual address = virtual page number + page offset (low 12 bits for 4 KiB pages).
  • The offset is copied verbatim; only the page number is translated.

Multi-level page tables (x86-64)

A flat page table would be enormous, so tables are hierarchical and sparse. Standard x86-64 uses 4 levels; opt-in 5-level paging (LA57) extends VA to 57 bits:

LevelName (Linux)x86 name
5 (opt-in)PGD → P4DPML5
4PGDPML4
3PUDPDPT
2PMDPD (page directory)
1PTEPT (page table)

The MMU walks 4 (or 5) levels, each indexed by 9 bits of the virtual address, ending at a page table entry (PTE). CR3 holds the physical address of the top-level table; it's reloaded on every context switch, which is what makes address spaces per-process.

The page table entry (PTE)

A PTE holds the physical frame number plus control bits: Present (in RAM?), R/W, User/Supervisor, Accessed, Dirty, NX (no-execute), and — when Present=0 — swap location or "not mapped." A cleared Present bit is exactly what triggers a page fault.

⚠ Trap

Virtual addresses are not physical addresses. Two processes can hold the same virtual address pointing at completely different physical frames — or the same frame (shared library, shared memory). Never reason about aliasing from virtual addresses alone.

04 The TLB & shootdowns

A full page-table walk is 4–5 memory accesses per translation. The TLB (Translation Lookaside Buffer) is a small CPU cache of recent virtual→physical translations that makes the common case one lookup.

  • TLB hit: translation cached → near-zero cost.
  • TLB miss: the MMU (hardware, on x86) performs a page-table walk, then caches the result. Costs tens–hundreds of cycles.
  • Coverage: a few thousand entries × 4 KiB is only a few MiB of memory "reachable" without a walk — huge pages exist largely to extend this.

Flush on context switch — and PCID/ASID

TLB entries belong to a specific address space. Naively, switching processes (reloading CR3) must flush the entire TLB, so the new process doesn't read stale translations. That refill cost hurts.

PCID (Process-Context ID, x86) / ASID (Address Space ID, ARM) tag each TLB entry with an address-space identifier, so entries survive a context switch and only entries matching the current PCID are used. This avoids full flushes and cuts switch cost. (KPTI/Meltdown mitigation reintroduced some flushing pressure, partly recovered via PCID.)

TLB shootdown

The TLB is per-CPU. When one core changes a mapping (unmaps a page, changes permissions, migrates a page), other cores may hold stale TLB entries for it. The originating core sends an IPI (inter-processor interrupt) to force other cores to invalidate the relevant entries. This "TLB shootdown" is a real scalability cost — heavy munmap/mprotect/migration on many-core boxes shows up as time in smp_call_function.

Q&A

Q: Why does frequent mprotect on a busy multithreaded server hurt?

A: Each permission change may require invalidating stale TLB entries on every core that ran the process — a TLB shootdown storm of IPIs, serializing cores.

05 Page faults & demand paging

A page fault is a CPU trap raised when a virtual access can't be satisfied by the current page tables. The kernel's fault handler decides: fix it up, fetch from disk, or kill the process. Three flavors matter.

Fault typeCauseKernel actionCost
Minor (soft)Page is already in RAM (page cache, zero page, COW) but not mapped in this PTEJust install/adjust the PTE — no I/OCheap (µs)
Major (hard)Page must be read from disk: swap-in or first read of a file-backed page not in page cacheBlock the thread, issue I/O, map when doneExpensive (ms — disk I/O)
InvalidAccess with no valid VMA or wrong permissions (write to r/o, NULL deref)Deliver SIGSEGV (or SIGBUS)Process usually dies

Demand paging & lazy allocation

Linux does not physically back memory until it's touched. malloc/mmap only reserve virtual address space and set up VMAs; the first write to each page triggers a minor fault that allocates a physical frame (or maps the shared zero page on first read). This is why:

  • A 10 GiB malloc can succeed instantly with no free RAM movement — nothing is backed yet.
  • Newly allocated memory reads as zeros (zero-fill on demand) without cost until written.
  • RSS grows as pages are actually touched, not at allocation time.

Anonymous vs file-backed pages

Anonymous

Heap, stack, BSS, MAP_ANONYMOUS. No file behind them. When reclaimed, they must go to swap (or be dropped if clean/zero). Under memory pressure with no swap, these are unreclaimable → OOM.

File-backed

Program text, mmap'd files, page cache. Backed by a file. Clean pages can be dropped and re-read from disk for free; dirty pages must be written back first.

⚠ Trap

A thread stuck in a major fault shows up as D state (uninterruptible sleep) in ps/top — it's blocked on I/O, not on CPU or a normal signal. A pile of D-state threads usually means swap thrashing or slow storage, not a deadlock. SIGKILL won't touch a D-state task until the I/O completes.

Q&A

Q: How do you tell minor vs major faults for a process?

A: ps -o min_flt,maj_flt -p <pid>, /proc/<pid>/stat fields 10/12, or /usr/bin/time -v ("major/minor page faults"). High major fault rate = disk/swap I/O in the hot path → latency killer. High minor is normal (COW, first-touch).

06 Page cache, dirty pages & writeback

Linux caches file data in RAM in the page cache. Reads hit RAM instead of disk; writes land in RAM first and are flushed later. This is why "free" memory always looks low — the kernel uses spare RAM as cache.

  • Clean pages match disk — instantly reclaimable (just drop them).
  • Dirty pages were written and not yet flushed. They must be written back before reclaim. Tracked in /proc/meminfo as Dirty: and Writeback:.
  • Writeback is done by kernel flusher threads, driven by vm.dirty_ratio, vm.dirty_background_ratio, and expiry timers, or forced by fsync/sync.
# Drop clean page cache (test/benchmark only — not a fix for prod)
sync; echo 1 > /proc/sys/vm/drop_caches   # page cache
echo 3 > /proc/sys/vm/drop_caches         # + dentries/inodes
⚠ Trap

Page cache is reclaimable — it is not "used up" memory. Low free with high buff/cache is healthy. Judge headroom by available, not free (see §13).

07 Copy-on-write, fork & exec

COW lets fork() be cheap: parent and child share physical pages read-only until one writes. The write faults, the kernel copies that single page, and each side gets its private copy.

fork + COW

  1. fork() duplicates the page tables, not the pages. All writable private pages are marked read-only in both parent and child, with a COW flag.
  2. Either process reading a page: fine, shared.
  3. Either process writing: a minor (protection) fault fires; the kernel allocates a fresh frame, copies the page, marks it writable, and resumes. Only the touched page is copied.
⚠ Trap

COW is why a large process can fork() quickly, but also why a forking server (e.g. Redis BGSAVE) can see RSS balloon if the parent keeps writing — every written page gets duplicated. Under strict overcommit this can even OOM despite "just forking."

exec replaces the address space

execve() does not create a process — it replaces the current one's image. The old address space (VMAs, mappings, heap, stack) is torn down and a fresh one is built from the new ELF: new text/data mapped from the binary, fresh BSS/heap/stack, ASLR re-rolled. The PID and open file descriptors (unless O_CLOEXEC) survive; the memory image does not. Hence the classic fork()-then-exec() pattern: COW-cheap fork, then blow it all away with exec.

08 Swap, swappiness & thrashing

Swap is disk (or a swap file / zram device) used to hold anonymous pages evicted from RAM under pressure. It buys capacity at the cost of latency — a swapped page costs a major fault to get back.

  • Swap-out: the reclaim path (kswapd / direct reclaim) writes cold anonymous pages to swap and frees the frames.
  • Swap-in: touching a swapped page → major fault → disk read → remap. Slow.
  • Thrashing: the working set exceeds RAM, so the system spends most of its time swapping in/out instead of doing work. Symptoms: high major-fault rate, huge si/so in vmstat, near-zero throughput, everything in D state.

vm.swappiness

A 0–200 knob (100 = "1:1") biasing reclaim between evicting file-backed page cache vs swapping anonymous pages. Default is 60. Lower → prefer dropping page cache, swap less; higher → swap anonymous more readily. 0 does not fully disable swap; it only avoids swapping until the file cache is nearly exhausted.

cat /proc/sys/vm/swappiness         # 60 by default
sysctl -w vm.swappiness=10          # DB boxes often lower it
free -h                             # see Swap: total/used/free
vmstat 1                            # watch si/so (swap in/out KB/s)

zram & zswap

zram

A compressed block device in RAM used as swap. Pages are compressed and kept in memory instead of hitting disk — effective on RAM-constrained systems (containers, embedded, ChromeOS). No disk needed.

zswap

A compressed writeback cache in front of a real swap device. Pages are compressed in a RAM pool; only when that fills do they spill to disk swap. Reduces disk swap I/O and latency.

09 Overcommit & the OOM killer

Linux hands out more virtual memory than it can physically back, betting most is never touched. When that bet fails and reclaim can't free enough, the OOM killer picks a victim.

vm.overcommit_memory modes

ModeNameBehavior
0 (default)HeuristicAllow "reasonable" overcommit; reject wildly large single requests. Most allocations succeed.
1AlwaysNever refuse. Even absurd allocations succeed. Used by workloads that sparsely map huge regions (some scientific/HPC, certain DBs).
2Never (strict)Commit is capped at swap + overcommit_ratio% × RAM. malloc can genuinely fail — no OOM surprises, but you must size it right.
sysctl vm.overcommit_memory      # 0 default
sysctl vm.overcommit_ratio       # 50 (used only in mode 2)
grep -i commit /proc/meminfo     # CommitLimit / Committed_AS
⚠ Trap

On default Linux, malloc almost never returns NULL — it only reserves virtual space. You discover you're out of memory later, when a write touches a page and there's no frame to give → the OOM killer fires. So "check malloc for NULL" is nearly useless as an OOM defense on Linux; the failure surfaces at first-touch, not at allocation.

The OOM killer

When the kernel truly can't reclaim enough to satisfy a fault, it invokes the OOM killer to free memory by killing a process. Selection is by oom_score, roughly proportional to the process's memory footprint (favoring big anonymous consumers), adjusted by oom_score_adj.

  • /proc/<pid>/oom_score — the computed badness (higher = more likely victim).
  • /proc/<pid>/oom_score_adj — tunable bias, range −1000 … +1000. −1000 makes a process effectively immune (score floored). Positive values make it a preferred target.
  • The kill and the chosen victim are logged to the kernel ring buffer — check dmesg / journalctl -k for "Out of memory: Killed process ..." and the per-task RSS table.
# Protect a critical daemon from the OOM killer
echo -1000 > /proc/<pid>/oom_score_adj
# See who died and why
dmesg | grep -i -E "oom|killed process"
journalctl -k | grep -i "out of memory"
⚠ Trap

The OOM killer does not simply kill "the biggest process." It kills the highest oom_score, which factors in oom_score_adj, and it's constrained to the memory context that triggered it (a cgroup-v2 memory.max breach kills within that cgroup, not system-wide). A tuned oom_score_adj or a container boundary can make a smaller process the victim.

10 Huge pages (HugeTLB & THP)

Bigger pages mean fewer PTEs and far better TLB coverage — one 2 MiB entry replaces 512 × 4 KiB entries. Great for large working sets (databases, JVMs, VMs); the tradeoffs differ between the two mechanisms.

Explicit HugeTLBTransparent Huge Pages (THP)
Sizes (x86-64)2 MiB, 1 GiB2 MiB (PMD-level)
SetupReserve pool up front (vm.nr_hugepages), app opts in via mmap(MAP_HUGETLB) / hugetlbfsAutomatic — kernel promotes/collapses transparently; khugepaged merges
GuaranteePinned, never swapped, deterministicBest-effort; can be split back to 4 KiB
ControlExplicit reservations/sys/kernel/mm/transparent_hugepage/enabled: always/madvise/never

THP tradeoffs

  • Win: fewer TLB misses, less page-table walking, better throughput for large contiguous allocations.
  • Latency risk: allocating a 2 MiB page may require compaction (defragmenting physical memory) — a stall. khugepaged also uses CPU. Latency-sensitive DBs (MongoDB, Oracle, Redis guidance) frequently recommend never or madvise to avoid unpredictable pauses.
  • Bloat: promoting a mostly-empty 2 MiB region wastes RAM (internal fragmentation).
cat /sys/kernel/mm/transparent_hugepage/enabled   # [always] madvise never
grep -i huge /proc/meminfo                          # AnonHugePages, HugePages_*
echo madvise > /sys/kernel/mm/transparent_hugepage/enabled

madvise is the common middle ground: THP only where the app explicitly asks via madvise(MADV_HUGEPAGE), avoiding surprise stalls elsewhere.

11 NUMA (Non-Uniform Memory Access)

On multi-socket servers, each CPU socket has its own local memory bank (a NUMA node). Accessing local memory is faster than reaching across the interconnect to another node's memory. Placement matters for throughput and latency.

  • Locality: a thread should run on the same node whose memory it uses. Cross-node access adds interconnect latency and contends for a shared link.
  • First-touch policy: a page is allocated on the node of the CPU that first writes it, not where it was malloc'd. So initialize data on the thread/node that will use it.
  • AutoNUMA: the kernel (numa_balancing) samples access patterns and migrates pages/threads toward locality automatically. Helps general workloads; can add migration overhead for carefully hand-placed ones.
numactl --hardware            # nodes, sizes, inter-node distances
numactl --cpunodebind=0 --membind=0 ./app   # pin CPU + memory to node 0
numastat -p <pid>             # per-node memory usage of a process
cat /proc/sys/kernel/numa_balancing         # 1 = AutoNUMA on
Q&A

Q: A DB is slow on a 2-socket box despite low CPU. Where do you look?

A: NUMA remote-access ratio (numastat). If memory is skewed onto one node while threads run on both, cross-node traffic serializes. Fix with numactl pinning, interleave for shared buffers (--interleave=all), or per-node worker affinity + first-touch init.

12 Allocators — kernel & user space

Two allocator layers matter: the kernel allocating physical frames and small kernel objects, and user-space malloc carving up virtual memory it gets from the kernel.

Kernel: buddy + slab/slub

The buddy allocator hands out physically contiguous power-of-two page blocks. The slab/slub allocator sits on top for small, frequent, same-size kernel objects (inodes, task_struct, dentries), keeping per-type caches to avoid fragmentation and speed alloc/free. SLUB is the modern default. Inspect via /proc/slabinfo and slabtop.

User space: glibc malloc

Gets memory from the kernel two ways: brk/sbrk grows the heap for small allocations; mmap is used for large ones (default threshold ~128 KiB, dynamic) and returned directly to the kernel on free. Multiple arenas (per-thread heap regions) reduce lock contention in threaded programs. Freed small memory often stays mapped in the process (returned to malloc's free lists, not the OS) — RSS may not drop after free.

⚠ Trap

Calling free() rarely shrinks RSS. Small freed chunks go back to the allocator's free lists (still mapped); only mmap-backed large allocations are unmapped immediately. "Memory not released after free" is usually allocator retention or fragmentation, not a leak. malloc_trim() can hint some back.

13 Measuring memory

The single most confused topic in interviews. Know exactly what each number counts — especially around shared pages and reclaimable cache.

Per-process metrics

MetricMeansWatch out
VSZ (virtual size)Total virtual address space reserved (mapped VMAs), touched or notWildly overcounts real usage — includes untouched malloc, mapped-but-unused files, guard regions
RSS (resident set)Physical pages currently in RAM for this processDouble-counts shared pages — every process mapping libc counts libc's RSS in full
PSS (proportional set)Private pages + shared pages ÷ number of sharersThe honest per-process number; PSS across all processes ≈ total RAM used. From /proc/<pid>/smaps
USS (unique set)Only private pages — what's freed if the process diesBest "how much does THIS process cost me" figure
# PSS/USS breakdown
grep -E "^(Rss|Pss|Private|Shared)" /proc/<pid>/smaps_rollup
# Sorted proportional set sizes across processes
smem -tk -c "pid user command pss rss uss"
⚠ Trap

VSZ overcounts, RSS double-counts. Summing RSS across processes overstates real RAM because shared libraries are counted once per process. For "how much memory is this actually using," use PSS (fair share of shared) or USS (private only). VSZ tells you almost nothing about physical usage.

System-wide: free — available vs free

free -h
#               total   used   free   shared  buff/cache  available
# Mem:           31Gi   8Gi    1Gi    0.5Gi   22Gi        21Gi
  • free = RAM sitting completely idle. Almost always low — Linux fills spare RAM with page cache on purpose.
  • buff/cache = page cache + buffers, mostly reclaimable.
  • available = the kernel's estimate of what a new workload could get without swapping — free + reclaimable cache − low-watermark reserves. This is the number that matters.
⚠ Trap

"Only 1 GiB free!" on a 32 GiB box is not a problem if available is 21 GiB. Overcommit + page cache mean free ≠ available. Alert on available (and swap-in rate), never on free.

cgroup v2 memory (containers)

In containers/systemd, memory is bounded and measured per-cgroup, not per-host. This is where container OOMs actually get decided.

FileMeaning
memory.currentCurrent bytes charged to the cgroup (anon + page cache + kernel)
memory.maxHard limit — exceeding it triggers reclaim, then the cgroup OOM killer
memory.highSoft limit — over it, allocations are throttled (reclaim pressure) but not killed
memory.statDetailed breakdown: anon, file, slab, pgfault, pgmajfault, etc.
memory.eventsCounters incl. oom and oom_kill — proof a container was OOM-killed
cat /sys/fs/cgroup/<path>/memory.current
cat /sys/fs/cgroup/<path>/memory.max        # "max" = unlimited
grep -E "anon|file|pgmajfault" /sys/fs/cgroup/<path>/memory.stat

14 Interview traps & Q&A — rapid fire

The recurring gotchas senior interviewers probe. If you can explain each in one breath, you're solid.

⚠ Trap — malloc doesn't touch pages

Under demand paging, malloc/mmap only reserve virtual space. No physical frame is allocated until the first write (first read maps the shared zero page). So RSS after a big malloc is ~unchanged, and initialization (memset/calloc touch) is what actually costs RAM and time.

⚠ Trap — VSZ vs RSS vs PSS

VSZ = virtual reservation (overcounts). RSS = resident pages but shared libs double-counted across processes. Use PSS for a fair per-process share, USS for what dies with the process.

⚠ Trap — free ≠ available

Low free is normal (page cache eats it). Overcommit means committed virtual > physical. Judge memory health by available and swap-in rate, not free.

⚠ Trap — OOM killer target

It kills highest oom_score (weighted by oom_score_adj, −1000…+1000, and by the triggering cgroup) — not automatically the largest process. Check dmesg for the actual victim and the RSS table.

⚠ Trap — D-state on major faults

Uninterruptible sleep (D) usually means blocked on I/O — often a major fault / swap-in. Not a hang, not SIGKILL-able until I/O returns. A cluster of D-state = thrashing or slow storage.

Q&A

Q: Minor vs major page fault?

A: Minor = page already in RAM (page cache / zero page / COW), just fix the PTE — no I/O. Major = must read from disk (swap-in or uncached file page) — blocks on I/O, orders of magnitude slower.

Q&A

Q: Why is vm.swappiness default 60, and when lower it?

A: 60 balances reclaiming file cache vs swapping anon pages. Latency-sensitive DBs lower it (e.g. 1–10) to keep their working set in RAM and prefer dropping cache; it doesn't fully disable swap even at 0.

Q&A

Q: Process freed gigabytes but RSS didn't drop — leak?

A: Usually not. glibc keeps freed small chunks on its free lists (still mapped); only large mmap-backed allocations unmap on free. Fragmentation or arena retention. malloc_trim() may return some.

Q&A

Q: Why huge pages, and why might they hurt?

A: They cut TLB misses and page-table walks (one 2 MiB entry = 512 × 4 KiB). Hurt via THP: compaction stalls and khugepaged CPU cause latency jitter — hence madvise/never for latency-critical services.

Q&A

Q: What's a TLB shootdown and when does it bite?

A: TLBs are per-core; changing a mapping means IPI-ing other cores to invalidate stale entries. Heavy munmap/mprotect/page migration on many-core systems serializes cores in the shootdown path.

Q&A

Q: How does fork() avoid copying all of memory?

A: Copy-on-write — page tables are duplicated, pages shared read-only; the first write faults and copies just that one page. Cheap fork; RSS grows only as pages diverge.
