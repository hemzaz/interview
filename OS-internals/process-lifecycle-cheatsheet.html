The fork+exec pattern is deliberate: fork() gives you a new process that's a clone of the parent, then execve() swaps in the program you actually want to run. Splitting them lets the child tweak its environment (redirect fds, set up pipes) between the two calls — which is exactly how a shell launches commands.
🔧The Core Callswhat each one actually does
🍴 fork() — the twin
Returns twice0 in the child, the child's PID in the parent, -1 on error.
Copy-on-writeAddress space isn't really copied — pages are shared read-only and duplicated only when written. Fork is cheap.
InheritsOpen fds, cwd, env, signal dispositions. Not: pending signals, locks, timers.
Under the hoodReally clone() with flags controlling what's shared.
🔄 execve() — the transplant
Replaces the current process image — code, data, heap, stack — with a new program loaded from disk. Same PID, brand-new contents. On success it never returns (there's nothing to return to); the old program is gone. Open fds survive unless marked close-on-exec.
pid_t pid = fork(); if (pid == 0) { // child execve("/bin/ls", argv, envp); _exit(127); // only if exec failed } else { // parent waitpid(pid, &status, 0); }
🚪 exit() — leaving
Flushesexit() runs atexit handlers & flushes stdio; _exit() skips them.
FreesKernel releases memory, closes fds — but keeps a tiny stub (exit status) until reaped.
StatusLow 8 bits available to parent via wait; convention: 0 = success.
🧹 wait() — reaping
Parent's dutywait()/waitpid() retrieves the child's status and lets the kernel drop the stub.
SIGCHLDKernel signals the parent when a child dies; handlers often reap there.
Blockingwait blocks; WNOHANG polls without blocking.
🧟Zombies & Orphansthe two failure modes everyone gets asked
🧟 Zombie — child died, parent didn't reap
A finished child stays as a zombie (Z / defunct) until the parent calls wait(). It holds no memory — just its exit status in the process table. Harmless one-off; dangerous in bulk: a parent that never reaps leaks PT entries until the table fills and no new process can start.
Fix: parent must wait(), or handle SIGCHLD, or ignore it (signal(SIGCHLD, SIG_IGN)) so the kernel auto-reaps.
👻 Orphan — parent died first
If the parent exits while the child still runs, the child is orphaned and re-parented to init/systemd (PID 1). That new parent reaps it when it finishes — so orphans, unlike zombies, get cleaned up automatically. Intentional orphaning is how daemons detach from their launching shell.
Containers: PID 1 inside a container must reap orphans too. A shell as PID 1 usually won't → use an init like tini, or docker run --init, to avoid zombie pile-up.

🧵 Process vs Thread
AspectProcess
Address spaceOwn, isolated (threads: shared)
Created byfork (threads: pthread_create→clone)

🔗 Shared memory → needs sync
Because threads share the heap and globals, concurrent access must be coordinated or you get data races:
PrimitiveGuards
MutexExclusive access to a critical section
SemaphoreCounted access to N resources
Cond varWait/signal on a condition
AtomicsLock-free single-word updates
In Linux, threads are just tasks that share more via clone() flags. The scheduler treats a thread and a process as the same unit — a schedulable task. pthread_exit() ends one thread; the process ends when the last thread (or an exit()) does.
Interview trap: "What does fork() do in a multithreaded program?" Only the calling thread survives in the child — all other threads vanish. Any mutex they held is now locked forever in the child. That's why post-fork you should only exec() or call async-signal-safe functions.
