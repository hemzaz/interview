The ⇄ arrows are the scheduler's job. A task oscillates between runnable and running as it's given and loses the CPU, and drops to sleeping when it waits on I/O or a lock — freeing the core for someone else. This mirrors the sleep()/yield() → schedule() → context switch loop.
R · Running/Ready
On a CPU or waiting in a run queue for one. ps shows R.
S · Interruptible
Sleeping, wakeable by signal or event. Most idle processes. ps shows S.
D · Uninterruptible
Blocked in kernel (usually disk I/O), can't be interrupted. Drives load average. D.
Z · Zombie
Exited but not yet reaped by parent's wait(). Holds only its exit status.

Then: the scheduler picks the next task, does a context switch, and that task runs on the CPU until the next trigger — exactly the cycle in your linux_scheduler sequence.
🔀 Context Switch — the cost
What savesRegisters, program counter, stack pointer of the outgoing task → its task_struct.
Then loadsThe incoming task's saved state; if it's a different process, also swaps the address space (page tables).
Direct cost~1–5 µs of pure switch work.
Hidden costCold caches & TLB flush on the new address space — often the bigger hit.
Interview trap: context switch vs mode switch. A mode switch (syscall) changes privilege ring but stays in the same task — cheap. A context switch changes which task runs, may swap page tables, and trashes caches — much more expensive. Thread↔thread switches skip the address-space swap, so they're cheaper than process↔process.
⚖️CFS — The Completely Fair SchedulerLinux's default policy for normal tasks
💡 The Idea
Instead of fixed time slices, CFS tracks each task's vruntime (virtual runtime — CPU time consumed, weighted by nice). It always runs the task with the lowest vruntime: whoever's had the least share of the CPU goes next. Tasks are ordered in a red-black tree keyed by vruntime, so "pick next" is O(log n).
🎛️ Knobs & Behavior
nice / weightnice -20 (highest) … +19 (lowest). Lower nice = vruntime grows slower = more CPU.
I/O-bound tasksSleep a lot → low vruntime → get scheduled promptly when they wake. Good for interactivity.
GranularityTarget latency is sliced across runnable tasks; more tasks = shorter slices.
SuccessorNewer kernels (6.6+) ship EEVDF, refining CFS's fairness with deadlines.

🧩 SMP — one run queue per CPU
Per-CPU rqEach core has its own run queue → no global lock contention.
Load balancingPeriodically migrates tasks from busy cores to idle ones (your diagram's "load balancing" step).
AffinityKeep a task on its core to preserve warm cache; set with taskset / sched_setaffinity.
NUMAPrefer running near the task's memory node to avoid slow remote access.
Interview trap: "High load average but low CPU usage — how?" Load average counts runnable + uninterruptible (D) tasks. Processes stuck in D on slow disk I/O inflate load without burning CPU. Load ≠ CPU utilization.
