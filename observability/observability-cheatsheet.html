The Three Pillars — Metrics, Logs, Traces

Each answers a different question. Mature observability correlates all three (a metric alert → jump to logs → follow the trace). "Three pillars" is the interview framing; note the modern critique that they're really three views over one event stream (OpenTelemetry unifies them).

SignalWhat it isAnswersCost / cardinalityRetention
MetricsNumeric time series, regularly sampled & aggregated (name + labels + value + timestamp)"Is something wrong? How much / how fast?" — trends, SLOs, alertingCheap per-sample; cost driven by label cardinality (# series)Long (weeks–years cheaply)
LogsTimestamped discrete events, ideally structured (JSON)"What exactly happened?" — detailed per-event contextHigh volume; cost driven by ingest bytes & index cardinalityMedium (days–weeks)
TracesCausally-linked spans following one request across services"Where in the system & why is it slow?" — latency breakdown, dependenciesHigh if unsampled; usually sampledShort (hours–days)

Rule of thumb: alert on metrics, diagnose where with traces, find the exact cause in logs. Correlate by injecting trace_id into logs and exemplars into metrics.

Q&A — "Observability vs monitoring?" Monitoring = watching known failure modes with predefined dashboards/alerts ("known-knowns"). Observability = ability to ask arbitrary new questions about internal state from external outputs without shipping new code — critical for "unknown-unknowns" in distributed systems. High-cardinality, wide structured events are what make a system observable.

Prometheus Architecture

Prometheus is a pull-based metrics system with a local time-series database (TSDB). The server scrapes HTTP /metrics endpoints on an interval, stores samples locally, and evaluates rules. It is a CNCF graduated project.

Core components & flow

  • Scrape / pull model: Prometheus initiates HTTP GET to each target's /metrics every scrape_interval (commonly 15–60s). Targets don't push.
  • Service discovery (SD): targets discovered dynamically — Kubernetes, Consul, EC2, file_sd, DNS, etc. — instead of static lists. relabel_configs filter/rewrite targets before scrape; metric_relabel_configs act on scraped samples (drop noisy metrics).
  • TSDB: local, append-only. Recent samples in an in-memory head block + WAL (crash recovery); every ~2h compacted to immutable on-disk blocks, then merged. Not clustered — single node stores its own data.
  • Exposition format: plain text, one sample per line: metric{label="v"} 1.27 [timestamp]. OpenMetrics is the standardized successor.
  • Exporters: sidecar processes that translate third-party stats into Prometheus format — node_exporter (host CPU/mem/disk/net), blackbox_exporter (probe endpoints), cAdvisor (containers), plus DB/queue exporters.
  • Pushgateway: for short-lived batch jobs that die before a scrape. Push metrics to it; Prometheus scrapes the gateway. Caveats: NOT for service metrics; it becomes a single point of failure, doesn't expire pushed metrics (stale values persist until deleted), and breaks up-based liveness. Use sparingly.

Scaling & long-term storage

  • remote_write: stream samples to an external system (Thanos Receive, Mimir, Cortex, VictoriaMetrics, cloud TSDB) for durable/centralized storage. remote_read queries them back.
  • Federation: a higher-level Prometheus scrapes aggregated series from many lower Prometheus servers via /federate. For hierarchical aggregation, not full data replication — don't use it to ship all raw series.
  • HA pairs: run two+ identical Prometheus servers scraping the same targets. They're independent (no shared state); a deduplicating layer (Thanos Querier, Mimir) merges them at query time.
  • Thanos / Cortex / Mimir: add global query view, long-term object-storage (S3/GCS), downsampling, and horizontal scale. Thanos = sidecar + object store + global querier; Cortex & its successor Grafana Mimir = horizontally-scalable, multi-tenant remote-write backends.

Recording vs alerting rules

Recording ruleAlerting rule
PurposePrecompute expensive/frequent expressions into a new seriesFire alerts when an expression is true over time
OutputA new time series (naming: level:metric:operation)Alerts sent to Alertmanager
Key fieldsrecord, expralert, expr, for, labels, annotations
 
# prometheus rule group
groups:
  - name: example
    rules:
      - record: job:http_requests:rate5m
        expr: sum by (job) (rate(http_requests_total[5m]))
      - alert: HighErrorRate
        expr: job:http_errors:rate5m / job:http_requests:rate5m > 0.05
        for: 10m          # must stay true 10m before firing (debounce)
        labels: { severity: page }
        annotations: { summary: "5xx > 5% on {{ $labels.job }}" }
⚠ Trap — pull vs push Interviewers love "why pull?" Pull gives Prometheus a built-in health signal (the synthetic up metric = did the scrape succeed), easy target discovery, and no client-side buffering. Downsides: firewalls/NAT/short-lived jobs (→ Pushgateway) and very ephemeral serverless (→ OTel push or agent). Prometheus is pull; don't claim it pushes.

Metric Types

Counter

Monotonically increasing (resets to 0 on restart). Only ever goes up. Never query the raw value — always rate()/increase().

e.g. http_requests_total, errors_total

Gauge

Value that goes up and down. Snapshot of current state. Query directly, or with avg_over_time, delta, deriv.

e.g. memory_bytes, queue_depth, temperature

Histogram

Samples into cumulative buckets (_bucket{le=...}) + _sum + _count. Buckets aggregate across instances → server-side quantiles via histogram_quantile().

e.g. request latency, response size

Summary

Client-computed streaming quantiles ({quantile=...}) + _sum + _count. Cheap to read, but quantiles can't be aggregated across instances.

Use when you need a preset percentile from one instance

Histogram vs Summary — the classic question

HistogramSummary
Quantiles computedAt query time (server)At ingest time (client)
Aggregatable across instances?Yes — sum buckets, then histogram_quantileNo — averaging quantiles is statistically invalid
Choose bucket/quantile ahead?Pick buckets ahead; any quantile queryablePick quantiles ahead; fixed φ & error
Client costCheapHigher (streaming quantile calc)
AccuracyBounded by bucket widthConfigurable per-quantile error

Default to histograms for anything you aggregate (SLOs across a fleet). Use summaries only for a single-instance exact quantile where you can't pick buckets. Modern native / exponential histograms (Prometheus 2.40+, still experimental as of 3.x) give high-resolution auto-bucketing at far lower cardinality than classic le buckets.

⚠ Trap — averaging percentiles You cannot average p99s from two instances to get the fleet p99. This is why summaries don't aggregate. With histograms you sum the raw bucket counts first, then compute the quantile once.

PromQL

Instant vs range vectors

  • Instant vector: one sample per series at the eval time — http_requests_total.
  • Range vector: a range of samples per series over a window — http_requests_total[5m]. Required by rate(), increase(), *_over_time().
  • Most functions take one and return the other. rate(counter[5m]) turns a range vector into an instant vector.

rate() vs irate() — and why they need a counter + range

# per-second average rate over the whole 5m window (smoothed) — USE THIS for alerting/graphs
rate(http_requests_total[5m])

# instantaneous rate from the LAST TWO samples in the window — spiky, for fast-moving dashboards only
irate(http_requests_total[5m])

# total count increase over 1h (rate * seconds, extrapolated) — for "how many in the last hour"
increase(http_requests_total[1h])
  • rate()/irate() take a range vector of a counter. They compute per-second increase and automatically correct counter resets (a drop is treated as a restart, not negative).
  • rate() = average over the window (robust, use for alerts & most graphs). irate() = last two data points only (reacts fast, but aliases/hides trends — never alert on it).
  • Pick a range ≥ 4× scrape interval so rate() always has ≥2 samples even with a missed scrape (e.g. 15s scrape → [1m]+). Too-short ranges → gaps/no data.

histogram_quantile with le buckets

# p95 request latency across ALL instances, per route
histogram_quantile(0.95,
  sum by (le, route) (rate(http_request_duration_seconds_bucket[5m]))
)
  • Input must be the _bucket series (cumulative, keyed by the le label). You must keep le in the by() grouping or the quantile is meaningless.
  • Wrap buckets in rate(...[5m]) first (they're counters), then sum by (le, ...) to aggregate across instances, then apply histogram_quantile.
  • Accuracy is bounded by your bucket boundaries; a quantile falling in the top open bucket (+Inf) can't be estimated precisely.

Aggregation, offset, subqueries

# sum by keeps only listed labels; sum without drops listed labels (keeps the rest)
sum by (job, code)   (rate(http_requests_total[5m]))
sum without (instance)(rate(http_requests_total[5m]))

# compare to one hour ago (offset shifts the eval time backward)
rate(http_requests_total[5m] offset 1h)

# subquery: max 5m-rate sampled every 1m over the last hour
max_over_time( rate(http_requests_total[5m])[1h:1m] )

Aggregators: sum, avg, min, max, count, count_values, stddev, topk, bottomk, quantile. by = keep these labels; without = keep all except these.

⚠ Trap — PromQL gotchas that fail candidates
  • rate() on a gauge → garbage. rate/increase assume monotonic counters; gauges go down and get miscorrected as "resets." Use deriv, delta, or *_over_time for gauges.
  • Counter resets: on restart a counter drops to 0. rate/increase handle this; the raw value or a naive delta do not.
  • Staleness: if a target disappears, its series get a "stale" marker and stop returning after ~5m (default). A missing series ≠ zero — up == 0 or absent() detect "gone," and increase won't invent data.
  • increase() is extrapolated — it can return non-integer values (e.g. 7.3) even for integer counters. Don't expect exact counts.
  • Range too short: [1m] with a 60s scrape may have <2 samples → empty result. Widen the range.

RED, USE & the Four Golden Signals

RED services / requests

  • Rate — requests/sec
  • Errors — failed requests/sec (or %)
  • Duration — latency distribution (p50/p95/p99)

Weaveworks / Tom Wilkie. Best for request-driven services. Same 3 metrics for every service = uniform dashboards.

USE resources

  • Utilization — % time resource was busy
  • Saturation — queued/extra work it can't service (run-queue, backlog)
  • Errors — error events

Brendan Gregg. Best for resources — CPU, memory, disk, network, pools. Answers "is this resource the bottleneck?"

Four Golden Signals Google SRE

  • Latency (split success vs error latency!)
  • Traffic — demand on the system
  • Errors — rate of failures
  • Saturation — how "full" the service is

From the Google SRE book. RED ≈ golden signals minus saturation, for user-facing services.

Q&A — "RED vs USE vs golden signals — when which?" They complement, not compete. Use RED / golden signals for request-serving services (the user's view); use USE for the underlying resources (the machine's view). A latency spike (RED) is often explained by resource saturation (USE). Key nuance: always separate success vs error latency — a fast 500 shouldn't flatter your p99.

Grafana (briefly)

  • Datasources: Grafana is a visualization layer over many backends — Prometheus, Loki, Tempo, Mimir, Elasticsearch/OpenSearch, SQL, cloud. It stores dashboards, not the metrics.
  • Dashboards & panels: panels run queries (PromQL/LogQL/TraceQL) against a datasource; time range is shared. Panels: time series, stat, gauge, table, heatmap, logs.
  • Template variables: $namespace, $instance etc. driven by label_values(metric, label) queries → reusable dashboards with dropdowns instead of one per host.
  • Alerting: Grafana has its own unified alerting (can also read Alertmanager). Exemplars let you jump from a latency spike on a graph straight to the trace that produced it.
  • Dashboards as code / provisioning (JSON + files/Terraform) for GitOps.

Alertmanager

Prometheus evaluates alerting rules and pushes firing alerts to Alertmanager, which handles deduplication, grouping, routing, inhibition, silencing, and delivery to receivers. Separation of concerns: Prometheus decides what is firing; Alertmanager decides who hears about it and how.

ConceptWhat it does
Routing treeHierarchical route config matches alerts by label → picks receiver & timing. Child routes inherit/override; continue: true to fall through to siblings.
Groupinggroup_by bundles related alerts (e.g. all pods of a service) into ONE notification. group_wait (initial buffer), group_interval (new alerts in a group), repeat_interval (re-notify unresolved).
InhibitionSuppress lower-priority alerts when a higher one fires (e.g. a whole-cluster-down alert mutes per-node alerts). Matches source/target label sets.
SilencesTime-bounded, label-matcher mutes set by a human (maintenance windows). Not in config — created via UI/API.
ReceiversIntegrations: PagerDuty, Opsgenie, Slack, email, webhook, etc. A route ends at exactly one receiver.
⚠ Trap — alert fatigue Too many noisy/non-actionable alerts train responders to ignore the pager. Fixes: alert on symptoms, not causes (page on SLO/user impact, not every CPU blip); use for: to debounce transient spikes; group + inhibit to collapse storms; route by severity (page vs ticket); and ensure every page is actionable (a runbook link in annotations). If you can't act on it, it's a dashboard, not a page.
Q&A — "Why for: in an alerting rule?" The expression must stay true for the whole for duration before the alert transitions pending → firing. It debounces flapping — a 3-second blip that self-heals never pages. It does not delay resolution.

Logging

Structured logging

  • Emit logs as structured key/value (JSON), not free-text. Machine-parseable, queryable, filterable without brittle regex.
  • Attach consistent context: service, level, trace_id, span_id, request_id, user_id. This is what enables log ↔ trace correlation — click a slow trace, pivot to its logs by trace_id.
  • Levels: TRACE < DEBUG < INFO < WARN < ERROR < FATAL. Prod usually at INFO; DEBUG is high-volume/expensive. Alert off ERROR rate, not individual lines.

Aggregation stacks

StackModelNotes
LokiIndex only labels, store log content compressed (grep-on-read)Grafana-native, cheap, Prometheus-style labels + LogQL. Keep label cardinality low — same trap as metrics.
ELK / EFKElasticsearch + Logstash/Fluentd/Fluent Bit + KibanaFull-text inverted index — powerful search, but storage/ops-heavy at scale. EFK = Fluentd/Fluent Bit shipper (common in k8s).
OpenSearchOpen-source fork of Elasticsearch + OpenSearch DashboardsDrop-in ELK alternative after Elastic's license change; AWS-backed.
⚠ Trap — log cardinality & cost In Loki, putting a high-cardinality value (user id, request id, trace id) in a label explodes the number of streams and destroys performance — keep those in the log line and filter at query time. In Elasticsearch/OpenSearch, "mapping explosion" from too many distinct fields blows up the index. Full-text logging is the biggest observability bill; sample/drop DEBUG, cap retention, and don't log PII.

Distributed Tracing & OpenTelemetry

Core concepts

  • Trace: the whole journey of one request across services, identified by a trace_id.
  • Span: one unit of work (an operation/service call) with start/end time, attributes, status, events. Spans form a tree via parent_span_id; the first is the root span.
  • Context propagation: trace/span ids passed across service boundaries (usually HTTP/gRPC headers) so spans link into one trace. The W3C Trace Context standard defines the traceparent header:
# W3C traceparent: version-traceid-spanid(parentid)-flags
traceparent: 00-4bf92f3577b34da6a3ce929d0e0e4736-00f067aa0ba902b7-01
             │    │                                │                 └ flags (01 = sampled)
             │    │                                └ parent span id (8 bytes)
             │    └ trace id (16 bytes, propagates unchanged)
             └ version
# tracestate: optional vendor-specific key/values, travels alongside

OpenTelemetry (OTel)

  • CNCF vendor-neutral standard + SDKs for all three signals — traces, metrics, logs — with a unified data model and the OTLP wire protocol. It's the de-facto instrumentation standard in 2026; it replaced OpenTracing + OpenCensus.
  • OTel Collector: a standalone pipeline of receivers → processors → exporters. Ingests OTLP (and Prometheus, Jaeger, etc.), batches/filters/samples/enriches, and fans out to backends. Run as an agent (per-host/sidecar) and/or gateway (central). Decouples apps from backend choice.
  • Semantic conventions standardize attribute names (http.request.method, service.name) so data is portable across tools.

Backends

  • Jaeger — CNCF distributed-tracing backend/UI (now OTLP-native).
  • Grafana Tempo — cheap, object-storage trace backend indexed only by trace id (+ TraceQL); pairs with Loki/Mimir.
  • Zipkin — the original open tracer, still around.

Sampling — head vs tail

Head-basedTail-based
Decision madeAt trace start, before outcome knownAfter the whole trace completes
Can keep all errors/slow?No — decision is blind to outcomeYes — that's the point (keep errors + high latency)
Cost / complexityCheap, simple, statelessExpensive — must buffer all spans of a trace (usually in the Collector) until complete
ConsistencyPropagate decision via sampled flag so all services agreeCentral component decides for the full trace
Q&A — "You sample 1% but need every error trace — how?" Head sampling can't guarantee that (the error isn't known at the start). Use tail-based sampling in the OTel Collector: buffer spans per trace, and after completion apply policies — keep 100% of traces with an error status or latency > threshold, sample the rest. Trade-off: memory + a stateful collector tier.

Cardinality Explosion — the #1 cost/perf trap

Cardinality = number of unique time series = product of all label-value combinations. Prometheus creates a separate series per unique label set. Add a label with 10k values and you multiply your series count by 10k — RAM, disk, and query time explode. This is the single most common way to take down a Prometheus.

# series count ≈ product of label-value counts, PER metric
http_requests_total{method, status, endpoint, user_id, request_id}
       5 methods × 6 status × 40 endpoints ×  1,000,000 users  =  1.2 BILLION series  💥

Label discipline — what belongs in a label

  • Good labels (bounded, low cardinality): method, status_code, region, service, route template. Ask: "is the set of values small and stable?"
  • Never a label: user id, email, request/trace id, session id, full URL with ids, timestamps, raw error messages, container id — unbounded values. Put those in logs or trace attributes, not metric labels.
  • Use a route template (/users/:id) not the raw path (/users/48213).
  • Detect it: topk(10, count by (__name__)({__name__=~".+"})), count(count by (label)(metric)), TSDB status page, or prometheus_tsdb_head_series. metric_relabel_configs can drop offending labels at scrape.
  • Cardinality applies to logs (Loki labels) and traces too — same discipline everywhere.
⚠ Trap — "just add a label" A single innocent-looking label (user_id, pod that churns every deploy, or an unbounded error string) can 100×+ your series count overnight and OOM the server. Before adding any label, ask "how many distinct values can this ever take?" If unbounded → it's a log/trace field, not a metric label.

Rapid-fire Traps & Interview Q&A

Q — Why rate() and not just subtract the counter?Raw subtraction breaks on counter resets (restart → drop to 0 → negative delta) and ignores the sampling interval. rate() handles resets and returns per-second, extrapolated over the window.
Q — rate() vs increase()?Identical math; increase(x[t]) ≈ rate(x[t]) × t. rate = per-second (alerting/graphs); increase = total over the window (human "how many"). Both extrapolate, so increase can be fractional.
Q — A metric disappeared. Is it 0?No. A missing series is "no data," not zero. After ~5m it goes stale and stops returning. Use absent(), up == 0, or or vector(0) to reason about "gone." Alerting on absence needs explicit handling.
Q — How does Prometheus know a target is down?Every scrape yields the synthetic up metric (1 = scrape ok, 0 = failed). up == 0 for 5m is the canonical target-down alert — a free benefit of the pull model.
Q — Is Prometheus HA / clustered?No native clustering or sharding. HA = run duplicate independent servers; dedup at query time (Thanos/Mimir). For global view + long-term + horizontal scale you add Thanos, Cortex, or Grafana Mimir on top.
Q — When Pushgateway?Only for short-lived batch/cron jobs that finish before a scrape. Not for long-running services (use pull). It doesn't expire metrics, is a SPOF, and hides individual instance liveness.
Q — Histogram or summary for latency SLOs?Histogram — buckets aggregate across instances so you can compute a fleet-wide p99 with histogram_quantile. Summaries pre-compute per-instance quantiles that can't be combined. Prefer native/exponential histograms in 2026.
Q — Metrics, logs, or traces for "checkout is slow for user X"?Metrics tell you it's slow (p99 up) and alert; a trace shows where (which downstream span dominates); logs (found via the trace_id) give the exact error/context. Correlation across all three is the answer they want.
Q — What is an exemplar?A sampled trace id attached to a metric sample (e.g. a histogram bucket) that lets you jump from an aggregate metric graph directly to a representative trace — the glue between the metrics and traces pillars.
Q — Head vs tail sampling in one line?Head = decide at the start (cheap, blind to outcome). Tail = decide after the trace finishes (keep all errors/slow, but must buffer spans — stateful & costly).
Q — Biggest observability cost driver?Cardinality (metrics/trace/Loki-label series count) and log ingest volume. Control with label discipline, sampling, retention limits, and downsampling — not by buying more storage.
