Architecture — GitOps Toolkit
// 7 modular controllers · each owns a concern · composes via CRDs
📡
source-controller
fetches and caches all sources
▶
What it doesFetches Git repos, OCI artifacts, Helm chart repos, S3 buckets. Stores downloaded content as Artifacts — tarballs with checksums. All other controllers consume Artifacts, never fetching sources themselves.
ArtifactsStored in source-controller's pod storage. Referenced via status.artifact.url. Controllers get the artifact tarball, not a live connection to Git. Decouples fetch from apply.
Reconcile triggerPoll interval (e.g., interval: 1m) + Git webhook for immediate trigger. Webhook receiver creates an Artifact immediately on push — no polling delay.
🗂️
kustomize-controller
applies Kustomizations
▶
What it doesWatches Kustomization CRDs. Fetches Artifact from source-controller, runs kustomize build (or plain kubectl apply), applies to cluster. Health checks resources after apply.
SOPSNative SOPS decryption built in. Set decryption.provider: sops with a Secret containing the age/PGP/KMS key. Decrypts secrets in-memory before apply — never writes plaintext to disk.
Pruneprune: true deletes resources removed from Git. Uses toolkit.fluxcd.io/prune: true label to track ownership. Only prunes resources it created — won't touch unrelated resources.
⎈
helm-controller
reconciles HelmReleases
▶
What it doesManages Helm releases declaratively. Runs helm install/upgrade. On chart version change or values change: auto-upgrades. Configurable auto-rollback on upgrade failure.
Helm v4 v2.8Flux v2.8 adds Helm v4 support: server-side apply for Helm-managed resources (proper field ownership), enhanced health checking. Use install.crds: CreateReplace with Helm v4 for CRD management.
RollbackSet upgrade.remediation.remediateLastFailure: true — helm-controller rolls back automatically if upgrade fails. Configurable retries, strategy, and cleanup of failed releases.
🔔
notification-controller
alerts, webhooks, receivers
▶
Three CRDsProvider (where to send: Slack/PagerDuty/GitHub), Alert (what triggers a notification: Kustomization failed/reconciled), Receiver (what receives webhooks: GitHub push → trigger reconcile).
PR comments v2.8New provider types: githubpullrequestcomment, gitlabmergerequestcomment, giteacomment. Post reconcile status directly on the PR that triggered the change. Closes the feedback loop.
🖼️
image-reflector-controller
scans registries for new tags
▶
ImageRepositoryPolls a container registry at interval. Stores the list of available tags in a status.lastScanResult. Requires imagePullSecret for private registries.
ImagePolicySelects the "latest" tag from an ImageRepository based on a policy: semver range, alphabetical, numerical. Writes the selected tag to status.latestImage. image-automation-controller reads this.
🤖
image-automation-controller
commits image tags back to Git
▶
What it doesWatches ImageUpdateAutomation CRDs. When ImagePolicy detects a new tag, automation-controller finds files in Git annotated with {"$imagepolicy": "..."} markers, updates the image reference, and commits back to Git.
Full loopCI builds + pushes image → image-reflector detects new tag → image-automation commits updated tag to Git → source-controller detects new commit → kustomize/helm controller applies → new pods running. Zero human intervention.

🌊 Flux Operator Recommended
What it isA Kubernetes operator that manages Flux installations declaratively. Instead of flux bootstrap, define a FluxInstance CRD — Flux Operator installs and manages Flux for you, including upgrades.
FluxInstance CRDDeclarative Flux config: which controllers to install, Flux version, storage settings, kustomize patches on Flux manifests. Operator handles upgrades when you change the version.
Web UIFlux Operator includes a Web UI (announced at KubeCon Atlanta 2025). Cluster dashboard, deep-dive for Kustomizations/HelmReleases, workload monitoring. Not as feature-rich as ArgoCD UI but substantially better than CLI-only.
ResourceSetNew Flux Operator CRD: deploy ephemeral preview environments from GitHub PRs / GitLab MRs. Similar to ArgoCD's Pull Request generator. Pair with the githubpullrequestcomment notification provider.
apiVersion: fluxcd.controlplane.io/v1 kind: FluxInstance metadata: name: flux namespace: flux-system spec: distribution: version: "2.x" # auto-patch within 2.x registry: ghcr.io/fluxcd components: - source-controller - kustomize-controller - helm-controller - notification-controller - image-reflector-controller - image-automation-controller cluster: type: kubernetes # or: k3s, openshift
Flux Operator is the recommended installation method for production clusters. It manages Flux upgrades without re-running flux bootstrap. The FluxInstance manifest is stored in Git → GitOps managing GitOps itself.

Source CRDs
// GitRepository · OCIRepository · HelmRepository · Bucket · ExternalArtifact
📁 GitRepository
apiVersion: source.toolkit.fluxcd.io/v1 kind: GitRepository metadata: name: fleet-infra namespace: flux-system spec: url: https://github.com/myorg/fleet-infra interval: 1m # poll interval ref: branch: main # or: tag, commit, semver # tag: v1.0.0 # commit: abc1234 # semver: ">=1.0.0 <2.0.0" secretRef: name: github-credentials # basic-auth or SSH verify: # commit signature verification provider: openpgp secretRef: name: gpg-public-keys ignore: | # exclude from artifact /*.md /docs/
SSH authCreate Secret with identity (private key), identity.pub (public key), known_hosts. Or use flux create secret git command.
Webhook triggerCreate a Receiver pointing to source-controller. Get the webhook URL and configure it in GitHub/GitLab. New commit → immediate reconcile, no polling wait.
GitRepository interval is per-source, not per-Kustomization. A Kustomization with interval: 10m still re-renders every 10m, even if the GitRepository was last fetched 30 seconds ago. The Kustomization interval controls reconcile frequency; the GitRepository interval controls how often new commits are fetched.
🐳 OCIRepository GA v2.6
apiVersion: source.toolkit.fluxcd.io/v1beta2 kind: OCIRepository metadata: name: my-oci-configs spec: url: oci://ghcr.io/myorg/k8s-configs ref: tag: latest # or: digest, semver # semver: ">=1.0.0" # digest: sha256:abc... interval: 5m secretRef: name: ghcr-credentials # imagePullSecret verify: # Cosign signature verification provider: cosign secretRef: name: cosign-public-key
Use casePackage Kustomize configs or Helm charts as OCI artifacts. Distribute configs via a container registry instead of Git. Enables "gitless" deployment for config distribution. Sign with Cosign for supply chain security.
Build with flux CLIflux push artifact oci://ghcr.io/myorg/configs:v1 --path ./k8s packages a directory as an OCI artifact. Pair with flux tag artifact to tag for environment promotion.
OCI artifacts for Helm charts: HelmRepository with type oci fetches charts from OCI registries — same CRD, different URL scheme: url: oci://registry.example.com/charts.
⎈ HelmRepository
# HTTP Helm repo kind: HelmRepository spec: url: https://charts.bitnami.com/bitnami interval: 1h # OCI Helm repo kind: HelmRepository spec: url: oci://ghcr.io/myorg/charts type: oci # required for OCI interval: 5m secretRef: name: ghcr-creds
intervalHow often to re-fetch the Helm repo index. 1h is fine for stable external repos. For your own internal charts where you push frequently: 5m.
🪣 Bucket
kind: Bucket spec: endpoint: s3.amazonaws.com bucketName: my-configs region: us-east-1 provider: aws # aws, gcp, azure, generic interval: 5m secretRef: name: s3-credentials ignore: | /*.txt
Use caseTeams storing config in S3, GCS, or Azure Blob. Useful for CI-generated configs that aren't committed to Git. Also: air-gapped clusters with an internal MinIO.
🔗 ExternalArtifact v2.7
kind: ExternalArtifact spec: url: https://example.com/configs.tar.gz interval: 1h certSecretRef: name: tls-certs
What it isFetch a tarball from any HTTPS URL. Useful for configs provided by vendors or external teams as direct downloads. Supports TLS client certificates and digest pinning.
ExternalArtifact is for static, versioned downloads. Not for frequently changing sources — use GitRepository or OCIRepository for those. Good for: vendor config bundles, release tarballs, compliance config packs.

Kustomize Controller
// Kustomization CRD · dependsOn · SOPS · post-build substitution · health checks
📄 Kustomization CRD — Full Anatomy
apiVersion: kustomize.toolkit.fluxcd.io/v1 kind: Kustomization metadata: name: apps namespace: flux-system spec: # ── SOURCE ────────────────────────────── interval: 10m # reconcile every 10m retryInterval: 2m # retry faster on failure timeout: 5m # max time for apply sourceRef: kind: GitRepository name: fleet-infra path: ./clusters/prod/apps # ── DEPENDENCY ORDERING ────────────────── dependsOn: - name: infra-configs # must be Ready first # ── APPLY BEHAVIOR ─────────────────────── prune: true # delete resources removed from Git force: false # true: replace (destructive) wait: true # wait for resources to be healthy # ── SOPS DECRYPTION ────────────────────── decryption: provider: sops secretRef: name: sops-age # contains age private key # ── POST-BUILD SUBSTITUTION ────────────── postBuild: substitute: cluster_name: prod-eu domain: mycompany.com substituteFrom: - kind: ConfigMap name: cluster-config # vars from ConfigMap - kind: Secret name: cluster-secrets # ── HEALTH CHECKS (custom) ─────────────── healthChecks: - apiVersion: apps/v1 kind: Deployment name: myapp namespace: myapp # ── TARGET NAMESPACE ───────────────────── targetNamespace: myapp # override namespace in manifests # ── SERVICE ACCOUNT (tenancy) ──────────── serviceAccountName: team-a-reconciler # run as this SA
🔗 dependsOn — DAG Ordering
What it doesdependsOn creates a dependency DAG. A Kustomization won't start reconciling until all its dependencies are Ready. This prevents applying CRD instances before CRD definitions exist.
Ready conditionA Kustomization is Ready when all its resources pass health checks and the reconcile completes without error. Until Ready: dependent Kustomizations block.
infra-crds # CRD definitions (wave 0)
↓ (waits for Ready)
infra-controllers # cert-manager, ingress-nginx CRD instances
↓ (waits for Ready)
infra-configs # ClusterIssuers, IngressClass etc.
↓ (waits for Ready)
apps # actual workloads
Circular dependsOn deadlocks both Kustomizations forever. If A depends on B and B depends on A, neither will ever become Ready. Flux doesn't detect cycles — it just hangs. Design your dependency graph as a strict DAG.
🔐 SOPS — Native Secrets
How it worksSOPS encrypts specific values in YAML/JSON files, leaving keys readable. Flux's kustomize-controller decrypts in-memory using a key from a Kubernetes Secret — never writes plaintext to disk or Git.
age (recommended)Fastest setup. Generate a keypair: age-keygen -o age.agekey. Store public key in .sops.yaml. Store private key in a K8s Secret. Encrypt: sops --age=pubkey -e secrets.yaml > secrets.enc.yaml.
KMS (cloud)AWS KMS, GCP KMS, Azure Key Vault. SOPS calls the KMS API to encrypt/decrypt. No private key to store in cluster — uses IAM/IRSA. Best for production. More complex setup.
# .sops.yaml (in repo root) creation_rules: - path_regex: .*/secrets/.*\.yaml age: age1xxxxxx... # public key - path_regex: .*/prod/.*\.yaml kms: arn:aws:kms:...:key/xx # KMS for prod # Create the age Secret in flux-system cat age.agekey | kubectl create secret generic sops-age \ -n flux-system \ --from-file=age.agekey=/dev/stdin # encrypted secrets.enc.yaml (committed to Git) apiVersion: v1 kind: Secret metadata: name: myapp-secrets data: password: ENC[AES256_GCM,data:abc...,type:str] sops: age: [...] # sops metadata, not sensitive
✏️ Post-Build Variable Substitution
What it isReplace ${variable} placeholders in rendered manifests at reconcile time. Lightweight parameterization without Helm. Use for cluster-specific values (cluster name, domain, region) across environments.
# In your manifest: name: myapp-${cluster_name} namespace: myapp-${environment} host: api.${domain} # In Kustomization: postBuild: substitute: cluster_name: eu-prod environment: prod domain: mycompany.com
Post-build substitution happens AFTER kustomize build. Variables in kustomization.yaml (Kustomize's own vars) are different from Flux's post-build substitution. Don't mix them — use postBuild.substitute for Flux variables, Kustomize overlays for Kustomize-level changes. Also: ${var} format — missing variable raises error unless you use substituteFrom.optional: true.

📄 HelmRelease — Full Anatomy
apiVersion: helm.toolkit.fluxcd.io/v2 kind: HelmRelease metadata: name: nginx-ingress namespace: ingress-nginx spec: # ── CHART SOURCE ──────────────────────── chart: spec: chart: ingress-nginx version: ">=4.0.0 <5.0.0" # semver range sourceRef: kind: HelmRepository name: ingress-nginx namespace: flux-system interval: 1h # check for new chart versions # ── RELEASE CONFIG ─────────────────────── releaseName: ingress-nginx # default: HelmRelease name targetNamespace: ingress-nginx storageNamespace: ingress-nginx # where Helm stores release secret interval: 1h # reconcile interval timeout: 10m # max upgrade time # ── VALUES ─────────────────────────────── values: # inline values controller: replicaCount: 3 resources: requests: { cpu: 100m, memory: 128Mi } valuesFrom: # values from ConfigMap/Secret - kind: ConfigMap name: nginx-values valuesKey: values.yaml # key in ConfigMap optional: false - kind: Secret name: nginx-secrets targetPath: controller.extraEnvs[0].value # ── INSTALL ────────────────────────────── install: remediation: retries: 3 # retry failed installs createNamespace: true crds: CreateReplace # manage CRDs on install # ── UPGRADE ────────────────────────────── upgrade: remediation: remediateLastFailure: true # auto-rollback on failure retries: 3 strategy: rollback # or: uninstall crds: Skip # don't upgrade CRDs (safer) cleanupOnFail: true # delete new resources on failure force: false # true: force-upgrade (destructive) # ── ROLLBACK ───────────────────────────── rollback: timeout: 5m cleanupOnFail: true recreate: false # ── TEST ───────────────────────────────── test: enable: true # run helm test after install/upgrade ignoreFailures: false # ── UNINSTALL ──────────────────────────── uninstall: keepHistory: false deletionPropagation: background # ── POST-RENDERER (Kustomize patch) ────── postRenderers: - kustomize: patches: - target: kind: Deployment name: ingress-nginx-controller patch: | - op: add path: /metadata/labels/team value: platform
⚡ Helm v4 Support v2.8
Server-side applyHelm v4 uses K8s server-side apply for releases. Proper field ownership tracking. No more "last-applied-configuration" annotation conflicts. Resources owned by Helm are clearly marked.
Enhanced healthHelm v4 waits for resources to be healthy (not just applied). HelmRelease health status more accurately reflects actual workload health, not just "helm install succeeded".
CRD behaviorWith Helm v4 and install.crds: CreateReplace: CRDs are created on first install and replaced (not skipped) on upgrades. More predictable CRD lifecycle management than Helm v3.
Helm v4 is opt-in per HelmRelease. Existing HelmReleases continue to use Helm v3. To use v4, set chart.spec.helmVersion: v4. Test in non-prod first — field ownership changes can cause transient diffs during initial upgrade from v3 to v4.
🎨 Post-Renderers — Patching Helm Output
Why neededHelm chart almost fits but you need to add a label, change a container image, or patch a field the chart doesn't expose. Post-renderer applies Kustomize patches on top of the Helm-rendered manifests without forking the chart.
# Add tolerations to all Deployments in chart postRenderers: - kustomize: patches: - target: { kind: Deployment } patch: | - op: add path: /spec/template/spec/tolerations value: - { key: dedicated, operator: Equal, value: infra, effect: NoSchedule } # Override a value not exposed via chart values images: - name: docker.io/library/nginx newName: my-mirror.registry.com/nginx newTag: 1.25-alpine
Post-renderers run after Helm renders but before Flux applies. The output is validated as standard K8s manifests. Multiple post-renderers chain: each receives the output of the previous. Use for: cross-cutting concerns (security contexts, labels, node selectors) that shouldn't live in every chart's values.
⚙️ HelmRelease Operations
# Suspend HelmRelease (stop reconciling) flux suspend helmrelease nginx-ingress -n ingress-nginx # Resume flux resume helmrelease nginx-ingress -n ingress-nginx # Force immediate reconcile flux reconcile helmrelease nginx-ingress -n ingress-nginx # Force upgrade even if spec unchanged flux reconcile helmrelease nginx-ingress \ -n ingress-nginx --force # Check status flux get helmreleases -A flux get helmreleases -n ingress-nginx nginx-ingress # See Helm release history (via helm) helm history nginx-ingress -n ingress-nginx # Export rendered Helm values (what Flux actually used) kubectl get helmrelease nginx-ingress \ -n ingress-nginx -o jsonpath='{.status.history[0]}'
HelmRelease reconcile interval vs chart interval. Two separate intervals: spec.interval (how often HelmRelease reconciles — checks if values/config changed) and spec.chart.spec.interval (how often to check for new chart versions in HelmRepository). Both can trigger upgrades independently.

📸 ImageRepository + ImagePolicy
# 1. ImageRepository: scan the registry kind: ImageRepository spec: image: ghcr.io/myorg/myapp interval: 5m secretRef: name: ghcr-credentials # 2. ImagePolicy: select latest by semver kind: ImagePolicy spec: imageRepositoryRef: name: myapp policy: semver: range: >=1.0.0 # latest stable # Alphabetical (last alphabetically): alphabetical: order: asc # Numerical (largest number): numerical: order: asc # Filter before applying policy: filterTags: pattern: '^main-[a-f0-9]+-(?P.*)$' extract: '$ts' # extract timestamp for sort policy: numerical: order: asc # latest timestamp
🤖 ImageUpdateAutomation
kind: ImageUpdateAutomation spec: interval: 1m sourceRef: kind: GitRepository name: fleet-infra git: checkout: ref: branch: main commit: author: email: fluxbot@myorg.com name: Flux Bot messageTemplate: | Update {{range .Updated.Images}} {{.Reflected.Name}}:{{.NewTag}} {{end}} [ci skip] push: branch: main # or: a feature branch + PR update: strategy: Setters # find marker annotations path: ./clusters/prod
push to branchInstead of pushing directly to main, push to a new branch and open a PR. Combine with githubpullrequestcomment notification provider (v2.8) for full PR workflow: bot opens PR, CI validates, human approves, Flux deploys.
🏷️ Marker Annotations
How Flux finds image refsAdd a comment marker next to the image field in your manifests. automation-controller scans for these markers and updates the tagged line when the policy selects a new image.
# deployment.yaml — mark for auto-update spec: template: spec: containers: - name: myapp # {"$imagepolicy": "flux-system:myapp"} image: ghcr.io/myorg/myapp:v1.2.3 # ↑ Flux will update v1.2.3 to latest tag # Tag only (not full image): # {"$imagepolicy": "flux-system:myapp:tag"} image: ghcr.io/myorg/myapp:v1.2.3 # Name only (not tag): # {"$imagepolicy": "flux-system:myapp:name"} image: ghcr.io/myorg/myapp:v1.2.3 # Kustomization image override: images: - name: ghcr.io/myorg/myapp # {"$imagepolicy": "flux-system:myapp:tag"} newTag: v1.2.3
Marker format is JSON in a YAML comment — exact format matters. # {"$imagepolicy": "namespace:policyname"} — the comment must be on the line immediately above (or same line as) the image field. Wrong spacing, wrong quotes, or wrong namespace breaks image automation silently — check flux get imageupdateautomations status for errors.

Multi-Tenancy
// platform team · app teams · ServiceAccount per Kustomization · namespace isolation
🏗️ Tenancy Model
The modelPlatform team owns flux-system namespace. They configure source-controller, manage tenant onboarding Kustomizations, and set resource quotas. App teams own their namespaces and their Kustomizations — but only within the permissions of their ServiceAccount.
ServiceAccount keyEach tenant's Kustomization runs as a dedicated ServiceAccount with limited RBAC. The kustomize-controller impersonates this SA when applying resources. Tenant can't deploy to other namespaces or create cluster-scoped resources without explicit RBAC.
# Platform team: create tenant ServiceAccount apiVersion: v1 kind: ServiceAccount metadata: name: team-a-reconciler namespace: team-a --- apiVersion: rbac.authorization.k8s.io/v1 kind: RoleBinding metadata: name: team-a-reconciler namespace: team-a subjects: - kind: ServiceAccount name: team-a-reconciler namespace: team-a roleRef: kind: ClusterRole name: cluster-admin # or a more restricted Role apiGroup: rbac.authorization.k8s.io --- # Platform team creates tenant's root Kustomization kind: Kustomization metadata: name: team-a namespace: flux-system # platform team owns this spec: sourceRef: kind: GitRepository name: team-a-configs # team A's own Git repo path: ./clusters/prod serviceAccountName: team-a-reconciler # ← THE ISOLATION KEY targetNamespace: team-a
Without serviceAccountName, Kustomization runs as the flux-system service account — which typically has cluster-admin. Any team whose Kustomization runs as flux-system SA has effectively cluster-admin access. Always set per-tenant SAs in production multi-tenant clusters.
📁 Multi-Tenancy Git Structure
# Platform team: one repo (fleet-infra) fleet-infra/ ├── clusters/prod/ │ ├── flux-system/ # Flux itself │ │ └── kustomization.yaml │ ├── tenants/ # platform-managed tenant onboarding │ │ ├── team-a.yaml # Kustomization for team A (platform creates) │ │ ├── team-b.yaml │ │ └── rbac/ # SA + RoleBindings per tenant │ └── infra/ # platform-managed infra │ ├── cert-manager.yaml │ └── nginx-ingress.yaml # Team A: their own repo (team-a-configs) team-a-configs/ └── clusters/prod/ ├── kustomization.yaml # applies to team-a namespace only ├── deployments/ ├── services/ └── secrets/ # SOPS encrypted # Platform team GitRepository for fleet-infra # Team A GitRepository for team-a-configs # Both in flux-system namespace (platform team creates both)
⚠️ Tenancy Security Checklist
SA per tenantEach Kustomization must have its own ServiceAccount. The SA's RoleBinding should only grant access to the tenant's namespaces — not the whole cluster.
targetNamespaceSet targetNamespace to override any namespace in the tenant's manifests. Prevents tenant from deploying to kube-system or flux-system by accident.
No cluster-scopedRestrict tenant SA so it cannot create ClusterRoles, ClusterRoleBindings, PodSecurityPolicies, or other cluster-scoped resources. Only namespace-scoped resource access.
sourceRef cross-nsKustomizations in tenant namespaces can only reference sources in their own namespace (by default). Platform creates GitRepositories in flux-system. Tenants reference platform sources via cross-namespace access (must be explicitly enabled in Flux).
flux-system cross-namespace access: Set allow-cross-namespace-refs: true in flux-system to allow tenants to reference platform-managed GitRepositories. Or create a GitRepository in each tenant's namespace for their own repo (simpler but more SA credential management).

🚀 flux bootstrap
# Bootstrap to GitHub flux bootstrap github \ --owner=myorg \ --repository=fleet-infra \ --branch=main \ --path=clusters/prod \ --personal # personal token (or use App) # Bootstrap to GitLab flux bootstrap gitlab \ --owner=mygroup \ --repository=fleet-infra \ --branch=main \ --path=clusters/prod \ --token-auth # Bootstrap to Gitea (self-hosted) flux bootstrap gitea \ --hostname=git.myco.com \ --owner=myorg \ --repository=fleet-infra \ --path=clusters/prod # What it does: # 1. 🔍 Debugging
## Check resource status with conditions kubectl describe kustomization apps -n flux-system kubectl describe helmrelease nginx -n ingress ## Common Kustomization status messages: # "Applied revision: main/abc123" → OK # "dependency not ready" → dependsOn blocked # "decryption failed" → SOPS key issue # "timeout waiting for" → resource not healthy ## Watch controller logs kubectl logs -n flux-system \ deploy/kustomize-controller -f | jq . kubectl logs -n flux-system \ deploy/helm-controller -f | jq . kubectl logs -n flux-system \ deploy/source-controller -f | jq . ## Events kubectl get events -n flux-system \ --sort-by='.lastTimestamp' | tail -20 ## Force re-render source cache kubectl delete gitrepository fleet-infra -n flux-system # Flux recreates it from the Kustomization spec ## Test SOPS decryption locally sops --decrypt clusters/prod/secrets/myapp.enc.yaml ## Check what ImageUpdateAutomation committed flux get imageupdateautomations -A kubectl describe imageupdateautomation main -n flux-system
Kustomization Not Ready blocks all dependents. If infra-controllers is stuck, apps (which depends on it) will never reconcile. Always debug the root of the dependency chain first — find the most upstream failing Kustomization. Use flux trace to map which Kustomization owns a failing resource.

Notifications & Receivers

📨 Receiver — Webhook Triggers
What it isReceiver listens for incoming webhooks (GitHub push events, Harbor notifications, etc.) and triggers an immediate reconcile of target Kustomizations/HelmReleases. Eliminates polling delay.
kind: Receiver spec: type: github # github | gitlab | bitbucket # harbor | generic | gcr interval: 1h # token rotation interval events: [ping, push] secretRef: name: webhook-token # HMAC signature secret resources: # what to reconcile on event - apiVersion: source.toolkit.fluxcd.io/v1 kind: GitRepository name: fleet-infra namespace: flux-system # Get the receiver webhook URL kubectl get receiver my-receiver -n flux-system \ -o jsonpath='{.status.webhookPath}' # → /hook/sha256hash... # Configure this URL in GitHub → Settings → Webhooks # Prefix with your notification-controller ingress URL
Receiver URL format: https://flux.mycompany.com/hook/sha256:.... The path includes the HMAC of the secret, so it's safe to expose publicly. notification-controller verifies the GitHub webhook signature using the same secret.
PR Comments (v2.8)With githubpullrequestcomment provider: Flux posts a comment on the PR that triggered a reconcile, showing the result. Combined with ResourceSet (Flux Operator) for preview environments: PR open → environment created → Flux comments status on the PR → PR merge → environment destroyed.

Traps & Interview Quiz
// 15 operational gotchas · interactive quiz
01
Kustomization interval
spec.interval: 1m on Kustomization, spec.interval: 1h on GitRepository. How fast does a new commit apply?
Up to 1 hour without a webhook. The Kustomization reconciles every minute, but it can only detect a new commit when GitRepository re-fetches — which is every hour. Solution: configure a GitHub webhook → Receiver → triggers GitRepository reconcile on every push.
02
dependsOn cycle
Kustomization A depends on B, B depends on A. What happens?
Both Kustomizations block forever — silent deadlock. Flux doesn't detect cycles and doesn't error — it just never makes either Kustomization Ready. Design your dependency graph as a strict DAG. Detect via flux get kustomizations: both stuck in "dependency not ready".
03
Multi-tenancy
A tenant's Kustomization has no serviceAccountName. What access does it run with?
It runs as the kustomize-controller's ServiceAccount — which typically has cluster-admin. Any tenant without a dedicated SA has effectively unrestricted cluster access. Always set serviceAccountName to a tenant-scoped SA in every tenant Kustomization.
04
SOPS
You rotated the age key — stored the new private key in the K8s Secret, but forgot to re-encrypt secrets in Git. What happens?
Kustomization fails with "decryption failed" on next reconcile. The secrets in Git are still encrypted with the old public key. SOPS can't decrypt them with the new private key. Re-encrypt all affected secrets: sops --rotate --in-place secrets.enc.yaml. Then commit the re-encrypted files.
05
HelmRelease
HelmRelease spec.interval: 1h and spec.chart.spec.interval: 5m. A new chart version published. When does it deploy?
Within 5 minutes — the chart interval controls version discovery. When source-controller finds a new chart version matching the semver range within 5 minutes, helm-controller reconciles immediately on that new version — it doesn't wait for the 1h HelmRelease interval. The 1h interval is for re-checking unchanged configuration.
06
Post-build substitution
Manifest has ${undefined_var} and postBuild.substitute doesn't define it. What happens?
Kustomization fails with a substitution error. Missing variables are fatal by default. Set substituteFrom[].optional: true to skip missing ConfigMap/Secret references, or ensure all referenced variables are defined. Check with kubectl describe kustomization for the exact missing variable name.
07
Image automation
Image automation commits to Git, but Flux doesn't pick up the new image immediately. Why?
GitRepository hasn't re-fetched yet. image-automation-controller commits to Git. source-controller only sees it on the next poll interval (or webhook). Without a Receiver configured for push webhooks, there's an interval delay between commit and apply. Configure a GitHub webhook → Receiver for zero-delay image deployments.
08
Prune
You rename a Deployment in Git (old name → new name). What does Flux do with the old Deployment?
With prune: true: old Deployment is deleted, new one created. Flux tracks resources it applied using labels (kustomize.toolkit.fluxcd.io/name). The old Deployment no longer appears in the rendered output → Flux prunes it. Without prune: both old and new coexist. Brief downtime possible if old is deleted before new is healthy.
09
HelmRelease remediation
HelmRelease upgrade fails 3 times. remediateLastFailure: true. What is the HelmRelease status after?
HelmRelease is rolled back to previous version and marked Failed. Flux stops retrying after exhausting the retry count. The cluster runs the previously healthy version. The HelmRelease shows a remediation message. Fix the chart/values and run flux reconcile helmrelease --reset to reset the failure count and retry.
10
Beta API removal
You upgrade Flux from v2.6 to v2.7. Some Kustomizations stop working. What happened?
Flux v2.7 began removing deprecated beta API versions. If you're using kustomize.toolkit.fluxcd.io/v1beta2 Kustomization CRDs, they're removed in v2.7. Update all your Kustomization manifests to /v1 and HelmRelease to /v2 before upgrading. Use Flux Operator to automate CRD migration.
11
OCIRepository
You use ref.tag: latest on OCIRepository. Is this production-safe?
No — "latest" is mutable and not GitOps-compatible. The same way you shouldn't use :latest for container images in deployments, don't use latest as an OCI artifact tag. Use digest pinning (ref.digest: sha256:abc...) or semver tags (ref.semver: ">=1.0.0") for reproducible deployments. Digest pinning is the most secure option.
12
Suspend
You run flux suspend kustomization apps during an incident, fix the issue manually, and forget to resume. What happens?
Kustomization never reconciles again — manual changes persist indefinitely. Flux doesn't auto-resume. The cluster drifts from Git silently. Alert on gotk_suspend_status Prometheus metric — set an alert for any resource suspended for > 1 hour. Always resume explicitly: flux resume kustomization apps.
13
Image policy
ImagePolicy semver range >=1.0.0 — new tag 2.0.0 pushed with breaking changes. Flux deploys it automatically. How to prevent?
Narrow the semver range. >=1.0.0 <2.0.0 would reject 2.0.0. Or use >=1.0.0 <=1.9.9. Better: push to a staging-specific image tag (e.g., 1.x-staging) and use filterTags.pattern. Production should use tight semver ranges or require explicit version bump in Git.
14
Post-renderer
Post-renderer Kustomize patch fails — what does Flux do to the HelmRelease?
HelmRelease upgrade fails — Flux applies remediation strategy. Post-renderer failure is treated as an upgrade failure. If remediateLastFailure: true: Flux rolls back to the previous chart version. The post-renderer error appears in kubectl describe helmrelease. Validate post-renderer patches locally with helm template | kubectl kustomize.
15
Cross-namespace
Tenant Kustomization in team-a namespace references a GitRepository in flux-system. Does it work by default?
No — cross-namespace source references are blocked by default. Flux restricts Kustomizations from referencing sources in other namespaces. Enable with allow-cross-namespace-refs: true on the kustomize-controller args. Alternatively, create a GitRepository in each tenant namespace for their own repo, managed by the platform team.
