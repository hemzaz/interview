🔴 v3 Breaking Changes from v2
Logs RBAC BREAKINGLogs access is now enforced by default. In v2 it was opt-in. Users need explicit logs, get permission or they can't see pod logs in the UI. Role:admin and role:readonly still work fine.
Legacy metrics REMOVEDargocd_app_sync_status, argocd_app_health_status, argocd_app_created_time removed. Deprecated since v1.5. Use argocd_app_info labels instead. Update dashboards before upgrading.
resource.exclusionsv3 adds sensible defaults to resource.exclusions in argocd-cm: high-churn objects (Endpoints, EndpointSlice, Lease, TokenReview, etc.) excluded by default. Reduces API server load dramatically.
Health persistence CHANGEDHealth status no longer persisted in Application CR by default. status.health is now computed dynamically, not stored. Restore old behavior: controller.resource.health.persist: true in argocd-cmd-params-cm.
RBAC inheritanceserver.rbac.disableApplicationFineGrainedRBACInheritance defaults changed. Fine-grained RBAC inheritance is now on by default. Check your AppProject policies before upgrading.
Dex sub claimDex OIDC subject claim changed to use federated_claims.user_id instead of sub. Existing RBAC policies keyed on the old sub value may stop matching. Test SSO login after upgrade.
✨ v3.1 New Features
Native OCI NEWArgoCD can use OCI registries as config sources directly. source.repoURL can point to an OCI repo. No Git required for config distribution. Enables "gitless" GitOps with signed OCI artifacts.
Source Hydrator NEWPre-renders manifests into a "hydrated" Git branch before the Application Controller applies them. Decouples template rendering from apply, enables PR-based review of rendered output. New workflow for large teams.
CLI Plugins NEWExtensible argocd CLI via plugins. Install as binaries or shell scripts in PATH. Follows the kubectl plugin model. Third-party tools can integrate as argocd mycommand.
Config MgmtContinued improvements to Config Management Plugins (CMP). Sidecar-based CMP model (not init containers) is now the standard. Helm 3 and Kustomize versions independently configurable per-repo.
📐 Deployment Topologies
Single instanceOne ArgoCD manages all clusters (hub-and-spoke). Simplest. All cluster credentials in one place. Blast radius: if ArgoCD is down, no deployments across all clusters.
Per-clusterEach cluster runs its own ArgoCD. Fully isolated. More maintenance overhead. Good for regulated environments where clusters must be fully independent.
Hub + satelliteCentral ArgoCD manages platform clusters. Each app cluster has its own ArgoCD for tenant workloads. Hybrid: platform team owns hub, app teams own their ArgoCD instance.
HA modeMultiple replicas of API Server + App Controller. Redis Sentinel or cluster mode. Required for production. Single-replica ArgoCD = maintenance window = no deployments.
In-cluster riskArgoCD managing its own cluster's resources (including itself). A bad sync can break ArgoCD. Bootstrap it via external Terraform/Helm, then hand self-management to ArgoCD carefully.
Recommended production HA install:
kubectl apply -n argocd --server-side --force-conflicts -f https://raw.githubusercontent.com/argoproj/argo-cd/stable/manifests/ha/install.yaml

🚦 Application Status States
SYNC STATUS
● Synced
Git desired state = cluster actual state
● OutOfSync
Drift detected — cluster differs from Git
● Unknown
Cannot determine state (connectivity / render error)
HEALTH STATUS
● Healthy
All resources pass health checks
● Progressing
Deployment rolling out / pods starting
● Degraded
CrashLoopBackOff / pod failures
● Missing
Expected resource absent from cluster
● Suspended
Intentionally paused (CronJob, Rollout)
Healthy ≠ Synced. An app can be Synced + Degraded (Git applied but pods crashing) or OutOfSync + Healthy (manual change that happens to work). Watch both independently.
🔗 Multiple Sources (v2.6+)
WhyOne Application can reference multiple Git repos or Helm charts. Useful for: base chart from external Helm repo + values from your private Git repo.
spec: sources: # plural! (vs single source:) - repoURL: https://charts.bitnami.com/bitnami chart: postgresql targetRevision: 12.x.x helm: valueFiles: [$values/envs/prod/pg-values.yaml] - repoURL: https://github.com/org/configs targetRevision: HEAD ref: values # $values refs this source
Multi-source Applications are resolved atomically. If any source fails to fetch/render, the whole Application fails. Pin both sources to specific revisions for prod — using HEAD for both means two separate things can change simultaneously and you can't reason about the diff.
🏷️ ignoreDifferences Patterns
HPA replicasIgnore /spec/replicas on Deployments — HPA manages this. Without this, ArgoCD keeps trying to set replicas back to what's in Git.
Webhook CAIgnore /webhooks/*/clientConfig/caBundle — cert-manager injects this. ArgoCD sees it as drift every reconcile.
ESO-managedUse managedFieldsManagers: [external-secrets] to ignore fields owned by ESO. The secret data field changes on rotation but ArgoCD shouldn't fight ESO.
JQPath filterjqPathExpressions for complex paths: .metadata.annotations["last-applied"]. More powerful than jsonPointers for nested/conditional ignores.

ApplicationSet
// generators · templates · matrix · when to use · vs App-of-Apps
⚡ What ApplicationSet Does
An ApplicationSet is a generator + template. It iterates over a list (clusters, directories, files, PRs…) and stamps out one Application per item. One ApplicationSet CRD → N Application objects. Applications are created, updated, and deleted automatically as the generator inputs change.
apiVersion: argoproj.io/v1alpha1 kind: ApplicationSet metadata: name: all-clusters-myapp namespace: argocd spec: goTemplate: true # use Go templates (recommended) goTemplateOptions: [missingkey=error] generators: # one or more generators - clusters: selector: matchLabels: env: prod # only prod-labelled clusters template: # Application template (with generator variables) metadata: name: '{{.name}}-myapp' # cluster name injected spec: project: default source: repoURL: https://github.com/org/configs path: 'clusters/{{.name}}/myapp' # per-cluster path targetRevision: HEAD destination: server: '{{.server}}' # cluster API server URL injected namespace: myapp syncPolicy: automated: { prune: true, selfHeal: true } syncPolicy: # AppSet-level: controls Application lifecycle applicationsSync: sync # sync | create-only | create-update | create-delete preserveResourcesOnDeletion: false
Generators — click to expand
🖥️
Cluster
Iterate registered ArgoCD clusters. Filter by labels.
One Application per registered cluster. Variables: {{.name}}, {{.server}}, {{.metadata.labels.env}}. Add new cluster to ArgoCD → Application auto-created. Remove cluster → Application auto-deleted.
clusters: selector: matchLabels: region: us-east-1
Adding a new cluster to ArgoCD automatically creates Applications for it. Ensure your cluster labels are precise — an accidentally labeled cluster gets all Applications.
📁
Git Directory
One app per directory in a Git repo path.
Scans a Git path, creates one Application per directory found. Variables: {{.path}}, {{.path.basename}}, {{.path.basenameNormalized}}. Add a directory → Application appears. Delete directory → Application removed (with prune).
git: repoURL: https://github.com/org/apps revision: HEAD directories: - path: apps/* - path: apps/excluded exclude: true
📄
Git Files
One app per JSON/YAML file matching a glob pattern.
Scan Git for files matching a glob. File contents are parsed as JSON/YAML and injected as template variables. Enables rich per-app config — each file can define name, namespace, values, etc.
git: repoURL: https://github.com/org/configs revision: HEAD files: - path: apps/**/config.json # config.json: { "app": "myapp", "env": "prod", "replicas": 5 } # Template uses: {{.app}} {{.env}} {{.replicas}}
📋
List
Explicit static list of key-value maps.
Simplest generator. Define a static list of parameter maps. Good for a fixed set of environments or tenants. Not dynamic — adding an entry requires a Git commit.
list: elements: - { env: dev, ns: myapp-dev, replicas: "1" } - { env: staging, ns: myapp-stg, replicas: "2" } - { env: prod, ns: myapp-prod, replicas: "5" }
✖️
Matrix
Cross-product of two generators. N×M = N×M Applications.
Combines two generators: every element from gen A × every element from gen B. Example: 3 clusters × 5 apps = 15 Applications. Powerful but can create an explosion of Applications if not scoped carefully.
matrix: generators: - clusters: {} # gen A: all clusters - git: # gen B: all apps in git directories: [{ path: apps/* }] # Result: one Application per cluster×app combo
Matrix with Cluster (all) × Git directory (all apps) on a large fleet can create hundreds of Applications. Use cluster label selectors to scope.
🔀
Merge
Combine generators with a merge key — override specific values.
Merges results from multiple generators on a key. Use case: default values from a List generator, override specific fields for specific clusters from a second generator.
merge: mergeKeys: [server] # merge on cluster server URL generators: - clusters: {} # base: all clusters - list: # override: prod gets more replicas elements: - { server: https://prod.k8s, replicas: "10" }
🔁
Pull Request
One app per open PR. Auto-deletes when PR closes.
Scans GitHub/GitLab/Gitea for open pull requests. Creates one Application per PR — preview environments. Automatically deletes the Application when the PR is closed or merged.
pullRequest: github: owner: myorg repo: myapp labels: [preview] # only PRs with this label tokenRef: secretName: github-token key: token # Variables: {{.number}} {{.branch}} {{.head_sha}}
PR generator = ephemeral preview environments. Deploy every PR to its own namespace, run E2E tests, auto-cleanup on merge. Classic preview env pattern without Helm hook complexity.
🏢
SCM Provider
All repos in a GitHub Org / GitLab Group matching filters.
Scans a GitHub Org / GitLab Group / Gitea org for all repositories matching filters (label, topic, language). One Application per matching repo. Platform teams use this to auto-onboard new services.
scmProvider: github: organization: myorg tokenRef: secretName: github-token key: token filters: - repositoryMatch: ^svc-.* # repos starting with svc- labelMatch: gitops-enabled
⚙️ ApplicationSet Sync Policy
applicationsSyncControls what the AppSet controller does to generated Applications:
ValueCreateUpdateDelete
sync (default)✓✓✓
create-only✓✗✗
create-update✓✓✗
create-delete✓✗✓
Use create-only for tenant self-service. Platform team creates the AppSet. Tenants create a config file to trigger Application creation, but the AppSet controller won't overwrite any manual changes they make to their Application afterward.
preserveResourcesOnDeletionIf true: deleting the ApplicationSet or removing a generator element does NOT delete the Application's K8s resources. Useful for safety — don't cascade-delete production resources when a generator entry is accidentally removed.
Default sync mode + prune in Application template = full cascade delete. Remove a cluster from ArgoCD → AppSet removes Application → Application prunes K8s resources → all app resources deleted. Use preserveResourcesOnDeletion: true for production ApplicationSets until you're confident in your generator logic.
🎯 Progressive Sync (AppSet)
What it isRoll out a new Application template change progressively across clusters — like a canary deployment but for ApplicationSets. Syncs to a subset of clusters first, waits for healthy, then proceeds.
strategy: type: RollingSync rollingSync: steps: - matchExpressions: - { key: env, operator: In, values: [dev] } maxUpdate: 100% # all dev clusters first - matchExpressions: - { key: env, operator: In, values: [staging] } maxUpdate: 1 # one staging at a time - matchExpressions: - { key: env, operator: In, values: [prod] } maxUpdate: "10%" # 10% of prod clusters
Progressive sync stops if any Application in a step becomes Degraded. The remaining steps are blocked. Like a deployment rollout strategy — but for cluster-wide rollouts. Critical for multi-cluster fleet management.

App-of-Apps vs ApplicationSet
// pattern deep-dive · when to use each · tradeoffs · design
🌳 App-of-Apps
📁
A root Application syncs a Git path containing Application manifests. Those manifests create child Applications. Recursive hierarchy — ArgoCD syncs YAML that describes more Applications.
📝
Child Applications are plain YAML files in Git. Any valid Application manifest works. Full control over every field without templating constraints.
🔀
Adding a new app = commit a new Application YAML to the Git path. Remove it = delete the file. Explicit, auditable, and reviewable via PR.
⚙️
Each child Application is independently configured — different sources, policies, namespaces per Application. No template constraints.
🔐
RBAC: who can modify the root App controls the whole cluster. Who can merge to the root path controls deployment for all children.
🌍
Best for static, well-known set of applications where you want explicit control over each. One cluster, platform team manages everything.
⚡
No CRD needed. Uses only the standard Application CRD. Works with any ArgoCD version. No ApplicationSet controller required.
VS
⚡ ApplicationSet
🔄
A CRD that generates Application objects from a template + generator. Applications are machine-generated, not hand-written. Dynamic — generator input changes automatically produce new Applications.
🖥️
Cluster generator: register a new cluster → Application appears automatically. Deregister → Application auto-deleted. No manual file creation needed.
⚡
Adding a new cluster/environment = change generator inputs (register cluster, add Git directory, create config file). Template applies uniformly.
📋
All Applications share the same template. Per-app customization via generator parameters. Less flexibility per-app — if template doesn't support it, all apps can't have it.
⚠️
Template changes affect all generated Applications simultaneously (unless Progressive Sync is used). High leverage — one change, N applications updated.
🌍
Best for dynamic, scaling fleets: multiple clusters, many similar apps, PR preview environments, SCM provider scanning all repos in a GitHub org.
🏢
Self-service tenancy pattern: platform team controls the AppSet; dev teams add a config file to trigger onboarding. Zero platform team intervention per new service.
🌳 App-of-Apps — Full Pattern
# Git structure: clusters/prod/ ├── root-app.yaml # bootstrapped by Terraform/Helm └── applications/ ├── nginx-ingress.yaml # these ARE Application manifests ├── cert-manager.yaml ├── myapp-api.yaml └── myapp-frontend.yaml # root-app.yaml — the root Application kind: Application metadata: name: root spec: source: path: clusters/prod/applications # points to child App manifests syncPolicy: automated: { prune: true } syncOptions: [ServerSideApply=true] # nginx-ingress.yaml — a child Application kind: Application metadata: name: nginx-ingress spec: source: repoURL: https://kubernetes.github.io/ingress-nginx chart: ingress-nginx targetRevision: 4.10.x syncPolicy: automated: { prune: true, selfHeal: true }
With prune: true on root-app: deleting nginx-ingress.yaml from Git → root-app prunes the Application object → ArgoCD cascades: the Application is deleted and all its K8s resources are removed via the finalizer.
⚡ ApplicationSet Equivalent
# Git structure: apps/ ├── nginx-ingress/ │ └── values.yaml ├── cert-manager/ │ └── values.yaml └── myapp/ └── values.yaml # One ApplicationSet (not N Application files) kind: ApplicationSet spec: generators: - git: repoURL: https://github.com/org/configs revision: HEAD directories: - path: apps/* # scans for app directories template: metadata: name: '{{.path.basename}}' spec: source: path: '{{.path}}' # helm chart in that directory syncPolicy: automated: { prune: true, selfHeal: true } # Adding an app = add a directory to apps/ # Removing an app = delete the directory # No Application YAML files to manage
All Applications share the same template, so they all have the same sync policy, project, and cluster. If you need per-app differences, use Git Files generator with a config.json per app that provides per-app overrides.
🎯 Decision Guide: Which to Use?
Apps all similar + scaling to many clusters?
→ YES: ApplicationSet (Cluster generator)
PR preview environments?
→ YES: ApplicationSet (PR generator)
Self-service: teams create config files?
→ YES: ApplicationSet (Git Files/SCM)
Small fleet, apps are all different?
→ YES: App-of-Apps
Platform infra bootstrap (cert-manager, nginx)?
→ YES: App-of-Apps (explicit control)
You can combine both. App-of-Apps root manages: (1) an ApplicationSet for tenant workloads, (2) individual Applications for platform infra. The ApplicationSet handles the dynamic scaling; App-of-Apps handles the static platform layer.
App-of-Apps with orphaned Applications. If root-app prune is false, deleting a child Application YAML from Git leaves the Application object in ArgoCD — and it keeps running. Either enable prune on the root, or add the resources-finalizer to each child Application.

Sync, Health & Hooks
// sync lifecycle · waves · hooks · sync windows · health checks
🔁 Sync Lifecycle with Hooks & Waves
PreSync
PreSync Hooks
Run Jobs/Workflows before any resources are applied. DB migrations, config validation, pre-checks. Must complete before sync starts.
Wave -N
Low Wave Numbers
Negative waves first. Typically CRD definitions (wave -2), Namespaces (wave -1). Must be healthy before next wave.
Wave 0
Default Wave
All resources without a wave annotation. The bulk of your application resources. Wait for health before wave 1.
Wave 1+
Higher Waves
Resources that depend on wave 0. Example: HPA (depends on Deployment), Service Monitors (depend on Service being up).
Sync
Sync Hooks
Run alongside the sync. Rare use case — most prefer PreSync or PostSync. Uses same wave ordering.
PostSync
PostSync Hooks
Run after all resources are healthy. Smoke tests, notifications, cache warm-up, CDN invalidation. Runs only when full sync succeeds.
SyncFail
SyncFail Hook
Runs if sync fails at any point. Alert team, trigger rollback, send to PagerDuty. Only runs on failure.
# Hook + wave annotation pattern metadata: annotations: argocd.argoproj.io/hook: PreSync argocd.argoproj.io/hook-delete-policy: HookSucceeded argocd.argoproj.io/sync-wave: "-1" # hook-delete-policy options: # HookSucceeded (delete on success) # HookFailed (delete on failure) # BeforeHookCreation (delete before next run) # Sync wave — lower runs first argocd.argoproj.io/sync-wave: "-2" # CRDs argocd.argoproj.io/sync-wave: "-1" # Namespaces argocd.argoproj.io/sync-wave: "0" # App resources (default) argocd.argoproj.io/sync-wave: "1" # HPAs, ServiceMonitors argocd.argoproj.io/sync-wave: "5" # PostSync validation job
# Sync Window — block or allow syncs at times apiVersion: argoproj.io/v1alpha1 kind: AppProject spec: syncWindows: - kind: deny # or: allow schedule: "0 22 * * *" # cron: 10pm every day duration: 8h # block for 8 hours applications: ["*"] # all apps in project manualSync: true # allow manual even during window - kind: allow schedule: "0 10 * * 1-5" # 10am weekdays only duration: 8h applications: [myapp-prod]
deny window overrides allow window. If both apply at same time, deny wins. Use allow windows to restrict prod deploys to business hours. Sync windows only block automated syncs by default; manualSync: true still allows argocd app sync during the window.
🔍 Custom Health Checks (Lua)
Why neededArgoCD has built-in health checks for standard K8s resources. For CRDs (cert-manager Certificate, Argo Rollout, etc.), you define custom Lua scripts to determine health.
Built-in CRDsArgoCD includes health checks for: Deployment, StatefulSet, DaemonSet, ReplicaSet, Job, Ingress, PVC, cert-manager Certificate, Argo Rollout, and many others. Check the ArgoCD resource-customizations repo.
# argocd-cm ConfigMap: custom health check resource.customizations.health.mygroup_MyCRD: | hs = {} if obj.status == nil then hs.status = "Progressing" hs.message = "Waiting for status" return hs end if obj.status.phase == "Ready" then hs.status = "Healthy" elseif obj.status.phase == "Failed" then hs.status = "Degraded" hs.message = obj.status.message else hs.status = "Progressing" end return hs
⚙️ ignoreDifferences Patterns
# ignoreDifferences: common patterns ignoreDifferences: # 1. HPA manages replicas - group: apps kind: Deployment jsonPointers: [/spec/replicas] # 2. cert-manager injects caBundle - group: admissionregistration.k8s.io kind: MutatingWebhookConfiguration jqPathExpressions: - .webhooks[].clientConfig.caBundle # 3. ESO owns Secret data - kind: Secret managedFieldsManagers: [external-secrets] # 4. Karpenter drift annotation - group: karpenter.sh kind: NodePool jqPathExpressions: - .metadata.annotations["karpenter.k8s.aws/ec2nodeclass-hash"]
ignoreDifferences only suppresses OutOfSync status — it does not prevent ArgoCD from applying changes. If you add ignoreDifferences for HPA-managed replicas but also have selfHeal: true, ArgoCD may still fight HPA by reapplying the replica count. Combine with RespectIgnoreDifferences=true sync option.
⚙️ Refresh vs Hard Refresh vs Sync
RefreshFetches latest from Git repo and re-computes diff. Uses cached repo if within poll interval. Shows you what changed but doesn't apply.
Hard RefreshBypasses repo cache — forces a fresh Git fetch. Use when you just pushed to Git and want ArgoCD to pick it up immediately without waiting for the poll interval.
SyncApplies the diff to the cluster. Can be triggered manually or automatically (if syncPolicy.automated is set). Sync without refresh uses last cached state.
Force syncargocd app sync --force: replaces resources (uses replace instead of apply). Used when an apply is stuck due to immutable field change (e.g., Job selector). Destructive — deletes and recreates.
# CLI equivalents argocd app get myapp --hard-refresh # force fetch from Git argocd app diff myapp # show what would change argocd app sync myapp # apply changes argocd app sync myapp --force # replace (destructive) argocd app sync myapp --dry-run # server-side dry run argocd app sync myapp --resource-name deploy/myapp # specific resource

📋 Casbin RBAC Policy Format
# argocd-rbac-cm ConfigMap # Format: p, subject, resource, action, object, effect # Built-in roles policy.default: role:readonly # default for all users # Built-in: role:readonly (get only) # Built-in: role:admin (all) # Custom policy examples: p, role:dev, applications, get, */*, allow p, role:dev, applications, sync, */*, allow p, role:dev, applications, create, */*, deny # no create # Resources: applications, applicationsets, # clusters, repositories, projects, accounts, # certificates, gpgkeys, logs, exec # Actions: get, create, update, delete, # sync, override, action, invoke, terminate # Object format: project/application # Wildcard: */*, myproject/*, myproject/myapp # v3: logs now require explicit permission p, role:dev, logs, get, */*, allow # NEW in v3 # Group → role mapping g, github-org:platform, role:admin g, alice@company.com, role:dev
v3 breaking: logs, get must be explicitly granted. In v2 with server.rbac.log.enforce.enable: false (old default), all users with applications, get could see logs. In v3, this flag is removed and logs RBAC is always enforced. Add p, role:readonly, logs, get, */*, allow to restore previous behavior.
🔑 SSO Configuration (Dex)
OIDC providersDex supports: GitHub OAuth, GitLab, Google, Microsoft, LDAP, SAML 2.0, Bitbucket. Configure in argocd-cm under dex.config.
# argocd-cm: GitHub SSO via Dex dex.config: | connectors: - type: github id: github name: GitHub config: clientID: $GITHUB_CLIENT_ID # from Secret ref clientSecret: $GITHUB_CLIENT_SECRET orgs: - name: myorg # restrict to org teams: # optional: specific teams - platform-team - dev-team # argocd-rbac-cm: map GitHub teams to roles g, myorg:platform-team, role:admin g, myorg:dev-team, role:readonly
Direct OIDC (without Dex): Set oidc.config in argocd-cm for OIDC providers that don't need Dex (Okta, Auth0, Azure AD). Simpler for orgs with a central IdP. Use Dex for providers that need claim mapping or don't support OIDC natively (LDAP, SAML).

Traps & Interview Quiz
// 15 operational gotchas · interactive quiz
01
AutoSync
You delete a file from Git. 3 minutes later, 5 prod services vanish.
prune: true + automated sync. ArgoCD removed resources no longer in Git. Fix: set prune: false for prod, require manual prune. Or use an approval gate. Always test prune behavior in non-prod first. Restore from etcd snapshot or re-add the file to Git immediately.
02
selfHeal: true — engineer's hotfix gets reverted within minutes
selfHeal
selfHeal reverts all manual changes. This is correct GitOps behavior. Procedure: (1) disable AutoSync for that app, (2) apply hotfix, (3) commit fix to Git, (4) re-enable AutoSync. Never fight the reconciler. ArgoCD always wins.
03
HPA keeps scaling pods down to what Git says (2 replicas), even though HPA set it to 10.
ignoreDifferences
Missing ignoreDifferences for /spec/replicas. ArgoCD syncs the replica count from Git every reconcile, overwriting HPA. Add ignoreDifferences: [{group: apps, kind: Deployment, jsonPointers: [/spec/replicas]}] AND sync option RespectIgnoreDifferences=true.
04
App is Synced but Degraded. Sync doesn't fix it. What's happening?
Status
Synced + Degraded = resources applied but pods unhealthy. Git state is applied correctly (Synced), but the running pods are crashing (Degraded). Check kubectl get pods -n namespace and kubectl logs. Sync success ≠ app works. Health checks are separate from sync status.
05
ApplicationSet Cluster generator creates an Application for a cluster you didn't intend.
AppSet
Missing cluster label selector. Cluster generator without a selector matches ALL registered clusters. Add selector.matchLabels: {env: prod}. Label your clusters carefully. Also: set applicationsSync: create-only so no auto-delete if you accidentally remove a cluster label.
06
Users can no longer see pod logs in the UI after upgrading to ArgoCD v3.
v3 Breaking
v3 enforces logs RBAC by default. In v2, server.rbac.log.enforce.enable defaulted to false. In v3, this flag is removed and log access requires explicit p, role:yourole, logs, get, */*, allow in argocd-rbac-cm. Add the permission for affected roles.
07
PreSync hook succeeded, but sync fails with "hook is still running" forever.
Hooks
Missing hook-delete-policy. Without cleanup policy, the hook Job stays in the cluster. Next sync finds it still there, thinks it's still running. Add argocd.argoproj.io/hook-delete-policy: HookSucceeded. For failed hooks: add HookFailed too, or set both with BeforeHookCreation.
08
App-of-Apps root app prune=false — deleted child Application YAML from Git but Application persists.
App-of-Apps
This is expected with prune=false. The root app won't remove the child Application object. Either: (1) enable prune on root app, (2) manually delete the Application with argocd app delete myapp, or (3) delete via kubectl. Also: add the resources-finalizer to child Applications to ensure K8s resources cascade-delete when Application is deleted.
09
Helm chart targetRevision: * and ArgoCD keeps upgrading to latest chart version automatically.
Helm Source
Wildcard * or missing version = always latest. On every reconcile, ArgoCD fetches the latest chart matching *. If a new chart version is published, ArgoCD syncs it immediately. Always pin to a specific version or semver range in prod: targetRevision: 4.10.3 or 4.10.x.
10
Sync is stuck with "another operation is already in progress"
Operations
A previous sync operation is still pending/running. This happens when ArgoCD loses contact with the cluster mid-sync or a hook is stuck. Fix: argocd app terminate-op myapp to cancel the in-progress operation, then retry. If it keeps happening: check cluster connectivity and hook health.
11
Matrix generator: 5 clusters × 10 apps = 50 Applications. One cluster goes away. All 10 of its Applications delete their K8s resources.
AppSet Safety
Default AppSet sync mode + Application prune. Cluster removed → AppSet deletes those 10 Applications → each Application's finalizer cascades to K8s resources. Set preserveResourcesOnDeletion: true in AppSet syncPolicy for production, and applicationsSync: create-update to prevent auto-delete of Applications.
12
ArgoCD shows OutOfSync for caBundle on MutatingWebhookConfiguration every sync.
Drift Noise
cert-manager injects caBundle at runtime, which doesn't exist in your Git manifest. ArgoCD constantly sees this as drift and marks OutOfSync. Add ignoreDifferences: [{group: admissionregistration.k8s.io, kind: MutatingWebhookConfiguration, jqPathExpressions: [".webhooks[].clientConfig.caBundle"]}].
13
Sync window blocks automated sync — deployments silently queue up.
Sync Windows
Deny windows block automated syncs without error. GitOps operators don't notice because ArgoCD just shows OutOfSync without explaining why. Check: argocd proj windows list myproject. During deny window, automated syncs are queued — they'll apply when the window closes. For urgent deploys: use --force to override (if manualSync: true on the window).
14
Two teams both have a role:admin group binding — they can see and modify each other's Applications.
Multi-tenancy
Missing AppProject isolation. Without AppProjects, all Applications are in the default project, accessible to any user with applications access. Create an AppProject per team. Restrict: sourceRepos to their repos, destinations to their namespaces, and use project-scoped roles (proj:myproject:role) not global roles.
15
After SSO upgrade, existing users can't log in — "invalid RBAC policy"
v3 Dex
v3 changed Dex's sub claim behavior. RBAC policies keyed on the old sub value (Dex internal) now don't match. The new value is from federated_claims.user_id. Check what value ArgoCD now receives for the user: look at the JWT in browser dev tools. Update g, <new-claim-value>, role:admin in argocd-rbac-cm.
