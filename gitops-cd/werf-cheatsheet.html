Philosophy & Architecture
// what werf is · how it differs · where it fits · the CI-first model
🎯 What werf Actually Is
Core ideawerf is a CI-first delivery tool. Unlike ArgoCD/Flux (in-cluster agents), werf is a CLI binary that runs in your CI pipeline. It owns the entire delivery lifecycle: build image → push → deploy → cleanup.
vs ArgoCD/FluxArgoCD/Flux are pull-based cluster agents that watch Git. werf is push-based — runs in CI, has kubeconfig, pushes changes directly. Different model, different tradeoffs.
Key differentiatorsGiterminism (builds reproducible from Git), content-based tagging (no unnecessary rebuilds), Nelm (their Helm replacement with SSA), smart cleanup (knows what's still needed).
Single binaryOne werf binary does everything. No separate build tool + deploy tool + cleanup tool. One config (werf.yaml), one pipeline step (werf converge).
Helm compatibleUses .helm/ directory — standard Helm chart layout. Your existing Helm charts work with werf without modification. Nelm is a superset of Helm 3, not a replacement of the chart format.

🏷️ Image Naming in Helm Templates
The key linkwerf injects the built image references into your Helm chart via the werf_image() Sprig-like function. You never hardcode image tags — werf resolves them at deploy time based on content hash.
# .helm/templates/deployment.yaml containers: - name: backend image: {{ werf_image "backend" }} # ↑ werf resolves to: registry.example.com/myapp/backend:sha256-abc123 # The tag is the content hash — deterministic, immutable - name: frontend image: {{ werf_image "frontend" }} # werf.yaml image name = werf_image() argument name # They MUST match exactly # werf.yaml: image: backend # .helm: {{ werf_image "backend" }}
Image name mismatch = deploy fails. If werf.yaml says image: backend but the template says werf_image "app-backend" — werf throws an error at deploy time. The names must be identical. Also: image names must be lowercase alphanumeric with hyphens only.
🔗 Dependencies Between Images
Build orderWithout dependencies, werf builds all images in parallel. Add dependencies to enforce ordering: image B won't start building until image A is complete.
Import from imageimports under dependencies: inject another image's name/ID as an environment variable into the Dockerfile build args. Used when one image needs to know the tag of another (e.g., a migration image that references the app image).
# werf.yaml: make e2e-tests depend on backend image: e2e-tests dockerfile: ./Dockerfile.e2e dependencies: - image: backend imports: - type: ImageName # full image:tag reference targetEnv: BACKEND_IMAGE - type: ImageID # sha256 digest targetEnv: BACKEND_DIGEST - type: ImageTag # just the tag targetEnv: BACKEND_TAG # In Dockerfile.e2e: ARG BACKEND_IMAGE RUN echo "Testing against ${BACKEND_IMAGE}"
🌐 Go Templating in werf.yaml
Template supportwerf.yaml supports Go templating via {{ }} syntax. Access environment variables with env "VAR_NAME". Enables dynamic config based on CI environment.
# Dynamic image selection based on CI env image: app {{ if eq (env "CI_ENVIRONMENT") "staging" }} dockerfile: Dockerfile.staging {{ else }} dockerfile: Dockerfile {{ end }} context: . args: ENV: {{ env "CI_ENVIRONMENT" | default "production" }} # Named templates (reuse config blocks) {{- define "common-labels" }} team: platform project: {{ .Values.project }} {{- end }}
Giterminism by default restricts env usage in werf.yaml. Using environment variables in werf.yaml breaks giterminism (result depends on env, not just Git). You must explicitly allow it in werf-giterminism.yaml: werf: {config: {goTemplateRendering: {allowEnvVariables: ["CI_ENVIRONMENT"]}}}.

Stapel Builder
// werf's own builder · stages · Git mappings · mounts · artifact images
📋 What is Stapel?
DefinitionStapel (Dutch for "stack") is werf's own builder, an alternative to Dockerfile. It uses an explicit stage model with fine-grained cache invalidation. Stages are invalidated only when their specific inputs change — not the whole layer chain.
vs DockerfileDockerfile: any change invalidates all subsequent layers. Stapel: you declare which files each stage depends on via stageDependencies — change a JS file only rebuilds stages that depend on JS. Better cache efficiency for monorepos.
Git-nativeStapel's git directive is a first-class feature. Source files are mapped from Git into the image at specified paths. werf tracks Git patches between builds — only changed files trigger re-execution of dependent stages.
Use whenComplex multi-stage builds where Dockerfile cache invalidation is too coarse. Monorepos where multiple services share dependencies. Need to import artifacts from other images mid-build.
Dockerfile builder is simpler and usually sufficient. Use Stapel when you need fine-grained stage-level cache control or Git-aware incremental builds. Both coexist — use Dockerfile for simple images, Stapel for complex ones, in the same werf.yaml.

from
From
Base image layer. Cache key: base image digest.
▶
from: node:20-alpine fromImage: tools # use another werf image as base fromCacheVersion: "v2" # force cache bust by changing this

Before Install
System setup before app deps. Rarely changes (apt, adduser).

install
Install
App dependencies. Keyed on dependency files (package.json, go.mod).
▶
git: - add: / to: /app stageDependencies: install: [package.json, package-lock.json] # ↑ install stage only reruns when these change shell: install: - cd /app && npm ci --only=production
beforeSetup
Before Setup
Pre-compile / asset prep. Between install and setup.
▶
Use caseRun TypeScript compilation, asset preprocessing, proto-gen. Anything that transforms sources before the main setup step. Keyed on files listed in stageDependencies.beforeSetup.
setup

Main app setup. Keyed on app source files.
▶
git: - add: / to: /app stageDependencies: setup: ["**/*.go", go.sum] # rebuild when Go files change shell: setup: - cd /app && go build -o /usr/local/bin/app ./cmd/app
🗺️ Git Mappings
What it isThe git directive maps files from your Git repo into the image. werf tracks content changes and only re-adds changed files between builds — not a full COPY each time.
git: # Map everything from repo root to /app - add: / to: /app # Map only src/ subdirectory - add: /src to: /app/src owner: app # set file ownership group: app includePaths: # include only these paths - **/*.go excludePaths: # exclude these paths - **/*_test.go - **/*.md stageDependencies: install: [go.mod, go.sum] setup: ["**/*.go"]
Git mappings only work with committed files. Uncommitted/untracked files are invisible to Stapel git mappings (giterminism). Use contextAddFiles for non-gitignored local files, or commit everything before building.
🗻 Mounts
What it isMounts allow sharing directories across stages or between builds — like persistent volume mounts during build. Cache dependencies (node_modules, Go cache) without baking them into the image.
mount: # Share build cache between builds (not in final image) - from: build_dir # werf-managed cache dir to: /root/.cache/go # mounted during build stages # Share /tmp between stages - from: tmp_dir to: /tmp # Reference another image's path - fromPath: /usr/local/bin to: /tools from: my-tools-image
build_dirPersists between builds on the same host. A Go module cache in build_dir means modules are downloaded once and reused across many builds — dramatically faster CI builds.
Mounts are build-time only — they're not present in the final image. Their contents don't affect the final image tag (content hash). Perfect for caches that should speed up builds without changing the final artifact.
🏺 Artifact Images
What it isArtifact images are intermediate images used only during build — they're never published to the registry or referenced in Helm charts. Used to share files between images without final image bloat.
# werf.yaml: define an artifact image artifact: builder # key: "artifact" not "image" from: golang:1.23-alpine git: - { add: /, to: /src } shell: install: - cd /src && go build -o /bin/app ./cmd/app # Final image: import from artifact image: app from: scratch # or distroless, alpine import: - artifact: builder add: /bin/app to: /app before: setup # import at which stage # Result: tiny final image with just the binary # The Go toolchain never enters the final image
Artifacts are the Stapel equivalent of Docker multi-stage builds — but more powerful because they can be shared across multiple final images, cached independently, and imported at specific stages.

Giterminism — The Core Principle
// Git-determinism · werf-giterminism.yaml · what's allowed · reproducibility
🔒 What is Giterminism?
The Problem Werf Solves
Traditional CI builds are non-deterministic. The same commit can produce different images depending on: environment variables in the shell, local files that aren't committed, the state of an external service, or the time of day. You can't reproduce yesterday's build from the same commit.
Giterminism means:
Given the same Git commit SHA
+ the same werf version
+ the same registry
= always the same image tag
= always the same deployed state
regardless of: env vars, local files,
machine state, who runs it
The Rules
📁
All config files must be committed
werf.yaml, werf-giterminism.yaml, and all .helm/ files must be tracked by Git. Uncommitted changes are ignored by default.
🚫
No env vars in config (by default)
Using env "VAR" in werf.yaml or Helm templates introduces non-determinism. Must be explicitly allowed in werf-giterminism.yaml.
🔍
Docker context = committed files only
Build context only includes Git-committed files. Local modifications not staged in Git are invisible to the build. Guarantees reproducibility.
⚡
Escape hatch: werf-giterminism.yaml
Explicitly list exceptions — env vars, local files, untracked Go templates. Each exception is documented, auditable, and intentional.
📄 werf-giterminism.yaml — Relaxing Rules
giterminismConfigVersion: 1 config: # Allow env variables in werf.yaml Go templates goTemplateRendering: allowEnvVariables: - CI_ENVIRONMENT - CI_REGISTRY - ^APP_.*$ # regex: all APP_ prefixed vars # Allow reading files that aren't committed to Git allowUncommitted: # must be listed explicitly dockerfiles: - ./Dockerfile # allow uncommitted Dockerfile werf.yaml files: - ./werf.yaml files: # for contextAddFiles - .env.production helm: # Allow env vars in Helm templates allowEnvVariables: patterns: [^HELM_.*$] # Allow Go template file-includes from outside .helm/ allowUncommittedFiles: - .helm/templates/_helpers.tpl # Dev mode: allow ALL uncommitted files (CI usage only) # werf converge --dev (command-line flag, not giterminism.yaml)
werf-giterminism.yaml must itself be committed. It's the contract — this file documents every exception from strict giterminism. If it's not in Git, the exceptions aren't giterministic either. Think of it as an auditable list of "intentional non-determinism".
⚡ Dev Mode
The problemGiterminism is great for CI/prod. But during local development, you're constantly editing files that aren't committed yet. Strict giterminism would block your local builds.
Dev flagwerf converge --dev disables giterminism checks. All local files (even uncommitted) are used. Never use --dev in CI — it breaks reproducibility. Only for local development.
Dev watchwerf converge --dev --follow watches for file changes and re-runs converge automatically. Instant feedback loop during development without needing to commit.
# Local dev: any local file change triggers rebuild + redeploy werf converge --dev --follow \ --kube-config ~/.kube/config \ --namespace dev-myname # Build only (no deploy), dev mode werf build --dev # Check what giterminism errors you'd get without --dev werf converge 2>&1 | grep "giterminism" # CI always runs WITHOUT --dev: werf converge \ --repo registry.example.com/myapp \ --env production
Never commit a CI script that uses --dev. If a CI pipeline runs with --dev, the resulting image is built from uncommitted local state — it's not reproducible. If you need to include a non-committed file in CI builds, use contextAddFiles + explicit werf-giterminism.yaml permission.

⎈ Nelm — werf's Deployment Engine
What is NelmNelm (New Helm) is werf v2's deployment engine, replacing Helm 3. Backward compatible with Helm charts — no changes needed to your .helm/ directory. Nelm 1.0 released August 2025 as a standalone CLI too.
Server-Side ApplyNelm uses Kubernetes SSA instead of Helm's 3-way merge. Proper field ownership tracking. Eliminates "last-applied-configuration" annotation conflicts. Multiple managers (e.g., HPA + Nelm) coexist cleanly.
Resource trackingHeavily parallelized resource tracking during deploy. Nelm watches all deployed resources simultaneously and reports status in real-time. Old Helm: sequential waits.
Auto rollbackIf any resource fails health checks during deploy, Nelm automatically rolls back to the previous release. Configurable — can be disabled per resource with annotations.
werf planNew command in werf v2: werf plan shows exactly what Nelm would change in the cluster — without applying anything. The equivalent of terraform plan for K8s deployments.
# The full converge lifecycle werf converge \ --repo registry.example.com/myapp \ --env production \ --namespace myapp-prod \ --release myapp-prod # Helm release name (default: project+env) --kube-config ~/.kube/config # What converge does internally: # 1. werf build — build images (skip if content unchanged) # 2. werf publish — push images to registry # 3. werf deploy — Nelm applies .helm/ chart with built image refs # Plan before apply (dry-run with diff) werf plan \ --repo registry.example.com/myapp \ --env production # Deploy only (skip build, use existing images) werf deploy \ --repo registry.example.com/myapp \ --env production # Build only (no deploy) werf build \ --repo registry.example.com/myapp # Rollback: dismiss + re-converge with old commit # (no native rollback command — re-run converge at old SHA) git checkout v1.2.3 werf converge --env production
werf has no native rollback command. The "rollback" is running werf converge at the previous Git commit. This is intentional — giterminism means every deploy is fully reproducible from Git. However, Nelm does auto-rollback within a single converge run if the new version fails health checks.
📁 .helm/ Directory Structure
# Standard Helm chart layout .helm/ ├── Chart.yaml # chart metadata ├── values.yaml # default values ├── values_production.yaml # werf env-specific values ├── values_staging.yaml ├── secret-values.yaml # encrypted secrets (werf encrypt) ├── .helmignore ├── templates/ │ ├── _helpers.tpl │ ├── deployment.yaml │ ├── service.yaml │ └── ingress.yaml └── charts/ # chart dependencies └── redis/ # werf loads values in order (later overrides earlier): # values.yaml # → values_{env}.yaml (e.g., values_production.yaml) # → secret-values.yaml (decrypted) # → --set / --values flags
Env-specific valueswerf automatically loads values_{env}.yaml based on --env flag. Pass --env production → loads values_production.yaml on top of values.yaml. Zero extra configuration needed.
🔐 Secret Values
How it workswerf has a built-in secrets mechanism. Encrypt secret values with a symmetric key (WERF_SECRET_KEY env var or .werf_secret_key file). Encrypted values live in secret-values.yaml committed to Git.
# Generate a secret key werf helm secret generate-secret-key # → store in CI secret as WERF_SECRET_KEY # → or in .werf_secret_key (gitignored!) # Encrypt a value werf helm secret values encrypt \ -f secret-values.yaml # secret-values.yaml (committed to Git) db: password: 1000c5c03f39a6c4f02d5f7ba29e9b7... # encrypted # In Helm templates — access like any value env: - name: DB_PASSWORD value: {{ .Values.db.password }} # werf decrypts in-memory before rendering
Never commit .werf_secret_key to Git. Gitignore it. The secret key must be injected via environment variable (WERF_SECRET_KEY) in CI. Without the key, werf converge fails. Rotate the key: generate new key → re-encrypt all secret files → update CI secret.

Content-Based Tagging
// how werf computes tags · cache invalidation · no unnecessary rebuilds · immutability
🏷️ How Content-Based Tags Work
Instead of tagging images with Git branch names (main), commit SHAs, or timestamps, werf computes a cryptographic hash of the image content. The tag is derived from the actual layers that make up the image — not from when or where it was built.
Base image digest
+
Dockerfile/Stapel instructions
+
Build args
+
Source files (via Git tree SHA)
=
sha256(all inputs) → image tag
Same content = same tagIf nothing in the image's inputs changed between two commits, the tag is identical. werf skips the build entirely — image already in registry. No wasted CI time, no unnecessary pod restarts.
Different content = different tagAny change to source files, Dockerfile, or base image produces a new tag. The old tag is never overwritten. Every tag is immutable. You can always reproduce the exact image from the tag.
Tag formatsha256-{hash} — e.g., registry.example.com/myapp/backend:sha256-abc123def456. Human-readable content addressing. Not guessable, not sequential, not tied to time.
Deployment effectIf frontend changed but backend didn't — werf converge only rebuilds and redeploys the frontend pods. Backend pods are not restarted (same image tag = K8s does nothing). Precise, minimal blast radius.
🔬 Stage-Level Caching (Stapel)
For Stapel images, caching is even more granular — each stage has its own content hash:
from stage hash
→
beforeInstall hash
→
install hash
→
setup hash
=
final tag
Stage cache hitIf you change a Go source file, only the setup stage (which depends on *.go via stageDependencies) is re-run. beforeInstall and install stages hit cache — their inputs haven't changed.
stageDependenciesExplicitly declare which files invalidate each stage. Change in package.json → install stage reruns (npm ci). Change in src/main.go → only setup stage reruns (go build). Independent cache per stage.
Registry as cacheStage caches are stored in the OCI registry — same registry as final images. Any CI runner can reuse the cache. No shared Docker daemon needed, no EFS/S3 for cache. The registry IS the cache.
## What werf logs during build $ werf build ✓ [stage from] CACHED (no change since last build) ✓ [stage beforeInstall] CACHED ✗ [stage install] REBUILDING (package.json changed) running: npm ci ✗ [stage setup] REBUILDING (depends on install) running: npm run build ## Result: 2 of 4 stages rebuilt (targeted cache miss) ## Total build time: 45s instead of 3 minutes
⚡ Why This Matters in Practice
No stale imagesUnlike :latest or branch tags, content-based tags are immutable. A tag pushed once never changes. No "latest was different on Tuesday" confusion.
Free deduplicationIf 5 branches all use the same base image and same npm dependencies, they all share the same install stage cache. One set of layers in the registry, referenced by 5 different final image tags.
Auditable historyGiven a tag (sha256-abc123), you can reverse-lookup exactly which Git commits produced that image. werf stores this provenance in image labels.
Safe concurrent buildsTwo CI jobs building the same commit simultaneously? Both compute the same tag. One pushes, the other finds the tag already exists — skips push. No race condition, no overwrite.

🧹 Smart Cleanup — How It Works
The problemContent-based tagging means a new tag is pushed on every build. Without cleanup, your registry fills with orphaned images. But you can't delete blindly — some "old" images are still deployed to production.
werf's approachwerf scans both Git history and live cluster state to determine which images are still needed. An image is "in use" if it's referenced by a Git branch, tag, or commit — or is currently running in the cluster. Only truly orphaned images are removed.
What cleanup checksGit strategy: keep images for the last N commits per branch, keep images for all Git tags, keep images for HEAD of each branch. Kubernetes strategy: scan all cluster workloads — any image currently referenced in a pod spec is kept regardless of Git state.
Full history neededCleanup requires fetch-depth: 0 in git checkout. Shallow clones (default in most CI) only have recent history — cleanup would incorrectly mark older images as unreferenced and delete them.
# Run cleanup (schedule daily in CI) werf cleanup \ --repo registry.example.com/myapp \ --kube-config ~/.kube/config # needed to check cluster state # Dry run (see what would be deleted) werf cleanup --dry-run \ --repo registry.example.com/myapp
⚙️ Cleanup Policies
Default policiesBy default, werf keeps: last 10 builds per branch, all Git tag builds, HEAD of each branch. Configure in werf.yaml under cleanup.keepPolicies.
# werf.yaml: custom cleanup policies configVersion: 1 project: myapp cleanup: keepPolicies: # Keep images for Git tags - references: tag: { limit: { last: 10 } } # last 10 release tags imagesPerReference: last: 1 # 1 image per tag # Keep recent builds for main branch - references: branch: main imagesPerReference: last: 5 # last 5 main builds # Feature branches: keep only last 2 builds - references: branch: { limit: { last: 20, in: 336h } } # 20 branches active in 14 days imagesPerReference: last: 2 # 2 builds per feature branch # KubernetesBased: keep images running in cluster - references: kubernetes: {} # check live cluster (requires --kube-config)
Run cleanup with --kube-config. Without cluster access, werf can't determine which images are currently deployed. It may delete images still running in production (only knowing their Git history is gone). Always give cleanup access to all clusters that run your images.

📦 werf bundle publish
What it createsA bundle is an OCI artifact containing: the rendered Helm chart (with image references pre-resolved to content-based tags) + metadata about which images are included. Published to the same OCI registry as images.
# CI step: build + publish bundle werf bundle publish \ --repo registry.example.com/myapp \ --tag 1.2.3 # bundle tag (semver or branch) --tag latest \ # can add multiple tags --env production # What gets published: # registry.example.com/myapp:1.2.3 # → bundle manifest with: # - rendered chart (values resolved) # - image refs: backend:sha256-abc, frontend:sha256-def # Copy bundle between registries werf bundle copy \ --from registry-internal.myco.com/myapp:1.2.3 \ --to registry-prod.myco.com/myapp:1.2.3
bundle copy is great for air-gapped environments. Build in internet-connected CI, push bundle to internal registry, CD environment pulls from internal registry. Zero internet access required for production deploys.
🚀 werf bundle apply
What it doesPulls a published bundle from OCI registry and deploys it to the cluster. No Git access, no build tools, no werf.yaml needed. Just: registry credentials + kubeconfig.
# CD environment: deploy from bundle (no Git needed) werf bundle apply \ --repo registry.example.com/myapp \ --tag 1.2.3 \ --namespace myapp-prod \ --env production # Override values at deploy time werf bundle apply \ --repo registry.example.com/myapp \ --tag 1.2.3 \ --set "replicaCount=5" \ --set "ingress.host=prod.myco.com" # Use with ArgoCD: configure ArgoCD to # use OCI source pointing at bundle # (requires Nelm Operator — coming) # Export bundle as tar for fully air-gapped: werf bundle export \ --repo registry.example.com/myapp \ --tag 1.2.3 \ --destination ./bundle.tar.gz
Bundle tag ≠ image content tag. The bundle tag (e.g., 1.2.3) is a human-readable label you assign. The image tags inside the bundle are still content-based (sha256-abc). The bundle tag is mutable (you can re-publish to latest) but image tags inside are immutable.

Traps & Interview Quiz
// 15 operational gotchas · interactive quiz
01
Giterminism
You edited werf.yaml locally but didn't commit. werf build uses the old version. Why?
Giterminism: uncommitted files are invisible to werf. werf reads config from Git (committed state), not the filesystem. This is intentional — ensures builds are reproducible. Use werf build --dev during local development to include uncommitted changes, but never in CI.
02
Content tagging
You push two identical commits to different branches. werf builds both. What tags are produced?
The same tag. Content-based tagging is independent of Git branch or commit. Two builds with identical inputs (same Dockerfile, same source files, same base image) produce the same content hash = same tag. The second build finds the image already in the registry and skips pushing. Zero wasted work.
03
Image naming
werf.yaml defines image: app-backend but the Helm template uses werf_image "backend". What happens?
Deploy fails — image name mismatch. werf_image argument must exactly match the image: field in werf.yaml. werf throws an error: "image 'backend' is not defined in werf.yaml". The name comparison is case-sensitive and exact. Check all werf_image "..." calls against your werf.yaml image definitions.
04
Cleanup
Cleanup deletes an image that production is still using. How did this happen?
Cleanup ran without cluster access OR without full Git history. Without --kube-config, werf can't see live pods and may delete in-use images. Without fetch-depth: 0 in git checkout, shallow history makes werf think old images are orphaned. Always: full Git fetch + cluster access for cleanup.
05
Stapel stages
You change go.mod. Which Stapel stages rebuild if install: [go.mod, go.sum] and setup: ["**/*.go"]?
install and all subsequent stages (setup). go.mod is in install's stageDependencies → install stage cache busts → since setup depends on install's output, setup also reruns. beforeInstall is NOT affected (it has different stageDependencies). This is the power of fine-grained stage dependencies.
06
Secret key
werf converge fails with "cannot decrypt secret values". CI worked yesterday. What changed?
WERF_SECRET_KEY environment variable missing or wrong in CI. werf uses a symmetric key for secret-values.yaml decryption. If the key was rotated and the CI secret wasn't updated, or if the CI secret expired/was accidentally deleted, decryption fails. Check: echo $WERF_SECRET_KEY | wc -c should be non-zero in CI.
07
Dev mode in CI
A developer adds --dev to the CI pipeline "to fix a flaky build". What's the risk?
Builds become non-reproducible. --dev includes uncommitted local filesystem state. In CI, this means the runner's local state affects the build. Two CI runs on the same commit can produce different images. Content-based tagging may assign different tags to identical code. This completely breaks giterminism and audit trail.
08
Nelm rollback
A deploy fails. How do you rollback with werf?
Re-run werf converge at the previous Git commit. There's no werf rollback command. The GitOps-compatible rollback is: git checkout v1.2.2 && werf converge --env production. Or revert the Git commit and let CI redeploy. Nelm does auto-rollback within a single failing converge run, but historical rollback is via Git.
09
werf_image
Your Helm chart uses image: "{{ .Values.image.repository }}:{{ .Values.image.tag }}" — standard Helm pattern. Does this work with werf?
It works but you lose content-based tagging benefits. The standard Helm pattern injects a tag from values — you'd need to manually pass the content tag. The correct werf pattern is {{ werf_image "myapp" }} — werf injects the content-based reference automatically. Without it, you're responsible for passing the right tag to Helm values.
10
Env in werf.yaml
werf.yaml uses {{ env "CI_ENVIRONMENT" }} but werf errors with "using the env function is not allowed".
Env variables in werf.yaml require explicit giterminism permission. Add to werf-giterminism.yaml: config: {goTemplateRendering: {allowEnvVariables: ["CI_ENVIRONMENT"]}}. Each allowed env variable must be explicitly listed — this documents the intentional non-determinism. Commit werf-giterminism.yaml before re-running the build.
11
Bundles
You werf bundle apply --tag latest. The next day the tag points to a new bundle. Old deploy is silently replaced. How to prevent?
Pin to immutable tags or digests, not latest. Bundle tags (like latest or main) are mutable — they can be overwritten by new publishes. Use semver tags (1.2.3) or OCI digest (@sha256:abc) for production applies. Treat bundle tags like Docker image tags: latest in prod = asking for surprise deploys.
12
Values loading
Values in values_production.yaml are not being loaded during werf converge --env production.
File must be committed to Git (giterminism). If values_production.yaml exists locally but isn't committed, werf ignores it. Also: the file must be named exactly values_{env}.yaml — no exceptions. Check: git status .helm/values_production.yaml. If untracked, add it to Git or allow it in werf-giterminism.yaml.
13
Stapel mounts
You use mount: from: build_dir for Go cache. The CI runner is different each run. No cache benefit. Why?
build_dir mounts are local to the host. If your CI uses ephemeral runners (new VM per job), build_dir is empty on every run. The cache only persists on the same host. Solution: use the OCI registry as cache storage — Stapel stage caches are always stored in the registry and work across any CI runner.
14
werf vs GitOps
Your security team says "CI must never have cluster credentials". Can you use werf?
Yes — use werf bundle publish in CI, werf bundle apply or Nelm Operator in CD. The upcoming Nelm Operator enables a pull-based model where CI only publishes a bundle to the registry, and the in-cluster operator deploys it. Or use ArgoCD/Flux pointing at the OCI bundle. werf + GitOps agent is a valid architecture that satisfies both requirements.
15
Nelm vs Helm
You migrate from werf 1.2 (Helm 3) to werf 2 (Nelm). Your existing Helm charts work but there are "stricter validation" errors. Why?
Nelm enforces stricter Helm chart validation than Helm 3. Invalid values types, extra keys, schema mismatches that Helm 3 silently ignored — Nelm reports as errors. Fix: run helm lint .helm/ or werf render to find issues. Also: Nelm uses SSA instead of 3-way merge — first apply may show diff for field ownership migration. Expected and safe.
