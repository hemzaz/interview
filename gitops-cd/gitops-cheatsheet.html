Why GitOps
// the problem it solves · benefits · vs traditional CI/CD
📜 The 4 OpenGitOps Principles (CNCF)
1
Declarative
The entire system (infrastructure and apps) must be described declaratively. Not "run these commands" — describe "what should exist." YAML manifests, Helm charts, Kustomize overlays = the desired state expressed as data, not scripts.
2
Versioned & Immutable
Git is the single source of truth. Every desired state is stored in Git with full history. You know exactly what was deployed when, who changed it, and why (commit message). State cannot be changed without a Git commit.
3
Pulled Automatically
Software agents in the cluster pull the desired state from Git and apply it. The cluster reaches out to Git — Git doesn't push to the cluster. This is the fundamental architectural difference from traditional CI/CD pipelines.
4
Continuously Reconciled
Agents continuously compare desired state (Git) with actual state (cluster) and correct drift. If someone manually deletes a resource, the agent recreates it. Self-healing by design — the cluster always converges toward what Git says.
✅ Why GitOps Wins
AuditabilityEvery deployment is a Git commit. Full audit trail: who, what, when, why. No more "I wonder who changed that Pod resource" — it's in the Git log. Essential for compliance (SOC2, PCI, HIPAA).
Rollback = git revertRollback a deployment? git revert HEAD and the reconciler applies the old state. No special rollback tooling, no runbooks. The history IS the rollback mechanism.
Cluster credential-free CICI never holds cluster credentials. CI pipeline writes to Git (image tag bump). The in-cluster agent picks it up. Compromised CI ≠ compromised cluster. Massive security improvement.
Drift detectionManual kubectl changes are detected and reverted automatically. Prevents "snowflake clusters." Every cluster converges to the same state from the same Git repo.
Disaster recoveryCluster destroyed? Point new cluster at the same Git repo. The agent recreates everything automatically. DR becomes: provision cluster → bootstrap GitOps agent → wait for convergence.
Multi-clusterOne Git repo can drive N clusters. Each cluster runs its own agent pointing at the same (or different) Git path. Environments become Git paths, not manual config.
⚠️ When GitOps Is NOT the Right Choice
Tiny teamsTeam of 2 with one cluster? GitOps overhead (Git repo structure, agent setup, PR workflow) may outweigh benefits. Helm + CI/CD push may be simpler.
Stateful app configDatabase schema migrations, runtime config that must be updated without restarts — GitOps forces a reconcile cycle. Sometimes direct mutation is needed.
Very fast iterationDev environments where you're changing things 50 times a day — the PR/commit overhead slows iteration. Use GitOps for staging/prod, not dev.
Non-K8s targetsArgoCD and FluxCD are Kubernetes-native. Non-K8s targets (VMs, on-prem servers) need Ansible/Puppet/Salt — different tools. GitOps principles apply but the tooling differs.

How GitOps Works
// reconciliation loop · sync · drift detection · the agent
🔁 The Reconciliation Loop
build · test · push image GitHub Actions / Jenkins / etc bump image tag commit to Git DRIFT SCENARIO Someone runs: kubectl delete svc api ↓ Agent detects drift on next cycle ↓ Reconciles: recreates svc api Cluster self-heals automatically
🔁 Reconciliation Details
Poll intervalArgoCD: configurable, default 3 min. FluxCD: GitRepository interval (e.g., interval: 1m). Webhook-based push for immediate trigger.
Webhook triggerBest practice: configure Git webhook → triggers immediate reconciliation on commit. Eliminates the polling delay. ArgoCD and FluxCD both support it.
Sync statusSynced = Git and cluster match. OutOfSync = drift detected. Unknown = agent can't determine state (connectivity issue).
Health statusHealthy = resources ready. Progressing = Deployment rolling. Degraded = pods failing. Missing = expected resource absent from cluster.
Server-side applyModern GitOps tools use SSA (not client-side apply). Proper field management — each manager owns its fields. Prevents "last-apply-configuration" annotation pollution.
Diff algorithmComputes diff between Git manifest and live cluster state. Ignores runtime-only fields (status, managedFields, resourceVersion). Shows only meaningful changes.
🔐 Secrets in GitOps
The problemGitOps wants everything in Git. But you can't commit plaintext secrets to Git. These approaches solve it:
Sealed SecretsEncrypt secrets using cluster public key → commit ciphertext to Git → controller in cluster decrypts. Simple. Works with any GitOps tool. Key rotation is painful.
SOPSMozilla SOPS encrypts individual YAML values (not the whole file) using KMS/PGP/age. Human-readable keys, encrypted values. FluxCD has native SOPS support. ArgoCD needs argocd-vault-plugin.
External SecretsDon't store secrets in Git at all. ExternalSecret CRD references AWS Secrets Manager / Vault / GCP Secret Manager. Agent syncs at runtime. Git has only references, not values.
Vault + AgentHashiCorp Vault Agent sidecar or Vault Secrets Operator injects secrets at runtime. Secrets never touch Git. Most complex but most secure and feature-rich.
Never commit plaintext secrets to Git. Even in private repos — access spreads, repos get forked, CI runners log output. Use one of the above patterns. ESO (External Secrets Operator) is the most cloud-native recommendation today.
🚀 Progressive Delivery
What it isGitOps + controlled rollout strategies: canary, blue-green, A/B. Route a % of traffic to new version, validate, increase gradually.
Argo RolloutsCRD-based progressive delivery for K8s. Replaces Deployment for rollout-managed workloads. Supports canary + blue-green. Integrates with ArgoCD natively.
FlaggerFluxCD's progressive delivery companion. Works with any service mesh (Istio, Linkerd) or SMI. Automated canary analysis via metrics (Prometheus, Datadog). Auto-rollback on metric breach.
Canary flowDeploy new version to 5% traffic → validate (error rate, latency) for N minutes → increment to 20% → 50% → 100%. Each step is a Git commit to the weight value.
kind: Rollout spec: strategy: canary: steps: - setWeight: 10 - pause: { duration: 10m } - setWeight: 50 - pause: { duration: 10m } - setWeight: 100

Core modelApplication CRD — one Application = one source (Git path) + one destination (cluster + namespace). ApplicationSet generates many Applications from templates (one per cluster, one per environment).
Sync policiesautomated = auto-sync on Git change. selfHeal = revert manual changes. prune = delete resources removed from Git. All opt-in — you control aggression level.
App-of-appsRoot Application points to a Git directory of Application manifests. Those Applications manage other repos. Hierarchical management of the entire cluster via a single root sync.
Multi-clusterOne ArgoCD instance manages multiple clusters. Each cluster registered with its kubeconfig. Central control plane model — ArgoCD pushes to remote clusters.
RBACPolicy-based RBAC via Casbin. SSO (Dex) integration. OIDC/SAML/GitHub/GitLab/LDAP. Fine-grained: team A can sync app X but not delete it.
Resource hooksPreSync, Sync, PostSync, SyncFail, PostDelete hooks. Run Jobs/Workflows at specific sync lifecycle phases (DB migrations, smoke tests).
ArgoCD manages cluster resources — if ArgoCD itself is in the cluster and it syncs a broken config, it can break itself. Bootstrap ArgoCD out-of-band (Helm install via CI), then hand control over to ArgoCD to manage its own config ("App-of-apps for ArgoCD").
# ArgoCD Application CRD apiVersion: argoproj.io/v1alpha1 kind: Application metadata: name: myapp-prod namespace: argocd spec: project: default source: repoURL: https://github.com/org/k8s-configs targetRevision: HEAD path: apps/myapp/prod destination: server: https://kubernetes.default.svc namespace: myapp syncPolicy: automated: prune: true selfHeal: true syncOptions: - CreateNamespace=true - ServerSideApply=true retry: limit: 5 backoff: duration: 5s maxDuration: 3m
# ApplicationSet: one app per cluster kind: ApplicationSet spec: generators: - clusters: {} # iterate all registered clusters template: spec: source: path: clusters/{{name}} # cluster name in path destination: server: {{server}}

ArchitectureModular controllers (GitOps Toolkit): source-controller (fetches Git/OCI/Helm repos), kustomize-controller (applies Kustomizations), helm-controller (reconciles HelmReleases), notification-controller (alerts/webhooks), image-reflector + image-automation (auto image tag updates).

Multi-tenancyKustomization CRD has serviceAccountName — each tenant's sync runs with a specific SA (with limited RBAC). Platform team owns Flux; app teams own their Kustomizations within their namespace permissions.

Werf
CNCF Sandbox · Full CI/CD + GitOps in one tool · Build + Deploy + Cleanup · Deterministic builds
CNCF Sandbox
Build + Deploy
Helm-based
Image cleanup
▶
What's uniqueWerf combines image build + deploy + GitOps in a single tool. Not just a deploy agent — it builds images, pushes to registry, and manages the full delivery pipeline with Git as the state. Suited for teams that want one tool to own everything.
GiterminismWerf's core concept: the result of a build or deploy must be deterministically reproducible from a Git commit alone. No "it worked on my machine" — same commit always produces same result.
werf.yamlDefines images (Dockerfile or Stapel builder), build context, and build stages. Werf tracks content hashes — if nothing changed in an image's files, no rebuild (efficient).
DeploymentUses Helm charts under the hood. werf converge builds, pushes, and deploys in one command. Uses Nelm (Werf's reimplemented Helm) for proper 3-way merge and resource tracking.
CleanupAutomatic image cleanup: scans Git history + cluster state → determines which images are no longer referenced → removes from registry. Prevents registry bloat.
vs ArgoCD/FluxWerf is more CI-oriented — runs in CI pipeline. ArgoCD/Flux are cluster agents — run inside Kubernetes. Werf fits "run from CI" mental model; Argo/Flux fit "cluster pulls from Git" model.
# werf.yaml (define images) configVersion: 1 project: myapp image: backend dockerfile: ./backend/Dockerfile context: ./backend image: frontend dockerfile: ./frontend/Dockerfile context: ./frontend # CI pipeline usage # .gitlab-ci.yml (or GitHub Actions) deploy: script: - werf converge --env production --repo registry.example.com/myapp # werf: builds → pushes → deploys helm chart # → all in one command → git state = truth # Helm chart: .helm/ # .helm/templates/deployment.yaml image: {{ .Values.werf.image.backend }} # ↑ Werf injects the built image reference

ArgoCD Deep Dive
// architecture · sync · ApplicationSet · RBAC · hooks · CLI
🏗️ ArgoCD Architecture
API ServergRPC/REST server. CLI + UI communicate with it. Handles Application CRUD, sync triggers, user auth. The "control center."
Repository ServerFetches and caches Git repos. Renders templates (Helm, Kustomize, plain YAML, Jsonnet). Returns rendered manifests. Stateless, scalable independently.
Application ControllerWatches Application CRDs. Compares desired (rendered manifests) vs actual (live cluster state). Triggers sync when OutOfSync. Uses Kubernetes informer cache for live state.
RedisCaches cluster state, rendered manifests, repo metadata. Critical for performance — without Redis, every API call would re-query etcd and re-render templates.
DexOptional OIDC provider/proxy. Connects ArgoCD to GitHub/GitLab/LDAP/SAML for SSO. Not required if you use external OIDC directly (Okta, Auth0).
ApplicationSet ctrlGenerates Application objects from templates + generators (cluster list, Git directory, matrix, merge). One ApplicationSet → many Applications.
# Common ArgoCD commands argocd login argocd.mycompany.com # List applications argocd app list # Sync an app manually argocd app sync myapp --prune # Get app details + sync status argocd app get myapp # Watch sync in real time argocd app wait myapp --sync --health # Diff what would change argocd app diff myapp # Rollback to previous version argocd app rollback myapp 3 # Force hard refresh (re-fetch Git) argocd app get myapp --hard-refresh # Delete app (and its resources) argocd app delete myapp --cascade
AutoSync with prune=true will delete resources removed from Git. If you accidentally delete a manifest from Git, the resource is gone from the cluster within minutes. Always test changes in non-prod with prune enabled. Production: consider prune=false + manual review.
📋 AppProject — Tenancy + RBAC
What it isAppProject scopes what an Application can do: which source repos, which destination clusters/namespaces, which K8s resources, which cluster resource whitelist.
kind: AppProject spec: sourceRepos: - https://github.com/myorg/* destinations: - { namespace: team-a-*, server: https://prod-cluster } clusterResourceWhitelist: - { group: '', kind: Namespace } namespaceResourceBlacklist: - { group: '', kind: ResourceQuota } roles: - name: dev policies: - p, proj:team-a:dev, applications, sync, team-a/*, allow
🪝 Sync Hooks & Waves
Hooksargocd.argoproj.io/hook: PreSync — Job/Workflow runs before sync. PreSync (migration), Sync (main apply), PostSync (smoke tests), SyncFail (alerting), PostDelete (cleanup).
Sync wavesargocd.argoproj.io/sync-wave: "1" — within a sync, resources with lower wave number are applied first and must be healthy before higher waves start. Order-dependent deploys.
# Run DB migration before app deploys metadata: annotations: argocd.argoproj.io/hook: PreSync argocd.argoproj.io/hook-delete-policy: HookSucceeded argocd.argoproj.io/sync-wave: "-1" # Wave -1 runs before wave 0 (default)

FluxCD v2 Deep Dive
// source controller · kustomize controller · helm controller · image automation
📡 Source Controller
GitRepositoryFetches and caches a Git repo. Produces an Artifact (tarball) on each new commit. All other controllers reference this artifact — they don't fetch Git themselves.
HelmRepositoryFetches and indexes a Helm chart repository. Caches index. HelmRelease objects reference this.
OCIRepositoryFetches an OCI artifact (container image or OCI Helm chart) from a registry. Flux can use OCI registries as config stores — no Git required.
BucketFetches from S3-compatible object storage. For teams storing configs in S3, Minio, GCS.
# Bootstrap FluxCD flux bootstrap github \ --owner=myorg \ --repository=fleet-infra \ --branch=main \ --path=./clusters/prod \ --personal # Create a GitRepository source flux create source git apps \ --url=https://github.com/myorg/apps \ --branch=main \ --interval=1m # Check source status flux get sources git
FluxCD reconciles per Kustomization independently. If one Kustomization fails, others continue. This is different from ArgoCD where one bad Application doesn't affect others either — but the mental model differs. Design Kustomization boundaries carefully.
🗂️ Kustomize + Helm Controllers
KustomizationReferences a GitRepository + path. Runs kustomize build on the path, applies the output. With prune: true, deletes resources removed from Git. Health checks track readiness.
DependenciesdependsOn field: Kustomization B won't sync until A is healthy. Chain ordering: infra → crds → apps. Prevents race conditions on CRD-dependent resources.
HelmReleaseFlux manages a Helm release. Reconciles on chart version change or values change. Rollbacks: upgrade.remediation.remediateLastFailure: true rolls back on upgrade failure automatically.
PostRenderersApply Kustomize patches on top of Helm-rendered output. Fixes "this Helm chart almost does what I want but I need to patch X." No chart fork needed.
apiVersion: helm.toolkit.fluxcd.io/v2 kind: HelmRelease spec: chart: spec: chart: nginx-ingress version: ">=4.0.0 <5.0.0" sourceRef: kind: HelmRepository name: ingress-nginx install: remediation: retries: 3 upgrade: remediation: remediateLastFailure: true postRenderers: - kustomize: patches: - target: { kind: Deployment } patch: | - op: add path: /metadata/labels/team value: infra
🖼️ Image Automation
ImageRepositoryWatches a container registry for new image tags. Polls at interval. Requires read access to registry (imagePullSecret).
ImagePolicySelects which tag to use from ImageRepository. Supports semver ranges (>=1.0.0), alphabetical (latest alphabetically), numerical (latest numerically).
ImageUpdateAutomationWhen a new tag matches ImagePolicy, commits the updated image reference back to Git. Triggers a reconcile. CI pushes image → Flux detects → commits to Git → Flux deploys. Fully automated image upgrade loop.
# deployment.yaml — mark for auto-update spec: template: spec: containers: - image: myrepo/myapp:v1.2.3 # {"$imagepolicy": "flux-system:myapp"} # ImagePolicy: use latest semver patch kind: ImagePolicy spec: imageRepositoryRef: { name: myapp } policy: semver: { range: >=1.0.0 } # Result: CI tags v1.2.4 → Flux detects # → commits image: myrepo/myapp:v1.2.4 to Git # → reconciles → new pods deployed automatically
Image automation = fully closed GitOps loop. Human never touches the image tag in Git. CI pushes an image → Flux commits the new tag → Flux deploys it. Git history shows every image promotion with timestamp.

🏗️ Where Terraform Stops
Terraform roleTerraform provisions the platform that GitOps runs on top of. Create the K8s cluster, configure cloud resources, install the GitOps agent (ArgoCD/Flux) itself — then hand off.
The handoffTF creates cluster → TF installs ArgoCD (Helm provider or kubectl exec) → TF creates the "root" Application pointing at Git → from that point, Git drives everything.
TF managing K8sThe Terraform kubernetes provider can manage K8s resources — but for app-level resources, this is an anti-pattern. TF reconciles on apply only (not continuously). No drift detection.
IAM + IRSAIAM roles for pods (IRSA) = Terraform creates the IAM role. The ServiceAccount annotation referencing the role ARN = GitOps. Each tool does what it's good at.
# Terraform: provision cluster + bootstrap resource "aws_eks_cluster" "this" { ... } resource "helm_release" "argocd" { chart = "argo-cd" namespace = "argocd" # TF installs ArgoCD... } resource "kubectl_manifest" "root_app" { yaml_body = templatefile("root-app.yaml", { git_url = var.git_url }) # ...then creates the root Application # From here: Git drives the cluster depends_on = [helm_release.argocd] }
Don't manage the same K8s resource from both Terraform and GitOps. Example: TF creates a Namespace AND Argo syncs a Namespace manifest for the same name. They'll fight — TF updates on next apply, Argo reverts drift. Pick one owner per resource.
✅ Where GitOps Takes Over
The principle"If it runs inside Kubernetes, GitOps manages it. If it runs in the cloud outside Kubernetes, Terraform manages it." This is the cleanest mental model.
CRDsCRD definitions = GitOps. CRD instances = GitOps. But if the CRD controller itself needs cloud resources (e.g., cert-manager needs Route53 IAM role), that IAM role = Terraform.
Platform addonscert-manager, external-dns, nginx-ingress, kyverno — should be managed by GitOps (ArgoCD App-of-Apps or Flux bootstrap Kustomization). Not Terraform. This way they're continuously reconciled.
The bootstrap problemArgoCD must be installed before it can manage itself. Solution: Terraform installs ArgoCD via Helm, then creates a GitOps Application for ArgoCD's own config. ArgoCD then manages its own updates.
Namespaces gray zoneRecommendation: Let GitOps create namespaces (CreateNamespace=true in ArgoCD, or Namespace manifests in Git). This keeps namespace lifecycle in Git history. Terraform can create "core" namespaces (kube-system, argocd) during bootstrap only.
⚠️ Gray Zone Decision Framework
Is the resource a cloud API resource?
→ YES: Terraform
→ NO: Is it a K8s resource?
→ YES: Is it platform bootstrap?
→ YES: Terraform (one-time)
→ NO: GitOps (always)
→ NO: Rethink your design
EKS addonsGray zone recommendation: manage EKS addons via Terraform (aws_eks_addon) for lifecycle coupling with cluster. ArgoCD won't know if the addon fails at the AWS API level.
RBACClusterRoles for platform operators (access to cluster-admin) = bootstrap via Terraform. App-level Roles/RoleBindings (scoped to namespace) = GitOps.
StorageClassManage StorageClass via GitOps — it's a K8s resource with no cloud API call on creation. The underlying EBS volume is cloud API (created by CSI on PVC binding), but StorageClass itself = GitOps.

📁 Repo Structure Patterns
Monorepo
fleet-infra/ ├── clusters/ │ ├── dev/ │ │ ├── flux-system/ # Flux CRDs │ │ ├── apps/ │ │ └── infra/ │ ├── staging/ │ └── prod/ ├── apps/ │ ├── myapp/ │ │ ├── base/ # shared config │ │ └── overlays/ │ │ ├── dev/ │ │ ├── staging/ │ │ └── prod/ └── infra/ ├── cert-manager/ ├── ingress-nginx/ └── monitoring/
Monorepo: everything in one place. Easy to see full system state. Good for small-medium teams. Harder to grant per-team access.
Multi-repo
fleet-infra/ # platform team ├── clusters/ └── infra/ team-a-configs/ # team A owns ├── myapp/ │ ├── dev/ │ ├── staging/ │ └── prod/ └── another-app/ team-b-configs/ # team B owns └── service/
Multi-repo: each team owns their config repo. Better separation of concerns, better RBAC (teams can only see their repo). Harder to see whole-system state.
🌿 Environment Promotion Strategies
Branch per envdev branch → dev cluster. staging branch → staging cluster. prod branch → prod cluster. Promote: PR from dev to staging. Simple but branch divergence is painful to manage long-term.
Directory per envRecommended. Main branch, different paths: clusters/dev, clusters/staging, clusters/prod. Promote: copy/Kustomize overlay update. Single branch, clear diffs.
Kustomize overlaysbase/ has shared config. overlays/prod/ patches replicas, resource limits, image tags. Promotion = updating the image tag in the overlay. Clear diff between envs.
Image tag promotionCI builds image → tags with :dev-$SHA → deploys to dev → passes tests → PR to bump image tag in staging overlay → merge → ArgoCD deploys to staging → repeat for prod.
# Kustomize overlay for prod # clusters/prod/myapp/kustomization.yaml bases: - ../../../apps/myapp/base images: - name: myrepo/myapp newTag: v1.2.3 # ← THIS gets bumped on promotion patches: - target: { kind: Deployment } patch: | spec.replicas: 5 # prod: 5 replicas resources.limits.cpu: 2000m
Avoid per-environment Git branches for long-lived environments. Branch drift causes merge conflicts and "works in dev, broken in prod" — because the branches diverged in unexpected ways. Use directories (Kustomize overlays) instead.
🌳 App-of-Apps (ArgoCD)
A root Application points to a Git path containing other Application manifests. ArgoCD syncs the root → creates child Applications → each child syncs its own resources. The entire cluster config is bootstrapped from one root sync.

Structure: root-app → syncs apps/*.yaml → each is an ArgoCD Application → each manages its own namespace/resources.

When to use: any multi-app ArgoCD cluster. It's the standard ArgoCD pattern for managing many applications.
🌊 Flux Kustomization Tree
Platform Kustomization (infra) runs before App Kustomization (apps) via dependsOn. Within infra: infra-crds → infra-controllers → infra-configs. This layered dependency graph prevents race conditions (CRDs must exist before CRD instances).

Key: dependsOn in Kustomization CRD creates a DAG. Flux respects it during reconciliation.
🌐 Multi-Cluster GitOps
ArgoCD hub-and-spoke: one ArgoCD in "management cluster" manages all other clusters. Each target cluster is registered. ApplicationSets generate Applications per-cluster from a generator.

Flux pull model: each cluster runs its own Flux agents. Clusters are independent — one failing cluster doesn't affect others. Config paths in Git differ per cluster.

Fleet: purpose-built for 100+ clusters. One GitRepo → ClusterGroup → N clusters. Built for scale.
🔐 Secrets Pattern: ESO
External Secrets Operator (ESO) is the modern recommendation. ExternalSecret CRD references a secret in AWS Secrets Manager / HashiCorp Vault / GCP Secret Manager. ESO fetches and syncs it to a Kubernetes Secret automatically.

In Git: only the ExternalSecret manifest (references, not values). In cluster: ESO creates real Secret. Benefits: secrets rotate in the source, ESO syncs without Git commits. Perfect GitOps story.
🔄 CI/CD + GitOps Split
CI does: lint, test, build image, push to registry, update image tag in Git (via PR or direct commit to non-prod branches).

GitOps does: detect Git change → reconcile → deploy.

Key rule: CI never has cluster credentials. CI only has Git write access and registry push access. The cluster pulls from Git. This is the security model that makes GitOps genuinely safer than push-based CI/CD.
📦 OCI-based GitOps
Instead of Git repos, store configs as OCI artifacts in a container registry. Flux v2 supports OCIRepository as a source. Package Helm charts or Kustomize configs as OCI artifacts → version + sign → reference from Flux.

Benefits: no Git server needed for config distribution, same registry for images + configs, Cosign signing for supply chain security.

Use case: distributing configs to many clusters from an air-gapped registry.

GitOps Traps
// operational gotchas that bite everyone adopting gitops
01
AutoSync + Prune
Accidentally deleted a manifest from Git — production resource vanished within minutes
AutoSync with prune=true automatically deletes resources removed from Git. This is a feature, but it bites teams who make typos in Git. Production safeguard: disable prune on prod, require manual sync for deletions. Or: use branch protection + PR review before merging to prod path.
02
Secrets in Git
Team committing plaintext secrets "just for now" — they persist in Git history forever
Git history is permanent. Even if you delete the commit, the secret may be in PR comments, CI logs, or git reflog on clone. You must rotate any secret committed to Git. Use pre-commit hooks to block secrets: git-secrets, detect-secrets, truffleHog. Set these up before the first commit.
03
Drift + Manual Changes
Engineer hotfixes prod with kubectl — ArgoCD reverts it 3 minutes later
This is GitOps working correctly — but it breaks the "hotfix now, document later" workflow. Establish a procedure: 1) disable AutoSync for the affected app 2) apply hotfix 3) commit the fix to Git 4) re-enable AutoSync. Never fight the reconciler — work with it.
04
Bootstrap Chicken-Egg
ArgoCD can't manage itself because it doesn't exist yet
Bootstrap paradox: ArgoCD must exist to manage its own config, but something must install ArgoCD first. Solution: Terraform installs ArgoCD via Helm (one-time bootstrap), then creates an ArgoCD Application pointing to the ArgoCD Helm values in Git. From then on, ArgoCD manages its own config via GitOps.
05
Dual Management
Terraform and ArgoCD both managing the same Namespace — constant drift
Pick ONE owner per resource. TF manages cloud resources. GitOps manages K8s resources. If TF creates a K8s Namespace AND ArgoCD syncs a Namespace manifest → they fight. The winner on each reconcile is unpredictable. Remove the K8s provider from TF, or add ignore_changes, or delete the GitOps manifest for that resource.
06
Git as Bottleneck
During an incident, can't deploy a hotfix because Git PR approval is blocking it
GitOps requires Git to be the change path — which is a bottleneck in emergencies. Pre-plan an emergency bypass: designate "break-glass" ArgoCD users who can sync without PR, or maintain a CLI procedure to temporarily disable AutoSync. Document this before the incident happens.
07
Large State Files
ArgoCD UI loads slowly — 2000 resources in one Application
Monolithic Applications with thousands of resources slow ArgoCD's reconciliation loop and UI. Split large Applications into smaller ones by team, namespace, or component. Use AppProject to group them logically. Target: < 500 resources per Application for good performance.
08
Helm + ArgoCD Drift
argocd app get myapp shows OutOfSync for fields managed by Helm
Helm adds labels/annotations that ArgoCD doesn't know about — constant "OutOfSync" noise. Fix: add ignoreDifferences for Helm-managed fields, or use ServerSideApply=true which uses field managers. Also: use Flux HelmRelease instead of ArgoCD + Helm for cleaner Helm integration.
09
CRD Ordering
GitOps syncs CRD instances before CRD definitions exist — resources stuck in Unknown state
K8s won't accept resources for unknown CRDs. In Flux: use Kustomization dependsOn — infra-crds Kustomization must be Ready before apps Kustomization runs. In ArgoCD: use sync waves — wave 0 for CRDs, wave 1+ for CRD instances. Also: use Replace=true sync option for CRD resources.
10
Image Tag "latest"
Using image: myapp:latest in manifests — GitOps can't track what's deployed
:latest is not GitOps-compatible. The deployed image changes without a Git commit. You lose the audit trail and the ability to roll back. Always use immutable tags: SHA (sha256:abc123) or semver (v1.2.3). Flux image automation can automate this correctly with proper tags.
11
Multi-team Access
Team A can see Team B's Application in ArgoCD — and accidentally syncs it
ArgoCD Projects (AppProject) define access boundaries. Create one Project per team, restrict to their source repos and destination namespaces. Assign RBAC roles to Git groups (GitHub team → ArgoCD project role). Without proper Projects, ArgoCD is a single-tenant tool in a multi-tenant cluster.
12
Flux Deprecation
Flux v1 vs v2 — team running Flux v1 which is EOL
Flux v1 reached EOL in 2022. Flux v2 (GitOps Toolkit) is a complete rewrite with different CRDs and concepts. The migration path is non-trivial. If you're on Flux v1: migrate now. Flux v2 is significantly better in every dimension and is CNCF graduated.

🚨 The Top 5 GitOps Mistakes
1
Managing the same resource from both Terraform and GitOps — dual ownership creates perpetual drift fights.
2
No secrets strategy before going live — teams commit plaintext secrets "temporarily" and they stay forever.
3
prune=true on prod without an emergency bypass procedure — accidental Git deletion = production outage.
4
No AppProject/namespace isolation in multi-tenant ArgoCD — teams can accidentally or maliciously affect each other.
5
Using :latest image tags — destroys the GitOps audit trail and makes rollback impossible.
